# 领域审计

## 触发场景

- 用户要求检查 DDD 边界、跨域调用、事件、数据模型或某个领域的完成度。
- 用户要求判断某段代码是否把业务规则放错了模块。

## 输入

- 目标领域、变更文件、业务流程、事件名称、数据表或验收条目。
- 相关领域文档、代码入口、测试和现有 evidence。

## 必读依据

- 对应 `docs/领域/*.md`。
- `docs/03-领域架构总览.md`、`docs/04-事件契约总表.md`、`docs/06-数据模型总表.md`。
- `AGENTS.md` 的模块边界和当前任务对应验收文档。

## 步骤

1. 探索代码前先使用 code-review-graph：按需调用 `detect_changes`、`semantic_search_nodes`、`query_graph`、`get_impact_radius`；图谱不可用时明确记录并再用文本检索补充。
2. 从业务入口追踪到 Application、Domain、Gateway、事件和持久化，核对调用方向与文档职责。
3. 检查典型越权：订单域算提成或最终归属、配置域执行规则、分析域重算归属、前端硬编码权限/状态机、SDK 泄漏业务语义。
4. 对每个发现区分代码事实、文档事实、测试证据和推论；不因类名或目录名直接判定架构合规。

## 输出

按 `严重度 | 位置 | 事实 | 影响 | 证据 | 建议` 输出。没有明确问题时写“未发现明确越权”，并列出尚未覆盖的风险。

结论使用 `PASS`、`PARTIAL`、`BLOCKED` 或 `FAIL`；架构边界通过不等于业务闭环通过。

## 验证

- 发现可定位到文件、类、函数、事件或 SQL。
- 关键结论至少有一个代码证据和一个文档/测试证据，或明确说明缺口。
- 修改代码后按项目 Definition of Done 构建、重启、健康检查并生成 evidence；只做审计时明确未执行项。
