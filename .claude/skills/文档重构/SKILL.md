# 文档重构

## 触发场景

- 用户要求清理冗余文档、合并入口、归档历史材料或修复文档链接。
- 用户要求把复杂规则改写成开发者能直接执行的说明。

## 输入

- 目标目录、文档主源、当前阶段口径、引用关系和保留期限。
- 文档中的命令、路径、状态、证据和 ADR；不能只按文件名判断价值。

## 步骤

1. 先读 `AGENTS.md`、`CLAUDE.md`、`docs/README.md` 和 `harness/README.md`，再盘点引用、重复主题、过期事实和失效链接。
2. 按当前事实、执行手册、决策记录、历史归档四类处理；不把 V1/V2/历史口径混在同一结论中。
3. 先确认无引用、无唯一证据、无当前运行职责，再归档或删除；归档必须保留 manifest/索引，不能直接覆盖原文。
4. 主入口只保留可发现、可执行、可验证的内容；复杂背景移到维护者 Runbook，避免新增平行规范。
5. 发现事实冲突时写入 ADR 或冲突清单，不替用户拍板；中文路径和命令必须保持可复制。

## 安全边界

- 不删除唯一证据、密钥、运行配置或未确认的业务合同。
- 不把历史文档改写成当前事实，不把 `PENDING`、`BLOCKED` 或 `PARTIAL` 改成 `PASS`。
- 文档/Harness-only 变更使用 `Scope=docs`，不虚报构建、容器和远端发布结果。

## 输出与验证

输出变更表：文件、动作、依据、保留/归档原因、引用检查和剩余风险。

- 运行 `git diff --check`、文档链接/结构检查和 `check-harness-limits.ps1 -BaselineRef HEAD`。
- 生成 evidence，记录实际执行项和未执行项；删除或归档的内容必须可追溯。
- 结论使用 `PASS`、`PARTIAL`、`BLOCKED` 或 `FAIL`，不能用“清理完成”替代验证结果。
