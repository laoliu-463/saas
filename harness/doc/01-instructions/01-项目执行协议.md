# 项目执行协议

## 项目定位

本仓库是抖音团长 SaaS V1 工程，服务商品管理、达人 CRM、寄样、订单归因、业绩统计和看板分析。

当前实际技术栈以源码和部署脚本为准：

- 后端：Spring Boot / Java 17
- 前端：Vue 3 / Vite / Pinia / Naive UI / TypeScript
- 数据库：PostgreSQL
- 缓存：Redis
- 部署：Docker Compose
- 环境：`test`、`real-pre`、远端 `real-pre`

旧文档中的 FastAPI、Celery、Python 爬虫、旧 V2.2 全量方案只作为历史背景，不作为当前运行事实。

## 执行主线

工程任务必须围绕以下链路：

```text
明确现象
-> 收集证据
-> 判断领域和 V1 边界
-> 最小修改
-> 构建
-> 重启对应 Docker 服务
-> 健康检查
-> 业务验证
-> 生成 evidence report
-> 复盘并更新 Harness
```

## 修改代码后的强制动作

修改业务代码后必须：

- 构建通过。
- 重启对应 Docker 容器。
- 执行健康检查。
- 执行相关业务验证。
- 生成 `harness/reports/evidence-*.md`。
- 生成 retro summary，或说明本次无需 Harness 升级。

`Scope=docs` 可跳过构建、重启和业务 E2E，但仍需执行安全检查、结构检查和 evidence report。

## real-pre 基线

real-pre 是真实联调环境，必须保持：

- `APP_TEST_ENABLED=false`
- `DOUYIN_TEST_ENABLED=false`
- `DOUYIN_REAL_UPSTREAM_MODE=live`

real-pre 禁止清库、禁止 mock、禁止 `docker compose down -v`、禁止删除 PostgreSQL / Redis volume。

## 结论口径

- 没有证据时，只能写阶段性结论。
- `BLOCKED`、`PENDING`、`PARTIAL` 不能写成 `PASS`。
- test/mock 结果不能证明 real-pre 真实闭环。
- 订单入库不等于渠道闭环；必须继续验证归因、寄样、业绩和看板。
