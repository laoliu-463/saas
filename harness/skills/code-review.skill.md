# Skill: code-review

## 使用场景

用于合并前复查、AI 生成代码审查、风险审计和回归范围判断。

## 必读文件

- `AGENTS.md`
- `harness/AGENT_CONTRACT.md`
- `harness/FORBIDDEN_SCOPE.md`
- `docs/09-测试验收总览.md`
- 当前变更涉及的领域、流程、接口、数据和权限文档

## 禁止事项

- 禁止只看代码风格，不看行为风险。
- 禁止忽略 real-pre / test 契约差异。
- 禁止把未验证的 AI 生成代码当作可交付代码。
- 禁止用 mock 结果证明真实业务闭环。

## 标准流程

1. 先看 `git diff`，确认变更文件和影响范围。
2. 按领域边界检查是否越界。
3. 检查接口请求、响应、错误处理和权限边界。
4. 检查状态机是否合法、是否有幂等和审计。
5. 检查是否需要同步 docs、evals、runbooks 或 evidence report。
6. 检查敏感信息是否进入 diff。
7. 给出按严重度排序的问题。

## 验证方式

- 至少给出构建、单测、E2E、API/SQL 或日志中的一种验证方式。
- 对未验证项必须标为 `PENDING`，不能写成通过。

## 输出格式

```md
发现的问题：
P0：
P1：
P2：
开放问题：
建议验证：
剩余风险：
```
