# Sample Domain DDD Instruction

## 领域职责

- 负责寄样申请、审核、发货、物流信息、签收、待交作业、交作业完成和状态机。
- 基于订单域发布的订单已同步事实判断交作业完成。

## 领域不负责

- 不负责订单事实同步。
- 不负责业绩归属、提成、冲正。
- 不负责用物流 API 阻塞寄样主流程。

## V1 规则

- 状态变更必须有操作者、时间、前后状态。
- 交作业完成必须基于订单已同步事件和可追溯匹配条件。
- real-pre 缺少真实归因订单样本时只能写 BLOCKED / PENDING。

## 禁止越界

- 禁止寄样域直接同步抖音订单。
- 禁止寄样域计算业绩或提成。
- 禁止寄样域绕过状态机直接覆盖最终状态。

## 允许调用的 Facade

- 优先使用寄样域应用服务、Facade、寄样 API 或订单已同步事件消费入口。
- 订单域只提供订单事实，不直接改寄样结论。

## 必须执行的测试

- 寄样状态机单测 / 集成测试。
- 订单已同步事件消费测试。
- 重复消费幂等验证。
- 寄样申请、审核、发货和交作业 E2E。

## 完成后必须更新的 state

- `harness/rules/state/snapshots/DOMAIN_STATUS.md` 的寄样域状态。
- 寄样状态、订单事件、真实样本和自动完成相关 evidence。

## 失败后必须写入 feedback

- 状态机非法流转、订单样本不足、事件消费失败或幂等缺口写入稳定 evidence；需持续跟踪时登记 `KNOWN_ISSUES.md`。
