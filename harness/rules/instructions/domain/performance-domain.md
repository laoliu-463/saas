# Performance Domain DDD Instruction

## 领域职责

- 负责消费订单 final user/source 后的提成、冲正、双轨金额和汇总刷新。
- 消费订单事实和配置参数，不回查商品映射、活动或当前角色重算归因。
- 输出 `performance_records`、归属结论、冲正记录和汇总表。

## 领域不负责

- 不负责抖音订单原始事实同步。
- 不负责商品转链或 `pick_source_mapping` 生成。
- 不负责采集第三方事实。

## V1 规则

- 订单域是 final 渠道/招商用户和 source 的归因事实来源，业绩域只消费并透传该事实。
- 招商 PERSONAL 数据范围按订单 final recruiter 过滤；招商链接业绩归创建链接的招商账号，活动招商不能覆盖可用链接归因。
- 计算结果必须保留输入来源，支持追溯。
- 服务费收入双轨公式必须区分预估 / 结算：预估服务费收入 = 预估订单额 × 服务费率（未扣除技术服务费）；结算服务费收入以官方结算服务费字段为准，不用订单额重算且不扣技术服务费。
- 服务费收益双轨公式必须区分预估 / 结算：预估服务费收益 = 预估服务费收入 - 预估服务费支出 - 技术服务费；结算服务费收益 = 结算服务费收入 - 结算服务费支出。
- 技术服务费只参与预估服务费收益扣减；结算服务费收益不扣减技术服务费。

## 禁止越界

- 禁止业绩域同步抖音订单。
- 禁止业绩域修改订单原始事实。
- 禁止业绩域用活动招商、商品负责人、商品映射或当前角色重算归因。
- 禁止把分析模块的展示查询写成业绩计算规则。
- 禁止把旧“服务费收入 - 技术服务费”作为预估 / 结算两轨统一服务费收益公式。

## 允许调用的 Facade

- 优先使用业绩域应用服务、Facade、查询 API 或业绩事件。
- 分析模块读取业绩结果时只能读 API、汇总表或只读查询能力。

## 必须执行的测试

- 业绩计算单测。
- 冲正和退款消费测试。
- 汇总刷新验证。
- 权限数据范围和重复消费幂等验证。

## 完成后必须更新的 state

- `harness/rules/state/snapshots/DOMAIN_STATUS.md` 的业绩域状态。
- 业绩计算、冲正和 dashboard 对账相关 evidence。

## 失败后必须写入 feedback

- 计算输入不完整、重复消费、冲正错误或汇总漂移写入稳定 evidence；需持续跟踪时登记 `KNOWN_ISSUES.md`。
