# Harness 报告风格政策（report-style-policy）

> 适用：`harness/reports/latest-*.md`、`harness/reports/current/*-final.md`、`harness/reports/current/*-status-before.md`、各治理/审计/盘点/合规报告。
> 不适用：`harness/archive/**` 历史报告（豁免，但鼓励按本风格补写摘要）。
> 限制：单文件 ≤ 200 行；超出必须 SPLIT。

## 1. 强制四节结构

每个报告必须包含以下四节，**顺序固定**：

1. **结论** — 一段话讲清"任务/盘点/审计最终结论是什么"，不超过 10 行。
2. **关键结果** — 表格 / 列表，量化结果（数量、占比、文件路径）。
3. **主要变更 / 处置** — 表格列出 KEEP / MERGE / SPLIT / ARCHIVE / DELETE / RENAME 动作与目标路径。
4. **风险与下一步** — 列出未消除的风险、待办、下次治理入口。

可选第五节 **证据** — 引用本次操作的实际命令输出、文件路径、行号截图。**强烈推荐**在 GC 报告中保留。

## 2. 标题与目录

- 一级标题：报告主题英文 / 中文均可，但全文档必须统一。
- 不允许出现"前言 / 引言 / 写在前面"等无信息节。
- 不允许出现"参考资料"占位空节。
- 不允许出现"附：聊天记录原文"等长引文块（长原文放 archive）。

## 3. 表格使用

- 比较、列举、统计优先使用表格。
- 表格首行必须为表头；表头单元格 ≤ 4 个英文单词 / 6 个汉字。
- 表格内不允许出现代码块或大段文字（超过 1 行的内容放代码块或单独段落）。

## 4. 链接

- 内部链接用相对路径：`../rules/structure-policy.md`。
- 链接文本带文件名可点击： `[structure-policy.md](../rules/structure-policy.md)`。
- 不允许使用"详见附录"等无具体路径的引用。

## 5. 代码块 / 命令

- 命令必须可复制粘贴执行；用 ` ```bash ` 或 ` ```powershell ` 标注语言。
- 一次性命令列表（GC 计划、文件操作）超过 10 条时拆分为"主操作"和"备选"两表。
- 禁止在报告中贴长 SQL 全文 / 长 JSON 原文 / 长日志原文（>30 行的去 `archive/`，报告里只贴关键摘要 + 引用路径）。

## 6. 行数控制

- 单报告 ≤ 200 行。
- 表格行 + 段落行总和不超 200。
- 超长时强制 SPLIT：
  - 主报告保留 4 节结构与摘要；
  - 详细数据 / 命令清单 / 引用清单拆到 `harness/reports/<topic>-detail.md` 或 `harness/archive/YYYYMMDD/<topic>-detail.md`。

## 7. 元信息（首部必须）

报告首部（紧跟一级标题之后）必须有以下元信息块：

```text
> 生成时间：YYYY-MM-DD
> 范围：本次报告覆盖的目录 / 任务 / 治理对象
> 用途：说明本报告被谁、何时引用
> 关联：相关 policy / manifest / 任务卡路径
```

## 8. 命名

- 在线报告（`reports/` 根 或 `current/`）必须以 `latest-` 前缀或 `*-final.md` / `*-status-before.md` 后缀命名。
- 历史报告一律在 `archive/YYYYMMDD/`，文件名带 `YYYYMMDD-HHMMSS` 或 `YYYYMMDD-<seq>`。
- 一次性 retire / retro 类只允许出现在 `archive/`，不允许常驻 `reports/` 根。

## 9. 反模式（绝对禁止）

- ❌ 标题写"~~以下内容仅供参考~~"等无信息副标题。
- ❌ 把"操作流水账" / "agent 思考过程"留在报告里。
- ❌ 在报告中再写一份"规则"（规则必须放 `rules/`）。
- ❌ 用"已完成"、"通过"代替证据引用。
- ❌ 用占位符 `<!-- TODO -->`、`...` 替代真实内容。
- ❌ 把多个主题的证据拼成一份"大杂烩"报告（必须按主题拆分）。
