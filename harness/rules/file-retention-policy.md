# Harness 文件保留政策

> 本文件是 `harness/` 文件生命周期的唯一主源。

## 生命周期

| 类别 | 内容 | 处理方式 |
| --- | --- | --- |
| 长期 | 规则、脚本、模板、manifest、当前探针 | 被替代后归档，不直接删除 |
| 当前 | 任务卡、稳定 evidence、当前状态摘要 | 同主题覆盖更新，结案后按价值归档 |
| 临时 | 调试日志、失败中间报告、空占位 | 证据已迁移后删除 |

## 目录规则

- `rules/`、`scripts/`、`templates/`、`probes/`：只保留当前版本。
- `tasks/`：任务结案后 7 天内归档。
- `reports/`：根目录目标不超过 20 个直接文件；新任务摘要写入 `reports/current/latest-<topic>.md`。既有固定工具 `latest-*` 可留根但不得按运行次数增长。
- `manifests/`：长期保留，记录每次归档和删除依据。
- `archive/`：按日期和主题分桶并遵守 50/50；历史不可变证据原样迁移可豁免行数追溯，新摘要仍受 200 行上限约束。

## 报告保留

- 每个任务只维护一份稳定 evidence；retro 默认内联，只有包含责任人、改进动作和验证方式时才单独生成。
- 原始日志、长 JSON、长 SQL 结果放在 `runtime/qa/out/<run-id>/` 或归档包中，报告只保留摘要和路径。
- 稳定摘要由 Git 历史保留普通版本；仅发布、审计或唯一真实证据进入 `archive/`。
- 无结论、无证据、仅重复脚本输出的报告可以删除。

## 删除保护

以下内容不得直接删除：

- 当前任务卡和当前生效探针。
- manifest 和已存在的归档索引。
- 被 `INDEX.md` 引用的权威文件。
- 真实验收证据的唯一副本。
- 源码、环境配置、数据库 migration、Docker Compose 和密钥文件。

## 操作要求

归档或删除必须：

1. 确认替代主源存在。
2. 检查当前引用。
3. 创建 manifest，记录路径、原因和替代文件。
4. 执行后运行结构限制和 Git diff 检查。
5. 在 evidence 中如实记录删除、未验证项和剩余风险。
