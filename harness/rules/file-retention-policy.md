# File Retention Policy

## 1. 当前有效文档保留规则
- 当前正在执行或高频查阅的任务、说明文档、最新验证报告，应留存在 `tasks/`, `reports/`, `rules/` 等一级有效目录。
- 文档必须保持最新，过期信息应及时被剔除或转移。
- 每目录限制 10 个文件，如果有效文档过多，应合并精简，或提炼建立 `INDEX.md`，按逻辑拆分至合理子目录。

## 2. 历史报告归档规则
- 过期报告、历史遗留执行结果、超过有效期的架构草案必须归档到 `archive/`。
- 归档命名建议：`archive/reports-YYYYMMDD/`。
- 归档目录同样受每级最多 10 个文件/10 个子目录的约束。如果归档积压，应按月或更粗粒度打包，或仅保留核心索引、删除中间过程文件。

## 3. 临时文件删除规则
- Agent 运行过程中产生的无效临时说明、测试失败的废弃报告、无结论的流水账，应当果断删除。
- 严禁将 `reports/` 作为临时文件的堆积池。

## 4. 证据文件保留规则
- 关键节点的安全检查、架构重构证据报告，允许保留最新版本于 `reports/` 中。
- 旧证据归档，并在新的执行中覆盖更新。所有清理动作必须留存在 `manifests/` 作为证据。

## 5. 归档命名与结构
- 一致使用连字符命名：`feature-name-YYYYMMDD.md`。
- 归档内必须有简短摘要，避免巨型日志留存。
