# Skill: domain-alignment

## 使用场景

用于任何涉及业务规则、状态机、权限、归因、看板或跨模块修改的任务。目标是先确认任务属于哪个领域，避免把规则写错层。

## 必读文件

- `CLAUDE.md`
- `docs/README.md`
- `harness/DOMAIN_MAP.md`
- `docs/01-V1交付范围与边界.md`
- `docs/02-业务闭环总览.md`
- `docs/03-领域架构总览.md`
- 当前任务对应的 `docs/领域/*.md` 和 `docs/流程/*.md`

## 禁止事项

- 禁止把前端展示规则写成后端业务事实。
- 禁止让订单域计算提成或最终归属。
- 禁止让配置域执行具体业务规则。
- 禁止让分析模块重算业绩归属。
- 禁止用旧 V2.2 范围扩大当前 V1。

## 标准流程

1. 复述任务现象或需求。
2. 判断所属领域和涉及的上下游领域。
3. 对照 `harness/rules/governance/forbidden-scope.md` 排除当前不做项。
4. 找到对应 API、数据模型、事件和验收文档。
5. 明确修改层：前端、后端、SDK、数据库、测试或文档。
6. 若发现文档与代码冲突，记录证据并写入 ADR-002，而不是自行裁决。

## 验证方式

- 能指出主责领域和辅助领域。
- 能指出不能修改或不能扩大解释的边界。
- 能列出至少一种脚本、API/SQL 或日志证据。
