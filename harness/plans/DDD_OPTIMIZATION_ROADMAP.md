# DDD Optimization Roadmap

## 定位

本文件记录 DDD 优化总路线。它是 Harness 执行计划，不代表代码已经完成对应重构；业务事实仍以 `docs/01-V1交付范围与边界.md`、`docs/03-领域架构总览.md`、`docs/领域/*.md` 和当前源码证据为准。

## 总原则

- 不新建第二套 Harness，不覆盖既有 Harness 文件。
- 优先更新现有主源；没有主源时才创建最小新增文件。
- 一次任务只推进一个主责领域任务卡。
- 先边界盘点，再代码修改，再验证证据，再 state / feedback 同步。
- V1 不引入 FastAPI、微服务、外部 MQ、独家覆盖、复杂财务结算或差异化提成；经营毛利（Y-04）已按 2026-06-05 用户决策纳入 V1 指标。
- 没有构建、测试、API/SQL、日志或 evidence report 的任务不能写成完成。

## 总顺序

0. 当前状态盘点与 V1 口径冻结
1. Harness 工程护栏
2. DDD 架构护栏与包结构
3. 用户域
4. 配置域
5. 订单域
6. 业绩域
7. 分析模块
8. 商品域
9. 达人域
10. 寄样域
11. Outbox 领域事件
12. 前端领域化
13. E2E 验收与垃圾回收

## 为什么按这个顺序执行

- 用户域先行：权限、角色、组织和 `self/group/all` 是所有列表、看板和业务操作的前置边界。
- 配置域第二：配置只提供参数，避免后续订单、业绩和商品优化时混入规则执行。
- 订单域第三：订单事实和归因输入必须稳定，后续业绩、寄样和分析才能验证。
- 业绩域第四：最终归属、提成、冲正和汇总依赖订单事实、商品映射、达人关系和配置参数。
- 分析模块第五：看板只读汇总，必须在业绩边界清楚后再治理。
- 商品、达人、寄样随后推进：三者影响渠道链与招商链，但应在权限、配置、订单、业绩口径稳定后收口。
- Outbox、前端领域化、E2E 和 GC 最后推进：这些是跨域治理和回归保障，依赖前面边界明确。

## 阶段目标与验收

| 阶段 | 目标 | 验收标准 |
| --- | --- | --- |
| 0 当前状态盘点与 V1 口径冻结 | 确认当前技术栈、V1 范围和旧文档冲突 | `CURRENT_STATE.md`、`FORBIDDEN_SCOPE.md`、ADR 索引一致 |
| 1 Harness 工程护栏 | 固定读取、执行、验证、证据和反馈入口 | `TASK_ROUTING.md` 能路由 DDD 任务，docs scope 可生成 evidence |
| 2 DDD 架构护栏与包结构 | 建立跨域访问、Controller 业务逻辑和 Repository 穿透检查口径 | `ddd-boundary-check.skill.md` 可执行，禁止项进入 `FORBIDDEN_SCOPE.md` |
| 3 用户域 | 收口身份、角色、菜单、组织和数据范围 | admin/group/self 权限证据可复现，业务域不复制权限规则 |
| 4 配置域 | 收口规则参数、模板和审计事实 | 配置变更可审计，业务规则仍由业务域执行 |
| 5 订单域 | 收口订单事实、退款事实、归因输入和事件 | 订单同步有真实响应、入库事实、幂等和事件证据 |
| 6 业绩域 | 收口最终归属、提成、冲正和汇总刷新 | `performance_records` 与汇总可追溯到订单、配置和映射输入 |
| 7 分析模块 | 确认 dashboard / 报表只读事实或汇总 | 看板 API/SQL 对账通过，不重算归因 |
| 8 商品域 | 收口商品库、活动商品、展示规则、转链和映射 | 转链后 `pick_source_mapping` 可追溯，商品库展示规则可验证 |
| 9 达人域 | 收口达人资料、认领、保护期、地址和标签 | 达人 API/E2E 可验证，达人域不替代订单或业绩归属 |
| 10 寄样域 | 收口申请、审核、发货、签收、待交作业和自动完成 | 状态机合法，交作业完成基于订单已同步事件 |
| 11 Outbox 领域事件 | 收口事件记录、幂等键、消费状态和失败重试 | 事件生产者、消费者、幂等键和证据路径完整 |
| 12 前端领域化 | 前端目录、API client、store 和页面按领域边界收口 | 前端不硬编码核心业务规则或状态机 |
| 13 E2E 验收与垃圾回收 | 补齐主链路验收和旧内容维护 | E2E / API / SQL / evidence 可复现，旧内容候选进入 GC 流程 |

## 执行要求

每个阶段开始前必须：

1. 读取本文件和 `DDD_DOMAIN_TASK_MATRIX.md`。
2. 读取对应 `harness/instructions/*.md`。
3. 读取对应 `docs/领域/*.md`、相关流程、API、数据和验收文档。
4. 对照 `FORBIDDEN_SCOPE.md` 排除 V1 不做项。

每个阶段完成后必须：

1. 运行当前 Scope 的固定命令。
2. 生成或更新 evidence report。
3. 更新 `harness/state/DOMAIN_STATUS.md`。
4. 将失败、阻塞、新风险或规则缺口写入 `harness/feedback/`、`harness/state/KNOWN_ISSUES.md` 或任务 evidence。
5. 未验证项只能写 `PENDING` / `BLOCKED` / `PARTIAL`，不能写 `PASS`。
