## 3. 订单域合同（当前事实）

### 3.1 订单域不变量（CLAUDE.md / ADR-003）

- **订单域只存事实**，不算提成，不应用独家覆盖
- **业绩域**负责最终归属、提成、冲正、双轨金额
- **分析模块**只读汇总，不重算业绩归属
- **不变量**：订单域不写 `performance_records`，不调 `PerformanceCalculationService`

### 3.2 同步双轨与归因

| 接口 | 用途 | 模式 | 水位 Redis key | 排程 |
| --- | --- | --- | --- | --- |
| `buyin.instituteOrderColonel`（6468） | 事实订单源 | `time_type=update` 24h 滚动 | `order:sync:institute_recent_last_time` | `OrderSyncJob.syncInstituteOrdersRecent` 10min |

> 6468 / 2704 / PAY_RECENT **三轨并存**，互不覆盖（已通过 `syncSource` 字段保护）。

### 5.3 归因失败根因

`pick_source_mapping` 表当前仅 13 条映射，且**本轮 real-pre 上游抖店返回的 100 条 6468 订单样本 `pick_source` 全部为空**（既往 `order-p0-dual-source-remote-verify-20260603-205719.md` 报告已确认）。

**这与代码 bug 无关**，而是：
- 当前样本由**非通过系统转链**产生（用户未走 `pickSourceMappingService` 生成的转链 URL 下单）
- pick_source 是抖店只在用户通过带货链接下单时才会回传的字段

**判断**：归因底座已具备，缺真实 `pick_source` 业务样本。归因**样本不足**，不是代码问题。

### 9.3 跨域违规（关键风险）

`DataApplicationService` 注入 `PerformanceRecordMapper` 与 `SysUserMapper` 直接从 `colonelsettlement_order` 表 join 业绩域：

```java
// DataApplicationService.java line 26-27
import com.colonel.saas.mapper.PerformanceRecordMapper;
import com.colonel.saas.mapper.SysUserMapper;
// line 402-419
List<String> orderIds = orders.stream().map(...).toList();
Map<String, PerformanceRecord> perfMap = loadPerformanceMap(orderIds);
Map<UUID, String> userNameMap = loadUserNameMap(perfMap.values());
```

**判断**：
- 订单**明细展示**需要在订单表 join 业绩域，这是**展示层**的合理需要（订单明细 VO 设计本就跨域）
- 订单域**主流程**（`OrderService` / `OrderSyncService` / `OrderAttributionService`）**没有越界**：仅读 `colonelsettlement_order` + `pick_source_mapping`，**不写** `performance_records`，**不调** `PerformanceCalculationService`
- 跨域 join 由 `DataApplicationService`（**分析模块**）执行，符合 ADR-005 "分析模块只读汇总"

**结论**：**未越界**。订单域边界完整，分析模块职责清晰。
