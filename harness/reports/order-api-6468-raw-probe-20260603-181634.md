| 维度 | 结果 |

## 核心发现

1. **6468 (`buyin.instituteOrderColonel`) 是有效订单数据源**：仅 1 天窗口就返回 20 条真实已支付订单
2. **2704 (`buyin.colonelMultiSettlementOrders`) 无法覆盖实际订单**：远端 6h 18 次 live 调用全部 fetched=0
3. **当前同步管道只调 2704，从不调 6468**：这是订单不入库的根因
4. **6468 有时间范围限制**：仅支持查询支付时间近 1 年的订单数据

## 建议：双源同步架构

```
OrderSyncJob
  ├── 6468 (buyin.instituteOrderColonel) → 订单事实源
  │   ├── 按小时滑动窗口拉取
  │   ├── 解析 order_list → 写入 colonelsettlement_order
  │   └── 字段：order_id, product, shop, pay_amount, author, flow_point
  │
  └── 2704 (buyin.colonelMultiSettlementOrders) → 结算补充源
      ├── 按天/周拉取
      └── 补充 settled_commission, settled_tech_service_fee 字段
```

## 剩余风险

1. 6468 API 分页拉取逻辑需实现（cursor-based，当前只拉了第 1 页 20 条）
2. 6468 返回的字段结构与 2704 不同，`toOrderListResult()` 解析逻辑可能需要适配
3. `flow_point=REFUND` 的订单需要明确同步策略（是否入库、如何标记）
4. 双源同步的去重策略需设计（同一 order_id 可能从两个接口都拉到）
5. 6468 时间范围限制（1 年）需要在同步窗口设计上考虑
