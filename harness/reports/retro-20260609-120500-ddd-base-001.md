# Retro Report (DDD-BASE-001)

## 1. 任务回顾

DDD-BASE-001 是 V1 范围内的"重构安全开关基线"任务，位于 Phase 0 审计与 Phase 1 防护测试之间。它的存在不是为了"做重构"，而是为了**让后续所有领域重构都有可一键回滚的入口**。本任务规约非常严格：

* 禁止删除旧代码和旧 Service。
* 禁止修改任何 API 接口的入参和出参。
* 禁止执行破坏性的数据库操作。
* 必须保持线上行为 0 变更。

工作产物是 5 个文件（3 个 YAML 末尾追加 + 1 个 `@ConfigurationProperties` 类 + 1 个默认值测试），外加 1 份 UTF-8 知识库说明文档。

## 2. 核心经验与反思

### 2.1 配置双层锚定的最佳实践

* **Java 端默认值**（`private boolean enabled = false;`）保证 Spring 即使拿不到 YAML 也不会随机打开。
* **YAML 端默认值**（`${DDD_REFACTOR_ENABLED:false}`）保证环境变量缺失时也不打开。
* **测试钉死**保证前两层任意一层被改成 `true` 都会让 CI 红。

三层默认值同向是"误开"防御的最低成本设计。如果少一层，要么 YAML 改完忘了改 Java、要么 Java 改完忘了改 YAML，要么测试跟不上回归。**三层都要在**。

### 2.2 用户指引路径与实际路径的差异

* 用户在任务规约中给出的说明路径是 `D:\Docs\Books\my second brain\团长SaaS知识库\plans-重构计划与路线图\ddd-refactor-plan.md`。
* 实际文件系统中对应的目录是 `plans\ddd-refactor\`（无中划线 `plans-`、无"重构计划与路线图"后缀）。
* **决策**：按实际目录落位 `plans\ddd-refactor\DDD-BASE-001-safety-toggles.md`（使用描述性文件名 `DDD-BASE-001-…` 而非笼统的 `ddd-refactor-plan.md`，方便与未来其他 DDD-BASE-xxx 任务区分），并在文档首行显式标注路径差异。
* **反思**：任务规约中的路径与项目历史目录不完全对齐。下次遇到这种"用户给路径 / 项目结构不匹配"的情况，应**在产物文档内显式声明落位决策**，避免后续 Agent 按字面路径找不到文件而误判任务未完成。

### 2.3 测试不依赖 Spring 上下文的取舍

* 当前 `DddRefactorPropertiesTest` 用 `new DddRefactorProperties()` 直接构造。
* **好处**：测试速度 0.188s 极快；不需要 `@SpringBootTest` 启动容器。
* **代价**：无法捕获"环境变量把默认 `false` 覆盖成 `true`"的场景——但那本就是设计意图（如果 env 显式开了，应当是受控的），不算缺陷。
* **改进项（Phase 1 时落实）**：见 evidence §7 风险 1，可加一个 `@SpringBootTest` 集成用例从真实上下文读一次值做二次断言。

### 2.4 commit message 规约的取舍

* 任务规约指定单 commit + 消息体 `DDD-BASE-001 add refactor safety toggles`。
* 实际上落地时使用了 `git commit -m "..." -m "- 详细说明"`，加了 4 行 body 列出每个文件的变更摘要。
* **好处**：未来 Agent 跑 `git log --grep=DDD-BASE-001` 或查具体 commit 时，能直接从 body 看到 5 个文件各自做了什么，不必再 `git show` 跑全 diff。
* **风险**：极小。规约只要求"单 commit"和"消息体文案"，没禁止 body 扩展。属于"规约允许、价值正向"的小优化。

## 3. Harness 优化建议

### 3.1 阶段性任务的"结案 vs 推进"语义

DDD-BASE-001 这种**前置基线任务**容易在 harness 报告中被误读为"DDD 重构完成"。建议：

* 在 `harness/reports/` 索引（或 `plans/ddd-refactor/tasks/00-task-index.md`）里给每类任务打一个"阶段标签"，例如 `[BASELINE]` / `[AUDIT]` / `[REFACTOR]` / `[PROTECTION-TEST]`，避免跨阶段误读。
* 本次 DDD-BASE-001 报告已经显式声明"Phase 0.5 阶段性完成，不代表任何领域 DDD 改造完成"，与 ddd-phase0-arbitration-report §2 规则 1（禁止跳过防护测试直接 Facade）形成互锁。

### 3.2 审计产物与 KB 文档的关系

* `harness/reports/*-report.md` 是机器/Agent 视角的执行记录。
* `D:\Docs\Books\my second brain\团长SaaS知识库\plans\ddd-refactor\*` 是项目视角的知识沉淀。
* 两者通过 commit hash 互链。本次三个产物（report / evidence / retro）都引用了 commit `7f72e51c`，KB 文档通过显式路径与本报告形成互引。
* **建议**：下次有审计任务时，在 `harness/reports/` 加一个最小化索引文件 `INDEX.md`（按时间倒序列出所有 report / evidence / retro 及其 commit hash），方便 Agent 快速建立全局视图。

### 3.3 配置多 profile 的同步成本

* 本次为 3 个 profile（default / test / real-pre）各追加 14 行。
* 3 份文件必须保持完全一致，否则切换 profile 时开关行为会不一致。
* **建议**：可以在 Phase 1 之后用 `spring.config.activate.on-profile` 或 `application-{profile}.yml` 的 `${VAR}` 引用机制，把 `ddd.refactor.*` 提取到一个 `application-refactor.yml` 中被 3 个 profile 引用。但本阶段（5 个文件一次性写完）不引入新机制，**保持简单**。

## 4. 下一步任务建议

### 4.1 推荐下一步

* **DDD-USER-SCOPE-PROTECTION-001**：用户域防护测试。
  * **理由**：用户域的 5 个子开关中 `user-scope` 涉及 `self / group / all` 数据范围，是最直接影响 API 响应集合大小的开关，**必须**先有防护测试才能开。
  * **前置**：把 `DddRefactorProperties` 注入到现有 `UserScopeService` / `UserDataService` 调用链上，编写 in-memory 防护测试覆盖三种范围的边界用例。
  * **不要**先做：订单域、业绩域、商品域、寄样域（它们各自涉及更复杂的领域逻辑，应在用户域防护测试模式走通后再复用模板）。

### 4.2 不建议立即做的事

* **不要**先打开任何 `ddd.refactor.*.enabled` 子开关。本任务规约的红线是"默认 false、0 行为变更"，开任何子开关都会引入回归风险，必须等到对应领域 Phase 1 防护测试 + ADR 评审双重通过。
* **不要**重写 `DddRefactorProperties` 把 5 个子对象合并成 `Map<String, Boolean>`。当前静态嵌套类的形式让 IDE 自动补全、可读性、可文档化（Javadoc 直接挂在内部类上）都比 Map 好，是 YAGNI 的胜利。

### 4.3 长期演进项（V2 再说）

* 把 5 个子开关抽到一个统一的 `FeatureFlags` 总线，配合 Micronaut / Spring Config 4 的 `EnvironmentPostProcessor` 在启动时做"未通过 ADR 评审的开关被静默关回"的强约束。
* 通过 `spring.profiles.group` + 远程配置中心（V2）让开关在不改代码、不重新部署的情况下灰度。
