- **Controller**: `OrderController`, `OrderAttributionController`

- **Event/Listener**: `OrderSyncedEvent`, `OrderSyncedEventListener`

## 4. 订单同步入口
- **增量自动同步**: 触发 `syncLatestWindow`，推进 `order:sync:last_time`。
- **补拉回扫 (PAY_RECENT)**: 触发 `syncPayRecentWindow`，回扫 6 小时已付款订单，推进 `order:sync:pay_recent_last_time`。
- **6468 事实源增量**: 触发 `syncInstituteOrdersRecentWindow`，推进 `order:sync:institute_recent_last_time`。
- **6468 实时热同步 (INSTITUTE_HOT)**: 触发 `syncInstituteOrdersHotRecent`，每分钟追最新付款单，推进 `order:sync:institute_hot_last_time`。
- **按订单号精确同步**: 手动精确查询并持久化。

## 5. 6468 链路
- 接口为 `buyin.instituteOrderColonel`。调用 `fetchInstitute` 并解析 `has_more`。数据以 `INSTITUTE` 记入，调用 `applyInstituteFactToOrder` 写入预估轨，保护已有结算轨金额。

## 6. 2704 链路
- 接口为 `buyin.colonelMultiSettlementOrders`。负责更新订单结算状态和结算金额，在更新时调用 `mergeEstimateSnapshot` 保护已有的预估轨字段不被覆盖。

## 7. hot recent 链路
- 小窗口、每分钟追付款单，具有独立的 `JobLockKeys.ORDER_SYNC_INSTITUTE_HOT` 锁以及 `order:sync:institute_hot_last_time` 水位，失败时不推进，交由普通增量增补。

## 8. cursor 分页
- `resolveNextCursor` 从 Map 响应体（支持 `data.cursor`、`data.data.next_cursor`）解析游标，退出机制完善（含空页、游标重复、以及 `maxPages` 限流）。

## 9. Redis checkpoint / lock / freshness
- 水位 key 包括增量、补拉、6468、热同步独立 key。
- lock 包括 `ORDER_SYNC`、`ORDER_SYNC_PAY_RECENT`、`ORDER_SYNC_INSTITUTE_HOT` 和 `ORDER_SYNC_INSTITUTE`。
- 水位更新已具备非 test 模式下的严格异常抛出，热同步日志会输出 `freshnessLagSeconds`。

## 10. Gateway / SDK 适配
- `RealDouyinOrderGateway` 在 contract 模式下通过硬编码返回夹具。
- 缺点：`OrderItem` 的 `rawPayload` 仍是一个 Map 并在 Service 中直接被 `pickNested`，导致 SDK 的 Json 模型泄漏到了业务层，亟需 ACL。

## 11. 落库与幂等
- 使用 claim 去重表抢占行锁防并发。
- 数据已存在时使用乐观锁 `updateSyncedById` 更新，不存在则 `insertIgnoreByOrderId` 写入。

## 12. 默认归因
- 优先级：独家商家 -> 独家达人 -> 原生团长 buyin_id -> pick_source 映射。
- 包含达人认领防冲突保护（`hasTalentClaimOwnerConflict`），有冲突时归为 `UNATTRIBUTED`。

## 13. 双轨金额字段
- 包含 `pay_amount`（实付）、`settle_amount`（结算）、`estimate_service_fee`（预估服务费）与 `effective_service_fee`（实际服务费，已完成结算扣减）。由 `OrderDualTrackAmountResolver.resolve` 负责映射。
- 金额可能在 `settle_amount` 缺失时回退到实付，存在回退逻辑污染性能表的已知风险。

## 14. 订单状态映射
- 映射 `flow_point` / `order_status` 字符串为本地 Integer，`PAY_SUCC` 对应 1，`REFUND`/`CLOSED` 对应 4。

## 15. OrderSyncedEvent 事件链路
- 在订单持久化成功后，通过 `publishAfterCommit` 采用 `TransactionSynchronization` 机制，在事务提交后异步发布 `OrderSyncedEvent`，防止导致原订单事务死锁或失败。

## 16. 跨域依赖
- 强依赖 `SysUserMapper`（用户域）以补齐用户名。
- 依赖 `PickSourceMappingService`（转链/商品域）以补齐转链。
- 依赖 `MerchantService`（商品/招商域）以沉淀商家。
- 依赖 `SampleLifecycleService`（寄样域）以触发寄样状态变迁。

## 17. 分层问题
- `OrderSyncService` 违反了单一职责原则，聚合了定时器、Json 解析、SDK 映射与编排多重功能。
- `DataApplicationService` 虽然继承 `BaseController`，但被放到了 `service` 包中，命名混乱。

## 18. God Service / 胖 Service
- `OrderSyncService` (1147行, 52KB) 具备典型的 God Service 属性，应予解耦。

## 19. DDD 拆分建议
1. 提取 `OrderAttributionPolicy` 以解耦归因算法。
2. 重构 `DouyinOrderGateway` 适配器（ACL），将 Json Map 转换为 Domain Entity，阻止 SDK 数据泄露。
3. 封装 `CheckpointStore` 端口以隔离 Redis 基础设施。
4. 剥离 `OrderSyncApplicationService` 用于业务编排，并将定时调度独立。
