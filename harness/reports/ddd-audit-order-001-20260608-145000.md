## 5. 6468 查询团长订单链路

当前实现事实：

- Gateway 入口：`RealDouyinOrderGateway.listInstituteOrders()` 调用 `OrderApi.listSettlement()`，后者对应 `buyin.instituteOrderColonel`。
- 同步入口：`syncInstituteOrdersHotRecent()`、`syncInstituteOrdersRecentWindow()`、`syncInstituteFullBackfillWindow()` 和 dry-run。
- 时间窗口：
  - Hot：`resolveInstituteHotStartTime(endTime)`，有 hot 水位时 `lastHot - overlap`，否则 `endTime - instituteHotWindowSeconds`。
  - Recent：有 institute 水位时 `last - overlap`，否则回退 24h；窗口上限 24h。
  - Full backfill：固定 24h 回补。
- lag / overlap：Hot 使用 `instituteHotLagSeconds`、`instituteHotOverlapSeconds`；Recent 使用普通 `lagSeconds` 和 `overlapSeconds`。
- 分页：`syncItemsWithLimits()` 支持 `hasMore` 或可遍历 cursor 继续翻页，记录 `pagesFetched`、`uniqueOrders`、`inserted`、`updated`、`failed` 和 `stopReason`。
- 6468 金额落库：`mapOrder()` 解析 raw payload 后，`SyncSource.INSTITUTE` 走 `OrderDualTrackAmountResolver.applyInstituteFactToOrder()`，只写 `order_amount`、`actual_amount`、`estimate_*`、`estimate_service_fee_expense`，不写 `settle_amount`、`effective_*`、`settle_colonel*`。
- raw payload：`order.setExtraData(rawPayload)`。
- checkpoint 推进：上游调用或 fetch 抛出 RuntimeException 时不会执行持久化水位；`syncItemsWithLimits()` 正常返回后推进 endTime，即使存在单订单处理失败计数。

DDD 问题：

- `OrderSyncService` 同时承担窗口计算、Gateway 调用、cursor 循环、raw payload 到 Entity 映射、归因调用、批量用户名补齐和持久化编排。
- Gateway 已有接口名，但 raw Map 仍穿透到 Service，未形成真正 ACL。

风险：

- hot / recent / full 的 checkpoint 行为需要测试锁定，尤其是单订单失败但整体推进水位的场景。
- 当前真实 6468 样本仍缺系统转链产生的 `pick_source` 命中，不能宣称渠道归因闭环。

## 6. 2704 团长结算订单链路

当前实现事实：

- Gateway 入口：`RealDouyinOrderGateway.listSettlement()` 调用 `OrderApi.listColonelMultiSettlementOrders()`，接口为 `buyin.colonelMultiSettlementOrders`，`time_type` 使用 `update`。
- 同步入口：2704 普通增量、PAY_RECENT、Webhook 精确订单同步。
- 2704 金额落库：`SyncSource.SETTLEMENT` 走 `OrderDualTrackAmountResolver.applyToOrder()`，写 `settle_amount`、`effective_service_fee`、`effective_tech_service_fee` 等结算轨字段，同时写预估轨字段。
- 与 6468 关系：`OrderSyncPersistenceService.mergeBySource()` 在 2704 更新已有订单时调用 `mergeEstimateSnapshot(existing, incoming)`，保护已有 `estimate_*` 和 `order_amount`。
- 结算样本：历史 evidence 显示本地 real-pre 曾 `settle_nonzero=0`，结算轨仍缺真实非零样本验证。

阶段性结论：

- 2704 代码路径存在且保留；但如果真实上游返回的 `settle_amount` / `settle_time` 缺失，则 `OrderDualTrackAmountResolver.resolve()` 会把 `settleAmount` 回退到 `payAmount`，这是需要测试保护的结算轨污染风险。
- 不能把订单到业绩 anti-join=0 写成“结算轨已验证完成”。

## 7. hot recent 热同步链路

当前实现事实：

- 调度：`OrderSyncJob.syncInstituteOrdersHot()`，默认每分钟。
- 独立锁：`JobLockKeys.ORDER_SYNC_INSTITUTE_HOT`。
- 独立 checkpoint：`order:sync:institute_hot_last_time`。
- 目标窗口：hot 水位 / hot window + hot lag + hot overlap。
- safety caps：使用 `instituteHotPageSize`、`instituteHotMaxPages`、`instituteHotMaxOrders`。
- freshness：成功路径调用 `logInstituteHotFreshness()`，输出 `freshnessLagSeconds`、`latestPayTimeBefore`、`latestPayTimeAfter`、`stopReason`。
- 失败行为：上游 RuntimeException 不推进 checkpoint；但单订单处理失败计入 `failed` 后仍可能正常返回并推进 checkpoint。
- 与普通 recent：hot 使用独立 lock / checkpoint；large recent 每 10 分钟补偿。

历史证据：

- `order-sync-freshness-optimize-001` 实现后曾因上游 `isv.signature-invalid` 阻塞 SLA 验证，后续 `douyin-upstream-reconnect-local-001` 记录本地上游恢复。

需要补的测试：

- hot fetch 失败不推进 checkpoint。
- hot 单订单失败是否允许推进 checkpoint 的业务接受性。
- `MAX_PAGES` / `MAX_ORDERS` stopReason 下 checkpoint 推进行为。

## 12. 默认归因

当前归因流程：

```text
订单 raw payload + order
 -> 提取 activityId / productId / pickSource / pickExtra / talentUid
 -> 查询 ProductOperationState 获取活动商品 assigneeId（招商输入）
 -> exclusive.enabled=true 时尝试独家商家 / 独家达人（V1 禁用能力，默认 false）
 -> colonel_order_info / colonel_buyin_id 原生团长映射
 -> pick_source / pick_extra 映射
 -> talent claim owner conflict guard
 -> ATTRIBUTED / UNATTRIBUTED + reason
```

阶段性结论：

- 代码保留独家商家/独家达人分支，但 `application.yml` 默认 `exclusive.enabled=false`，且 V1 禁止启用独家覆盖；审查时应写成“配置关闭的历史/预留风险分支”，不能写成 V1 当前业务能力。
- 招商输入来自 `ProductOperationState.assigneeId`，订单域保存为 `colonelUserId`，最终业绩归属由业绩域处理。
- 无真实系统转链 `pick_source` 样本时，渠道归因闭环仍只能写 `BLOCKED_BY_SAMPLE` / `PENDING`。

DDD 问题：

- `AttributionService` 直接依赖 `PickSourceMappingMapper`、`ProductOperationStateMapper`、`TalentMapper`、`TalentClaimMapper`、`ExclusiveTalentService`、`ExclusiveMerchantService`，跨商品/达人/历史独家能力。
- 后续应提取 `OrderAttributionPolicy`，并通过 Facade / Port 获取必要输入。

## 13. 双轨金额字段

字段事实：

- `pay_amount` / `order_amount`：订单实付/预估轨输入。
- `settle_amount`：结算轨输入，2704 路径写入。
- `estimate_service_fee`：预估服务费收入。
- `effective_service_fee`：结算服务费收入，代码在非 rate 计算场景会扣减 effective tech service fee。
- `estimate_tech_service_fee` / `effective_tech_service_fee`：双轨技术服务费。
- `estimate_service_fee_expense` / `effective_service_fee_expense`：当前 raw payload 无字段，解析器默认 0。

风险：

- `resolve()` 在结算金额缺失时把 `settleAmount` 回退为 `payAmount`；6468 路径不写 settle 字段，但 2704/settlement 路径可能带来结算轨污染。
- `applyToOrder()` 兼容旧字段时 `settle_colonel_commission` 回退到预估服务费，需测试保护避免下游误用旧字段。
