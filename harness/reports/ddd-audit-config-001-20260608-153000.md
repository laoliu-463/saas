# 配置域 DDD 只读审查报告 (DDD-AUDIT-CONFIG-001)

## 1. 配置域代码结构

配置域现有的类结构包括：
* **数据模型与映射器：**
  * `SystemConfig.java` / `SystemConfigMapper.java`：系统通用配置模型与 DB 映射。
  * `CommissionRule.java` / `CommissionRuleMapper.java`：提成（佣金）维度规则与 DB 映射。
  * `SystemConfigChangeLog.java` / `SystemConfigChangeLogMapper.java`：配置变更审计日志。
* **业务服务：**
  * `SysConfigService.java`：配置基础维护，提供新增、更新、删除及批量更新接口，包含操作审计日志、敏感数据遮蔽、outbox领域事件和Spring事件发布。
  * `BusinessRuleConfigService.java`：针对具体业务参数（如寄样重复限制天数、达人保护期、安全锁定）的只读读取服务，集成了 `ShortTtlCacheService`（5 分钟 TTL 本地缓存）。
  * `CommissionRuleService.java`：按照 `product > activity > user > global` 顺序多级回退解析提成比例。
* **监听器：**
  * `ConfigChangedCacheInvalidationListener.java`：监听 `ConfigChangedApplicationEvent` 并在配置事务成功提交后刷新对应缓存。

---

## 2. 配置读取与更新路径

### 2.1 读取路径
* 核心业务规则均通过 `BusinessRuleConfigService` 代理读取，先查找 `ShortTtlCacheService` 的本地缓存，若缺失则通过 `SystemConfigMapper` 查询数据库并存入缓存。
* 提成比率在计算时，会触发 `CommissionRuleService#resolveRatio` 检索 `commissions` 规则库。

### 2.2 更新路径
管理员在 UI 或 API 提交批量更新 -> `SysConfigService#batchUpdateByKeys` 拦截校验 -> 更新 DB -> 记录 `SystemConfigChangeLog` -> 领域事件存入 `domain_event_outbox` -> 发布 Spring 事件 `ConfigChangedApplicationEvent` -> 事务提交后，监听器监听到事件并调用 `BusinessRuleConfigService#invalidate` 逐项失效缓存。

---

## 3. 缓存机制与失效一致性
* **缓存：** 5 分钟 TTL 的本地强引用内存缓存。
* **前缀：** `biz-config:raw:`。
* **失效：** 使用 `TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)` 强一致保护。仅在数据库更新事务成功提交后才执行驱逐，防范了因更新回滚导致缓存与数据库不一致的潜在漏洞。
* **风险：** 本地缓存在多节点部署时无法通过单机 Event 实现多实例同步，但在抖音团长 SaaS V1 单节点/短TTL模式下风险可控。

## 4. 跨域直接直查 system_config 风险

审计中发现 **4 处** 严重的跨域直查表行为，绕过了配置域的缓存与接口隔离：

1. **`ExclusiveTalentService.java`**
   * 直接使用 `JdbcTemplate` 执行原生 SQL 读取 `talent.exclusive.service_fee_ratio`。
   * 直接使用 `JdbcTemplate` 执行原生 SQL 读取 `talent.exclusive.monthly_samples`。
2. **`ExclusiveMerchantService.java`**
   * 直接使用 `JdbcTemplate` 执行原生 SQL 读取 `merchant.exclusive.service_fee_ratio`。
3. **`CommissionService.java`**
   * 在业绩高频计算中，直接使用 `JdbcTemplate` 执行原生 SQL 读取 `commission.business_default_ratio`、`commission.channel_default_ratio` 及活动比例覆盖配置。
4. **`TalentPresetTagsBootstrap.java`**
   * 启动时直接注入并使用 `SystemConfigMapper` 读写 `talent.preset_tags` 配置。

## 6. DDD 拆分建议 (Phase 2 与 Phase 4)

* **`ConfigDomainFacade` (Facade门面)**
  * 提取跨域读取接口，收敛业绩、达人、招商对全局配置和提成比例的查询。
* **`ConfigValidationPolicy` (写入验证策略)**
  * 收敛 `ConfigDefinitionRegistry` 的校验职责，确保在配置被更改时，值类型和业务范围正确。
