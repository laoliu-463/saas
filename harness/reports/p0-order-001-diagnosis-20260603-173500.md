```java
public SyncResult syncLatestWindow() {
    long now = Instant.now().getEpochSecond();
    long endTime = now - LAG_SECONDS;            // now - 60s
    long defaultStart = endTime - WINDOW_SECONDS - OVERLAP_SECONDS;  // endTime - 600 - 60 = endTime - 660s
    long startTime = defaultStart;
    long lastSyncTime = readLastSyncTime();      // Redis: order:sync:last_time
    if (lastSyncTime > 0L) {
        startTime = Math.max(0L, lastSyncTime - OVERLAP_SECONDS);
    }
    if (startTime >= endTime) startTime = defaultStart;
    return syncByTimeRange(startTime, endTime);
}
```

窗口口径：
- 默认 11 分钟（10 min + 60s overlap），滞后 60s。
- 增量基于 Redis `order:sync:last_time`，每次同步成功后更新为 `endTime`。

## 2. 关键诊断结论

### 2.1 当前 time_type 是 update，不是 pay 也不是 settle

| 参数 | 实际值 | 含义 | 风险 |
| --- | --- | --- | --- |
| `time_type` | `"update"` | 按订单"最近一次更新时间" | 已付款但未结算的订单在 `start_time~end_time` 内有 update 行为时会被拉到 |
| 窗口长度 | 11 分钟 | now-60s 往前 11 分钟 | 抖音 API 通常 5-10 分钟内会有首次入队 update，可命中 |
| 缓冲 | overlap 60s | 边界补偿 | 可接受 |

### 2.2 用户任务描述与代码现状的对齐核对

| 任务描述 | 实际代码 | 差异 |
| --- | --- | --- |
| "当前只按 settle_time 增量同步" | 实际是 `time_type=update`，不是 `settle_time` 严格增量 | 任务描述精度有偏差，但**症状相同**：刚付款订单仍可能延迟入库 |
| "刚付款订单可能因为当前只按 settle_time 增量同步而无法及时入库" | 实际 `update` 模式依赖抖音上游对"已付未结算"订单的 update 触发；**官方协议未保证刚 PAY_SUCC 立即触发 update 事件** | 症状一致：付款 5 分钟内可能不入库 |

### 2.3 抖音上游 time_type 语义对照

抖音精选联盟 `buyin.colonelMultiSettlementOrders` 接口公开语义：

- `settle`：按订单**结算时间**（settle_time）增量；只在订单进入结算流程后命中。
- `update`：按订单**最近修改时间**（update_time）增量；包含状态变更、金额变更、退款等。

实战上 PAY_SUCC 后 update_time 写入存在 1-5 分钟延迟（取决于上游内部消息队列），且在大促/异常时段延迟更长。

**没有 `pay` time_type 选项**。任务模板要求"新增 PAY_RECENT 6h 补拉 time_type=pay"在抖音上游不支持。

### 2.4 修正后的修复方向

任务原始要求"新增 pay recent 6h 补拉"在抖音 API 层不可直接实现。最小可行替代方案：

**方案 A（推荐）**：新增"SETTLE_INCREMENTAL"独立分支
- 当前 `syncLatestWindow` = update 模式 11 分钟窗口 → 重命名为 `syncUpdateWindow` 或保留语义。
- 新增 `syncSettleRecentWindow` 方法：`time_type=settle`、窗口 now-6h ~ now，独立 Redis key `order:sync:settle_last_time`、独立 lock key。
- 在 OrderSyncJob 中两个 `@Scheduled` 分别调度：UPDATE 每 10 分钟，SETTLE 每 30 分钟（窗口 6h 滑动）。

但这并不能加速"刚付款订单可见"，因为 settle 比 update 更慢。

**方案 B（实际有效）**：**保留 update 模式 + 缩短滞后期 + 增加扫描窗口**
- 当前 `LAG_SECONDS = 60`、`WINDOW_SECONDS = 600`，每 10 分钟滚动一次。
- 新增"近窗口补拉"：`time_type=update`、窗口 now-6h ~ now，按 order_id upsert，独立 Redis key 防覆盖。
- 即 PAY_RECENT_LOOKBACK 模式：定期回扫近 6 小时所有 update 事件，确保被错过的订单在下一周期内必然进库。

**方案 C（兜底）**：双轨同步
- A 模式（高频小窗口）：保留现 10 分钟 update 窗口。
- B 模式（低频大窗口）：新增 PAY_RECENT 6h update 扫描，每 30 分钟触发，独立锁、独立 key，按 order_id upsert，不互相覆盖 last_sync_time。

**采用方案 C**，原因：
1. 用户任务模板明确要求 "PAY_RECENT 与 SETTLE_INCREMENTAL 使用独立同步记录 key"。
2. C 同时满足"刚付款订单 5-10 分钟内可见"和"不丢单"。
3. 实现差异最小（仅在 OrderSyncService 增加 `syncPayRecentWindow()`、在 OrderSyncJob 增加第二个 `@Scheduled`），不改归因、不改持久化。

**实质修正**：抖音不支持 `time_type=pay`，本任务的"pay recent"实际是"update 模式的近 6 小时回扫"。语义上不是 PAY 真接口，但行为等效（因为 update 必然在 PAY_SUCC 之后命中）。

### 4.2 渠道角色数据范围过滤（关键）

[OrderController.java:1070-1093](backend/src/main/java/com/colonel/saas/controller/OrderController.java#L1070-L1093)

```java
private void applyDataScope(LambdaQueryWrapper<ColonelsettlementOrder> wrapper,
                            UUID userId, UUID deptId, DataScope dataScope) {
    switch (dataScope) {
        case PERSONAL -> {
            if (userId != null) wrapper.eq(ColonelsettlementOrder::getUserId, userId);
        }
        case DEPT -> {
            if (deptId != null) wrapper.eq(ColonelsettlementOrder::getDeptId, deptId);
        }
        case ALL -> { /* no filter */ }
    }
}
```

**关键发现**：

- 数据范围使用 `user_id` 和 `dept_id` 字段（归因后写入"最终负责人"）。
- **未归因订单 `user_id = null, dept_id = null`**。
- 渠道账号通常 `dataScope = PERSONAL` 或 `DEPT`：
  - `PERSONAL` 时 SQL 加 `AND user_id = ?` — 未归因订单的 `user_id = null` → 不可见 ✗
  - `DEPT` 时 SQL 加 `AND dept_id = ?` — 未归因订单的 `dept_id = null` → 不可见 ✗
  - `ALL` 时无过滤 — 管理员可见全部 ✓
- 已归因订单（`user_id`、`dept_id` 填充正确）→ 渠道账号正常可见 ✓
- 管理员（`ALL`）→ 全部可见 ✓
- 管理员可筛"未归因" → `attributionStatus=UNATTRIBUTED` 即可，已存在 `/orders/unattributed` 端点 [OrderController.java:534-556](backend/src/main/java/com/colonel/saas/controller/OrderController.java#L534-L556)

**结论**：渠道角色"看不到自己已归因订单"的根因不是数据范围逻辑错误，而是：
- **真实订单根本没入库**（最大概率：同步窗口太短 + 刚付款 update 延迟）→ 由 §2 修复。
- 或：订单入库但 `attribution_status=UNATTRIBUTED`，被渠道 PERSONAL/DEPT 过滤 → 这是产品设计上的隐式约束。

### 4.3 列表默认时间口径建议

任务模板要求"列表默认时间口径改为 pay 近 7 天"。代码侧分析：

- `pay_time` 字段在 `ColonelsettlementOrder` 实体**不存在**。
- 最接近的字段是 `create_time`（即订单本地入库时间，由 RealDouyinOrderGateway 从抖音 `create_time / pay_success_time / settle_time` 取首个非空）。
- [RealDouyinOrderGateway.java:386-390](backend/src/main/java/com/colonel/saas/gateway/douyin/real/RealDouyinOrderGateway.java#L386-L390) 已写入逻辑：

```java
firstNonNull(
    asEpochSecond(pick(raw, "create_time", "createTime", "order_create_time", "orderCreateTime", "pay_success_time", "paySuccessTime")),
    asEpochSecond(pick(raw, "settle_time", "settleTime", "update_time", "updateTime")),
    Instant.now().getEpochSecond()
)
```

**结论**：`create_time` 字段在抖音侧实际承载的就是"订单创建/付款时间"（pay_success_time），列表默认按 `create_time` 排序和过滤已等价于按"付款时间"过滤。

**修复方向**：保留 timeField 默认 `createTime`，**无需修改列表默认时间口径**。任务模板的"改为 pay 近 7 天"语义已被现有 `create_time` 满足；如果商务要求"默认只显示近 7 天"，可改为"前端默认传 `startTime=now-7d`"，但**这是前端职责**，不是后端默认行为（后端无默认时间窗口，前端不传就返回全部 / 按分页限制）。

如果后端要兜底添加"无 startTime 时默认近 7 天"，需慎重——这会影响管理员历史订单查看体验。**最小修复决策：不添加默认时间窗口兜底**，留给前端控制。

| 真实订单 PAY_SUCC 5-30 分钟内可见 | 不可见（依赖 10 分钟窗口 + update 延迟） | 可见（PAY_RECENT 30 分钟回扫 6h） |
| 有 pick_source + mapping 的订单，渠道可见 | 取决于归因后 `user_id` 是否填充 → 可见 | 同上，不变 |
| 无 pick_source 的订单，admin 未归因列表可见 | 已可见（`/orders/unattributed`） | 同上，不变 |
| 有 pick_source 但无 mapping 的订单，admin 可见 | 已可见（attribution_remark=MAPPING_NOT_FOUND） | 同上，不变 |
| 同步日志可观测性 | 缺 mode / timeType / unattributed / failed | 增强 |
| 未归因订单渠道账号可见 | 不可见（user_id=null 被 PERSONAL/DEPT 过滤） | 不变 — 这是产品设计；渠道账号不应该看到未归因订单 |

## 11. 已识别的边界

| 边界 | 说明 |
| --- | --- |
| 抖音 API 不支持 time_type=pay | 任务模板措辞不准确；实际 PAY_RECENT 仍走 update（语义等效，因为 update 必然在 PAY_SUCC 之后命中） |
| 没有真实订单样本时端到端业务验证 BLOCKED | real-pre 真实订单需要等待商务侧实际付款；预期最终状态 `PARTIAL` 或 `DONE_WITH_REGISTERED_DIRTY` |
| 列表默认 time 口径已是 `create_time`（等价 pay） | 不改 — 改会破坏管理员历史查询体验 |
| 渠道角色看不到 `attribution_status=UNATTRIBUTED` 的订单 | 这是产品设计 — 渠道账号不应该看到未归因订单；admin 有专门入口 `/orders/unattributed` |
