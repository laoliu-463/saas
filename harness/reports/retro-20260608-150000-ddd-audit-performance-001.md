# Retro - DDD-AUDIT-PERFORMANCE-001

## 1. 本次做对了什么
- **严格遵循 A 类只读边界**: 没有任何核心源码的改动，保证了业务基准的一致性。
- **梳理清晰的公式和关系**: 理清了预估轨和结算轨提成的底层计算机制、退款零提成策略，为接下来的单测回归建立了保障。

## 2. 本次发现了什么业绩域风险
- **配置域的跨域直接依赖**: `CommissionService` 绕过 Facade 概念，直接通过 `jdbcTemplate` 查询 `system_config` 配置表，增加了配置模型变迁时的跨域脆弱性。
- **归因重用与泄漏**: 业绩直接全量继承订单域的归属（final = default），未保留独家分配的能力，且没有在事件载荷中实现完整的防腐。

## 3. 哪些结论仍需后续测试确认
- **配置更改后的月度重算时序问题**: 管理员在重算月份业绩时，活动级比例覆盖的变更是否会正确在 `PerformanceMonthRecalculationService` 中生效，这需要通过 `DDD-TEST-PERFORMANCE-CALC-001` 回归验证。

## 4. 下一步为什么做寄样域审查
- **寄样状态与订单的业务闭环**: 渠道链闭环的最后一环是“订单同步 -> 渠道业绩 -> 寄样自动完成”。既然已经审清了订单域和业绩域，下一环应当切入寄样域，盘点其状态机如何响应订单同步事件并完成签收自动关闭的逻辑。因此下一步推进 `DDD-AUDIT-SAMPLE-001` (寄样域只读审查)。

## 5. 后续 Agent 应避免什么
- **严禁擅自修改公式和 SQL**: 在没有防护测试就绪前，千万不要去改 `CommissionService` 里的公式或者用 SQL 直接修复数据库，必须维持当前只读的审查事实。
