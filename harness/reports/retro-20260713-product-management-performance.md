# Retro：商品管理性能优化

## 本次完成

- 先用 real-pre SQL、接口日志和同步日志建立基线，再改代码。
- 将商品状态刷新从“整轮上游同步结束后”前移到“每页上游结果落库后”。
- 增加按活动串行锁、短 TTL 缓存和进行中前端刷新。
- 补充后端调用顺序回归测试与前端 500ms 轮询测试。
- 完成构建、容器重启、健康检查和真实定时同步观测。

## 证据结论

- 活动商品查询 SQL 实测 5.366ms，活动状态计数 0.814ms。
- 真实同步页返回后的状态刷新实测约 97–342ms。
- real-pre 业务预检因 admin login HTTP 401 阻塞，完整登录后 E2E 未通过也未伪造通过。

## 暴露的问题

- 上游接口单页耗时有 431–1041ms 波动；本轮只收敛了返回后的落库到状态刷新尾延迟。
- 商品库复杂筛选仍存在内存回退路径，不能用活动页查询结果代替其性能证据。
- Harness 的自动提交入口在业务预检失败时只生成失败证据，不会自动提交；本轮需人工确认后再提交本次代码。

## Harness 升级判断

- 暂不需要新增 Harness 规则。
- 建议后续在同步日志补充统一 `upstreamRequestStartAt`、`snapshotPersistedAt`、`libraryStateVisibleAt` 三个时间点，以便严格验证端到端 1 秒指标。
- 本次限制检查发现 `harness/reports` 有 82 个直接文件，历史资产超出 50 文件上限；未在本轮擅自归档，需单独安排清理窗口。
