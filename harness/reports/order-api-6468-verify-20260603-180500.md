## 7. 结论

### 最终分类：B — 代码有 6468 实现，但运行态没有调用

| 维度 | 2704 (`colonelMultiSettlementOrders`) | 6468 (`instituteOrderColonel`) |
|---|---|---|
| 代码实现 | ✅ OrderApi + Gateway + Service + Job | ✅ OrderApi + Gateway |

| Service 调用 | ✅ `fetchSettlement()` → `listSettlement()` | ❌ 未被 Service 调用 |
| Job 触发 | ✅ 每 10 分钟 | ❌ 无定时触发 |
| 远端运行态 | ✅ 18 次/6h | ❌ 0 次/6h |
| 用途 | 主同步路径 | 仅联调 RAW 探针 |
| 数据落库 | `colonelsettlement_order` (0 行) | 无 |

### 当前订单同步真实数据源

```
系统仅接入 2704 (buyin.colonelMultiSettlementOrders) 作为订单同步唯一数据源。
6468 (buyin.instituteOrderColonel) 代码已实现但未接入同步链路。
```

### 是否能覆盖"刚产生实际订单"

```
不能。
2704 是"团长分次结算订单"，仅返回已进入结算阶段的订单。
6468 是"查询团长订单"，覆盖抖音/火山/头条/西瓜实际订单，但未被调用。
```

## 8. 最小修复建议

### 8.1 双源同步架构

```
OrderSyncJob
  ├── syncInstituteOrdersRecent()          ← 新增
  │     ├── 调 /buyin/instituteOrderColonel (6468)
  │     ├── 拉最近付款/创建/更新订单
  │     ├── upsert 基础订单事实到 colonelsettlement_order
  │     └── 做 pick_source 默认渠道归因
  │
  └── syncColonelMultiSettlementOrders()    ← 已有
        ├── 调 /buyin/colonelMultiSettlementOrders (2704)
        ├── 拉分次结算订单
        ├── UPDATE 结算金额 / 服务费 / 技术服务费 / settle_time
        └── 触发业绩重算
```

### 8.2 具体改动点

1. **OrderSyncService** 新增 `fetchInstituteOrders()` 方法，调用 `douyinOrderGateway.listInstituteOrders()`
2. **OrderSyncJob** 新增一个同步方法或在现有 `syncLatestWindow()` 中先调 6468 再调 2704
3. **去重机制**：使用 `order_sync_dedup_claim` 表确保同一 `order_id` 不重复插入
4. **upsert 策略**：6468 先 INSERT，2704 按 `order_id` UPDATE 结算字段

### 8.3 验证步骤

1. 先用远端 RAW 探针 `POST /api/douyin/order-sync-probes/raw` 调用 6468，确认抖音后台是否有实际订单
2. 如果 6468 能返回订单，则实施双源同步改造
3. 如果 6468 也返回 0，则问题在抖音侧（店铺/团长确实无订单）
