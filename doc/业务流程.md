## 二、订单归因流程

### 2.1 流程说明

订单归因决定每笔成交订单的佣金归属，是团长结算的核心逻辑。

### 2.2 归因优先级

按以下优先级依次判断（详见 `doc/rules/attribution-logic.md`）：

| 优先级 | 类型 | 判断条件 | 归属 |
|--------|------|----------|------|
| 1 | 独家商家 | 商家在独家商家表中，且在活动有效期 | 该独家商家配置 |
| 2 | 独家达人 | 达人在独家达人表中，且在有效期 | 该独家达人配置 |
| 3 | pick_source 映射 | 商品有渠道来源映射记录 | 对应 pick_source |
| 4 | 默认 | 无以上匹配 | 系统默认比例 |

### 2.3 归因流程图

```
订单入库
    │
    ▼
┌─────────────────────────┐
│ 匹配独家商家？            │
│ (商家ID + 活动时间)       │
└─────────┬───────────────┘
    │ 是                         │ 否
    ▼                            ▼
┌──────────────┐    ┌─────────────────────────┐
│ 归属独家商家  │    │ 匹配独家达人？             │
│ (按商家比例)  │    │ (达人ID + 有效期)         │
└──────────────┘    └─────────┬───────────────┘
                              │ 是               │ 否
                              ▼                   ▼
                    ┌──────────────┐   ┌─────────────────┐
                    │ 归属独家达人 │   │ 匹配pick_source │ → 有 → 按渠道比例归属
                    │ (按达人比例) │   └─────────────────┘
                    └──────────────┘          │ 否
                                              ▼
                                      按系统默认比例结算
```

### 2.4 关键约束

- 独家商家和独家达人均需判断活动时间是否在有效期内
- 归因结果写入 `colonelsettlement_order` 表的 `attribution_type` 和 `attribution_id` 字段
- 归因为幂等操作，已归因的订单不再重新计算

## 三、寄样闭环流程

### 3.1 流程说明

寄样闭环描述从达人申请样品到达人完成作业的完整生命周期。

### 3.2 状态机定义

| 状态 | 状态值 | 说明 | 可执行动作 |
|------|--------|------|-----------|
| 申请 | `APPLIED` | 达人提交寄样申请 | 审核 |
| 待审核 | `PENDING` | 等待团长审核 | 通过/拒绝 |
| 审核通过 | `APPROVED` | 待发货 | 发货 |
| 已发货 | `SHIPPED` | 样品已发出 | 确认签收 |
| 已签收 | `RECEIVED` | 达人已收到样品 | 交作业 |
| 待交作业 | `PENDING_TASK` | 等待达人提交作业证明 | 交作业/超时关闭 |
| 已完成 | `COMPLETED` | 达人已提交作业 | — |
| 已关闭 | `CLOSED` | 超时/拒绝/取消 | — |

### 3.3 状态流转图

```
申请(APPLIED)
    │
    ▼
┌──────────────────┐
│  审核拒绝 ──→ 已关闭 │
│  (REJECTED)     │
└────────┬─────────┘
         │ 审核通过
         ▼
┌──────────────────┐
│  待发货(APPROVED) │
└────────┬─────────┘
         │ 发货
         ▼
┌──────────────────┐
│  已发货(SHIPPED)  │
└────────┬─────────┘
         │ 确认签收
         ▼
┌─────────────────────────────┐
│  已签收(RECEIVED)           │
└────────┬────────────────────┘
         │ 交作业
         ▼
┌──────────────────────────────┐
│  待交作业(PENDING_TASK)      │
└────────┬────────────────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
 交作业    超时30天
    │         │
    ▼         ▼
┌─────────┐ ┌─────────┐
│ 已完成  │ │ 已关闭  │
│COMPLETED│ │ CLOSED  │
└─────────┘ └─────────┘
```

### 3.4 自动闭环规则

**规则一：订单驱动自动完成**
- 当订单入库时，系统自动查找匹配的「待交作业」状态寄样记录
- 匹配条件：达人 + 商品 + 活动
- 自动将状态推进为「已完成」

**规则二：超时自动关闭**
- 「待交作业」状态持续 30 天无订单，自动关闭
- 关闭原因记录：`TIMEOUT_30D`
- 状态变更写入 `sample_log` 表

### 3.5 关键约束

- 状态变更必须记录日志（`sample_log` 表）
- 每个状态只能执行该状态下允许的动作（状态机校验）
- 订单自动完成逻辑仅触发一次，不重复推进

## 四、独家机制流程

### 4.1 机制说明

独家机制是团长的核心竞争力，通过与商家或达人建立独家合作关系，获得更高的佣金比例。

### 4.2 独家商家

**业务流程：**
1. 团长与商家签订独家合作协议
2. 在系统中录入独家商家信息（有效期、佣金比例）
3. 商家参与团长活动后，订单归因自动使用独家比例

**关键字段：**
- `merchant_id` — 商家 ID
- `exclusive_start_date` — 独家起始日期
- `exclusive_end_date` — 独家结束日期
- `commission_rate` — 专属佣金比例
- `service_rate` — 专属服务费比例

### 4.3 独家达人

**业务流程：**
1. 团长与达人签订独家合作协议
2. 在系统中录入独家达人信息（有效期、佣金比例）
3. 达人带货订单归因自动使用独家比例

**关键字段：**
- `talent_id` — 达人 ID
- `exclusive_start_date` — 独家起始日期
- `exclusive_end_date` — 独家结束日期
- `commission_rate` — 专属佣金比例

### 4.4 月度评估定时任务

**执行频率**：每月 1 日凌晨执行

**评估逻辑：**
1. 统计上个月每个独家商家/达人的带货数据
2. 考核指标：订单数量、GMV、佣金金额
3. 生成月度评估报告
4. 触发续签/降级/终止流程（预留接口）

### 4.5 归因优先级的业务含义

```
独家商家 > 独家达人 > pick_source
```

- **独家商家优先**：保护团长与核心商家的独家协议
- **独家达人其次**：保护团长与核心达人的独家协议
- **pick_source 兜底**：普通渠道按来源归因

## 五、订单同步流程

### 5.1 滑窗同步机制

```
定时任务触发（每5分钟）
    │
    ▼
获取Redis分布式锁
    │
    ▼
计算同步时间窗口
（上次同步时间 → 当前时间）
    │
    ▼
调用抖音订单API
(alliance.colonelActivityOrderList)
    │
    ├── 有数据 ──→ 批量写入数据库 ──→ 更新同步时间戳
    │
    └── 无数据 ──→ 直接更新同步时间戳
    │
    ▼
释放Redis锁
```

### 5.2 关键约束

- 使用 Redis SETNX 实现分布式锁，防止多实例重复执行
- 滑窗大小可配置，默认 5 分钟
- 批量写入采用 MyBatis-Plus `saveBatch`，批量大小可配置
- 同步异常时记录错误日志，不更新同步时间戳，下次继续尝试

## 六、数据看板口径

### 6.1 核心指标

| 指标 | 数据来源 | 计算方式 |
|------|----------|----------|
| GMV | `colonelsettlement_order` | SUM(order_amount) |
| 订单数 | `colonelsettlement_order` | COUNT(*) |
| 预估佣金 | `colonelsettlement_order` | SUM(colonel_commission) |
| 结算佣金 | `colonelsettlement_order` | SUM(settle_colonel_commission) |

### 6.2 数据隔离

- 所有查询必须携带 `merchant_id` 过滤条件（DataScope 切面自动注入）
- 超级管理员可查看全部数据
