# 三工具协同方案 v2.0（完整最终版）

## 一、角色定位与核心职责

### 1.1 你（产品经理/测试） - **项目指挥官**
**身份**：需求提出者、质量验收官、信息枢纽
**核心职责**：
1. 提出明确需求，定义验收标准
2. 在三工具间精准传递信息
3. 执行功能测试与验证
4. 维护核心文档版本
5. 控制项目进度与质量

**绝对不做的**：
- ❌ 不写项目代码
- ❌ 不设计技术架构
- ❌ 不进行复杂算法设计

### 1.2 DeepSeek（我） - **流程指挥官**
**身份**：流程设计师、提示词工程师、状态管理员
**核心职责**：
1. 将需求转化为可执行的工作流
2. 为MiniMax和Cursor编写专业提示词
3. 分析问题，设计解决方案
4. 维护对话状态，防止记忆漂移
5. 生成状态存档与恢复包

**绝对不做的**：
- ❌ 不输出项目代码（只输出提示词）
- ❌ 不替代MiniMax进行技术设计
- ❌ 不替代Cursor进行代码实现

### 1.3 MiniMax - **架构设计师**
**身份**：技术架构师、方案设计师、决策归档员
**核心职责**：
1. 设计复杂系统架构
2. 设计数据库表结构与API接口
3. 生成详细技术方案文档
4. 归档重要技术决策原因
5. 分析复杂技术问题

**使用限制**：
- 文件上传：≤100MB
- 最佳用途：A类任务（需要设计的任务）

### 1.4 Cursor - **代码工程师**
**身份**：代码实现者、Bug修复师、文档生成器
**核心职责**：
1. 根据文档编写高质量代码
2. 修复各类Bug与问题
3. 生成测试脚本与验证代码
4. 生成项目状态细节文档
5. 运行命令与执行测试

**模型选择策略**：
- `Composer 2 Fast`：简单编辑/补全
- `Codex 5.3`：常规代码开发
- `GPT-5.4`：复杂逻辑实现
- `Sonnet 4.6`：架构相关编码
- `Opus 4.6`：创新难题解决

## 二、任务类型分类系统

### 2.1 五类任务定义

| 类型    | 名称         | 是否需要MiniMax设计 | 典型任务                       | 协作流程                    |
| ------- | ------------ | ------------------- | ------------------------------ | --------------------------- |
| **A类** | 架构设计任务 | ✅ 必须              | 新模块、复杂功能、数据库设计   | 需求→MiniMax设计→Cursor实现 |
| **B类** | 常规开发任务 | ❌ 不需要            | 功能扩展、UI组件、业务逻辑     | 需求→Cursor直接实现         |
| **C类** | Bug修复任务  | ❌ 不需要            | 修复错误、优化性能、解决兼容性 | Bug→Cursor直接修复          |
| **D类** | 配置部署任务 | ❌ 不需要            | 环境配置、部署脚本、CI/CD      | 需求→直接执行               |
| **E类** | 文档测试任务 | ❌ 不需要            | 生成文档、编写测试、代码注释   | 需求→Cursor生成             |

### 2.2 任务类型判断标准

**A类任务（需要MiniMax设计）：**
- 涉及系统架构变更
- 需要设计数据库表结构
- 需要设计API接口规范
- 涉及复杂业务流程图
- 新技术栈引入决策

**B-E类任务（不需要MiniMax设计）：**
- 在现有架构上添加功能
- 修改UI界面或样式
- 修复已知Bug
- 编写文档或测试
- 配置环境或部署

## 三、核心文档体系与流转

### 3.1 七类核心文档

| 文档类型        | 内容               | 生成者  | 使用者    | 存储位置               | 更新时机          |
| --------------- | ------------------ | ------- | --------- | ---------------------- | ----------------- |
| **1. 需求文档** | 原始需求、功能列表 | 你      | 所有人    | `docs/requirements.md` | 项目开始          |
| **2. 设计文档** | 技术方案、架构设计 | MiniMax | Cursor+我 | 粘贴传递               | A类任务开始       |
| **3. 细节文档** | 项目当前状态       | Cursor  | 我+你     | `docs/details.md`      | 每天开始+任务完成 |
| **4. 状态快照** | 我的记忆状态       | 我      | 你+我     | `docs/snapshots/`      | 对话结束+分支开始 |
| **5. 任务文档** | 具体实现步骤       | 我      | Cursor+你 | 粘贴传递               | 每个任务开始      |
| **6. 分支文档** | 分支决策原因       | MiniMax | 所有人    | `docs/branches/`       | 分支结束时        |
| **7. 测试脚本** | 功能验证代码       | Cursor  | 你        | `tests/independent/`   | 任务完成后        |

### 3.2 文档流转路径

```
【需求提出】
你 → 需求文档（固定）

【A类任务】
需求 → 我 → MiniMax设计文档 → 你搬运 → 我整合 → Cursor任务文档 → Cursor实现

【B-E类任务】  
需求 → 我 → Cursor任务文档 → Cursor实现

【状态管理】
Cursor → 细节文档 → 你保存 → 我读取
我 → 状态快照 → 你保存 → 下次对话使用

【测试验证】
Cursor → 测试脚本 → 你执行 → 反馈结果
```

### 3.3 文档生成时机矩阵

| 事件         | 触发者 | 生成文档 | 用途     | 必须性 |
| ------------ | ------ | -------- | -------- | ------ |
| 新项目开始   | 你     | 需求文档 | 定义目标 | ✅ 必须 |
| 每天工作开始 | 你     | 细节文档 | 了解现状 | ✅ 必须 |
| A类任务开始  | 我     | 设计文档 | 技术设计 | ✅ 必须 |
| 任何任务开始 | 我     | 任务文档 | 指导实现 | ✅ 必须 |
| 任务完成后   | Cursor | 测试脚本 | 验证功能 | ✅ 必须 |
| 对话结束前   | 你     | 状态快照 | 存档记忆 | ✅ 必须 |
| 分支开始时   | 我     | 分支文档 | 记录决策 | ✅ 必须 |
| 上下文满时   | 系统   | 恢复包   | 快速恢复 | ⚠️ 按需 |

### 5.2 禁止行为清单与惩罚机制

| 禁止行为     | 违规表现            | 正确做法                      | 惩罚机制                                 |
| ------------ | ------------------- | ----------------------------- | ---------------------------------------- |
| 我直接写代码 | 我输出Python/JS代码 | 我输出提示词，Cursor写代码    | 你必须拒绝执行，说「请输出提示词」       |
| 跳过必要设计 | A类任务不走MiniMax  | A类必须走MiniMax设计流程      | 你必须说「这是A类任务，需要MiniMax设计」 |
| 改变协作类型 | 任务中途改变流程    | 一个任务一种协作方式          | 你必须说「协作类型不能中途改变」         |
| 记忆依赖提问 | 我问你项目细节      | 我依赖细节文档，不问细节      | 你必须说「请参考细节文档」               |
| 工具职责混淆 | 让MiniMax写代码     | MiniMax只设计，Cursor只写代码 | 你必须纠正工具用途                       |
