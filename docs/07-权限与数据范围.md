# 权限与数据范围

> 范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 状态边界

- [当前事实] V1 仍使用固定角色注解、JWT 角色/全局数据范围和现有数据范围切面；下文 V1 条目用于描述迁移基线，不代表长期安全目标。
- [V2 长期治理] [ADR-012](决策/ADR-012-用户域内建设权限中心化授权能力.md) 已采纳权限中心化 RBAC + 领域数据范围：接口检查权限编码，用户域返回 `allow/deny` 与当前权限的领域范围，业务域继续判断对象归属和状态。
- [V2 长期治理] 详细 Schema、JWT、灰度和验证设计见 [DDD RBAC 长期治理设计](superpowers/specs/2026-07-13-ddd-rbac-long-term-governance-design.md)；在实施证据完成前不得把目标状态写成已落地。

## 数据范围不变量

- [V1 必做] 用户域统一提供 `self/group/all` 数据范围。
- [V1 必做] 业务域不得自行发明数据范围语义。
- [V1 必做] 查询侧必须能说明当前用户、角色、组织和数据范围如何影响结果。
- [V1 简化] V1 以当前权限注解、后端过滤和 E2E 验证为主，复杂 ABAC / PBAC 放入 V2。

## V1 多角色合并规则

- [V1 必做] 同一账号拥有多个内置业务角色时，动作权限按并集合并，不能因新增一个角色而丢失已有业务能力。
- [V1 必做] `纯招商`、`纯渠道`、`纯运营`等收缩视图只在账号的内置角色集合中确实只有对应角色时生效；自定义菜单角色不改变岗位判断。
- [V1 必做] 数据范围仍由用户域统一解析；多角色取已授权范围中的最大范围，`ops_staff` 现有 V1 基线解析为 `all`。
- [V1 必做] 全流程业务账号可组合招商、渠道和运营角色，但不得因此获得用户管理、角色管理、系统配置等管理员能力。
- [V2 长期治理] 迁移到权限编码后，业务动作改由 permission bundle 表达，岗位角色不再作为唯一授权依据。

## 范围定义

| 范围 | 含义 | 典型使用 | 验收方式 | 范围 |
| --- | --- | --- | --- | --- |
| `self` | 仅当前用户负责的数据 | 渠道个人商品、达人、订单、寄样 | 使用不同账号比对列表 | V1 必做 |
| `group` | 当前组织或团队范围 | 主管查看组内订单、业绩 | 组织维度 API/SQL 比对 | V1 必做 |
| `all` | 管理员或授权全量范围 | 管理端看板、审计、配置 | 管理员账号 E2E | V1 必做 |

## 模块接入要求

| 模块 | 接入要求 | 风险 | 范围 |
| --- | --- | --- | --- |
| 商品域 | 列表、转链、映射查询按数据范围过滤；活动列表非 admin 强制 `mine`（`recruiter_user_id = self`），活动商品读接口校验活动归属 | 越权看到他人商品或映射 | V1 必做 |
| 达人域 | 达人资料、跟进、地址按角色和组织过滤 | 越权查看达人隐私 | V1 必做 |
| 寄样域 | 申请、审批、发货权限分离 | 越权审批或修改状态 | V1 必做 |
| 订单域 | 订单事实查询按数据范围过滤 | 越权查看订单事实 | V1 必做 |
| 业绩域 | 业绩明细和汇总按数据范围过滤 | 越权查看提成和归属 | V1 必做 |
| 分析模块 | dashboard 根据当前用户范围读汇总 | 看板越权 | V1 必做 |
| 配置域 | 配置修改需要管理权限，读取按业务需要开放 | 低权限修改规则 | V1 必做 |

## 验证清单

- [V1 必做] 至少使用管理员、主管、普通渠道账号验证同一 API 返回差异。
- [V1 必做] E2E 覆盖登录后菜单、列表数据、操作按钮和越权错误提示。
- [V1 必做] 后端测试或 SQL 证据能证明过滤条件生效。
- [V1 不做] 不用前端隐藏按钮替代后端权限校验。
- [V2 长期治理] 缺失用户、组织、权限或领域范围时默认拒绝，禁止以“不加过滤”作为兼容降级。
- [V2 长期治理] 进入 `ENFORCE` 前必须完成权限矩阵确认、影子差异归因、负向安全测试和旧令牌失效测试。
