# 02 架构设计

更新时间：2026-04-25

## 当前架构目标

当前架构不是围绕“如何尽快接真实 SDK”设计，而是围绕：

> 如何让内部业务闭环先稳定跑通，并且后续切真实 SDK 时不重写业务层。

## 核心分层

```text
前端 Vue
  -> 系统内部 API Controller
  -> 业务 Service
  -> DouyinGateway 接口
     -> MockDouyinGateway
     -> RealDouyinGateway
  -> Repository / Mapper
  -> PostgreSQL / Redis
```

## 分层职责

### Controller

负责：

- 接收前端请求
- 参数校验
- 返回系统内部 API 结果

不负责：

- 拼第三方 Mock 数据
- 编排核心业务状态

### Service

负责：

- 编排业务流程
- 维护状态流转
- 持久化和操作日志
- 串联商品、活动、达人等内部业务关系

不负责：

- 关心当前是真接口还是 Mock
- 直接处理第三方原始字段差异

### Gateway

负责：

- 抽象第三方能力
- 定义统一方法、入参、出参
- 隔离 Mock 和 Real 的实现差异

### Mock Gateway

负责：

- 按官方字段模拟第三方返回
- 覆盖成功和失败场景

### Real Gateway

负责：

- 调用真实抖店 / 百应接口
- 把第三方返回转换为内部 DTO

### Repository / Mapper

负责：

- 数据持久化
- 查询、更新、分页

## 技术栈

| 层级 | 技术 | 说明 |
| --- | --- | --- |
| 前端 | Vue 3 + Naive UI + Vite | 页面与交互 |
| 后端 | Spring Boot 3 + Java 17 | 主业务服务 |
| 持久化 | MyBatis-Plus + PostgreSQL | 数据存储 |
| 缓存 | Redis | token、临时状态、缓存 |
| 认证 | JWT + Spring Security | 登录、权限、DataScope |
| 第三方集成 | 抖音开放平台 SDK / HTTP 能力 | 网关封装层 |
| API 文档 | Knife4j / OpenAPI 3 | 联调与导入 Apifox |
| 部署 | Docker Compose | 本地和部署编排 |

## 当前已落地的架构收口

- `DouyinAuthGateway`
- `DouyinColonelActivityGateway`
- `DouyinPromotionGateway`
- `DouyinProductGateway`

当前效果：

- `ColonelActivityController` 已不再直接拼 Mock 活动和商品数据
- `ProductService` 已不再直接补 Mock 转链结果
- `DouyinTokenService` 已改为依赖 `DouyinAuthGateway`

## 当前仍需收口的点

1. `DouyinProductGateway` 仍未独立抽出
2. 商品详情 / SKU 的 Real Product Gateway 仍未接入真实接口
3. 订单相关第三方能力尚未统一到完整 Gateway 契约
4. 商品主链路状态机尚未完全统一
5. 达人跟进还未与商品主链路完全打通

## 当前设计约束

1. Mock / Real 必须走同一套 Gateway 契约
2. Service 只能依赖契约，不能依赖具体实现
3. 内部 DTO 和第三方原始字段不能在业务层混用
4. Mock 场景必须覆盖失败分支
5. 业务状态、日志、落库属于 SaaS 内部职责，不属于第三方接口职责
