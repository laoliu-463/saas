# 02-架构设计

更新时间：2026-04-27

## 一、总体架构

项目当前采用前后端分离的单体架构：

- 前端：Vue 3 + Vite + Naive UI + Pinia
- 后端：Spring Boot
- 数据库：PostgreSQL
- 缓存：Redis（在 Test 模式下降级处理）

当前核心设计目标不是做复杂微服务，而是：

> 用统一分层和统一 Gateway 契约，把 Test 演示能力和后续 Real 联调能力同时保住。

## 二、后端分层

当前后端主要分为：

### 1. Controller

职责：

- 暴露内部 API
- 处理请求参数
- 返回统一响应

约束：

- 不直接调用第三方 SDK
- 不直接写复杂业务逻辑

### 2. Service

职责：

- 承担核心业务编排
- 串联商品、订单、寄样、达人等流程

约束：

- 只依赖 Gateway 接口
- 不感知 Test / Real 具体实现细节

### 3. Gateway

职责：

- 吸收第三方平台差异
- 屏蔽真实 SDK 与 Test 数据差异

分类：

- Test Gateway
- Real Gateway

约束：

- 对上提供统一 DTO
- 对下可自由适配第三方字段

### 4. Repository / 数据访问层

职责：

- 落库
- 查询
- 支撑回放和排查

## 三、前端结构

当前前端主要围绕业务页面组织：

- `/product`
- `/orders`
- `/dashboard`
- `/sample`
- `/talent`
- `/dev/test`

当前前端原则：

- 只调用内部 API
- 不直连第三方接口
- 不把演示数据写死在页面
- 页面围绕业务闭环而不是围绕技术结构组织

## 四、核心架构原则

### 1. Gateway 契约隔离

最关键原则：

- Test / Real 共用一套 Gateway 契约
- 切换环境时替换实现，不替换上层逻辑

### 2. Test 与 Real 解耦

当前项目必须同时满足：

- `test` 环境可以独立演示
- `real` 环境可以逐步联调

所以不允许：

- 在 Service 里写大量 `if test`
- 在前端里感知真实 SDK 字段差异

### 3. 业务链路优先

当前设计不是单纯做接口集合，而是围绕业务闭环设计：

- 商品转链
- 订单归因
- 寄样推进
- 达人沉淀

### 4. 排查能力前置

系统不仅要“能跑”，还要“能查”。

因此当前架构特别保留：

- 订单详情排查能力
- 推广映射关系
- 寄样与订单联动关系
- 达人与订单 / 寄样关系

## 五、当前关键数据关系

### 1. 商品与活动

- 商品归属于活动
- 商品可被审核、分配、转链

### 2. 推广链接与归因映射

- 推广链接生成后写入 `promotion_link`
- 归因标识写入 `pick_source_mapping`

### 3. 订单与归因

- 订单通过 `pick_source` 匹配映射
- 匹配成功则归因
- 匹配失败则进入未归因排查

### 4. 寄样与订单

- 寄样记录与商品、达人关联
- 订单命中后可自动完成寄样单

### 5. 达人与业务结果

- 达人和寄样、订单产出联动
- 达人 CRM 展示当前归属和业务结果

## 六、当前环境设计

### 1. Test 环境

用于：

- 演示
- 回归
- 本地开发验证

特征：

- `/dev/test`
- `/api/test/*`
- 可以 reset / seed / 生成订单 / 模拟物流

### 2. Real 环境

用于：

- 真实第三方 SDK 联调

特征：

- 不破坏 Test 基线
- 按 Gateway 逐个联调

## 七、当前已知边界

当前架构暂不覆盖：

- 微服务拆分
- 复杂调度中心
- 完整财务结算引擎
- 复杂权限矩阵平台化
- 真实达人爬虫体系

## 八、当前结论

当前架构的核心价值在于：

> 一套业务逻辑，两套环境能力。Test 保证演示和回归，Real 保证后续真实联调扩展。

这也是当前项目能够稳定推进的根基。
