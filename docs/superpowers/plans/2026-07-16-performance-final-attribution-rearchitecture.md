# 业绩最终归属重构执行计划

> 状态：执行中。业务来源：用户已确认的《完整梳理后的结论》；本计划取代同日“订单 final 归属”口径，不删除旧计划，保留其作为历史证据。

## 目标与边界

将“订单默认归属 → 业绩最终归属 → 业绩汇总 / 看板”收敛为可审计闭环：

1. 订单域仅保存订单事实和默认渠道 / 招商归属；不计算独家覆盖、手工调整、提成或最终业绩归属。
2. 业绩域基于订单默认归属、订单发生时间和已生效的归属规则，计算并固化最终渠道 / 招商归属、规则来源、计算版本及金额结果。
3. 看板只消费业绩计算完成 / 汇总刷新结果，不再从订单同步事件自行形成业绩汇总。
4. 历史订单的映射解析冻结在订单事实中；受控单笔重放产生新的订单版本，可靠触发对应业绩重算。
5. 退款作为订单事实独立进入业绩冲正账本；不得把退款静默改写为原业绩记录失真。

本轮为用户明确通过的跨域修复，选择 **Gate 4**。它跨订单、商品、业绩、分析、用户和 Outbox；不新增外部 MQ、不批量裸改 real-pre 数据、不直接部署远端。

## 已确认的现状证据

- `PerformanceCalculationApplicationService#buildRecord` 将订单的渠道 / 招商默认字段直接复制到 `final_*`；现有 `PerformanceAttributionPolicy` 未在生产调用链执行。
- 订单只有聚合 `attribution_status`，无法区分“渠道已归属、招商未归属”。
- 普通 `OrderSynced` 使用订单 ID / 行 ID 作为 Outbox 幂等键；订单版本改变后不能保证形成新的可消费事件。
- `PerformanceRecordSyncListener` 的普通订单消费会吞掉计算异常，Outbox 可被标记为已发布而业绩未更新。
- `PerformanceAccessScope` 对双角色用户优先命中渠道分支，未实现两个归属维度的并集。
- 商品域已经固化 `attribution_owner_type`，但双角色创建链接没有显式归属类型输入；不能靠角色集合猜测业务意图。
- 当前领域文档仍把订单归属写为 final，和本次已通过的设计冲突；必须同步修正，并在 ADR 中记录口径变更。

## 目标模型与兼容策略

### 订单默认归属

- 保留 `channel_user_id` / `colonel_user_id` 作为既有 API 兼容字段，语义明确改为默认渠道 / 默认招商归属。
- 新增独立的 `channel_attribution_status`、`recruiter_attribution_status`、来源映射 ID / 版本 / 有效时间快照和归因审计记录。
- `attribution_status` 仅为兼容汇总：两个维度均归属为 `ATTRIBUTED`、任一维度归属为 `PARTIAL`、均未归属为 `UNATTRIBUTED`；不得再作为某一维度的判断输入。
- 正常同步仅更新上游订单事实，且不覆盖已有默认归属快照；单笔重放才可更新默认归属并记录前后值。

### 业绩最终归属

- `performance_records.default_*` 保存从订单复制的默认事实；`final_*`、最终归属类型、规则版本、决策快照和计算时间仅由业绩域写入。
- 调用 `PerformanceAttributionPolicy`：默认归属、已生效独家商家、已生效独家达人、已批准人工调整，优先级和生效时间均写入决策快照。手工调整高于系统规则。
- 不创建第二套独家配置能力；只消费已有独家事实。若既有独家功能开关关闭，最终归属自然回退默认归属。
- 提成计算以最终招商归属为规则输入，并持久化命中的规则标识 / 版本；金额双轨口径保持不变。

### 可靠事件、退款与看板

- 所有订单事实事件使用 `orderId + orderVersion`，退款使用 `refundId + refundVersion`；Outbox 失败必须保留 `FAILED/DEAD` 并可重试，禁止监听器吞异常。
- 新增业绩计算执行记录（事件键、订单版本、状态、失败信息、重试时间），使补偿任务能重放失败计算而不是静默漏单。
- 新增业绩调整账本：退款与人工调整以不可变明细记录，业绩明细/汇总由该账本和计算结果刷新。
- 分析模块仅订阅业绩计算完成 / 汇总刷新事件；不再直接消费订单同步或退款事件形成业务汇总。

## 实施步骤

1. 先为目标合同写失败测试：默认归属与最终归属分离、独家/手工优先级、双维度状态、双角色权限并集、版本化 Outbox、失败重试、退款账本和看板消费边界。
2. 增加幂等 PostgreSQL migration、实体和 Mapper：订单双维度快照/审计、业绩决策与执行记录、业绩调整账本；补充索引和回滚说明。迁移只新增字段/表和兼容回填，不批量改变归属值。
3. 修改商品链接创建契约：双角色调用者必须显式选择 `CHANNEL` 或 `RECRUITER`，服务端校验选择与角色匹配并固化到链接与映射。
4. 修改订单默认归属解析与重放：两个归属维度分别输出状态/来源/映射快照；常规同步冻结既有默认事实；重放产生审计记录和版本化订单事件。
5. 修改 Outbox 和业绩消费：普通同步和单笔重放都以版本事件入队；业绩监听器同步执行、失败抛回 dispatcher；补偿任务读取执行记录并重试。
6. 修改业绩计算与退款：通过 Facade 读取订单事实和独家规则，调用归属策略，保存决策快照、佣金规则输入、账本冲正和 `PerformanceCalculated` 事件。
7. 修改权限和看板：个人范围对 `final_channel_user_id OR final_recruiter_user_id` 取并集；主管按各自最终归属部门范围过滤；看板只由业绩事件刷新。
8. 更新订单、业绩、商品、用户、分析、事件、数据、流程、验收文档及 ADR；更新领域状态，说明真实订单验证仍需要受控样本。
9. 运行目标测试、完整构建、Docker 重启、健康检查、本地 real-pre API / SQL 最小闭环、Harness evidence 与 Git push。远端部署继续等待单独明确授权。

## 验收矩阵

| 场景 | 必须证明的结果 |
| --- | --- |
| 招商链接由壮云创建并成交 | 订单默认招商=壮云；业绩最终招商=壮云（无覆盖）；壮云 PERSONAL 可查询；组长不因活动分配获得个人归属 |
| 渠道链接成交 | 只写默认渠道，不伪造招商；渠道 PERSONAL 可查询 |
| 无链接活动订单 | 仅允许活动招商作为默认招商 fallback，且独立状态为渠道未归属 / 招商已归属 |
| 双角色人员 | 创建链接时显式选择类型；查询订单和业绩取得两个维度并集 |
| 单笔归因重放 | 默认归属变更有审计；订单版本递增；新 Outbox 事件进入业绩重算；旧事件不误去重 |
| 业绩消费失败 | Outbox / 计算执行记录为 FAILED，可重试后成功；不得标记已发布后静默漏算 |
| 独家或人工调整 | 最终归属与默认归属可不同；决策来源、规则版本与有效时间可追溯 |
| 部分退款 | 退款事实和业绩调整账本独立存在；原订单金额事实不被改写；汇总随业绩事件刷新 |

## 风险、回滚与未授权事项

- 历史数据仅迁移结构和兼容性字段，不自动批量重算或改归属。历史重算必须通过后续受控任务、范围确认和 dry-run。
- 已有正常订单事件将不会被批量重发；新逻辑只影响新版本订单事件和明确的单笔重放。
- 若迁移或应用回滚，新增列/表可保留，应用回退仍使用既有兼容字段；不得删除 real-pre volume 或清库。
- 真实订单 `6927995582750227729` 的角色变更、映射回填和单笔重放属于生产数据动作，必须在代码部署和验证完成后，按此前用户列出的受审计流程单独执行。
