# DDD RBAC 长期治理实施路线图

**状态：** 已批准设计，分阶段实施

**设计基线：** `docs/superpowers/specs/2026-07-13-ddd-rbac-long-term-governance-design.md`

**决策基线：** `docs/决策/ADR-012-用户域内建设权限中心化授权能力.md`

## 实施原则

- Authorization 作为用户限界上下文内的子域能力建设，不新增微服务。
- 每个阶段独立计划、独立测试、独立 evidence，不跨阶段一次性替换。
- 旧链与新链仅允许 `LEGACY`、`SHADOW`、`ENFORCE` 三态切换，禁止宽松 OR 放行。
- 权限编码与旧角色映射必须由业务负责人确认，技术实现不得代替业务拍板。
- 数据库迁移、远端部署和领域进入 `ENFORCE` 分别取得明确授权。
- 回滚优先降级运行模式并保留新增事实表，不通过删表回滚。

## 阶段顺序

| 阶段 | 交付物 | 进入条件 | 完成门禁 | 回滚边界 |
| --- | --- | --- | --- | --- |
| 1. 休眠式授权基础 | Schema 合同、权限值对象、授权决策策略、授权快照端口与只读适配器、窄 `AuthorizationFacade` | 本设计和 ADR 已批准 | 新内核单元/集成测试通过；现有请求决策未接入；本地 real-pre 构建、重启、健康检查完成 | 直接回退代码；现有数据库未执行迁移时无数据回滚 |
| 2. 基础设施激活与影子链 | 手工迁移、`authzVersion`、版本化缓存、JWT 版本校验、`AuthorizationPrincipal`、三态配置、差异日志 | 阶段 1 PASS；数据库变更获明确授权 | 旧令牌失效、事务版本递增、401/403/503、影子差异分类测试通过 | `SHADOW` 降为 `LEGACY`，保留新增表和日志 |
| 3. 授权管理与前端投影 | 权限目录只读管理、角色权限/领域范围配置、审计、`UserDirectoryFacade` 拆分、当前用户权限投影、前端菜单/按钮消费 | 阶段 2 PASS；候选权限矩阵已由业务负责人确认 | 管理 API、审计、前端路由和负向越权测试通过 | 管理写入口关闭；运行模式保持 `SHADOW` |
| 4. 寄样域试点 | 寄样入口权限、对象归属与状态机策略、查询范围映射 | 阶段 3 PASS；寄样权限矩阵确认 | 影子差异未解释项为零；寄样角色×范围×状态 E2E 通过 | 寄样域单独降为 `SHADOW` |
| 5. 达人域与商品域 | 达人/商品命令与查询迁移 | 寄样域试点稳定 | 两域分别完成差异清零、负向越权和 E2E | 按域独立降级 |
| 6. 订单域与业绩域 | 高风险读写和数据范围迁移 | 前三业务域稳定；性能基线已采集 | 归属、金额、状态冲突、范围和性能回归通过 | 按域独立降级，禁止更改归属事实兜底 |
| 7. 系统管理、菜单与旧链退出 | 系统管理入口迁移；移除角色切面、JWT 旧声明、JSON 权限和全局范围链 | 全部业务域 `ENFORCE` 稳定一个观察周期 | 无业务域新增角色字符串；旧链调用为零；清理迁移获批准 | 清理前保留可回退版本；删列迁移不与切换同批执行 |

## 计划分拆规则

阶段 1 的可执行计划见 `2026-07-13-ddd-rbac-foundation.md`。后续阶段在前一阶段 evidence 形成后分别编写实施计划，原因是当前代码图谱显示授权核心影响 500 个节点、至少 141 个附加文件；提前把后续阶段写成文件级修改清单会把尚未验证的接口当成事实。

后续计划固定按以下名称生成：

1. `2026-07-13-ddd-rbac-shadow-runtime.md`
2. `2026-07-13-ddd-rbac-management-projection.md`
3. `2026-07-13-ddd-rbac-sample-pilot.md`

这些名称代表独立规划门，不是可直接执行的实施任务；只有对应计划完成评审后才允许实施。

## 业务确认门

进入阶段 3 前，业务负责人必须逐项确认：

- 权限编码、中文名称、所属领域、风险等级和是否需要数据范围。
- 每个现有角色应授予的权限集合。
- 每个角色在每个业务域的 `SELF/GROUP/ALL` 范围。
- 冲突角色和职责分离规则；未确认时不创建推测性限制。
- 紧急访问流程和审批人；不得以代码级管理员绕过替代。

AI 仅可从现有注解、Controller、菜单和 JSON 权限生成候选矩阵，并标注证据来源与冲突，不能把候选矩阵直接写入生产授权表。

## 全程证据门禁

每个阶段都必须形成“现象/目标 → 代码与数据证据 → 测试结果 → 阶段结论”链路，并至少包含：

- 聚焦单元测试、数据库集成测试、架构守卫和编译结果。
- 本地 `real-pre` 对应容器重启、健康检查与相关业务验证。
- `harness/reports/latest-evidence-20260713.md` 与 retro summary。
- 当前分支提交和上游推送结果。
- 未执行项明确标记为 `PENDING`、`BLOCKED` 或 `PARTIAL`，不得写成 `PASS`。
- 未明确授权时，`DeployRemote=false`，不执行远端数据库迁移或远端部署。
