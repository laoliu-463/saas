# 角色感知的推广链接业绩归属设计

**日期：** 2026-07-16

**状态：** 业务口径已批准，待设计复核

**范围：** 推广链接归属快照、订单默认归因、业绩来源、个人数据范围、历史安全重放

## 目标

推广链接产生订单时，按链接账号在生成链接时的业务身份归属业绩：招商账号归招商业绩，渠道账号归渠道业绩。活动绑定招商只在没有有效招商链接归属时兜底。归属账号必须能在个人数据范围内查询自己的业绩。

以上证据能确定“为何显示组长”。由于订单原始载荷缺少 `pick_source`，只能通过活动、商品、团长原生键、链接有效期和唯一账号做安全推断，不能把时间相邻本身当作精确点击链路证据。

## 统一语言

- **链接归属账号**：生成推广链接并写入 `promotion_link` / `pick_source_mapping` 的系统用户。
- **归属维度**：链接账号对应的业绩维度，取值 `CHANNEL` 或 `RECRUITER`。
- **精确映射**：订单携带 `pick_source` / `pick_extra`，可直接命中有效映射。
- **原生唯一映射**：订单缺失 `pick_source` 时，通过团长百应 ID、活动、商品和生效时间得到唯一归属账号及维度。
- **歧义映射**：候选映射包含多个不同的 `(userId, ownerType)`；不得自动选择最新记录或活动组长冒充链接归属。
- **活动招商兜底**：没有有效招商链接归属时，使用活动绑定招商。

## 方案决定

采用“链接创建时快照归属维度，订单时按精确映射或原生唯一映射读取”的方案。

不在订单重放时按用户当前角色临时推断，因为用户改角色会篡改历史语义。不采用“所有链接创建人都算招商”，因为渠道链接会被错误计算为招商。不采用“商品 + 活动取最后一个链接”，因为多人生成同一活动商品链接时无法证明订单属于最后操作人。

## 归属不变量

1. 招商链接账号优先于活动招商：`RECRUITER link owner > activity recruiter > null`。
2. 渠道链接账号只写渠道维度，不覆盖招商维度；招商链接账号只写招商维度，不伪造渠道归属。
3. 订单任一维度有有效归属时可标记为已归因；渠道和招商来源必须分别记录，不能再用一个状态推断两个维度。
4. 同一查询键下多条映射若最终归一为同一个 `(userId, ownerType)`，仍视为唯一；出现不同账号或不同维度即为歧义。
5. 映射必须在订单业务时间之前已生效，且订单业务时间不晚于映射失效时间；禁止使用订单之后补建的映射回填历史。
6. 链接账号角色不明确、同时拥有渠道和招商两类业务角色，或没有业务角色时，不得静默猜测归属维度。
7. 历史修复必须先 dry-run、再按明确订单 ID 或受控批次执行；禁止直接修改 `performance_records`。
8. 订单同步与历史重放必须调用同一个默认归因策略。

## 归属维度快照

`pick_source_mapping` 和 `promotion_link` 增加 `attribution_owner_type`：

- `CHANNEL`：账号仅拥有 `channel_staff` / `channel_leader` 业务维度。
- `RECRUITER`：账号仅拥有 `biz_staff` / `biz_leader` 业务维度。
- `NULL`：历史未分类记录或无法安全判断的记录，不参与角色感知链接归属。

生成链接时由后端从可信用户域读取角色，不接收前端任意指定归属维度。管理员、运营等非业绩角色不形成链接业绩归属。若账号同时具备渠道和招商维度，接口返回稳定的冲突错误，待角色配置纠正；本期不引入双归属和前端维度选择器。

壮云必须先通过现有用户管理能力纠正为实际招商角色，再对其历史链接执行受控分类。不得把线上用户 ID 硬编码进 SQL 或业务代码。

## 默认归因算法

### 第一步：解析候选映射

1. 优先使用 `pick_source`，其次使用 `pick_extra` / short ID 精确查找。
2. 精确字段缺失或没有有效结果时，使用原生键查找：
   - 首选 `colonel_buyin_id + activity_id + product_id`；
   - 首选无结果时，可按 `activity_id + product_id` 回退，但必须记录百应 ID 是否不一致；
   - 仅按 `colonel_buyin_id` 的通用回退只在候选归一后仍唯一时允许。
3. 所有候选按映射状态、删除标记、`valid_from`、`valid_until` 和订单业务时间过滤。
4. 按 `(user_id, attribution_owner_type)` 去重。唯一则返回；多组则返回 `AMBIGUOUS`；零组则没有链接归属。

订单业务时间优先使用 `pay_time`，其次 `order_create_time`，最后才使用本地 `create_time`。缺少可验证业务时间的原生回退不得用于历史重放。

### 第二步：分别计算两个维度

| 映射结果 | 默认渠道 | 默认招商 | 渠道来源 | 招商来源 |
| --- | --- | --- | --- | --- |
| `CHANNEL` 链接账号 | 链接账号 | 活动招商兜底 | `pick_source` 或 `native_unique_link_owner` | `activity_owner` 或 `unattributed` |
| `RECRUITER` 链接账号 | 不因该映射赋值 | 链接账号 | `unattributed` | `pick_source` 或 `native_unique_link_owner` |
| 无链接归属 | 空 | 活动招商兜底 | `unattributed` | `activity_owner` 或 `unattributed` |
| 歧义映射 | 空 | 不使用歧义链接账号；允许活动招商兜底但必须保留歧义诊断 | `ambiguous` | `activity_owner` 或 `ambiguous` |

独家商家、独家达人等最终覆盖继续由业绩域处理，不在订单域重算提成。它们必须保留自己的最终来源，不得被默认来源字符串覆盖。

### 第三步：持久化可审计来源

`colonelsettlement_order` 增加：

- `channel_attribution_source`
- `recruiter_attribution_source`

稳定来源值至少包括：`pick_source`、`native_unique_link_owner`、`activity_owner`、`ambiguous`、`unattributed`。`attribution_status` 表示订单是否至少有一个有效业绩维度；两个来源字段负责表达各维度是否完整。

`PerformanceCalculationApplicationService` 从订单复制默认归属和来源，不再把所有非空招商统一写成 `activity_owner`。最终归属仍由业绩域策略覆盖，个人查询继续读取 `final_channel_user_id` / `final_recruiter_user_id`。

## 账号角色与个人查询

- 壮云在执行历史分类和重放前必须具备实际招商角色及正确数据范围；当前 `channel_staff` 配置不能作为招商身份事实。
- `biz_staff + PERSONAL` 查询必须限定 `final_recruiter_user_id = 当前用户`。
- 通用 PERSONAL 范围保留“渠道或招商任一维度等于当前用户”的最小权限语义。
- 前端只展示后端返回的渠道和招商名称，不再用团长名称补招商名称；无招商归属时显示未归属。

## 历史记录修复

### 历史映射分类

提供受控的管理员 dry-run/apply 能力，只处理 `attribution_owner_type IS NULL` 的映射：

1. 按指定用户或受限批次读取当前有效角色。
2. 只有唯一业务维度时给出候选分类；多维度或无维度列为冲突，不更新。
3. dry-run 返回映射数、候选账号、候选维度、潜在订单数和冲突数。
4. apply 必须显式 `confirm=true`，写操作日志，并保持幂等。

账号角色纠正与映射分类是两个独立审计动作。分类结果一旦写入就是历史快照，后续角色变化不自动改写。

### 订单重放

复用现有管理员 `POST /api/orders/replay-attribution` 的 dry-run/apply 入口，但内部改为调用与新订单同步相同的默认归因解析器，并继续执行映射时间安全检查。重放后必须同步 upsert 业绩记录，不能只改订单。

首轮仅指定订单 `6927995582750227729`：

1. dry-run 证明原生唯一映射候选为壮云、维度为 `RECRUITER`、映射早于订单。
2. apply 后订单默认招商与业绩最终招商均为壮云，招商来源为 `native_unique_link_owner`。
3. 以壮云身份查询个人招商业绩，必须能查到该订单。
4. 对相同活动商品构造两名不同归属账号的映射时，dry-run 必须报告歧义且不覆盖历史记录。

扩大历史批次前必须对歧义数、映射晚于订单数、仍未归因数逐项人工确认。

## 数据库变更

- 绿场脚本、汇总迁移和独立增量脚本同时增加四个字段：两张链接映射表各自的 `attribution_owner_type`，以及订单表的两个维度来源字段。
- 为原生回退增加支持状态、来源类型、团长百应 ID、活动、商品和有效期的查询索引；具体索引以 `EXPLAIN` 为依据，不能凭空增加重复索引。
- 新字段先允许空值以兼容滚动部署；新写入链稳定并完成历史分类后，再评估是否收紧约束。
- 增量脚本必须幂等，不重置账号角色，不硬编码壮云或任何线上用户 ID。

## 错误与诊断语义

- `ATTRIBUTION_OWNER_TYPE_AMBIGUOUS`：链接账号同时具有渠道和招商业务维度。
- `NATIVE_MAPPING_AMBIGUOUS`：原生键命中多个不同账号或维度。
- `MAPPING_AFTER_ORDER`：映射生效时间晚于订单业务时间。
- `MAPPING_OWNER_TYPE_MISSING`：历史映射尚未安全分类。
- 日志记录订单 ID、活动、商品、候选数量和稳定原因码；不得记录 Token、完整原始链接或敏感请求头。

## 测试与验收

### 单元测试

- 招商链接账号覆盖活动招商。
- 渠道链接账号只影响渠道，活动招商仍作为招商兜底。
- 精确 `pick_source` 优先于原生回退。
- 多行同账号同维度归一为唯一；不同账号或维度报告歧义。
- 映射时间晚于订单、角色维度缺失和多角色均不得猜测。
- 业绩记录复制真实渠道/招商来源，不再固定写 `activity_owner`。

### 集成与接口测试

- 新链接同时写入两张表的相同归属维度快照。
- 订单缺少 `pick_source` 时，唯一原生映射可归属；歧义映射不可归属给链接账号。
- 重放 dry-run 不写订单和业绩；apply 幂等更新两者。
- 招商个人范围可查询自己的最终招商记录，不能查询其他招商记录。
- 前端订单详情缺少招商时不回退展示团长名称。

### real-pre 验收

- 构建、对应容器重启和健康检查通过。
- 先纠正壮云账号角色，再 dry-run 分类历史链接。
- 指定订单 dry-run 结果无歧义且通过时间安全检查后，才允许 apply。
- DB、订单接口、业绩接口和壮云账号页面四方结果一致。
- evidence 明确记录远端是否部署、角色变更、映射分类、订单重放和剩余风险；任何未执行项不得标为 PASS。

## 回滚

- 代码回滚后保留新增字段，旧版本忽略它们；不通过删列回滚。
- 新映射分类出错时，按操作日志定向恢复该映射快照并重放明确订单，不全表覆盖。
- 订单或业绩重放异常时，以指定订单的变更前快照恢复，再排查策略；禁止清库或批量手工改业绩表。
- 远端部署和线上数据动作仍需用户单独明确授权。

## 非目标

- 不建设通用佣金结算平台。
- 不引入一个链接同时归属渠道和招商的双重提成。
- 不根据达人昵称做核心归属；昵称只能辅助诊断，不能替代稳定 ID。
- 不修正所有历史订单；本期先闭环已确认订单，再基于 dry-run 证据决定批次范围。
- 不把活动招商负责人、商品负责人或前端展示字段当作链接账号事实。
