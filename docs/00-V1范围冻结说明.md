## 一、一句话口径

> **领域设计文档可以写全，但 V1 实现只按 [01-V1交付范围与边界](./01-V1交付范围与边界.md) 的「做 / 简化 / 不做」列，以及技术落地裁剪执行。**

V1 业务目标不变：**跑通渠道链、招商链、管理链**；独家达人/商家、个别品负责人覆盖等 **整体延后 V2**。服务费收入和收益按 2026-06-06 用户口径区分预估 / 结算两轨：预估服务费收入 = 预估订单额 × 服务费率（未扣除技术服务费）；结算服务费收入以官方结算服务费字段为准，不用订单额重算且不扣技术服务费；预估服务费收益 = 预估服务费收入 - 预估服务费支出 - 技术服务费，结算服务费收益 = 结算服务费收入 - 结算服务费支出。技术服务费在结算轨只展示，不扣减结算服务费收益。毛利（Y-04）已按 2026-06-05 用户决策纳入 V1 经营指标，口径为 `服务费收益 - 招商提成 - 渠道/媒介提成`，不扩展为财务结算或多账期治理。

---

## 二、强制按领域对齐的三条规则

1. **每个领域必须先钉 V1 裁剪结论**，不能把 V2.2 全量领域设计当实现清单。
2. **领域顺序固定**：用户/配置 → 商品/达人 → 寄样 → 订单 → 业绩 → 分析 → 跨域验收。
3. **逐域通过后必须做三条链 E2E**，否则单域「绿灯」不算 V1 对齐完成。

## 三、整体执行顺序

```text
0. V1 范围冻结（本文 + 裁剪表 + 不做清单）
↓
1. 用户域
↓
2. 配置域
↓
3. 商品域
↓
4. 达人域
↓
5. 寄样域
↓
6. 订单域
↓
7. 业绩域
↓
8. 分析模块
↓
9. 跨域事件与三条链验收
```

**排序原因**：用户域、配置域是底座；商品/达人提供主数据；寄样依赖商品与达人；订单提供事实；业绩依赖订单；分析消费订单与业绩结果。领域间以 **事件协作** 为主，不跨域直改他域库表（见 [02-V1业务流程与领域设计](./02-V1业务流程与领域设计.md)）。

## 四、必须冻结的全局裁剪（V1）

| 项目 | V1 结论 |
|------|---------|
| 独家达人 / 独家商家 | **不做** |
| `final_channel` | **等于** `default_channel` |
| 推广链接归属 | **创建时按渠道/招商角色固化 owner type；不可按当前角色回推** |
| `final_recruiter` | **优先招商链接所有者；活动绑定招商仅为无链接归因时的 fallback** |
| 个别品负责人覆盖招商 | **不做** |
| ~~毛利（Y-04）~~ | ~~**不做**~~ **已撤销**（2026-06-05 用户决策毛利纳入 V1） |
| 商品负责人变更重算（Y-05） | **不做** |
| 寄样 30 天自动关闭（S-06） | **不做** |
| 一人多角色 | **做**（权限并集，数据范围取最宽） |
| 看板整体导出（A-08） | **不做** |
| 差异化提成 / 独家阈值配置 | **不读、不启用** |

细节编号以 [01-V1交付范围与边界](./01-V1交付范围与边界.md) 为准。

## 七、领域对齐完成定义

某领域「对齐完成」需同时满足：

1. 该域所有 **V1 做** 项在总表中有 **当前实现** 与 **验收方式**；
2. 所有 **V1 不做 / V2** 项在代码与接口中 **无默认开启路径**；
3. **差异** 已标为：已修 / 接受差异 / 待补证 / 明确后移；
4. 跨域阶段（步骤 9）三条链用例可复现。
