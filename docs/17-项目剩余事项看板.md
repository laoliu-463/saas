### 6.3 P1-1 当前治理结论

当前环境边界统一为：

- `local-mock`：本地人工业务联调
- `test`：自动化测试 / 隔离测试栈
- `prod`：真实联调 / 生产式运行口径
- `dev`：兼容保留，不再作为推荐默认入口

当前最小治理动作已经明确为：

1. 根目录 `.env` 默认切到 `local-mock`
2. `application.yml` 默认兜底切到 `local-mock`
3. `docker-compose.yml` 继续作为 `local-mock` 标准入口
4. `docker-compose.test.yml` 继续作为 `test` 覆盖入口
5. `application-dev.yml` 标记为兼容 profile，不再作为本地联调推荐路径

### 6.4 P0/P1 安全治理补充

#### 6.4.1 凭据治理口径

- 仓库只保留 example 模板文件，不保留真实值
- 真实联调凭据只允许放在未跟踪 env 文件、CI/CD Secret 或服务器环境变量
- 若 `DOUYIN_CLIENT_SECRET` 等值曾在共享工作区、截图、历史提交或多人机器中出现，建议轮换

#### 6.4.2 前端 API 口径

- 前端业务请求统一使用 `/api/**`
- 本地开发通过 Vite 代理决定 `/api` 指向后端
- 不再推荐让 `VITE_API_BASE_URL=http://localhost:8080/api` 承担主请求拼接语义

#### 6.4.3 `/api/test/**` 口径

- `TestController` 仅允许在 `local-mock` / `test` profile 下启用
- `OPTIONS /api/test/**` 允许通过，避免预检请求被 JWT 拦截
- `POST /api/test/**` 仍保持鉴权，避免本地造数接口无边界裸放

## 11. 本地优化路线（业务逻辑 + 前端设计）

当前第三方 SDK 尚不能稳定联调，后续开发不应停滞，也不应无序扩功能。默认按下面路线推进：

### 11.1 优化目标

- 在不破坏现有 `test MVP` 主链路的前提下，让本地业务流程更像真实团队在操作
- 让前端页面更容易支持“推进、排查、复盘”三类动作

### 11.2 业务逻辑优化优先项

优先补“商品主链路的过程证据”，而不是继续增加新页面：

1. **商品推进责任起点**
   - 明确谁在什么时候把商品分配给了谁
   - 为后续推进动作提供统一起点

2. **商品推进动作时间线**
   - 优先围绕商品详情页建立最小时间线
   - 首条事件至少包括：分配时间、分配人、被分配人
   - 后续再逐步补优先级调整、推进备注等事件

3. **商品推进判断**
   - 让商品不只是“已分配”，还要逐步具备“主推 / 次推 / 暂缓 / 放弃”等业务判断语义
   - 后续变更需要带原因和时间线

4. **订单复盘衔接**
   - 当前阶段先不追求完整“订单倒推全部前置动作”
   - 但新增的商品推进信息应为后续订单复盘留好挂接点

### 11.3 前端设计优化优先项

前端设计优化应服务业务操作，不单独追求装饰性调整：

1. **商品详情页**
   - 优先作为商品推进信息的主展示面
   - 重点展示：当前责任人、最近动作、推进时间线

2. **商品列表页**
   - 增强快速扫读能力
   - 重点补“谁在跟 / 最近有没有动作 / 当前是否值得继续推进”的摘要感知

3. **订单详情页**
   - 强化归因结果与排查原因的可理解性
   - 为未来接入前置动作复盘预留信息位置

4. **关联入口**
   - 商品、达人、寄样、订单之间的入口要更顺手
   - 减少“要切多个页面才能判断当前业务状态”的摩擦

### 11.4 执行边界

当前本地优化阶段，明确不优先做：

- 新增无直接业务收益的独立页面
- 纯视觉美化但不提升信息可读性的改动
- 再造新的 Mock 机制
- 为了“更真实”而破坏现有 Test Gateway / Real Gateway 契约

### 11.5 推荐最小落地顺序

1. 完成 P0 回归与文档收口
2. 商品详情页补商品推进时间线（先落“分配事件”）
3. 商品列表补责任人与最近动作摘要
4. 视本地验证结果，再扩优先级调整和原因记录
5. 最后再处理更广泛的拟真数据与环境治理问题
