## 一、开发步骤

### 1. 接口资产盘点，输出对照表
- 扫描所有后端 Controller 的注解：`@Tag`、`@Operation`、`@RequestMapping`，以及前端 `src/api/*.ts`。
- 对照 `docs/05-接口与数据模型.md`，列出“当前实际分组、路径、参数命名、描述完备度”与“目标方案”的差异表。
- 产物：`/docs/接口整改-现状vs目标.md`，明确标注哪些是“直接可执行”，哪些是“待决策项”。

### 2. 统一分组方案（直接执行）
- **测试工具分组**：为 `TestController` 类级别添加 `@Tag(name = "测试工具")`，其下所有方法的 `@Operation(summary)` 统一加 `[测试]` 前缀。
- **抖音联调分组**：为 `DouyinController`、`DouyinWebhookController` 类级别添加 `@Tag(name = "抖音联调")`，方法 `summary` 统一加 `[联调]` 前缀。
- **废弃接口降噪**：保持 `ProductController` 类上已有 `@Tag`，但确保 `summary` 以 `[已废弃]` 开头；如不存在类级 Tag，补为 `@Tag(name = "商品管理（已废弃）")`。
- **说明**：方法级 Tag 不新增，避免分组分散。所有 Tag 修改均在 Controller 类上完成。

### 3. 中文化 summary 与 description（直接执行）
- 检查所有公开业务接口的 `@Operation`：`summary` 必须为中文（例如 “获取订单列表”）；`description` 必须补全，说明业务场景与注意事项。
- 联调接口额外注明“验证上游 SDK 的具体能力项”，数据页接口说明与主链路接口的区别。

### 4. 参数描述与示例补全（严格执行）
- 所有 `@RequestParam`、`@PathVariable`、`@RequestBody` 均添加 `@Parameter(description="...")` 或 `@Schema(description="...")`。
- **枚举值说明来源**：
  - 有常量类或枚举类定义的，直接引用；
  - 无代码定义的，描述写 “待确认：取值含义请联系产品”，**严禁臆造**。
- 补充典型请求示例，使用 `@Content(examples = ...)` 或 `@Schema(example = "...")` 提供真实 JSON 片段。

### 5. 分页参数统一（先确认，再执行）
- 盘点：业务接口中存在两套命名 `page/pageSize` 与 `page/size`。
- 与团队确认统一范围：对内业务 API 统一为 `page` + `size`；联调/透传接口保留 `pageSize`，并加文档说明。
- 确认后修改：相关 Controller 的 `@RequestParam` 名称，前端 `src/api/*.ts` 的请求参数名，以及测试用例中的断言。Gateway DTO 内部字段暂不更改。

### 6. 达人接口 `claims` / `release` 处理（不修改路径，只补描述）
- 保持 `POST /{id}/claims` 与 `POST /{id}/release` 路径不变。
- 在 `@Operation(description)` 中明确语义：“认领动作，生成认领记录”、“释放动作，解除锁定状态”。
- 同步更新 `docs/05`，记录当前路径并备注“动作语义，非资源集合”。

### 7. 生成 OpenAPI 并导入 APIFOX 验证
- 重新运行 SpringDoc 生成 OpenAPI JSON/YAML。
- 导入 APIFOX，逐项检查：
  - 无 default 分组，所有接口有明确中文 Tag；
  - 测试/联调/废弃接口通过前缀和分组隔离；
  - 参数描述完整，示例可展示。
- **截图留档**：分组列表页、测试分组页、联调分组页、废弃接口页各保留一张截图，便于验收核对。
- 未通过项回溯修正，直到满足标准。

### 8. 重写 `docs/05-接口与数据模型.md`
- 以整改后的实际状态全量重写，使用 UTF-8 编码保存（参照 `docs/12` 解决乱码）。
- 包含：接口总览表、统一返回格式、命名规范、各分组核心接口片段、废弃接口列表、前端封装对应关系。
- 为 `/douyin/webhook` 添加说明：“默认纳入抖音联调分组，后续可根据发布策略在 OpenAPI 暴露范围中单独控制”。

### 9. 输出决策备忘录与交付
- 整理未确认的枚举值清单、达人路径决策结论、分页统一的具体范围等。
- 最终交付物：更新的代码注解、OpenAPI 文档、`docs/05`、决策备忘录。

## 二、验收标准
- [ ] APIFOX 中无 “default” 分组，所有接口归属到正确的中文 Tag 下。
- [ ] 测试接口和联调接口可独立过滤（通过 `[测试]`、`[联调]` 前缀及分组）。
- [ ] 所有公开接口 `summary` 为中文，`description` 完整，无英文残留。
- [ ] 参数描述栏都有内容，枚举值说明有据可查（无臆造）。
- [ ] 分页参数在业务接口中统一为 `page`、`size`；联调接口差异有文档注明。
- [ ] 废弃接口带有 `[已废弃]` 前缀且 description 指明替代。
- [ ] `docs/05` 内容与 Swagger 完全一致，无乱码，无过期信息。
- [ ] 前端现有 `src/api/*.ts` 调用与整改后的参数名、路径保持一致，关键列表页可正常请求。
- [ ] 各页面截图已存档，与验收描述吻合。

## 三、风险备注
- 分页参数变更可能影响前端多页面及测试用例，需全面回归。
- `/douyin/webhook` 暴露风险：OpenAPI 生成时默认包含，如需隐藏请使用 `@Hidden` 注解或在配置中按路径排除。
- 枚举值若后续被产品/第三方更新，需同步更新 Swagger 描述。
- 达人路径语义待定，未来若修改路径需做重定向或版本控制，避免前端硬编码失效。
