# DDD Domain Task Matrix — Cross（跨域）

> 用户 / 配置 / 业绩 / 分析 / Outbox / 前端 / E2E+GC 7 个跨域任务的详细清单。
> 拆分自 `DDD_DOMAIN_TASK_MATRIX.md`（GC-OPTIMIZE-003 Step 2）

## 用户域

- U-1 盘点当前用户域代码、接口、表和测试。
- U-2 梳理用户、角色、菜单、组织和数据范围表结构。
- U-3 统一 CurrentUser 来源和调用边界。
- U-4 统一 PermissionContext 构造和传递。
- U-5 实现或收口 DataScopeResolver。
- U-6 实现或收口 PermissionChecker。
- U-7 统一 UserDomainFacade 或等价应用服务出口。
- U-8 改造订单域列表权限消费方式。
- U-9 改造寄样域列表权限消费方式。
- U-10 改造商品域权限消费方式。
- U-11 改造达人域权限消费方式。
- U-12 改造业绩域权限消费方式。
- U-13 改造分析模块权限消费方式。
- U-14 补齐用户域操作日志和审计证据。
- U-15 补用户域单测、权限 E2E 和越权负例。

## 配置域

- C-1 盘点配置域代码、接口、表和测试。
- C-2 梳理系统配置、规则参数、提成比例和模板来源。
- C-3 明确配置域只提供参数，不执行业务规则。
- C-4 收口配置读取 API 和缓存策略。
- C-5 收口配置保存、校验和版本记录。
- C-6 补齐配置变更审计。
- C-7 明确提成比例由业绩域消费和执行。
- C-8 明确复制模板、`pick_extra` 和展示参数的消费方。
- C-9 检查业务域是否反向写配置事实。
- C-10 检查配置域是否触发具体业务动作。
- C-11 补配置域单测。
- C-12 补配置审计 API/SQL 验证。
- C-13 补配置异常分支和权限验证。
- C-14 更新配置域 state、feedback 和 evidence。

## 业绩域

- Y-1 盘点业绩域代码、接口、表、任务和测试。
- Y-2 梳理 `performance_records` 生成入口。
- Y-3 梳理最终归属输入：订单、商品映射、达人关系、配置参数。
- Y-4 梳理提成计算规则来源。
- Y-5 梳理冲正处理入口和退款事实消费。
- Y-6 梳理双轨金额计算和审计字段。
- Y-7 确认业绩域不同步抖音订单。
- Y-8 确认业绩域不修改订单原始事实。
- Y-9 收口业绩计算幂等键。
- Y-10 收口汇总刷新入口。
- Y-11 定义业绩归属已计算事件。
- Y-12 定义业绩汇总已刷新事件。
- Y-13 验证配置参数变更后的消费方式。
- Y-14 验证退款冲正和汇总变化。
- Y-15 补业绩域单测。
- Y-16 补业绩计算集成测试。
- Y-17 补权限和数据范围验证。
- Y-18 补异常分支和重复消费验证。
- Y-19 更新业绩域 state、feedback 和 evidence。

## 分析模块

- A-1 盘点分析模块代码、接口、表和测试。
- A-2 梳理 dashboard 指标来源。
- A-3 梳理汇总表、事实查询和导出来源。
- A-4 确认分析模块不重算业绩归因。
- A-5 确认分析模块不写订单、寄样或业绩事实。
- A-6 收口 dashboard API 数据范围消费。
- A-7 收口趋势、排行和对账查询。
- A-8 验证 dashboard API 与汇总 SQL 一致。
- A-9 验证 admin/group/self 下看板差异。
- A-10 补分析模块单测。
- A-11 补 dashboard E2E。
- A-12 补导出验证。
- A-13 补异常分支验证。
- A-14 补只读边界审查记录。
- A-15 更新分析模块 state、feedback 和 evidence。

## Outbox 事件

- E-1 盘点当前本地事件、Outbox 表和消费状态。
- E-2 梳理订单已同步事件。
- E-3 梳理退款事实已同步事件。
- E-4 梳理业绩归属已计算事件。
- E-5 梳理业绩汇总已刷新事件。
- E-6 梳理寄样状态已变更事件。
- E-7 梳理商品转链已完成事件。
- E-8 统一事件命名、payload、幂等键和版本。
- E-9 定义消费失败和重试证据。
- E-10 确认 V1 不强制引入 Kafka / RabbitMQ。
- E-11 补事件生产单测。
- E-12 补事件消费单测。
- E-13 补重复消费幂等验证。
- E-14 补事件证据索引。
- E-15 更新事件 state、feedback 和 evidence。

## 前端领域化

- F-1 盘点前端页面、API client、store 和路由。
- F-2 按用户域收口登录、权限、菜单和数据范围展示。
- F-3 按配置域收口规则中心和配置页面。
- F-4 按订单域收口订单列表、同步和错误展示。
- F-5 按业绩域收口业绩明细、冲正和汇总展示。
- F-6 按分析模块收口 dashboard、趋势、排行和导出。
- F-7 按商品域收口商品库、活动商品和转链交互。
- F-8 按达人域收口达人资料、地址、标签和跟进。
- F-9 按寄样域收口寄样申请、审核、发货和交作业。
- F-10 检查前端是否硬编码核心业务规则或状态机。
- F-11 补前端构建、Vitest、E2E 和截图证据。

## E2E 与 GC

- G-1 梳理渠道链 E2E。
- G-2 梳理招商链 E2E。
- G-3 梳理管理链 E2E。
- G-4 补 admin/group/self 权限回归。
- G-5 补订单归因、寄样完成、业绩和 dashboard 对账。
- G-6 补 real-pre preflight 与阻塞证据。
- G-7 执行旧内容维护候选计划。
- G-8 更新 evidence report、retro summary 和 Harness changelog。
