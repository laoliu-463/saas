## 一、V1 范围

### V1 做

- O-01 定时同步团长订单
- O-02 双轨金额（预估 + 有效 / 结算服务费）
- O-03 `pick_source -> 默认渠道`
- O-05 订单状态及退款 / 失效更新
- O-06 订单列表多维筛选
- O-08 发出“订单已同步”后续处理信号
- O-09 首次联调校验 API 字段映射
- O-10 订单明细 + 业绩补充字段

### V1 简化

- O-07 订单列表 / 导出按当前项目口径简化收口

### V1 不做

- O-04 活动 + 商品 -> 默认招商（含个别品负责人）

## 三、与领域设计的主要偏差

| 类型 | 条目 | 说明 |
|---|---|---|
| 少做 | O-03 真实样本 | 当前缺真实 `pick_source` 订单样本，暂时无法把归因写成“已完全闭环” |
| 做法差异 | O-08 事件实现 | 当前更偏内部处理路径，不强调显式独立事件总线展示 |
| 范围差异 | O-04 默认招商覆盖 | 个别品负责人覆盖不纳入当前 V1 |
| 文档风险 | 独家归因旧口径 | 订单域文档不能再把独家逻辑当作 V1 主链 |

## 四、状态机与规则检查

| 状态 | 当前实现 | 结论 |
|---|---|---|
| 待结算 | 已有 | ✅ |
| 已结算 | 已有 | ✅ |
| 已退款 | 已有 | ✅ |
| 已失效 | 已有 | ✅ |

核心规则：

- 默认渠道归因依赖 `pick_source_mapping`
- 双轨金额是订单域事实来源
- 主订单事实源和结算事实源拆开：`buyin.instituteOrderColonel` 默认 `update`，`buyin.colonelMultiSettlementOrders / 2704` 默认 `settle`
- 2704 金额单位未通过真实 raw JSON 对表前，不把结算金额展示写成上线可信
- 当前 V1 不把个别品负责人覆盖带入默认招商主链
