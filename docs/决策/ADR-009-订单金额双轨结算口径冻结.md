# ADR-009 订单金额双轨结算口径冻结

> P0-ADR-001。范围标记：`V1 必做`、`待确认`、`历史归档`。

## 状态

- [V1 必做] 已采纳。
- [V1 必做] 日期：2026-06-14。

## 背景

- [V1 必做] 订单同步同时存在金额单位口径冲突、预估轨污染结算轨、1603 分页漏单风险和看板对账口径不清。
- [V1 必做] 当前代码、数据库字段和测试中金额字段以 long 保存，现行运行链路按人民币分存储和计算。
- [历史归档] 旧归档文档曾写入库转元或把部分技术服务费字段合并解释，不能覆盖当前可运行事实。
- [V1 必做] ADR-003 要求订单域只存上游事实；ADR-004 要求业绩域负责双轨金额和最终业绩计算。

## 决策

- [V1 必做] DB 金额单位以当前项目事实为准：订单域和业绩域金额字段全链路按分存储，暂不做除以 100。
- [V1 必做] 预估轨和结算轨是两套独立事实字段；`effective_*` 不得使用 `estimate_*`、实付金额或预估技术服务费兜底。
- [V1 必做] `effective_service_fee` 只能来自上游明确结算服务费字段；待结算缺字段时保持 0/null，不得用订单额重算或扣技术服务费推导。
- [V1 必做] `effective_tech_service_fee` 只能来自上游明确结算技术服务费字段；不得使用 `estimate_tech_service_fee` 或预估 `tech_service_fee` 冒充。
- [V1 必做] 已结算订单再次同步时只更新上游返回的 `effective_*` 结算轨事实，保留首次 `estimate_*` 快照。
- [V1 必做] 看板和业绩汇总必须显式区分时间 cohort 与金额轨道，不能静默用预估轨替代结算轨。
- [V1 必做] `effective_service_profit` = `effective_service_fee` - `effective_service_fee_expense`；结算轨技术服务费只展示，不扣减结算服务费收益。

## 影响

- [V1 必做] `OrderAmountMapperPolicy` 和 1603 结算链路禁止 estimate -> effective fallback。
- [V1 必做] real-pre 复审时，待结算订单的 `effective_service_fee`、`effective_tech_service_fee` 为 0/null 不视为异常。
- [V1 必做] 旧字段 `settle_colonel_commission`、`settle_colonel_tech_service_fee` 如仅用于兼容，不得把预估轨写成结算事实。
- [V1 必做] only-in-DB 订单不能直接删除，也不能直接参与结算看板；需按来源、同步日志和上游详情追溯。

## 验证方式

- [V1 必做] 单元测试覆盖待结算单 estimate 有值、effective 为空时不 fallback。
- [V1 必做] 单元测试覆盖已结算单 estimate/effective 独立映射。
- [V1 必做] 单元测试覆盖上游缺结算字段时 `effective_*` 保持 0/null。
- [V1 必做] 1603 分页测试覆盖 `has_more` 缺失或 false 但 `nextCursor` 存在时继续翻页，并覆盖重复 cursor 停止。
- [V1 必做] real-pre 复审报告需列出构建、容器、健康检查、业务验证和剩余风险；未采集项不得写 PASS。
