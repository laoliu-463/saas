# ADR-013 Harness 分层文件门禁

## 状态

- 已实施。
- 批准日期：2026-07-13。
- 实施日期：2026-07-13。
- 白名单扩展：2026-07-18 由 ADR-014 扩展为 13 个一级目录。

## 背景

当前 Harness 使用统一的 50/50/200 硬限制：任意目录直接文件数和直接子目录数不超过 50，非脚本文本不超过 200 行。

2026-07-13 的只读审计发现：

- `runtime/qa/out/` 根目录有 89 个文件，其中 evidence 33 个、retro 29 个、content-retire 15 个。
- evidence、retro、content-retire 脚本仍向该目录生成时间戳文件，正常流程会持续突破 50 文件限制。
- Harness 内 1118 个非脚本文本的 P95 为 138 行，仅 3 个文件超过 200 行。
- 因此，200 行限制仍能约束可读性；主要矛盾是统一目录上限与流水产物生命周期冲突。

OpenAI Harness Engineering 的公开实践支持机械执行边界、文件大小限制、渐进披露和周期性 GC，但没有规定适用于所有仓库的 50/50/200 数字：

- <https://openai.com/index/harness-engineering/>
- <https://openai.com/index/unrolling-the-codex-agent-loop/>
- <https://openai.com/academy/skills/>

## 已评估方案

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 保持统一硬限制 | 实现简单 | 流水文件持续制造假失败 |
| 删除数字限制 | 局部阻力最小 | 文档熵只能事后发现 |
| 分层、基线感知门禁 | 保留边界并避免历史债务阻断 | 检查器和生成流程需要调整 |

## 决策

采用分层、基线感知门禁。核心原则是：

1. 关键边界使用硬门禁。
2. 活跃知识使用数量和行数预算。
3. 流水产物使用生命周期和稳定路径管理。
4. 历史债务不阻断无关任务，但禁止新增或恶化。
5. 每次任务检查增量，周期任务负责全量 GC。

## ADR-014 白名单扩展

ADR-014 只扩展本 ADR 的一级目录白名单，不改变数量预算、文本行数、报告生命周期或基线感知语义。

当前 13 个一级目录为：`rules/`、`tasks/`、`probes/`、`reports/`、`scripts/`、`manifests/`、`archive/`、`templates/`、`engineering/`、`src/`、`contracts/`、`state/`、`tests/`。

- `src/`、`contracts/` 和 `tests/` 分别承载 Harness 核心源码、机器契约和自身测试。
- `state/` 只承载稳定发布、架构或验证基线，按需创建；运行时临时数据和原始证据继续进入 `runtime/qa/out/<run-id>/`，禁止进入 `state/`。
- 新目录只有存在真实职责和内容时才创建，禁止用空目录或占位 README 制造表面完成。

## 门禁分层

### 1. 全局硬不变量

以下情况不使用历史债务豁免，发现即失败：

- `harness/` 出现一级目录白名单之外的目录。
- 新增密钥、凭证或明文敏感信息。
- 新增断链、失效权威入口或重复事实主源。
- `reports/` 根目录新增时间戳 evidence、retro 或 content-retire。
- 新文件或修改后的活跃文档超过对应硬上限。

### 2. 活跃知识预算

适用于 `rules/`、`tasks/`、`probes/`、`templates/`、`engineering/`、`manifests/` 和 `reports/current/`：

| 指标 | 预警 | 硬上限 |
| --- | ---: | ---: |
| 直接文件数 | 40 | 50 |
| 直接子目录数 | 40 | 50 |
| 非脚本文本行数 | 160 | 200 |

- `README.md` 继续以 120 行为建议预算。
- 脚本不使用 200 行上限，改由测试、静态检查和职责边界约束。
- npm 标准生成的 `harness/package-lock.json` 仅按该精确路径豁免文本行数预算，其他 JSON 或 lockfile 不豁免；文件数量、结构和安全门禁继续适用。
- Git 忽略的 `harness/node_modules/` 属于可再生依赖，不参与目录和仓库健康统计且禁止提交；其他未知一级目录仍立即失败。
- manifest 接近 40 个时按日期或主题分桶，不等待突破 50。

### 3. 报告与归档

- `reports/` 根目录只允许 `README.md`、`INDEX.md`、`latest-*.md` 和受控子目录，目标直接文件数不超过 20。
- `reports/current/` 保存稳定命名的当前摘要，同一主题覆盖更新，不生成时间戳副本。
- 原始日志、长 JSON、长 SQL 和逐步命令输出进入 `runtime/qa/out/<run-id>/`，默认不提交 Git。
- 每个任务只生成一份 evidence 摘要；无可执行 Harness 改进项时，retro 合并为 evidence 中的一个小节。
- 只有包含责任人、改进动作和验证方式时，才允许单独生成 retro。
- Git 历史作为普通报告的历史版本；发布、审计或唯一真实证据才进入 `archive/`。
- `archive/` 的每个分桶继续遵守 50/50；历史不可变证据不追溯强制压缩到 200 行，新摘要仍不得超过 200 行。

## 基线与状态语义

- 本地默认以 `HEAD` 为基线，CI 以变更目标分支为基线。
- 基线已超限时，数量保持或下降不阻断任务；任何增加都失败。
- 新增或修改文件突破硬上限时失败，不因基线已有其他债务而豁免。
- 检查结果同时输出“本次任务门禁”和“仓库整体健康度”。
- 本次无新增债务时任务门禁可以通过；历史债务未清零前，仓库整体健康度保持 `PARTIAL`。
- 无法取得可靠基线时不得输出整体 `PASS`，并必须说明阻塞原因。

## 执行流程

1. 为任务创建唯一 `run-id`。
2. 将原始输出写入 `runtime/qa/out/<run-id>/`。
3. 对工作区与基线执行增量门禁。
4. 生成或覆盖一份 `reports/current/latest-<topic>.md`。
5. 通过显式文件集合只暂存当前任务拥有的文件并提交，禁止默认接管整个脏工作区。
6. 周期任务执行全量检查、陈旧报告清理、重复主源扫描和归档分桶。

## 失败处理

- 新增硬违规：任务 `FAIL`，输出具体路径、阈值和修复动作。
- 仅有历史债务：任务不阻断，仓库健康度为 `PARTIAL`。
- 基线缺失或扫描失败：不得假定通过，不自动删除文件。
- GC 只生成候选清单；归档或删除仍需 manifest 和引用检查。

## 实施结果

已完成：

- `2c304397`：分层、基线感知检查器与双状态模型。
- `b38c9405`：稳定报告、显式 OwnedFiles、scoped staging 和内联 retro。
- `d1ef7289`：按 manifest 将 75 份根报告分为 evidence、retro、content-retire 三组归档。
- `b3140bb0`：把维护脚本生成的报告与归档目标纳入任务 owned set，并覆盖未跟踪目录派生检查。
- Pester 覆盖边界、基线、稳定报告、路径穿越、分组归档和精确 owned set。

本 ADR 不直接决定业务规则、远端部署策略或现有业务代码结构。

## 验收场景

- 基线目录 89 个文件，当前仍为 89：任务不失败，整体为 `PARTIAL`。
- 基线目录 89 个文件，当前增加到 90：任务失败。
- 基线目录 89 个文件，当前减少到 88：任务通过，整体仍为 `PARTIAL`。
- 活跃文档从 199 行增加到 201 行：任务失败。
- 新增 `reports/evidence-<timestamp>.md`：任务失败。
- 覆盖 `reports/current/latest-<topic>.md` 且数量不增长：允许。
- 工作区存在其他任务的脏文件：当前任务 evidence 不纳入这些文件，提交也不得暂存这些文件。
- 归档旧证据超过 200 行但内容 blob 未修改：按历史不可变证据豁免，不阻断任务或仓库健康度。
- 新脚本超过 200 行：不触发行数门禁，但必须通过对应测试与职责审查。
- 标准格式化的 `harness/package-lock.json` 超过 200 行：只豁免行数门禁；同名嵌套 lockfile 仍失败。
- `npm ci` 生成且被 Git 忽略的 `harness/node_modules/`：不计入白名单和健康债务；若被纳入提交则由 Git/安全门禁阻断。

## 后果

- 50/50/200 从全目录统一规则变为活跃知识预算和局部硬门禁。
- 报告增长由“事后归档”转为“源头不产生重复文件”。
- 门禁结果更能区分本次回归和历史债务。
- 当前脚本、canonical 规则与本 ADR 对齐；后续改动必须继续通过自动化门禁。
