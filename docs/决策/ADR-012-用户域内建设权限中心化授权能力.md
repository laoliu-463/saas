# ADR-012 用户域内建设权限中心化授权能力

> 范围标记：[V2 长期治理] 权限模型与安全边界变更。
>
> 关联：ADR-001、PRD-DDD-MIGRATION-100、`2026-07-13-ddd-rbac-long-term-governance-design.md`。

## 状态

- [V2 长期治理] 已采纳，2026-07-13 经用户逐段确认。

## 背景

- 当前接口主要通过 `@RequireRoles` 和固定角色编码鉴权，自定义角色无法稳定获得业务入口权限。
- `sys_role.permissions`、`sys_menu.permission_code` 与角色切面形成多套授权事实。
- `sys_role.data_scope` 是全局最大范围，不能表达角色在不同业务域的差异。
- JWT 携带角色和数据范围，权限缓存失效不能立即撤销已签发令牌中的旧权限。
- 缺少用户或组织上下文时，现有数据范围链存在不附加过滤的路径。
- PRD-DDD-MIGRATION-100 原则上要求行为 1:1、无 Schema 变更，无法完成上述根因治理。

## 决策

### 领域归属

在用户限界上下文内建设 Authorization 子域能力，不新建微服务。用户域管理身份、组织、角色、权限目录、领域数据范围、授权版本和通用决策；业务域管理对象归属、状态机、幂等和动作合法性。

拆分跨域接口：

- `AuthorizationFacade` 输出授权主体与决策。
- `UserDirectoryFacade` 输出名称、组织和目录事实。
- 现有 `UserDomainFacade` 仅保留迁移适配，不新增职责。

### 授权模型

- 使用权限中心化 RBAC：用户—角色—权限。
- 使用领域化范围属性：角色—业务域—SELF/GROUP/ALL。
- 业务动作采用 RBAC 入口权限与领域 ABAC/策略组合。
- 默认拒绝；缺失上下文或授权基础设施故障不得放行。
- 第一阶段不采用用户直接授权、角色继承、显式拒绝规则和代码级管理员绕过。

### 持久化与令牌

增量新增 `sys_permission`、`sys_role_permission`、`sys_role_domain_scope`、`sys_authz_change_log`，并为 `sys_user` 增加 `authz_version`。旧 JSON 和全局数据范围字段在迁移期兼容，完成对账后通过独立迁移删除。

JWT 长期只承载认证声明和授权版本。服务端按当前有效关系生成授权快照；授权变更与受影响用户版本递增在同一事务完成。

### 执行与迁移

使用 Spring Security `AuthorizationManager` 与统一权限元注解作为入口执行点。各业务域以 `LEGACY/SHADOW/ENFORCE` 独立灰度，禁止把新旧规则做宽松 OR 合并。

权限编码语义和旧角色映射必须由业务负责人确认，AI 和迁移脚本只生成候选矩阵。

## 对既有决策的影响

- 保持 ADR-001 模块化单体，不引入独立 IAM 服务或外部策略引擎。
- 仅在 RBAC 长期治理范围内，取代 PRD-DDD-MIGRATION-100 的“无 Schema、行为 1:1”限制。
- 保留该 PRD 的外部 API 兼容、渐进迁移、证据门禁和禁止跨域 Mapper 原则。
- 不改变订单、业绩、商品、达人、寄样和配置域的既有业务职责。

## 结果

正向结果：

- 角色配置能够真实控制接口权限，授权事实单一且可审计。
- 数据范围按业务域和当前权限计算，消除全局最大范围扩权。
- 角色或权限变更可通过授权版本立即使旧 Access Token 失效。
- 业务状态规则继续留在所属领域，避免用户域膨胀为全局业务规则中心。

代价与风险：

- 需要新增 Schema、授权管理 API、前端权限投影和跨领域迁移。
- 角色权限变更需要递增关联用户版本，当前规模可接受，后续按监控评估版本向量。
- 旧角色矩阵映射属于业务判断，未确认项会阻塞对应领域进入 `ENFORCE`。
- 回退到 `LEGACY` 会恢复已知风险，只能限时使用。

## 验证与回滚

- 以权限矩阵确认、Schema 集成测试、负向安全测试、JWT 失效测试、影子差异清零、领域 E2E 和架构守卫作为强制门禁。
- 回滚优先从 `ENFORCE` 降为 `SHADOW`，保留新增表和审计数据，不执行破坏性删表。
- 详细测试、迁移顺序和错误语义以关联设计文档为准。
