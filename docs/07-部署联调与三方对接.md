# 07-部署联调与三方对接

更新时间：2026-05-19

> 本文档统一描述 **环境、部署、联调、真实三方对接边界**。  
> 当前目标不是直接推生产，而是先稳定 `test` 和 `real-pre` 两类环境，并围绕 V1 主链完成补证收口。

## 一、当前结论

- `test` 是当前功能验证、权限验证和本地业务联调基线
- `real-pre` 是当前真实 SDK / 真实上游联调环境
- `local-mock` 只保留为历史参考和回滚口径
- `prod` 只作为正式环境规划，不在本轮展开

## 二、环境矩阵

| 环境 | 用途 | 当前口径 | 说明 |
|---|---|---|---|
| `local-mock` | 历史 Mock 口径与回滚参考 | 历史参考 | 不再作为新的主线环境扩展 |
| `test` | 系统功能 / 权限测试 / 本地联调 | 当前默认验证环境 | 支撑大部分功能、规则和页面回归 |
| `real-pre` | 真实 SDK / 真实上游联调 | 当前真实联调环境 | 支撑 Token、活动、商品、转链、订单同步、Dashboard 取证 |
| `prod` | 正式环境规划 | 后续单独规划 | 本轮不展开 |

## 三、当前运行方式

### 统一约束

1. 同一时间只允许一个 active 环境
2. 固定容器名：
   - `saas-frontend`
   - `saas-backend`
   - `saas-postgres`
   - `saas-redis`
3. 固定端口：
   - 前端 `3000`
   - 后端 `8080`
   - PostgreSQL `5432`
   - Redis `6379`
4. `test` 和 `real-pre` 通过不同 `.env` 切换，不再混起双套容器

### 常用命令

启动 `test`：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-test.ps1
```

启动 `real-pre`：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-real-pre.ps1
```

查看状态：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\status.ps1
curl http://localhost:8080/api/actuator/health
```

停止当前环境：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\stop-all.ps1
```

### 当前 P0 主链

## 五、抖音联调边界

### 1. Token / 授权主体

当前要求：

- 只在 `real-pre` 做真实 Token 初始化和刷新验证
- 不把真实凭据写入文档或 Git
- 凭据只允许放在本机未跟踪 env、CI / CD Secret 或服务器环境变量

### 2. 活动 / 活动商品

当前目标：

- 能读取真实活动列表
- 能同步真实活动商品
- 能读取商品基础信息
- `buyin.productSkus.v2` 用于 SKU 规格补充；当前按商品域 P1 预留，不作为订单 / 结算 / V1 P0 主链阻断项

当前接口归类补充：

| 上游接口 | 文档 ID | 当前归类 | V1 优先级 | 说明 |
|---|---:|---|---|---|
| `buyin.instituteOrderColonel` | `1603` | 订单域主同步 / 默认结算主接口 | P0 | 团长主订单事实来源默认 `time_type=update`；结算同步默认 `time_type=settle` |
| `buyin.colonelMultiSettlementOrders` | `2704` | fallback / probe / 对照 | P0 | 不再作为默认写库主链路；仅显式配置或只读对照时使用 |
| `buyin.productSkus.v2` | `6468` | 商品域 SKU 补充 | P1 | 查询商品 SKU，不承担订单 / 结算口径 |

- `buyin.productSkus.v2` 需按上游主体满足对应达人授权、MCN 机构授权、抖客授权、团长授权等授权条件；联调前先确认当前 token 主体与接口授权类型匹配

### 3. 转链

当前目标：

- 调用真实转链接口
- 生成有效 `pick_source`
- 落 `pick_source_mapping`

### 4. 订单同步

当前目标：

- 通过 `buyin.instituteOrderColonel` 拉回真实团长订单
- 写入本地订单池
- 为默认归因、寄样完成和 Dashboard 提供事实来源

当前补充口径：

- 时间范围拉取、结算回扫与按订单号补偿的默认入口统一使用 `buyin.instituteOrderColonel`
- 主订单同步默认 `time_type=update`，维护订单事实、状态变化、预估看板和寄样“出单即完成”判断
- 结算同步默认 `time_type=settle`，同步来源标记为 `INSTITUTE_SETTLEMENT`，只读取上游明确返回的结算字段
- `buyin.colonelMultiSettlementOrders` 仅作为 fallback / probe / 对照；通过 `douyin.order.settlement-source=colonelMultiSettlementOrders` 显式回滚
- 1603 真实结算字段稳定性仍需 raw probe 与小窗口 dry-run 继续补证；未验证项不得写成业务闭环 PASS。

### 5. Webhook

当前口径：

- 收件箱、验签、幂等、重放已有底座
- 真实业务副作用仍依赖上游样本
- 当前不把完整 Webhook 业务消费当作 V1 P0 上线前提

## 六、物流 API 口径

### 当前 V1 口径

- 手动录入物流公司、单号和状态推进为主
- 物流 API 只保留底座
- 未配置真实物流密钥时，不把自动物流写成当前主交付

### 当前边界

- 快递鸟 / 其他物流供应商都属于后续增强位
- 真实物流轨迹接入不作为当前 P0 阻断项

## 七、达人外部数据口径

### 当前 V1 口径

- 达人基础字段 + 手动补录兜底
- 不把真实爬虫增强、Cookie 池、代理池作为当前主线前提

### 当前边界

- 真实外部采集能力不作为当前 V1 P0 阻断项
- 如果后续接入，需要单独补新证据，不复用当前手动兜底结论

## 九、当前对接边界总表

| 能力 | 当前状态 | 是否属于 V1 主链 |
|---|---|---|
| 抖音 Token / 授权主体 | 已进入主线 | 是 |
| 活动 / 活动商品 | 已进入主线 | 是 |
| 转链 / `pick_source_mapping` | 已进入主线 | 是 |
| 订单同步 | 已进入主线 | 是 |
| Dashboard / 数据验证 | 已进入主线 | 是 |

| 达人外部采集增强 | 手动兜底 | 否 |
| 店铺侧能力 | 不纳入当前主线 | 否 |

## 十、一句话结论

> 当前部署与联调口径只有一个目标：让 `test` 稳定承接功能验证，让 `real-pre` 稳定承接真实主链补证；自动物流、爬虫增强、店铺侧能力统一后置，不重新拉回当前 V1。
