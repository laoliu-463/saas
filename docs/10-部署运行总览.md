# 部署运行总览

> 范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 当前运行口径

- [V1 必做] 当前运行以 Docker Compose 管理 PostgreSQL、Redis、后端和前端。
- [V1 必做] 本机常用端口组为 `3000/8080` 与 `3001/8081`，不得混起第二个 `3001` Vite 或额外 `8080` 后端。
- [V1 必做] real-pre 典型公网/宿主机端口为前端 `3001`、后端 `8081`；PostgreSQL `5432` 与 Redis `6379` 仅在 Compose 网络内访问，不对公网暴露。
- [V1 必做] real-pre 必须关闭 mock：`APP_TEST_ENABLED=false`、`DOUYIN_TEST_ENABLED=false`。
- [V1 必做] 当前只保留两个环境：`test` 测试环境与 `real-pre` 真实上游 / 生产形态环境；旧 `dev`、`local-mock`、`real`、`prod` 入口已合并或删除。

## 环境清单

| 环境 | Compose / 配置 | 用途 | 验证 | 范围 |
| --- | --- | --- | --- | --- |
| test | `docker-compose.test.yml`、`.env.test` | 本地 mock / P0 回归 | E2E、API、SQL | V1 必做 |
| real-pre | `docker-compose.real-pre.yml`、`.env.real-pre` | 真实联调、上线前验收和当前生产形态部署 | preflight、真实接口、日志、迁移记录 | V1 必做 |

## 部署前检查

- [V1 必做] 环境变量不包含过期 Token、测试开关误开或端口冲突。
- [V1 必做] PostgreSQL 和 Redis 容器健康。
- [V1 必做] 后端 profile、数据库连接、Redis 连接、第三方配置在启动日志中可证。
- [V1 必做] 前端目标 API 地址指向当前环境后端。
- [V1 必做] real-pre 跑验收前执行预检命令。
- [V1 必做] 部署允许条件：构建产物、数据库迁移、环境变量、真实 API Token、Redis/PostgreSQL、日志目录、证据目录全部确认；real-pre 上游读、同步、刷新、回调和写入类开关默认开启，包括真实推广写双开关 `DOUYIN_REAL_PROMOTION_WRITE_ENABLED=true` 与 `ALLOW_REAL_PROMOTION_WRITE=true`。如因风控、上游冻结或只读排障临时关闭，必须记录原因、影响范围和恢复计划。
- [V1 必做] 数据库备份和迁移由 Jenkins 根据当前 SHA 到目标 SHA 的迁移文件 diff 决定；没有数据库变更时明确跳过，普通 Agent 不得强制执行远端数据库操作。

## 服务器 real-pre 受控部署结论

- [V1 必做] 当前状态达到“服务器 real-pre 受控部署验证”门槛，但未达到“real-pre P0 全量通过”门槛。
- [V1 必做] 部署目标是进入真实样本采集与真实订单闭环验证阶段，不是正式生产放量。
- [V1 必做] 允许：服务器 real-pre 受控部署。
- [V1 必做] 不允许：宣称 real-pre P0 完全通过 / 可正式全量上线。
- [V1 必做] 部署后第一轮验收固定执行 `npm run e2e:real-pre:p0:preflight`、`npm run e2e:real-pre:p0`、`npm run e2e:real-pre:roles`。

| 部署后结果 | 结论 | 范围 |
| --- | --- | --- |
| 无失败 + 无真实订单 | real-pre 环境部署成功，P0 仍因真实样本不足保持 PENDING | V1 必做 |
| 无失败 + 有真实订单 + 归因 / 寄样 / 业绩通过 | real-pre P0 可升级为通过 | V1 必做 |
| 出现失败 | 按失败项定级回滚或修复，不得放量 | V1 必做 |

## Jenkins 单通道 real-pre 受控部署

- [V1 必做] 当前采用单机 Docker Compose + Jenkins 唯一发布队列；不需要 Kubernetes，也不允许普通 Agent 手工 SSH 部署。
- [V1 必做] 服务器目录固定为 `/opt/saas/app`、`/opt/saas/env`、`/opt/saas/logs`、`/opt/saas/backups`、`/opt/saas/releases`、`/opt/saas/runtime/qa/out`。
- [V1 必做] `main` 是唯一集成主线，`release/real-pre` 是唯一部署来源；服务器运行内容由完整 SHA 镜像和 `/opt/saas/releases/*.json` 标识，不再以共享工作树反复 checkout/pull/reset 作为版本事实。
- [V1 必做] real-pre 部署复用 `docker-compose.real-pre.yml` 与只读的 `/opt/saas/env/.env.real-pre`，固定端口为前端 `3001`、后端 `8081`、PostgreSQL 容器内 `5432`、Redis 容器内 `6379`，数据库和 Redis 不对公网暴露。
- [V1 必做] real-pre 不是 `docker-compose.test.yml` 的 mock/test 基线，必须保持 `APP_TEST_ENABLED=false`、`DOUYIN_TEST_ENABLED=false`、`DOUYIN_REAL_UPSTREAM_MODE=live`，数据库必须为 `saas_real_pre`。real-pre 默认开启真实推广写双开关，商品库“复制简介”会触发真实 `buyin.instPickSourceConvert` 并写入 `pick_source_mapping`；验收和排障必须按真实上游副作用记录证据。
- [V1 必做] 发布顺序为任务分支 PR/CI、串行合并 `main`、发布提升 PR、`release/real-pre`、Jenkins 全局锁、顺序检查、不可变镜像、按需数据库步骤、backend/frontend 部署、版本一致性、real-pre 验收、发布清单和 evidence。
- [V1 必做] 端口验活使用公开探针 `/api/system/health` 和前端 `/login`，不使用需要鉴权的 `/api/actuator/health` 作为无鉴权健康检查。
- [V1 必做] Compose project 固定为 `saas-active`，与当前 real-pre 启动脚本、容器名和 QA 默认 DB 容器一致；不要切回旧 project 导致创建第二套数据卷。
- [V1 必做] 完整域名部署时，宝塔只负责站点、SSL、Nginx 反代、防火墙和监控；Docker Compose 仍负责 PostgreSQL、Redis、backend、frontend。
- [V1 必做] 完整域名部署的公网入口统一为 `https://real-pre.<domain>`，Nginx 将 `/api/` 反代到 `127.0.0.1:8081/api/`，将 `/` 反代到 `127.0.0.1:3001/`。
- [V1 必做] 抖店 OAuth 回调、OAuth 成功 / 失败页、快递100订阅回调和 CORS 必须使用同一个 HTTPS real-pre 域名；不得在百应后台继续保留临时 IP + 端口地址。
- [V1 必做] Nginx / SSL 验收通过后，云安全组和宝塔防火墙只保留 `80/443/22/宝塔面板受限端口`，关闭公网 `8081/3001/5432/6379`。
- [V1 必做] 部署允许 real-pre 短暂中断；不得加 `-v`，避免清空 real-pre 数据卷。
- [V1 必做] `npm run e2e:real-pre:p0:preflight`、`npm run e2e:real-pre:p0`、`npm run e2e:real-pre:roles` 执行结果必须进入部署报告；`BLOCKED` / `PENDING` 不能记为通过。
- [V1 必做] 每次部署必须归档 Compose 状态、后端 / 前端 / PostgreSQL / Redis 日志、`/opt/saas/logs/**`、`runtime/qa/out/**`、`playwright-report/**` 和 `test-results/playwright/**`。
- [V1 必做] 抖店 OAuth 回调到受保护页面时必须保留登录后回跳目标；未登录浏览器进入 `/system/douyin?oauth=success` 后跳 `/login?redirect=...`，登录成功后回到抖店联调页继续刷新 Token 状态。
- [V1 必做] 巨量百应官方授权管理页为 `https://buyin.jinritemai.com/dashboard/institution/power-manage`；它用于人工检查或开通授权主体能力，不替代 OAuth code 回调换 Token 链路。

## 当前代码到服务器发布步骤

- [V1 必做] 所有开发从 `main` 创建 `codex/<issue>-<slug>` 独立分支和 worktree。
- [V1 必做] 任务完成后提交 PR，CI 通过并串行合并到 `main`；禁止直接推送 `main` / `release/real-pre`。
- [V1 必做] 发布控制器创建从目标 `main` 提交到 `release/real-pre` 的提升 PR；一次只合并一个候选。
- [V1 必做] Jenkins Job `saas-real-pre-cd` 只接受 `DEPLOY_BRANCH=release/real-pre`，并通过 `saas-real-pre-deploy` 全局锁排队。
- [V1 必做] `.env.real-pre`、`.env.test`、`.env` 不进入 Git；服务器真实环境文件固定为 `/opt/saas/env/.env.real-pre`，发布任务不得修改。
- [V1 必做] 目标不是当前部署版本后继提交时默认拒绝，只有批准回滚才能设置 `ROLLBACK_APPROVED=true`。
- [V1 必做] 无迁移输入变化时 `RUN_DB_MIGRATIONS=false`，跳过远端备份、迁移和 Schema 预检；有变化时才执行受控数据库链路。
- [V1 必做] 发布完成后同时核对 Jenkins SHA、后端 `gitSha/imageDigest`、前端 `/version.json`、Docker SHA/摘要和 Flyway 成功版本。

发布指针：

```text
/opt/saas/releases/<完整SHA>/release.json
/opt/saas/releases/current.json
/opt/saas/releases/previous.json
```

只有健康、版本、镜像、数据库版本和业务验证全部通过后，Jenkins 才原子更新 `current.json`。

## 当前部署验证基线

- [V1 必做] 最近一次本地后端全量回归：`mvn test`，`1499 tests, 0 failures, 0 errors`。
- [V1 必做] 最近一次前端生产构建：`npm run build` 通过；仅存在 Vite chunk 体积警告。
- [V1 必做] 最近一次 real-pre 预检：`npm run e2e:real-pre:p0:preflight` 通过；证据目录 `runtime/qa/out/real-pre-preflight-20260531-200252/`。
- [V1 必做] 本地 real-pre 容器健康形态：`frontend-real-pre`、`backend-real-pre`、`postgres-real-pre`、`redis-real-pre` 四个服务均应为 healthy。

## 明确不做

- [V1 不做] 不把旧 FastAPI / Celery 部署方案写为当前部署事实。
- [V1 不做] 不使用 mock 数据证明 real-pre 闭环。
- [V1 不做] 不再维护独立 `prod` / `production` 环境入口；生产形态统一进入 `real-pre`。
- [V2 预留] 高可用、灰度发布、告警体系和容量规划另行建设。
