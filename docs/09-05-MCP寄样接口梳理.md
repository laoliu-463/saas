# 09-05-MCP寄样接口梳理

更新时间：2026-05-20

> 本文档是 [09-MCP接口梳理-第一版](./09-MCP接口梳理-第一版.md) 的第五份拆分稿，专门收口 **寄样申请 -> 限制校验 -> 审核 -> 发货/物流 -> 待交作业 -> 订单触发完成** 链路。  
> 本轮已按定版口径回写 **后端代码 / 前端权限 / 测试 / 文档**；V1 仍不把 V2 后移项重新拉回主验收范围。

## 一、梳理结论

1. business OAS 已暴露寄样主接口族：`/samples`、`/samples/{id}`、`/samples/{id}/status`、`/samples/{id}/status-logs`、`/samples/eligibility-check`、`/samples/batch-approve`、`/samples/batch-reject`、`/samples/batch-ship`、`/samples/{id}/logistics/refresh`、`/samples/board`、`/samples/exports`、`/samples/talent-candidates`、`/samples/product-candidates`。
2. 渠道发起寄样主入口是 `POST /samples`；入参实际是 `talentId`、`productId`、`quantity`、`remark`、达人快照字段，以及结构化 `recipientName` / `recipientPhone` / `recipientAddress` 收货字段；后端写入 `sample_request.recipient_*` 并在 `extra_data.addressSource` 标记来源。
3. `POST /samples` 会消费配置规则：`sample.restrict_enabled` 与 `sample.restrict_days`；普通渠道命中同渠道 + 同达人 + 同商品 + 限制天数内重复申请时抛业务异常。
4. 7 天限制豁免当前代码已收口为渠道申请侧口径：仅 `admin`、`channel_leader` 豁免；`biz_leader` 可审核本组寄样，但不豁免 7 天重复申请限制。
5. 被拒绝申请当前不计入 7 天重复限制：后端查询重复申请时排除 `REJECTED`。
6. 寄样审核没有独立 `/approve` / `/reject` 单条接口；当前单条审核通过 / 拒绝统一走 `PUT /samples/{id}/status`，批量审核走 `/samples/batch-approve`、`/samples/batch-reject`。
7. 当前审核角色实现已收口为：`admin` 可审核全部，`biz_staff` 可审核自己负责商品，`biz_leader` 可审核本组招商负责商品。
8. 发货与签收主口径是手动状态推进：单条走 `PUT /samples/{id}/status`，批量发货走 `POST /samples/batch-ship`；物流自动刷新底座是 `POST /samples/{id}/logistics/refresh` + `LogisticsGateway`。
9. 状态机内部字段和前端展示字段不完全同名：内部 `PENDING_HOMEWORK` 输出为 `PENDING_TASK`，内部 `COMPLETED` 输出为 `FINISHED`，内部 `SHIPPING/DELIVERED` 对前端都输出为 `SHIPPED`。
10. `completed_by_order` 没有独立业务接口；订单入库/更新后由 `OrderSyncPersistenceService` 调用 `SampleLifecycleService.completePendingHomeworkByOrder` 自动完成匹配寄样单，且必须订单 `pay_time != null`。
11. `completed_by_order` 的取证路径应写成：`POST /orders/sync` 结果 + `GET /samples` / `GET /samples/{id}` + `GET /samples/{id}/status-logs` + `GET /orders/{orderId}` 的 `sample.completedByOrderRule` + DB/日志补证。
12. S-09 物流 API 当前只保留底座，V1 仍以手动物流为主；S-10 Excel 批量导入物流单号不进 P0，即使前端有本地解析 Excel 并调用 `batch-ship` 的痕迹，也不能写成 P0 验收已通过。
13. 本轮业务编号 S-06（筛选/导出字段）在主范围文档对应 S-07：寄样分页、详情和导出补齐合作方 ID、合作方名称、收货人、手机号、收货地址；筛选支持 `partnerName`、`receiverAddress` / `address`。

## 三、寄样申请链路

当前链路：

```text
GET /samples/product-candidates
GET /samples/talent-candidates
POST /samples/eligibility-check
POST /samples
GET /samples
GET /samples/{id}
```

关键写实：

- `POST /samples` 必须是已入库商品：后端会检查活动商品是否 `selectedToLibrary=true`。
- `channel_staff` 发起申请前必须已认领达人；`admin` 和 `channel_leader` 不走该私海认领硬校验。
- 结构化地址字段当前已在 `SampleApplyRequest` 中暴露：`recipientName`、`recipientPhone`、`recipientAddress`，写入 `sample_request.recipient_*`。
- 后端 `buildSampleExtraData` 会记录 `addressSource`，可用于区分本次手工填写或认领地址兜底来源；是否已在 real-pre 完成“达人地址自动带入”仍需单独补证。
- 资格预检和创建都会读取寄样门槛；商品补充信息里的门槛优先于 `sample.default_standard`，不达标时需要填写 `remark`，否则后端返回业务异常。

## 四、寄样限制与豁免

### 1. 规则消费点

`POST /samples` 内部执行：

```text
checkSevenDaysLimit(userId, talentId, productId, roleCodes)
```

实际规则：

| 项 | 当前实现 |
|---|---|
| 开关 | `BusinessRuleConfigService.isSampleRestrictEnabled()` 读取 `sample.restrict_enabled`，默认 `true` |
| 天数 | `BusinessRuleConfigService.getSampleRestrictDays()` 读取 `sample.restrict_days`，默认 `7` |
| 匹配范围 | `channel_user_id + talent_id + product_id + create_time >= now - restrictDays` |
| 排除状态 | `REJECTED` 不计入重复限制 |
| 命中返回 | 抛业务异常：`Duplicate sample request is blocked within {restrictDays} days` |
| 豁免角色 | `admin`、`channel_leader` |

### 2. 角色影响分析

| 角色 | 能否发起 `POST /samples` | 是否受 7 天限制 | 当前结论 |
|---|---:|---:|---|
| `channel_staff` | 是 | 是 | 普通渠道命中限制会被拦截 |
| `channel_leader` | 是 | 否 | 当前确认豁免 |
| `admin` | 是 | 否 | 当前确认豁免 |
| `biz_staff` | 否 | 不适用 | 招商做审核，不发起申请 |
| `biz_leader` | 否 | 不豁免 | 可审核本组招商负责商品的寄样单，但不属于渠道申请侧豁免 |
| `ops_staff` | 否 | 不适用 | 运营做发货和签收推进 |

结论：

- P0-8 当前可以写成“接口与代码规则已确认”，不能写成“验收通过”。
- “组长豁免”当前准确表述为“渠道组长 / 管理员豁免”；招商组长只参与审核，不参与 7 天申请豁免。
- 被拒绝申请不计入限制已经由代码确认，可纳入后续 Apifox/DB 断言。

## 五、寄样审核链路

当前没有独立 `/samples/{id}/approve` 或 `/samples/{id}/reject`：

| 动作 | 单条接口 | 批量接口 | 当前角色 | 关键断言 |
|---|---|---|---|---|
| 审核通过 | `PUT /samples/{id}/status`，`action=APPROVED/PENDING_SHIP` | `POST /samples/batch-approve` | `admin` / `biz_leader` / `biz_staff` | `status=PENDING_SHIP`、`auditTime` 有值、状态日志记录 |
| 审核拒绝 | `PUT /samples/{id}/status`，`action=REJECTED` + `reason` | `POST /samples/batch-reject`，`remark` 必填 | `admin` / `biz_leader` / `biz_staff` | `status=REJECTED`、`rejectReason` 有值、状态日志记录 |

需要核对项：

- `biz_leader` 已按“招商组长”定版纳入审核角色，但审核范围只限本组招商负责商品。
- OAS 描述“仅招商角色”当前已与代码收口为 `biz_leader` / `biz_staff` / `admin`。

## 六、发货与物流链路

当前 V1 主链应按手动物流收口：

```text
PENDING_SHIP
PUT /samples/{id}/status action=SHIPPED + trackingNo
或 POST /samples/batch-ship
SHIPPED
PUT /samples/{id}/status action=SIGNED
PENDING_TASK
```

物流 API 底座：

- `POST /samples/{id}/logistics/refresh` 会调用 `LogisticsTrackService.refreshAndProgress`。
- `LogisticsTrackService` 只在寄样单为内部 `SHIPPING`，且存在 `trackingNo + shipperCode` 时查询物流轨迹。
- 当 `LogisticsGateway.queryTrack` 返回内部状态 `SIGNED` 时，系统自动推进到内部 `PENDING_HOMEWORK`，对前端输出 `PENDING_TASK`。
- `KdniaoLogisticsGateway` 仅在 `kdniao.enabled=true` 时启用；否则 `ManualFallbackLogisticsGateway` 返回未配置提示，不推进状态。
- `application-real-pre.yml` 中 test logistics 关闭，real-pre 不能把自动物流轨迹写成当前主交付。

## 七、寄样状态机字段对齐表

| 业务状态 | V1 口径 | 当前接口字段 | 内部状态 | 触发接口/动作 | 是否可读 | 是否可改 | 关联 P0 | 结论 |
|---|---|---|---|---|---|---|---|---|
| 待审核 | 渠道提交申请后进入 | `PENDING_AUDIT` | `PENDING_AUDIT(1)` | `POST /samples` | 是，`GET /samples` / `GET /samples/{id}` | 是，审核通过/拒绝 | `P0-1` / `P0-2` | 已确认主链 |
| 待发货 | 招商审核通过 | `PENDING_SHIP` | `PENDING_SHIP(2)` | `PUT /samples/{id}/status action=APPROVED/PENDING_SHIP`；`POST /samples/batch-approve` | 是 | 是，发货或自动关闭 | `P0-1` / `P0-2` | 已确认主链 |
| 快递中 | 录入物流/发货后进入 | `SHIPPED` | `SHIPPING(3)`，`DELIVERED(4)` 对外都映射为 `SHIPPED` | `PUT /samples/{id}/status action=SHIPPED/SHIPPING`；`POST /samples/batch-ship` | 是 | 是，签收/待交作业 | `P0-1` / `P0-6` | 已确认主链，但内部 `DELIVERED` 对前端不单独展示 |
| 待交作业 | 签收后等待出单 | `PENDING_TASK` | `PENDING_HOMEWORK(5)` | `PUT /samples/{id}/status action=SIGNED/PENDING_TASK/PENDING_HOMEWORK`；物流刷新签收自动推进 | 是 | 人工完成不允许；订单命中后系统自动完成 | `P0-6` | 已确认主链 |
| 已完成 | 订单匹配后完成 | `FINISHED` | `COMPLETED(6)` | `OrderSyncPersistenceService` -> `SampleLifecycleService.completePendingHomeworkByOrder` | 是 | 人工接口被权限拦截；系统自动 | `P0-6` | 接口可读，完成触发需订单样本补证 |
| 已拒绝 | 招商拒绝 | `REJECTED` | `REJECTED(7)` | `PUT /samples/{id}/status action=REJECTED`；`POST /samples/batch-reject` | 是 | 可由渠道/管理员删除；不计入 7 天限制 | `P0-8` / `P0-2` | 已确认主链 |
| 已关闭 | 运营/系统关闭为主 | `CLOSED` | `CLOSED(8)` | 当前人工 `CLOSED` 动作被权限拦截；超时自动关闭底座存在 | 是 | 系统自动为主 | 非 P0 阻断 | 待核对，不作为 V1 P0 主入口 |

状态机非法跳转结论：

- `ensureTransition` 会校验当前状态是否等于动作所需前置状态。
- `COMPLETED` / `CLOSED` 在 `ensureActionRolePermission` 阶段被人工角色拦截。
- 因此 Apifox 应包含“错误状态执行审核/发货/签收失败”的负向断言。

## 八、订单触发 completed_by_order

### 1. 当前实现路径

```text
POST /orders/sync
-> OrderSyncService 获取订单
-> OrderSyncPersistenceService.persistOrder
-> insert/update colonelsettlement_order
-> SampleLifecycleService.completePendingHomeworkByOrder(order)
-> 匹配 PENDING_HOMEWORK 寄样单并置 COMPLETED
```

匹配条件：

| 条件 | 当前实现 |
|---|---|
| 渠道 | `order.channelUserId = sample_request.channel_user_id` |
| 达人 | `order.extra_data.talent_uid`，无则 `author_id`，匹配 `sample_request.talent_uid` |
| 商品 | `order.productId` 匹配 `product.product_id`，再关联 `sample_request.product_id = product.id` |
| 支付时间 | `order.pay_time IS NOT NULL`，未付款订单不触发完成 |
| 状态 | 只处理内部 `PENDING_HOMEWORK(5)` |
| 完成动作 | `sample_request.status=6`，`complete_time=now` |
| 日志 | `sample_status_log.remark = auto complete by order: {orderId}` |

### 2. 取证路径

若 OAS 无独立接口，不能写成缺功能，也不能写成验收通过。当前应按以下组合补证：

1. `POST /orders/sync`：订单同步或更新成功。
2. `GET /orders/{orderId}`：先对 `pay_time=null` 样本断言不完成，再对 `pay_time!=null` 样本断言 `sample.matched=true`、`sample.sampleStatus=FINISHED`、`sample.completedByOrderRule=true`。
3. `GET /samples/{id}`：负向样本仍为 `PENDING_TASK`；正向样本对应寄样单 `status=FINISHED`、`completeTime` 非空。
4. `GET /samples/{id}/status-logs`：出现 `PENDING_TASK -> FINISHED`，备注含 `auto complete by order: {orderId}`。
5. DB/日志补证：`sample_request.status=6`，`sample_status_log.to_status=6`，必要时对 `channel_user_id + talent_uid + product_id`。

## 九、S-09 / S-10 边界

| 项 | 当前事实 | V1 口径 |
|---|---|---|
| S-09 物流 API | 有 `LogisticsGateway`、`KdniaoLogisticsGateway`、`ManualFallbackLogisticsGateway`、`/samples/{id}/logistics/refresh` | 只保留底座，手动物流为主，不作为 P0 阻断 |
| 手动物流 | `PUT /samples/{id}/status` 和 `POST /samples/batch-ship` 可录入物流单号并推进状态 | V1 主链 |
| 自动轨迹 | 仅在物流网关配置且返回 `SIGNED` 时自动推进待交作业 | 不拉回 V1 主验收 |
| S-10 Excel 批量导入 | 前端 `ops/Shipping.vue` 有 XLSX 解析并调用 `batch-ship` 的痕迹 | 不进 P0；只能算辅助能力或后续增强 |

## 十、Apifox 测试场景建议

| 场景 | 接口序列 | 重点断言 | 目的 |
|---|---|---|---|
| 渠道申请寄样 | `GET /samples/product-candidates` -> `GET /samples/talent-candidates` -> `POST /samples/eligibility-check` -> `POST /samples` -> `GET /samples/{id}` | 创建成功；状态 `PENDING_AUDIT`；`channelUserId` 为当前渠道；资格快照可见 | 覆盖 S-01 |
| 资格不达标要求原因 | `POST /samples/eligibility-check` -> `POST /samples` | `needReason=true` 时空 `remark` 被拒绝；补 `remark` 后可提交 | 覆盖 S-01/C-04/P0-3 |
| 普通渠道 7 天限制 | 同一 `channel_staff + talent + product` 连续两次 `POST /samples` | 第二次业务异常，提示限制天数；`sample.restrict_days` 影响提示 | 覆盖 P0-8 |
| 豁免角色重复申请 | `channel_leader` / `admin` 重复 `POST /samples` 或复用限制判断 | 不触发重复限制；申请角色仍以渠道侧为主 | 覆盖 P0-8 |
| 招商组长不豁免 | `biz_leader` 复用同渠道 + 同达人 + 同商品重复限制判断 | 仍被 7 天限制拦截；同时保留本组审核能力 | 覆盖 P0-8/S-03 |
| 拒绝单不计限制 | 创建后 `REJECTED`，再同组合 `POST /samples` | 新申请不被上一条拒绝单拦截 | 覆盖 P0-8 |
| 招商审核通过 | `PUT /samples/{id}/status action=APPROVED` -> `GET /samples/{id}` -> `GET /samples/{id}/status-logs` | 状态 `PENDING_SHIP`；日志可查；渠道角色越权失败 | 覆盖 P0-2 |
| 招商拒绝 | `PUT /samples/{id}/status action=REJECTED reason=...` | `rejectReason` 回显；日志记录；无 reason 失败 | 覆盖 P0-2/P0-8 |
| 运营发货 | `PUT /samples/{id}/status action=SHIPPED trackingNo=...` 或 `POST /samples/batch-ship` | 状态 `SHIPPED`；`trackingNo` 回显；`logisticsSource=MANUAL` | 覆盖 P0-1/P0-6 |
| 运营签收到待交作业 | `PUT /samples/{id}/status action=SIGNED` | 状态 `PENDING_TASK`；日志 `SHIPPED -> PENDING_TASK` | 覆盖 P0-6 前半段 |
| 非法状态跳转 | 对 `PENDING_AUDIT` 直接发货，或人工 `COMPLETED` | 返回权限/业务错误，不更新状态 | 固化状态机 |
| 物流刷新底座 | 已发货且有 `shipperCode/trackingNo` 后 `POST /samples/{id}/logistics/refresh` | 未配置网关不推进；签收返回时推进 `PENDING_TASK` | 覆盖 S-09 底座 |
| 订单触发完成 | `POST /orders/sync` -> `GET /orders/{orderId}` -> `GET /samples/{id}` -> `GET /samples/{id}/status-logs` | `pay_time=null` 不完成；`pay_time!=null` 时 `sample.completedByOrderRule=true`、寄样 `FINISHED`、日志含订单号 | 覆盖 P0-6 |
| 寄样列表/导出 | `GET /samples?status=...&partnerName=...&receiverAddress=...`、`GET /samples/exports` | 可按合作方名称和收货地址筛选；分页/CSV 含合作方名称与收货地址 | 覆盖 S-07（业务 S-06） |

| 编号 | 待核对项 | 当前依据 | 建议处理 |
|---|---|---|---|

| G-03 | 招商组长是否应豁免 7 天限制 | 已定版：`biz_leader` 不豁免，仅 `admin` / `channel_leader` 豁免 | 已落地代码与测试 |
| G-04 | 已关闭状态人工入口 | `CLOSED` 分支存在但权限前置拦截；自动关闭底座存在 | 不作为 P0 主验收 |

| G-07 | S-10 Excel 批量导入物流单号 | 前端有本地解析，后端有 `batch-ship`，但范围文档明确不进 P0 | 不写入 P0 通过条件 |
