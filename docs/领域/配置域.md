# 配置域领域合同

> 模板来源：`.claude/templates/领域合同模板.md`。范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 领域定位

- [V1 必做] 配置域负责配置项、规则参数、配置变更日志和配置查询。
- [V1 必做] 配置域只提供配置事实，不执行具体业务规则。

## 职责边界

| 类型 | 内容 | 范围 |
| --- | --- | --- |
| 负责 | 配置读取、保存、校验、审计 | V1 必做 |
| 负责 | 佣金规则参数、业务开关、展示规则参数 | V1 必做 |
| 不负责 | 商品转链、订单归因、提成计算、寄样状态流转 | V1 不做 |
| 不负责 | 直接触发业务动作 | V1 不做 |

## 核心事实

- [V1 必做] 规则执行属于读取配置的业务域，例如业绩域负责提成和冲正计算。
- [V1 必做] 配置变更必须有操作者、变更前后值和时间。
- [V1 简化] V1 以当前配置 API、规则中心和审计日志支撑验收。
- [V2 预留] 完整规则引擎、灰度配置和多租户配置隔离后续设计。

## 对外能力

| 能力 | 使用方 | 证据 | 范围 |
| --- | --- | --- | --- |
| 配置读取 | 商品、订单、业绩、分析 | 配置 API | V1 必做 |
| 配置修改 | 管理端 | 配置变更日志 | V1 必做 |
| 规则参数查询 | 业绩域、商品域 | 规则中心 API | V1 必做 |

## 事件契约

| 事件 | 触发 | 消费方 | 范围 |
| --- | --- | --- | --- |
| 配置已变更 | 配置保存成功 | 缓存失效、审计、业务域后续读取 | V1 必做 |

## 数据契约

- [V1 必做] 配置项、配置值、配置版本和变更日志属于配置域事实。
- [V1 必做] 业务执行结果不写入配置域。

## 验收与证据

- [V1 必做] 修改配置后，配置 API 返回新值，审计日志可查。
- [V1 必做] 业务域使用配置后的效果需在对应领域验收，不在配置域单独证明业务正确。

## dataScope 在配置域（含规则中心 / 佣金规则）不应用（t6-commission / 2026-06-02）

[V1 必做] **规则中心与佣金规则查询不应用 `dataScope`**，理由（4 条 + 端到端断言 + V2 扩展路径）：

1. **配置域全局可见** — `配置域` 领域合同第 4 条 `配置域只提供配置事实,不执行具体业务规则`；过滤规则中心 / 佣金规则会改变业绩域的输入集，造成相同订单在不同 admin 下产生不同提成，违反"分析模块只读汇总表,业绩域不重算"不变量。
2. **admin dataScope=ALL 隐式覆盖** — 规则中心 7 个端点（`/rule-center` 全集）统一 `@RequireRoles({RoleCodes.ADMIN})`；唯一可访问角色 `admin` 的 `dataScope=3 (ALL)` 已经隐式等价于"全部配置"，无需在 service 显式消费。
3. **审计入口需要全量** — 变更历史（`/rule-center/change-logs`）与事件消费状态（`/rule-center/events`）是事后追溯入口，按调用者过滤会破坏可审计性。
4. **端到端断言** — `RuleCenterControllerTest.controller_shouldNotConsumeDataScopeAttribute` 显式断言 `dataScope=self` 时 `currentValues()` 仍返回全量（`backend/src/test/java/com/colonel/saas/controller/RuleCenterControllerTest.java:235-250`）。

[V1 必做] 如果 V2 引入"按部门维护规则"或"按招商组配置提成"，应单独建 `RuleCenterDepartmentScopedController`（新接口），而**不**给现有 `/rule-center/**` 注入 dataScope 过滤——保持配置域事实全局可见。

## 冲突与待确认

- [V1 必做] 如果旧设计要求配置中心直接执行规则，应拆回业务域并记录 ADR-002。
