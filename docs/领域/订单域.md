# 订单域领域合同

> 模板来源：`.claude/templates/领域合同模板.md`。范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 领域定位

- [V1 必做] 订单域负责订单事实、订单明细、退款 / 售后事实和归因输入。
- [V1 必做] 订单域只存事实，不算提成，不应用独家覆盖。

## 职责边界

| 类型 | 内容 | 范围 |
| --- | --- | --- |
| 负责 | 抖店订单同步、订单入库、退款事实、幂等 | V1 必做 |
| 负责 | 保存并解析 `pick_source`、链接 owner type 等归因事实 | V1 必做 |
| 负责 | 发布订单已同步、退款事实已同步事件 | V1 必做 |
| 不负责 | 最终业绩归属、提成、冲正、双轨金额 | V1 不做 |
| 不负责 | 独家覆盖规则执行 | V1 不做 |

## 核心事实

- [V1 必做] 订单域是订单事实来源，不是业绩结论来源。
- [V1 必做] 订单同步必须具备幂等键、时间窗口、同步日志和错误证据。
- [V1 必做] 退款事实必须独立保存，供业绩域处理冲正。
- [V1 必做] 双源订单同步口径（ORDER-1603-SETTLEMENT-AS-PRIMARY-001）：
  - `buyin.instituteOrderColonel / 1603`：主订单事实源与默认结算订单主接口；事实同步默认 `time_type=update`，结算同步默认 `time_type=update`，写入普通订单 `settle_amount` / `effective_*` / `settle_time` / `flow_point`。
  - `buyin.colonelMultiSettlementOrders / 2704`：仅作为 fallback / probe / 对照；不得作为默认写库主链路；空结果或空结算字段不得清空 1603 / 历史有效字段。
- [历史归档] 旧文档中的 Celery 定时任务不是当前运行事实。

## 数据契约

- [V1 必做] 订单号、商品、达人/渠道归因输入、金额、状态、支付时间、同步时间属于订单域事实。
- [V1 必做] 订单域不得保存最终提成结果作为订单事实。
- [V1 必做] 订单主表 `colonelsettlement_order` 当前以 `extra_data JSONB` 保存上游原始扩展载荷；旧材料提到的 raw payload 在本表落地口径中对应该字段，不另行声明不存在的 `raw_payload` 主表列。
- [V1 必做] `order_id` 是订单同步幂等主键输入；`order_sync_dedup_claim.order_id` 为 primary key，`order_row_id`、`first_seen_at`、`last_seen_at` 记录首次和最近同步声明，用于并发同步去重和历史回填。
- [V1 必做] 链接创建时在 `promotion_link` 与 `pick_source_mapping` 固化 `attribution_owner_type`。订单解析时：`RECRUITER` 链接只写招商用户，`CHANNEL` 链接只写渠道用户；活动招商只能作为无可用链接归因时的招商 fallback，商品负责人不得作为默认招商。
- [V1 必做] 缺失 `pick_source` 时，按 `pay_time > order_create_time > create_time` 筛有效期内的原生键映射；只有唯一 `(user_id, owner_type)` 可采用，不同 owner/type 必须保留 `ambiguous`，不得选择最新记录。订单需保存渠道/招商 user 与各自 source（`pick_source`、`native_unique_link_owner`、`activity_owner`、`ambiguous`、`unattributed`）作为下游事实。
- [V1 必做] 订单域不得据此计算提成、重算业绩归属、应用独家覆盖或执行业绩冲正。

## 验收与证据

- [V1 必做] real-pre 订单同步需提供真实接口响应、订单入库 SQL/API、同步日志。
- [V1 必做] 验证订单域未应用独家覆盖和提成计算，相关结果应出现在业绩域。

## 冲突与待确认

- [V1 必做] 如果旧逻辑在订单同步时直接写提成或最终归属，必须进入 ADR-003/ADR-004 决策处理。
