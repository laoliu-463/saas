# 用户域领域合同

> 模板来源：`.claude/templates/领域合同模板.md`。范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 状态边界

- [当前事实] 下文 V1 角色注册表、JWT `roleCodes`、全局 `data_scope` 和 `@DataScope` 切面是迁移基线，其中自定义角色不能自动获得业务接口权限。
- [V2 长期治理] [ADR-012](../决策/ADR-012-用户域内建设权限中心化授权能力.md) 已采纳用户域内 Authorization 子域能力；目标模型、Schema 和灰度门禁见 [DDD RBAC 长期治理设计](../superpowers/specs/2026-07-13-ddd-rbac-long-term-governance-design.md)。
- [V2 长期治理] 目标尚未实施完成前，现状与目标必须分别标记，不得把设计文档当作运行证据。

## Authorization Phase 1 已验证状态（2026-07-14）

- [已验证资产] 用户域内已具备 dormant Schema 资产、canonical `PermissionCode`、fail-closed 决策策略、`AuthorizationSnapshotStore` port、PostgreSQL adapter 和 `AuthorizationFacade`；Phase 1 聚焦测试及 PostgreSQL 16 Testcontainers 集成测试已通过。
- [数据库迁移 PENDING] 本地 `real-pre` 已只读确认 RBAC 4 表为 0、`sys_user.authz_version` 列为 0；本次工作流未执行远端 migration 或部署，远端当前 Schema 状态未核验。本阶段不得执行或宣称完成数据库迁移。
- [运行激活 PENDING] 本次源码证据确认 facade 无 request-path 消费者，尚未接入 JWT、Controller、Aspect 或业务请求链；已知 legacy 基线仍由 role / JWT / `data_scope` / `@DataScope` 承担，本条只描述已验证的本地与源码事实，不断言远端当前运行形态。
- [Phase 2 门禁] 后续必须单独审批数据库 migration window，并另写 shadow runtime 计划，先影子对比再决定是否进入 enforcement；本合同不代替该审批和计划。
- [业务待确认] 当前没有经业务审批的 role-permission seed matrix；不得将 dormant foundation 表述为 RBAC 已上线或已替代 legacy。

## 领域定位

- [V1 必做] 用户域负责用户、角色、组织、菜单和统一数据范围。
- [V1 必做] 用户域对其他业务域提供 `self/group/all` 数据范围，不替业务域判断订单归属、业绩归属或业务状态。

## 职责边界

| 类型 | 内容 | 范围 |
| --- | --- | --- |
| 负责 | 登录用户上下文、角色、组织、菜单、数据范围解析 | V1 必做 |
| 负责 | 管理员、主管、普通渠道等角色的权限基础 | V1 必做 |
| 不负责 | 商品、订单、寄样、业绩的业务状态机 | V1 不做 |
| 不负责 | 订单最终归属、提成、独家覆盖 | V1 不做 |
| 负责 | 权限目录、角色权限、领域数据范围、授权版本和通用授权决策 | V2 长期治理 |
| 不负责 | 业务对象归属、状态机、幂等和最终动作合法性 | V2 长期治理 |

## 核心事实

- [V1 必做] 数据范围统一抽象为 `self/group/all`。
- [V1 必做] 其他领域查询必须能追溯当前用户、角色、组织和数据范围来源。
- [V1 简化] V1 以当前权限注解、后端查询过滤和 E2E 证据为主。
- [V2 预留] 更复杂的 ABAC / PBAC、细粒度字段权限和组织树治理可后续扩展。

## 数据契约

- [V1 必做] 用户、角色、菜单、组织和用户组织关系属于用户域事实。
- [V1 必做] 业务数据只引用用户或组织标识，不复制权限判断逻辑。

## 数据范围切面

- [V1 必做] `@DataScope` 是用户域公开 API，目标包为 `com.colonel.saas.domain.user.api`。
- [V1 必做] `DataScopeAspect` 位于用户域 infrastructure，触发点为标注 `@DataScope` 的 Mapper 方法。
- [V1 必做] 切面只消费 `UserDomainFacade.resolveDataScope(currentUserId)` 返回的 `userIds`。
- [V1 必做] `self/group` 通过 `userField IN (userIds...)` 追加行级过滤；`all` 以空 `userIds` 表示 no-op。
- [V1 必做] 无 request context、无 `userId`、scope 为空或 `userIds` 为空时直接放行，不在切面层抛业务异常。
- [V1 必做] `JwtAuthenticationFilter` 负责在认证后写入 request attribute `userId`；切面不直接解析角色或部门规则。

## 角色编码注册表（V1）

| role_code | 含义 | 典型 data_scope | 说明 |
| --- | --- | --- | --- |
| `admin` | 超级管理员 | 3 (all) | 系统内置，不可通过管理端新建/改码 |
| `biz_leader` | 招商组长 | 2 (group) | 活动分配、本组审核与看板 |
| `biz_staff` | 招商专员 | 1 (self) | 日常招商操作 |
| `channel_leader` | 渠道组长 | 2 (group) | 本组渠道与寄样 |
| `channel_staff` | 渠道专员 | 1 (self) | 选品、寄样申请 |
| `ops_staff` | 运营 | 1 (self) | 发货与物流；登录后 data_scope 可能有业务特例 |

**编码规则**

- 持久化字段：`sys_role.role_code`（唯一）。
- 格式：小写字母开头，仅 `a-z`、`0-9`、`_`，最长 50 字符；管理端可不传编码时由服务端自动生成。
- 运行时：JWT `roleCodes[]` 与 `@RequireRoles` 比较前做 `trim` + 小写规范化。
- 历史 `colonel_leader` 已于 2026-05-30 合并至 `biz_leader`（迁移脚本 `backend/src/main/resources/db/alter-role-code-merge-colonel-leader.sql`）；未刷新 JWT 前端仍映射为 `biz_leader`。

**自定义角色**

- 管理端可新建非预置编码，但 **V1 接口鉴权以 Java `RoleCodes` + `@RequireRoles` 为准**，自定义角色不会自动获得业务写接口权限；主要用于菜单绑定（`sys_role_menu`）与 data_scope 配置。
- 禁止通过 API 新建与上述 6 码同名的角色；系统内置角色禁止修改 `role_code`、禁止删除。
- 角色管理前端 **不展示** `role_code` 字段；新建时由服务端按角色名称自动生成编码，鉴权与 JWT 仍使用 `role_code`。

**登录账号**

- `POST /api/auth/login` 的 `username` 字段表示 **登录账号**：先按 `sys_user.username` 精确匹配，未命中再按 `real_name` 精确匹配。
- 多个用户同名时拒绝登录并提示使用用户名；登录失败计数与锁定仍按解析后的 `username` 归一化 key 统计。
