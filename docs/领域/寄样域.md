# 寄样域领域合同

> 模板来源：`.claude/templates/领域合同模板.md`。范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 领域定位

- [V1 必做] 寄样域负责寄样申请、审批、发货、物流、收货和交作业状态。
- [V1 必做] 寄样域通过订单已同步事件判断交作业完成。

## 职责边界

| 类型 | 内容 | 范围 |
| --- | --- | --- |
| 负责 | 寄样申请、审批、发货、状态机、操作日志 | V1 必做 |
| 负责 | 基于订单已同步事实判断交作业完成 | V1 必做 |
| 不负责 | 订单事实同步 | V1 不做 |
| 不负责 | 业绩归属、提成、冲正 | V1 不做 |

## 核心事实

- [V1 必做] 交作业完成判断必须以订单域发布的订单已同步事实为证据。
- [V1 必做] 寄样状态变更必须有操作者、时间、前后状态。
- [V1 简化] V1 优先寄样主链路和 P0 状态，复杂物流自动化可简化。
- [V1 简化] 物流轮询以 Redis 31 分钟跨实例限频为准；`QUERY_THROTTLED` 只表示本轮跳过，不是物流事实，不得改变寄样单现有物流状态。
- [V2 预留] 更复杂的样品库存、自动催办、异常赔付后续设计。

## 对外能力

| 能力 | 使用方 | 证据 | 范围 |
| --- | --- | --- | --- |
| 寄样申请 | 渠道、招商 | 寄样 API、E2E | V1 必做 |
| 审批 / 发货 | 管理端、招商 | 状态日志、物流信息 | V1 必做 |
| 交作业判断 | 订单域、寄样域 | 订单已同步事件、寄样状态 | V1 必做 |
| 导出 / 看板 | 管理端、分析模块 | 导出文件、汇总 API | V1 简化 |

## V1 动作权限矩阵

| 动作 | 允许角色 | 额外约束 |
| --- | --- | --- |
| 申请、资格预检、删除待审核/已拒绝申请 | `biz_leader`、`biz_staff`、`channel_leader`、`channel_staff`、`admin` | 渠道角色需满足达人私海认领；招商角色不套用渠道私海约束 |
| 审核通过、驳回 | `biz_leader`、`biz_staff`、`admin` | 仍需满足状态机和数据范围 |
| 发货、物流推进、物流同步 | `ops_staff`、`admin` | 仍需满足状态机；纯运营视图只展示物流阶段 |
| 完成、关闭 | 系统自动推进 | 人工接口拒绝 |

- [V1 必做] 多角色账号按动作权限并集处理；只有内置角色集合中仅含 `ops_staff` 时才启用纯运营收缩视图。

## 事件契约

| 事件 | 触发 | 消费方 | 范围 |
| --- | --- | --- | --- |
| 寄样状态已变更 | 申请、审批、发货、作业状态变化 | 分析模块、审计 | V1 必做 |
| 订单已同步 | 订单域同步订单事实 | 寄样域判断交作业 | V1 必做 |

## 数据契约

- [V1 必做] 寄样申请、寄样状态、物流单号、达人地址和操作日志属于寄样域事实。
- [V1 必做] 寄样域保存订单关联结果，但订单原始事实仍属于订单域。
- [V1 必做] 从活动商品快照发起寄样时，寄样申请必须固化快照中的活动 ID 与抖音商品 ID；历史申请未固化时，详情读模型可按寄样单保存的快照关联 ID 只读回填，不反向修改历史数据。
- [V1 必做] 复制推广文案以寄样详情中的活动 ID、抖音商品 ID 和达人推广标识为前置事实；事实齐全后才允许调用商品推广网关，事实缺失时返回可追踪的业务错误。

## 验收与证据

- [V1 必做] 申请、审批、发货、订单回流后交作业完成需有 E2E/API/SQL 证据。
- [V1 必做] real-pre 场景必须证明订单来自真实同步或明确 BLOCKED。
- [V1 简化] 物流批次汇总须区分 success / failed / skipped，限频结果只能进入 skipped。

## 冲突与待确认

- [V1 必做] 若旧文档存在寄样域自行判断订单事实的口径，应改为消费订单已同步事件。
