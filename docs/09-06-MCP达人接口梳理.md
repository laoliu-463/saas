# 09-06-MCP达人接口梳理

更新时间：2026-05-18

> 本文档是 [09-MCP接口梳理-第一版](./09-MCP接口梳理-第一版.md) 的第六份拆分稿，专门收口 **达人新增/查询 -> 认领 -> 保护期 -> 地址带入寄样 -> 出单后保护期观察** 链路。  
> 本轮只读取 **Apifox MCP / business 离线 OAS / 前端调用 / 后端源码**，不改业务代码，不跑测试，不把 V2 后移项重新拉回 V1。

## 一、梳理结论

1. business OAS 已暴露达人主接口族：`/talents`、`/talents/{id}`、`/talents/{id}/claims`、`/talents/{id}/release`、`/talents/pools/public`、`/talents/pools/private`、`/talents/{id}/manual-fill`、`/talents/{id}/refresh`、`/talents/refresh/weekly`、`/talents/{id}/enrich-task/latest`、`/talents/{id}/blacklist`、`/talents/{id}/unblacklist`、`/talents/{id}/override-assignee`、`/talents/{id}/exclusive-status`。
2. 达人基础新增 / 查询 / 编辑主入口是 `POST /talents`、`GET /talents`、`GET /talents/{id}`、`PUT /talents/{id}`；入参和出参字段集中在 `douyinUid/douyinNo/uid/secUid/profileUrl/nickname/fansCount/contactPhone/contactWechat` 等基础资料。
3. 达人认领实际路径是 `POST /talents/{id}/claims`，不是单数 `/claim`；释放路径是 `POST /talents/{id}/release`。
4. 认领会消费 `talent.protection_days`：`TalentService.claim` 读取 `BusinessRuleConfigService.getTalentProtectionDays()`，并写入 `talent_claim.expire_time`，接口出参通过 `protectedUntil` 暴露。
5. 公海 / 私海不是单独字段落在 `talent` 表里，而是由有效认领记录动态计算：列表和详情会输出 `poolStatus`、`ownerId`、`ownerName`、`claimedAt`、`protectedUntil`、`activeClaimCount`、`activeClaimOwners`。
6. 数据范围对达人域有直接影响：`PERSONAL` 只查当前用户有效认领达人，`DEPT` 查本部门有效认领达人，`ALL` 可查全量；`TEAM_PUBLIC` 视图会放宽到公海查询。接口存在不等于 P0-9 已通过，仍需多账号实跑补证。
7. 保护期到期释放有定时底座：`TalentClaimReleaseJob` 每天触发 `releaseExpiredClaims`；若保护期内已有出单，当前逻辑会跳过过期释放。源码未确认“出单后立即重置保护期到新截止日”的独立接口或显式更新点，需通过订单同步结果 + 达人详情/认领记录 + DB/日志补证。
8. 达人结构化地址当前未在 `Talent`、`TalentDetailResponse`、`SampleApplyRequest` 中确认；达人详情只读到联系方式与备注，寄样申请收货信息当前由前端拼到 `remark`，后端写 `extraData.addressSource=manual`。因此只能说“手工地址底座存在”，不能写成“达人地址自动带入已闭环”。
9. 达人外部数据刷新接口存在：`POST /talents/{id}/refresh`、`POST /talents/refresh/weekly`、`GET /talents/{id}/enrich-task/latest`；它们只作为补全 / 排查辅助，不作为 V1 P0 阻断。
10. T-07 批量导入达人为 V1 简化；T-09 预设标签库不进 V1；T-10 Cookie 池 / 代理池不进 V1；达人爬虫增强只保留辅助口径，不写成 V1 主链验收入口。

## 三、达人基础信息链路

当前基础链路：

```text
POST /talents
GET /talents
GET /talents/{id}
PUT /talents/{id}
PUT /talents/{id}/manual-fill
```

关键写实：

- `POST /talents` 至少需要能解析出 `douyinUid`，可由 `profileUrl/douyinNo/uid/secUid` 兜底解析；重复 `douyinUid` 会报错。
- `PUT /talents/{id}` 只更新一组基础字段，当前确认到 `nickname/fans/level/status/contactPhone/contactWechat/intro`。
- `manual-fill` 是 V1 兜底关键入口，可补 `avatarUrl/fans/likesCount/worksCount/ipLocation/contactPhone/contactWechat/intro` 等信息。
- `Talent` 实体没有确认到结构化 `address` 字段；详情 DTO 只有 `contactPhone` 和 `remark`，因此达人地址不能写成已自动带入寄样。

## 四、达人认领 / 公海私海

### 1. 认领关系

当前认领表事实：

| 字段 | 当前含义 |
|---|---|
| `talent_claim.talent_id` | 达人主键 |
| `talent_claim.user_id` | 认领人 |
| `talent_claim.dept_id` | 认领人部门 |
| `talent_claim.apply_time` | 认领时间，对外为 `claimedAt` |
| `talent_claim.expire_time` | 保护期截止，对外为 `protectedUntil` |
| `talent_claim.status` | `1=ACTIVE`、`2=EXPIRED`、`3=RELEASED` |

认领时：

- 先加 Redis 锁 `talent:claim:lock:{talentId}`，防重复并发认领。
- 当前用户已有有效认领时会报“你已认领该达人，无需重复认领”。
- 多人可同时认领同一达人；列表会用 `activeClaimCount` 和 `ownerName` 展示“多人认领”摘要。

### 2. 公海 / 私海判断

| 场景 | 当前实现 | 结论 |
|---|---|---|
| 公海列表 | 过滤掉所有存在有效认领的达人，并排除黑名单 | 已确认主链 |
| 我的私海 | 当前用户有效认领记录对应的达人 | 已确认主链 |
| 本组私海 | 分页接口在 `DataScope.DEPT` 下按 `dept_id` 的有效认领过滤 | 需多账号补证 |
| 管理员全量 | `RoleGuardAspect` 对 `admin` 放行；数据范围仍需结合登录返回验证 | 需多账号补证 |
| 公海全员可见 | `TEAM_PUBLIC` 视图会把基础查询放宽到 `ALL` 后再过滤公海 | 需实跑补证 |

P0-9 结论：

- 接口和源码具备数据范围底座。
- 但渠道只能看自己私海、渠道组长看本组、管理员看全部、公海是否全员可见，必须通过多账号 Apifox / 浏览器样本补证。

## 五、达人保护期

### 1. 配置消费点

保护期消费路径：

```text
POST /talents/{id}/claims
-> TalentService.claim
-> BusinessRuleConfigService.getTalentProtectionDays()
-> system_config.talent.protection_days，默认 30
-> talent_claim.expire_time = now + protection_days
-> GET /talents / GET /talents/{id} 返回 protectedUntil
```

归属覆盖同样会读取保护期配置：

```text
POST /talents/{id}/override-assignee
-> 旧有效认领置 EXPIRED
-> 新负责人 claim.expire_time = now + protection_days
```

### 2. 到期释放与出单影响

当前定时底座：

```text
TalentClaimReleaseJob.releaseExpiredClaimsDaily
-> TalentService.releaseExpiredClaims(now)
-> 找 status=ACTIVE 且 expire_time < now 的认领
-> 若 hasOutputSinceClaim(talent, claim) 为 true，则跳过释放
-> 否则 status=EXPIRED
```

当前可确认：

- 有 `talent.protection_days=30` 默认配置与业务消费点。
- 到期释放会检查保护期内是否有出单；有出单则不会释放该 claim。
- 源码未确认“订单同步后立即把 `protectedUntil` 重置为 now + protection_days”的独立接口或显式更新点。

因此当前文档口径：

> 出单重置保护期没有独立接口。本轮不能写成缺功能，也不能写成已验收通过；应通过订单同步结果 + 达人详情/认领记录 + `talent_claim` DB/日志补证来确认最终业务口径。

## 六、达人地址与寄样衔接

当前事实：

| 连接点 | 当前依据 | 结论 |
|---|---|---|
| 达人联系方式 | `Talent.contactPhone/contactWechat`，详情输出 `contactPhone` | 可读 / 可手动维护 |
| 达人结构化地址 | `Talent` / `TalentDetailResponse` 未确认 `address/receiver*` 字段 | 待核对 |
| 寄样申请地址 | `SampleApplyRequest` 无结构化地址字段；前端表单收 `receiverName/receiverPhone/receiverAddress` 后拼入 `remark` | 手动地址底座 |
| 后端地址来源 | `buildSampleExtraData` 写 `addressSource=manual` | 当前不是达人地址自动带入 |
| 达人候选 | `GET /samples/talent-candidates` 输出 `talentId/nickname/fansCount/creditScore/mainCategory/region` | 不含地址 |

结论：

- T-08 / S-07 当前只能写成“接口层有联系方式和手工地址兜底”。
- “选择达人 -> 地址自动带入 -> 提交寄样”还不能写成闭环；需要产品确认是否 V1 必须结构化维护达人收货地址。
- Apifox 场景可先断言：寄样 `remark` 中包含收货信息、`extraData.addressSource=manual`，但不能把它替代达人地址闭环。

## 九、保护期与地址带入影响分析

### 1. 保护期

P0-3 可写成：

- `talent.protection_days` 已有配置键。
- `POST /talents/{id}/claims` 已确认消费该配置，并写 `protectedUntil`。
- `GET /talents` / `GET /talents/{id}` 可读 `protectedUntil`。

但不能写成：

- 改配置后保护期规则已通过。
- 出单后保护期重置已通过。
- 数据范围保护期已通过。

原因：

- 仍缺改前 / 改后专项样本。
- 源码只确认“出单后到期释放时不释放”，未确认“订单同步即刻延长 expire_time”。
- 渠道、组长、管理员视角仍需多账号补 P0-9。

### 2. 地址带入

P0-1 可写成：

- 寄样申请页面可手动填写收货人、手机号、地址。
- 当前后端通过 `remark` 和 `extraData.addressSource=manual` 记录手工地址来源。

不能写成：

- 达人地址已维护。
- 达人地址已自动带入寄样。
- T-08 / S-07 已闭环。

原因：

- 达人实体和详情 DTO 未确认结构化地址字段。
- `GET /samples/talent-candidates` 不返回地址。
- `SampleApplyRequest` 不接收结构化地址字段。

## 十、出单重置保护期取证路径

如果业务验收需要证明“出单后保护期变化”，建议按以下路径取证：

1. `GET /configs` 或 `/configs/grouped`：记录 `talent.protection_days` 配置值。
2. `POST /talents/{id}/claims`：认领达人。
3. `GET /talents/{id}`：记录 `claim.protectedUntil`、`claim.activeClaimOwners[]`。
4. 制造或同步一条命中该达人 `douyinUid/talent_uid/author_id` 的订单。
5. `POST /orders/sync`：订单入库 / 更新。
6. `GET /orders/{orderId}`：确认订单 `talentId/talentName/channelUserId` 等归因结果。
7. `GET /talents/{id}`：再次记录 `protectedUntil/orderCount/naturalOrderTalent/orders[]`。
8. DB/日志补证：查询 `talent_claim.apply_time/expire_time/status`，确认是否延长、保持或仅跳过到期释放。

当前结论：

- 无独立“重置保护期”业务接口。
- 若验收口径是“有出单则保护不释放”，当前已有源码底座；若验收口径是“出单后立即重置到 now + 30 天”，本轮未确认显式实现，需要专项补证或决策。

## 十二、接口缺口与待核对项

| 编号 | 待核对项 | 当前依据 | 建议处理 |
|---|---|---|---|
| T-GAP-01 | 达人结构化收货地址字段 | `Talent` / `TalentDetailResponse` / `SampleApplyRequest` 均未确认 `address` 字段 | 不写成自动带入；如 V1 必须则另定方案 |
| T-GAP-02 | 出单后是否立即重置保护期 | 未见独立接口或显式更新 `talent_claim.expire_time`；只确认到期释放时有出单则不释放 | 通过订单同步 + 达人详情 + DB/日志补证 |

| T-GAP-05 | T-05 标签最多 3 个 | 本轮 OAS 未确认独立标签接口或后端上限规则 | 保留 `TASK-MUST-05` |

| T-GAP-08 | 独家达人接口 | `/talents/{id}/exclusive-status` 存在 | 标记 V1 不采用，避免回流 |
