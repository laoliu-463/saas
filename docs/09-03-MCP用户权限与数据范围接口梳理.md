# 09-03-MCP用户权限与数据范围接口梳理

更新时间：2026-05-18

> 本文档是 [09-MCP接口梳理-第一版](./09-MCP接口梳理-第一版.md) 的第三份拆分稿，专门收口 **登录 -> 菜单权限 -> 用户角色 -> 数据范围 -> U-09 一人多角色** 链路。  
> 本轮只读取 **Apifox MCP / business 离线 OAS / 前端调用 / 后端代码**，不改业务代码，不跑测试，不把 V2 后移项重新拉回 V1。

## 一、梳理范围

本轮只拆以下 5 组：

1. 登录与身份上下文
2. 菜单与权限接口
3. 用户管理接口
4. 角色管理接口
5. 一人多角色 U-09 与数据范围字段

统一边界：

- **主事实源**：`SAAS-business-offline` -> `out/openapi-business.json`
- **辅助事实源**：前端 `src/api/auth.ts`、`src/api/sys.ts`、`stores/auth.ts`、`router`、`layout/Sider.vue`、`system/UserList.vue`、`system/RoleList.vue`；后端 auth / user / role / menu / security / aspect 代码
- **不纳入主链**：
  - `/douyin/**` 和 `sdk-debug`
  - 新增 `/me` 一类当前不存在的接口需求
  - 把“接口存在”直接写成“验收通过”

## 二、当前结论

1. 当前 business OAS **没有** 独立 `/me` 接口；当前用户上下文按 `POST /auth/login` 返回值 + `GET /menus/tree` 后端菜单树核对。
2. `POST /auth/login` 返回 `token / refreshToken / userId / deptId / dataScope / roleCodes / username / realName`，JWT 内也写入 `deptId / dataScope / roleCodes / username`。
3. 后端用户创建与分配角色均使用 `roleIds[]`，服务层以 `sys_user_role` 多行关系保存，接口结构上支持一个用户多个角色。
4. 后端菜单树按用户所有角色取菜单 ID 并集；后端 `@RequireRoles` 拦截和前端 `hasAccess` 都按“任一角色命中即可访问”判断。
5. 后端登录时普通多角色 `dataScope` 取角色配置最大值，只有 `admin` 会被强制提升到 `3(ALL)`；`ops_staff` 保持角色配置数据范围。
6. 当前前端用户管理表单已使用多角色选择：编辑时保留完整 `roleIds[]`，提交 `assignUserRoles` 时带完整角色数组；它仍需要 U-09 专项样本作为验收证据。
7. 当前前端角色管理表单只展示 `1(仅本人)` 与 `3(全部数据)` 两个选项；business OAS / 后端支持 `2(本组)`，但前端自定义角色配置 `group` 入口仍待核对。
8. 当前前端侧边栏没有直接调用 `/menus/tree`，而是基于登录响应里的 `roleCodes` 和本地菜单配置渲染；`/menus/tree` 可作为 Apifox / 后端权限树核对接口。
9. U-09 当前只能写成：**API、后端规则与前端员工表单支持多角色、权限并集和数据范围最宽；real-pre 专项样本仍待补证**。

## 三、状态标记

| 状态 | 含义 |
|---|---|
| `已确认主链` | 已在当前 business OAS、controller / service 与前端调用或当前上下文链路中形成主口径 |
| `待核对` | 当前有实现迹象，但还不足以直接写成主链已闭环 |
| `辅助接口` | 对排查、配置或补证有用，但不是登录 / 权限 / 数据范围主链的直接入口 |
| `V1 不采用` | 当前 V1 不以该接口 / 能力作为主链验收口径 |

## 六、U-09 一人多角色核对结论

| 检查点 | 当前结论 | 证据路径 | 是否可写成验收通过 |
|---|---|---|---|
| 是否支持一个用户多个角色 | API / 后端 / 前端表单支持 | `SysUserCreateRequest.roleIds[]`、`SysUserAssignRolesRequest.roleIds[]`、`SysUserService.replaceUserRoles` 多行写 `sys_user_role`；`UserList.vue` 多选提交完整 `roleIds[]` | 否，real-pre 专项样本未补 |
| 权限是否取并集 | 后端与前端判断支持并集 | `SysMenuService.findUserTree` 对多角色菜单 ID 做并集；`RoleGuardAspect` 任一角色命中即放行；前端 `hasAccess` 使用 `some` | 否，仍需 Apifox / real-pre 组合样本 |
| 数据范围是否取最宽 | 普通角色代码支持取最大 `dataScope`；仅 `admin` 额外强制 `all` | `AuthService.login/refreshToken` 对角色 `dataScope` 取 `max`，并对 `admin` 置 3 | 否，前端 group 配置入口与业务数据范围专项仍待确认 |
| 当前用户上下文是否清楚 | 清楚，但没有 `/me` | `POST /auth/login` 返回值 + JWT claims + `GET /menus/tree` | 否，接口事实清楚不等于 P0-9 通过 |
| 前端是否完整支持 U-09 配置 | 员工表单已支持多角色；角色 group 配置入口仍缺 | `UserList.vue` 多选角色并提交完整 `roleIds[]`；表格展示多个角色标签 | 否，仍缺专项样本与 group 配置入口 |

当前建议口径：

1. **接口 / 后端事实**：支持 U-09 的核心规则。
2. **前端配置事实**：用户管理表单已按 U-09 开放多选；角色管理表单仍需确认 `group` 配置入口。
3. **验收事实**：P0-9 不能因接口存在而写通过，仍需补“一人多角色账号 -> 登录上下文 -> 菜单并集 -> 业务接口越权 / 数据范围”专项样本。

## 七、Apifox 测试场景建议

| 场景 | 接口序列 | 重点断言 | 目的 |
|---|---|---|---|
| 登录上下文 | `POST /auth/login` | `token` 非空；`userId/deptId/dataScope/roleCodes[]` 存在；无 `/me` 依赖 | 固化当前用户上下文 |
| 当前菜单树 | `POST /auth/login` -> `GET /menus/tree` | 菜单树只含该角色可见菜单；多角色时取并集 | 验证菜单权限 |
| 用户多角色写入 | `GET /roles/enabled` -> `POST /users` 或 `PUT /users/{id}/roles` -> `GET /users/{id}` | `roleIds[]` 多个 ID 能回读 | 验证 API 层 U-09 |
| 多角色登录回读 | 多角色用户 `POST /auth/login` | `roleCodes[]` 包含多个编码；`dataScope` 等于最宽值 | 验证身份上下文 |
| 接口权限并集 | 多角色用户分别访问两个单角色入口 | 命中任一允许角色即 200；无角色仍 403 | 验证后端 `@RequireRoles` |
| 数据范围最宽 | 准备 `self + group` 或 `self + all` 角色组合，查询 `/orders`、`/dashboard/summary` | 返回数据范围随 `dataScope` 最宽值变化；不把数据量当唯一证据，需配固定样本 | 验证 P0-9 |
| 运营 configured 回归 | `ops_staff` 登录 + 查业务接口 | 登录返回保持角色配置；寄样发货台可处理全量，订单 / Dashboard / 达人默认不可全量 | 验证运营只在寄样发货场景 all |
| 角色 group 配置入口 | `POST /roles` / `PUT /roles/{id}` 传 `dataScope=2`，再前端回看 | API 可写 2；前端是否可选择 / 展示 2 | 验证 U-03 / U-04 前端差异 |
