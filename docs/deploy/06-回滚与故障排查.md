# real-pre 回滚与故障排查

## 日常回滚入口

日常回滚仍然走 `main -> release/real-pre -> Jenkins`。回滚候选必须引用上一份已验证的 `repository@sha256:digest`，不能通过服务器 Git checkout 或重新构建镜像完成。

Jenkins 在部署前记录：

- 当前运行的 source SHA。
- 当前后端、前端镜像完整引用和本地 image ID。
- `current.json` / `previous.json`。
- 数据库备份和迁移判定。

readiness、frontend、smoke、E2E、调度恢复或最终健康检查失败时，Jenkins 在同一 `saas-real-pre-deploy` 锁内调用 `scripts/cd/rollback-real-pre.sh`，切回部署前已验证且仍在本机的 `repository@sha256:digest`，重新启用调度并执行健康检查；回滚结果写入 `deployment-started`、`rollback-completed` 等状态文件和 evidence。失败 hook 不依赖已结束的 GHCR 登录凭证，也不会在锁释放后用新镜像做补救。

## 证据优先排查

1. 查看 Jenkins artifact：`harness/reports/current/latest-jenkins-cd.md`。
2. 查看服务器归档：`/opt/saas/runtime/qa/out/jenkins-<build>/`。
3. 对照 `sourceMainSha`、OCI revision、后端 `/api/system/health` 的 `gitSha/imageDigest` 和前端 `/version.json`。
4. 查看 Docker Compose 状态和 backend/frontend 日志。
5. 如果涉及数据库，先确认备份目录和 Flyway history，再判断是否需要数据库恢复；应用回滚不等于数据库回滚。

常用只读检查：

```bash
docker compose --env-file /opt/saas/env/.env.real-pre \
  --project-name saas-active -f docker-compose.real-pre.yml ps
docker compose --env-file /opt/saas/env/.env.real-pre \
  --project-name saas-active -f docker-compose.real-pre.yml logs --tail=300 backend-real-pre
curl -fsS http://127.0.0.1:8081/api/system/health
curl -fsS http://127.0.0.1:3001/healthz
```

## Break-glass

SSH、服务器 `git pull`、现场构建和手工 Compose 只允许经过明确审批的紧急恢复。执行时必须获得与 Jenkins 相同的主机锁，先备份、再使用固定镜像引用、执行健康检查并补齐私有证据。它不是日常发布路径，不能替代后续正常 PR 收敛。

禁止 `docker compose down -v`，禁止删除 PostgreSQL / Redis volume，禁止切换到 test/mock 配置后宣称 real-pre 通过。
