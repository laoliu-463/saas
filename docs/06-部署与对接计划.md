# 06-部署与对接计划

更新时间：2026-05-12

## 一、当前目标

当前部署与对接工作的重点不是直接上生产，而是：

1. 保证本地 Mock 基线可稳定启动
2. 保证演示链路可复现
3. 为真实 SDK 联调保留清晰切换路径

当前阶段补充：

- `M1.7 部署验证` 已于 2026-05-12 完成首轮收口
- 本轮收口依据：
  1. `real-pre` 可按 compose 重新构建并恢复健康
  2. `3001 / 8081 / 5433 / 6380` 独立拓扑可稳定提供服务
  3. 重建后浏览器 smoke 与 real-pre 抖店联调 E2E 继续通过

## 二、当前环境划分

### 0. 当前标准拓扑

从 2026-05-03 起，本机环境启动统一按以下拓扑执行：

| 用途 | 前端 | 后端 | PostgreSQL | Redis | 来源 |
| --- | --- | --- | --- | --- | --- |
| `test` 人工联调 / 自动化基线 | `3000` | `8080` | `5432` | `6379` | `docker-compose.test.yml` + 本机单前端 |
| `real-pre` 真实上游联调 / 浏览器回归 | `3001` | `8081` | `5433` | `6380` | `docker-compose.real-pre.yml` |

约束：

- `3000` 只允许一个本机前端入口
- `3001` 不允许再起本机第二个 Vite；它应专属于 `real-pre`
- `8080` 只作为 `test` backend 暴露端口
- `8081` 只作为 `real-pre` backend 暴露端口
- 非明确需要时，不单独起本机 `redis-server` 占用 `6379`
- 每次启动完成后，应执行：`powershell -ExecutionPolicy Bypass -File .\scripts\check-env-topology.ps1`

| `saas-postgres-real-pre-1` | `real-pre` | `real-pre` PostgreSQL | `5433` |
| `saas-redis-real-pre-1` | `real-pre` | `real-pre` Redis | `6380` |
| `saas-backend-real-pre-1` | `real-pre` | `real-pre` 后端 | `8081` / `5006` |

说明：

- `test` 负责 `3000 / 8080 / 5432 / 6379`
- `real-pre` 负责 `3001 / 8081 / 5433 / 6380`

#### B. 按需拉起的前端容器

| 容器名 | 环境 | 是否默认必需 | 说明 |
| --- | --- | --- | --- |
| `saas-test-frontend-1` | `test` | 否 | 当前更推荐由本机 Vite 占用 `3000`，不要求长期保留该容器 |
| `saas-frontend-real-pre-1` | `real-pre` | 视回归需求而定 | 若要严格走 `real-pre` 全容器拓扑，则应拉起并占用 `3001` |

执行口径：

- `3000` 默认优先使用本机前端，不强制要求 `saas-test-frontend-1` 常驻
- `3001` 若执行 `real-pre` 浏览器回归，优先使用 `saas-frontend-real-pre-1`

#### C. 非当前标准口径的残留容器

若出现以下容器：

- `saas-backend-1`
- `saas-frontend-1`
- `saas-postgres-1`
- `saas-redis-1`

则默认视为旧启动残留或非当前标准拓扑产物。

处理原则：

- 不将其作为当前 `test` / `real-pre` 的正式环境来源
- 排查端口与容器来源时，优先核对 `saas-test-*` 与 `saas-*-real-pre-*`
- 在准备正式回归、联调或浏览器 smoke 前，应先确认残留容器不会混淆 `8080 / 8081 / 3000 / 3001`

### 0.2 环境口径收口与旧容器处理（2026-05-10）

当前不执行“容器合并”，也不直接删除旧 `test` 数据。容器只是临时运行物，真正要统一的是配置文件、启动命令、profile 命名、数据库名称、端口分配、Mock / Real 开关和文档记录。

当前保留两条环境线：

| 环境 | 职责 | Mock / Real | 数据隔离 |
| --- | --- | --- | --- |
| `test` | 本地业务联调、Mock 数据、回归测试、演示闭环 | `APP_TEST_ENABLED=true`，`DOUYIN_TEST_ENABLED=true` | `colonel_saas_test` / Redis DB 1 |
| `real-pre` | 三方 API 小范围验证、Token、活动、订单、转链冒烟 | `APP_TEST_ENABLED=false`，`DOUYIN_TEST_ENABLED=false` | `colonel_saas_real` / Redis DB 0 |

处理原则：

- `local-mock` 不再作为独立容器环境扩展；历史脚本、历史报告和 `application-local-mock.yml` 可继续作为本地 Mock 口径参考，但当前 Mock 联调环境收敛到 `test`
- 旧 `test` 容器如仍承载 Mock 回归数据，应先视为可回滚基线；需要释放端口时只执行 `docker compose -p saas-test -f docker-compose.test.yml down`
- 不要在收口阶段执行 `docker compose down -v` 或删除 `saas-test_*` volume
- 若发现 `saas-*` 非 real-pre 残留容器，先确认端口、project label 和 volume 名称，再停止容器；volume 是否删除单独决策

旧 `test` volume 只有在以下 5 项全部满足后，才进入清理评估：

1. 新 `test` 环境可以一键启动
2. `http://localhost:8080/api/actuator/health` 返回 `UP`
3. 登录、商品库、活动、订单、寄样、达人、Dashboard 主链路都能跑通
4. `backend mvn test` 通过
5. `frontend npm run build` 通过

### 1.1 Docker Desktop / Engine 排查口径（2026-05-04）

2026-05-04 上午，本机出现过一次“Docker Desktop 看起来已启动，但 `docker` CLI 无法连接 engine”的状态。本次排查得到的事实如下：

- `Docker Desktop Service` 状态为 `Running`
- `Docker Desktop.exe` 进程存在
- `wsl -l -v` 中 `docker-desktop` 为 `Running`
- 当前 `docker context` 指向 `desktop-linux`
- 但 `docker ps` / `docker version` / `docker info` 均报错：
  - `failed to connect to the docker API at npipe:////./pipe/dockerDesktopLinuxEngine`
  - `open //./pipe/dockerDesktopLinuxEngine: The system cannot find the file specified.`
- 当时直接检查命名管道：
  - `Test-Path \\\\.\\pipe\\dockerDesktopLinuxEngine` = `False`
  - `Test-Path \\\\.\\pipe\\docker_engine` = `False`
- Docker Desktop 日志尾部可见：
  - `C:\\Program Files\\Docker\\Docker\\resources\\com.docker.backend.exe services: exit status 1`

结论：

> 若出现“Docker Desktop UI / 服务 / WSL 都在，但 `docker ps` 仍提示找不到 `dockerDesktopLinuxEngine` 管道”，应优先按 **Docker Desktop backend / engine 异常** 处理，而不是直接判为项目容器或 compose 配置问题。

本机排查日志路径：

- `%LOCALAPPDATA%\\Docker\\log\\host\\com.docker.backend.exe.log`
- `%LOCALAPPDATA%\\Docker\\log\\host\\monitor.log`

建议恢复动作：

1. 先执行 `docker desktop status`
2. 若返回 `Could not retrieve status`，优先重启 Docker Desktop
3. 重启后再次确认：
   - `docker ps -a`
   - `docker context ls`
   - `Test-Path \\\\.\\pipe\\dockerDesktopLinuxEngine`
4. 只有在 CLI 已恢复连通后，再执行项目 compose 启停、容器排查和浏览器 smoke

补充约束：

- 在 `docker ps` 无法返回前，不要把 `8080 / 5432 / 6379` 的异常直接归因为本项目后端、数据库脚本或前端代理问题
- 依赖 compose 容器的 E2E / smoke 必须先确认 Docker engine 可连接
- 若 Docker Desktop backend 再次异常退出，先保留上述日志现场，再做重启或重新拉起

建议顺序：

## 七、切换到真实 SDK 环境前必须确认的事项

- [ ] Test 基线完整可回归
- [ ] Real 配置不覆盖 Test 配置
- [ ] Gateway 切换边界清楚
- [ ] token / secret 不写死在仓库中
- [ ] 回调地址和网络策略明确

## 八、当前不要做的事

- 直接删除 Test 数据链路
- 用 Real 联调临时改前端字段
- 为了跑通真实接口破坏现有 Service
- 把演示环境与真实联调环境混在一起
- 让 `3001` 同时存在 Docker 前端和本机 Vite 两个来源
- 让 `8080` 混成“手工 local-mock backend”和 `test backend` 双来源
- 在没有明确理由时同时保留 Docker Redis 和本机 Redis 争用 `6379`

## 九、`local-mock` 下线判定清单

当前结论不是“马上删除 `local-mock`”，而是：

> 先让 `test` 具备完整人工联调替代能力，再决定是否下线 `local-mock`。

只有同时满足以下条件，才建议把 `local-mock` 收敛到 `test`：

### 1. 人工联调可替代

- [ ] `test` 环境可稳定承接登录、商品、订单、寄样、达人、系统管理等人工走查
- [ ] `test` 环境下 `/dev/test` 或等价调试能力仍可用，且不会污染自动化测试基线
- [ ] 商品详情、寄样申请、角色 CRUD、用户 CRUD 等细交互已在 `test` 实测通过

### 2. 自动化与人工使用不互相污染

- [ ] 人工 `reset / seed / 造数` 不会影响 CI 或日常自动化回归
- [ ] 自动化执行窗口与人工联调窗口已隔离，至少在数据或实例层面有明确边界
- [ ] `test` 环境中的测试数据命名、清理、回放规则已经标准化

### 3. 启动与脚本入口完成替换

- [ ] 所有本地 SOP 已从 `local-mock` 切换到 `test`
- [ ] `runtime/qa/local-mock-supplement.cjs` 已有 `test` 等价脚本或已完成合并
- [ ] `scripts/run-local-mock-supplement.ps1` 的职责已迁移，且相关一键命令仍可直接使用

### 4. 文档与认知完成收口

- [ ] `README.md`、`docs/README.md`、`docs/04`、`docs/06` 不再把 `local-mock` 作为默认入口
- [ ] 归档文档中的 `local-mock` 历史表述已降级为“历史说明”，不再影响当前执行口径
- [ ] 团队已明确接受“`test` 同时承担自动化与人工联调”或已经拆出新的人工联调口径

### 5. 回退路径清楚

- [ ] 已保留一版可回退的 `local-mock` 配置、脚本和文档快照
- [ ] 若 `test` 承接人工联调后出现不稳定，可在半天内恢复到旧口径

在上述条件没有同时满足前，当前推荐策略调整为：

- `local-mock`：历史人工联调口径、脚本和报告名，保留为回滚参考
- `test`：Mock 联调 / 自动化测试 / 隔离测试栈 / 演示闭环
- `real-pre`：浏览器回归 / 权限验收 / 独立拓扑验证

## 十、当前结论

> 当前最稳的推进方式是：冻结旧 `test` 作为回滚基线，把 Mock 联调口径收敛到 `test`，保持 `real-pre` 回归可用，再逐个 Gateway 推进真实 SDK 联调。
