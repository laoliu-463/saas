# 06-部署与对接计划

更新时间：2026-04-25

## 阶段 1：Mock-first 开发阶段

目标：

- 内部业务闭环跑通
- 第三方能力统一通过 Mock Gateway 提供

要求：

- 前端只调用内部 API
- 关键业务动作真实落库
- 操作日志可回看

## 阶段 2：真实 SDK 测试接入

目标：

- 逐步用 Real Gateway 替换部分 Mock Gateway

优先顺序：

1. `DouyinAuthGateway`
2. `DouyinColonelActivityGateway`
3. `DouyinProductGateway`
4. `DouyinPromotionGateway`
5. `DouyinOrderGateway`

检查项：

- token 是否可用
- 团长权限是否正确
- 活动是否能查到
- 商品是否能同步
- 转链是否能成功
- 订单是否能回流

## 阶段 3：真实环境联调

目标：

- 用真实团长授权跑通主链路

联调重点：

- 权限异常处理
- token 过期处理
- 参数错误日志
- 限流与失败场景追踪

## 环境切换口径

### 测试环境

- `SPRING_PROFILES_ACTIVE=test`
- `DOUYIN_MOCK_ENABLED=true`
- `TALENT_PUBLIC_PAGE_CRAWL_ENABLED=false`
- 使用独立测试库和独立 Redis DB

### 生产环境

- `SPRING_PROFILES_ACTIVE=prod`
- `DOUYIN_MOCK_ENABLED=false`
- `TALENT_PUBLIC_PAGE_CRAWL_ENABLED=false`
- 使用真实授权、真实数据库和正式密钥

## 数据库初始化说明

当前仓库未接入 Flyway / Liquibase。

现状：

- `docker-compose.yml` 会在新建数据库卷时自动执行初始化脚本
- 商品主链路和订单归因相关脚本已挂载到 PostgreSQL 初始化目录
- 新增测试库时，可自动完成大部分基础结构初始化

限制：

- PostgreSQL 官方镜像只会在数据目录为空时执行初始化脚本
- 复用旧数据库卷时，新增脚本不会自动补执行

建议：

- 新环境联调优先使用全新数据库卷启动
- 若复用旧卷，需要手工执行新增 SQL，确保字段和表结构补齐

## 订单回流 MVP 补充说明

2026-04-25 测试环境联调中，已经出现过“代码已升级但旧测试库未补字段”的情况。

已实际遇到的缺失字段：

- `pick_source_mapping.promotion_link_id`
- `colonelsettlement_order.talent_name`

因此当前规则为：

- 新建测试库时，必须使用最新初始化脚本
- 复用旧测试库时，必须补执行订单归因相关增量脚本
- 如果前端订单页打开 `GET /orders` 或 `GET /orders/unattributed` 报 500，优先检查测试库字段是否与实体一致

## Compose 环境命名约定

### 测试 / Mock 环境

- 命令：`docker compose -f docker-compose.yml -f docker-compose.test.yml ...`
- Compose 项目名：`saas-test`

### 生产覆盖环境

- 命令：`docker compose -f docker-compose.yml -f docker-compose.prod.yml ...`
- Compose 项目名：`saas-prod`

说明：

- 基础 `docker-compose.yml` 已移除通用 `container_name`
- 容器、网络、卷会自动带项目名前缀，例如 `saas-test-*`、`saas-prod-*`
- 可直接通过 Docker 资源名称判断当前运行口径
