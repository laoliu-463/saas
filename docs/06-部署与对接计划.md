## 一、当前目标

当前部署与对接工作的重点不是直接上生产，而是：

1. 保证 `test` 默认功能与权限测试基线可稳定启动
2. 保证 `real-pre` 真实 SDK 联调可清晰切换
3. 避免 test 与 real-pre 容器混用导致排查结果不可信

当前阶段补充：

- `M1.7 部署验证` 已于 2026-05-12 完成首轮收口
- 2026-05-15 环境治理后，本机运行口径从“双环境并行”改为“单活环境切换”
- 本轮收口依据：
  1. `test` / `real-pre` 共用 `docker-compose.yml`
  2. 两个启动入口都使用 project name `saas-active`
  3. 启动任一环境前先停止所有 SAAS 容器
  4. 后端启动日志打印 active profile、Mock 开关和数据库名
  5. 前端登录后显示当前环境标识

### 1.1 Docker Desktop / Engine 排查口径（2026-05-04）

2026-05-04 上午，本机出现过一次“Docker Desktop 看起来已启动，但 `docker` CLI 无法连接 engine”的状态。本次排查得到的事实如下：

- `Docker Desktop Service` 状态为 `Running`
- `Docker Desktop.exe` 进程存在
- `wsl -l -v` 中 `docker-desktop` 为 `Running`
- 当前 `docker context` 指向 `desktop-linux`
- 但 `docker ps` / `docker version` / `docker info` 均报错：
  - `failed to connect to the docker API at npipe:////./pipe/dockerDesktopLinuxEngine`
  - `open //./pipe/dockerDesktopLinuxEngine: The system cannot find the file specified.`
- 当时直接检查命名管道：
  - `Test-Path \\\\.\\pipe\\dockerDesktopLinuxEngine` = `False`
  - `Test-Path \\\\.\\pipe\\docker_engine` = `False`
- Docker Desktop 日志尾部可见：
  - `C:\\Program Files\\Docker\\Docker\\resources\\com.docker.backend.exe services: exit status 1`

结论：

> 若出现“Docker Desktop UI / 服务 / WSL 都在，但 `docker ps` 仍提示找不到 `dockerDesktopLinuxEngine` 管道”，应优先按 **Docker Desktop backend / engine 异常** 处理，而不是直接判为项目容器或 compose 配置问题。

本机排查日志路径：

- `%LOCALAPPDATA%\\Docker\\log\\host\\com.docker.backend.exe.log`
- `%LOCALAPPDATA%\\Docker\\log\\host\\monitor.log`

建议恢复动作：

1. 先执行 `docker desktop status`
2. 若返回 `Could not retrieve status`，优先重启 Docker Desktop
3. 重启后再次确认：
   - `docker ps -a`
   - `docker context ls`
   - `Test-Path \\\\.\\pipe\\dockerDesktopLinuxEngine`
4. 只有在 CLI 已恢复连通后，再执行项目 compose 启停、容器排查和浏览器 smoke

补充约束：

- 在 `docker ps` 无法返回前，不要把 `8080 / 5432 / 6379` 的异常直接归因为本项目后端、数据库脚本或前端代理问题
- 依赖 compose 容器的 E2E / smoke 必须先确认 Docker engine 可连接
- 若 Docker Desktop backend 再次异常退出，先保留上述日志现场，再做重启或重新拉起

- [ ] 团队已明确接受“`test` 同时承担自动化与人工联调”或已经拆出新的人工联调口径

### 5. 回退路径清楚

- [ ] 已保留一版可回退的 `local-mock` 配置、脚本和文档快照
- [ ] 若 `test` 承接人工联调后出现不稳定，可在半天内恢复到旧口径

在上述条件没有同时满足前，当前推荐策略调整为：

- `local-mock`：历史人工联调口径、脚本和报告名，保留为回滚参考
- `test`：Mock 联调 / 自动化测试 / 隔离测试栈 / 演示闭环
- `real-pre`：浏览器回归 / 权限验收 / 独立拓扑验证

## 十、当前结论

> 当前最稳的推进方式是：冻结旧 `test` 作为回滚基线，把 Mock 联调口径收敛到 `test`，保持 `real-pre` 回归可用，再逐个 Gateway 推进真实 SDK 联调。
