# 06-部署与对接计划

更新时间：2026-04-27

## 一、当前目标

当前部署与对接工作的重点不是直接上生产，而是：

1. 保证 `test` 环境可稳定启动
2. 保证演示链路可复现
3. 为真实 SDK 联调准备清晰切换方案

## 二、当前环境划分

### 1. Test 环境

用途：

- 本地开发
- 回归验证
- 老板演示
- 联调前基线

当前口径：

- 后端端口：`8080`
- 前端统一前缀：`/api`
- 调试台页面：`/dev/test`
- 初始化接口：`/api/test/seed`

### 2. Real 环境

用途：

- 真实第三方 SDK 联调
- 后续生产前验证

原则：

- 不破坏 Test 基线
- 逐个 Gateway 切换

## 三、本地 / Test 启动方式

### 1. 后端

```bash
cd backend
mvn spring-boot:run -Dspring-boot.run.profiles=test
```

### 2. 前端

```bash
cd frontend
npm install
npm run dev
```

### 3. 测试与构建

```bash
cd backend
mvn test

cd frontend
npm run build
```

补充说明：

- 抖店 SDK 依赖当前通过 `backend/lib/maven-repo/` 提供
- 若清理本地仓库或迁移项目，请保留 `backend/lib/` 与 `backend/lib/maven-repo/`

## 四、Test 演示启动后检查项

启动完成后，至少检查：

1. `/login` 可正常打开
2. `admin / admin123` 可登录
3. `/dev/test` 可正常打开
4. `/api/test/seed` 可正常调用
5. `/product`、`/orders`、`/dashboard`、`/sample`、`/talent` 可访问

## 五、Docker 相关口径

当前仓库已存在：

- `docker-compose.yml`
- `docker-compose.test.yml`
- `docker-compose.prod.yml`

当前建议：

- Test 演示优先以当前本地可运行口径为主
- Docker 配置用于环境对齐，不应改变 `/api/test/*` 和 `/dev/test` 运行口径

## 六、真实对接计划

### 第一阶段：认证打通

- 获取 `AppKey / AppSecret`
- 获取授权账号
- 验证 token 获取与刷新

### 第二阶段：活动与商品打通

- 联调活动列表
- 联调活动商品
- 对齐商品详情字段

### 第三阶段：转链打通

- 联调真实推广链接生成
- 对齐归因参数口径
- 验证 `promotion_link` 与 `pick_source_mapping`

### 第四阶段：订单打通

- 联调订单回流
- 验证归因状态
- 验证订单详情排查字段

### 第五阶段：达人与物流打通

- 联调达人信息刷新
- 联调物流状态同步

## 七、切换到 Real 环境前必须确认的事项

- [ ] Test 基线完整可回归
- [ ] Real 配置不覆盖 Test 配置
- [ ] Gateway 切换逻辑清楚
- [ ] token / secret 不写死在仓库中
- [ ] 回调地址和网络策略明确

## 八、当前不要做的事

当前阶段不要：

- 直接删除 Test 数据链路
- 用 Real 联调临时改前端页面字段
- 为了跑通一个真实接口破坏现有业务 Service
- 把演示环境和真实联调环境混在一起

## 九、当前结论

当前部署与对接的正确顺序是：

1. 保持 Test 环境稳定
2. 固化演示与回归口径
3. 再逐个 Gateway 准备 Real 联调

这也是当前项目最稳的推进方式。
