# 商品库推广中自动入库修复报告

> 日期：2026-06-01
> 状态：代码修复已完成；real-pre 定时同步开关已补入本地配置并重建 backend 生效；repair 写入修复尚未执行，需部署后先 dryRun 并人工确认窗口。

## 1. 问题根因

现象是 real-pre 中大量上游 `status=1/推广中` 商品没有进入商品库展示。

代码证据：
- `ProductService.upsertSnapshotsWithStats` 在活动商品同步命中商品时，会按上游商品自身状态执行自动入库。
- `ProductBizStatusService.initStateIfAbsent` 对已存在 `product_operation_state` 直接返回，不会重建历史状态。
- `ProductDisplayRuleService.applyForActivityId` 原逻辑只查询 `selected_to_library=true` 的状态记录再做展示去重。
- 因此历史上已存在、但仍为 `selected_to_library=false / PENDING_AUDIT` 的 `status=1` 商品不会被展示规则纳入计算。
- real-pre 当前 `PRODUCT_ACTIVITY_SYNC_ENABLED` 未设置，后端日志出现 `ProductActivitySyncJob skipped (disabled by config)`，定时同步没有持续刷新该状态断链。

阶段性根因结论：不是前端展示问题，也没有证据证明上游状态码整体接错；主要根因是“上游推广中自动入库规则缺少历史状态 repair 入口”，并且定时活动商品同步未启用。

## 3. 修复方案

新增统一 repair 能力，所有写入必须走后端规则入口：

1. 对目标活动 / 商品 / 全量推广中快照加载 `product_snapshot + product_operation_state`。
2. 对每条记录调用统一判断 `shouldAutoEnterLibrary(snapshot, state)`。
3. `status=1/推广中` 且未本地拒绝、未本地暂停、未推广期过期：自动设置 `selected_to_library=true`、`audit_status=2`，空/待审核业务状态转为 `APPROVED`，并进入展示规则竞争。
4. 本地拒绝：隐藏，`hidden_reason=LOCAL_REJECTED`。
5. 本地暂停：隐藏，`hidden_reason=LOCAL_PAUSED`。
6. 非推广中：隐藏，`hidden_reason=UPSTREAM_NOT_PROMOTING`。
7. 写入后调用 `applyForActivityId` 做去重展示重算。

## 6. 状态规则说明

商品库展示口径：

```text
商品库展示 = 上游商品 status=1/推广中
          + 本地未拒绝
          + 本地未暂停
          + 未被推广期/去重规则隐藏
```

`status=1` 的历史 `PENDING_AUDIT` 商品可被 repair 自动入库；非推广中、本地拒绝、本地暂停和过期商品不得展示。

## 14. 风险与回滚方案

风险：
- `dryRun=false` 会批量改变历史 `product_operation_state`，必须先保存 dryRun 差异。
- 大活动执行展示重算可能超过 5s，需要根据日志评估是否拆批或异步化。
- `LOCAL_PAUSED` 替代历史 `PUBLISH_PAUSED` 作为暂停隐藏原因，前端若硬编码旧值需同步兼容。

回滚：
- 代码回滚：回退本次修改文件后重新部署。
- 数据回滚：执行前必须保存 dryRun 差异和数据库备份；若写入后发现异常，按 dryRun 差异中的旧值生成受控回滚脚本，经人工确认后执行。
- 操作回滚：保持 `PRODUCT_ACTIVITY_SYNC_ENABLED` 未开启或移除白名单，停止自动刷新。
