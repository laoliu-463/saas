## 二、根因分析

### 2.1 错误返回链路

`GlobalExceptionHandler.java`（第 65-69 行）：
```java
@ExceptionHandler(Exception.class)
public ApiResult<Void> handleGeneral(Exception e) {
    log.error("系统异常", e);
    return ApiResult.of(ResultCode.SERVER_ERROR, null);  // → code=500, msg="服务器异常"
}
```

所有未捕获异常统一走 `handleGeneral`，返回固定 businessCode=500。这意味着：

- 业务码 500 ≠ 一定是数据库 / NPE 等底层异常
- 也可能是 Service 层抛出的未处理 RuntimeException
- 具体堆栈只在 backend stdout 日志中可见

### 2.2 create 流程中可能的异常抛出点

`TalentService.create()`（第 183-254 行）事务内调用的关键服务：

```
talentMapper.insert(request)           // ← 可能：DB 约束/字段超长/枚举值非法
  → createEnrichTask(...)             // ← 可能：taskMapper.insert NPE
  → markEnrichTask(RUNNING)           // ← 可能：taskMapper.update NPE
  → talentEnrichOrchestrator.enrich() // ← 可能：外部 API 超时/null 返回/RuntimeException
  → enrichTalentInfo()                 // ← 可能：crawlerTalentInfoService 异常
  → talentMapper.updateById(request)  // ← 可能：乐观锁/ID 不存在
  → catch RuntimeException
```

**最可疑点**：enrich orchestrator 在 test 环境依赖抖店 API mock 数据，若 mock 服务异常或超时，会抛出 RuntimeException，被 catch 后设置 enrichStatus=FAILED 但事务不回滚（因为只在 catch 里改状态）。但若抛出的是非 RuntimeException（如 Error 或 checked Exception），则不会进入 catch，直接冒泡到 GlobalExceptionHandler。

### 2.3 日志缺失问题

**test 环境后端日志全部缺失**：

| 日志文件 | 最新修改时间 | 距今 |
|----------|------------|------|
| `.codex-run/backend-spring.out.log` | 2026-05-04 11:37 | 15 天 |
| `.codex-run/backend-jar.out.log` | 2026-05-04 11:42 | 15 天 |
| `.codex-run/backend-local.out.log` | 2026-05-04 11:07 | 15 天 |

`runtime/tmp/backend-real-pre.stdout.log` 是 real-pre 环境（2026-05-18），不是 test 环境。

**test 环境今日无日志输出**，可能原因：
1. test 后端进程未启动 / 已崩溃（runner 使用 `localhost:8080`，由外部进程提供）
2. test 后端启动时未重定向 stdout 到持久化路径
3. 日志输出到 Docker 容器 stdout（若 test 环境是 Docker 部署）
