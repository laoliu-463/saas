## 二、当前数据库建模结论

### 1. 已有骨架可以承接 V2.2 的模块

当前库表已经覆盖或部分覆盖以下数据域：

- 用户与权限：`sys_user`、`sys_role`、`sys_user_role`
- 活动与商品：`colonel_activity`、`product`、`colonel_activity_product`
- 商品主链路状态：`product_snapshot`、`product_operation_state`、`product_operation_log`
- 推广与归因：`promotion_link`、`pick_source_mapping`
- 达人 CRM：`talent`、`talent_claim`、`talent_enrich_task`、`talent_follow_record`
- 寄样：`sample_request`、`sample_status_log`
- 订单：`colonelsettlement_order`、`order_detail`、`order_decrypt_record`
- 独家机制：`exclusive_talent`、`exclusive_merchant`
- 商家：`merchant`
- 配置与审计：`system_config`、`operation_log`

### 2. 当前建模最重要的现实口径

V2.2 需求里“商品库”并不是一张表能解决的事情，当前代码实际采用的是三层建模：

- `product`：商品基础主数据
- `product_snapshot`：活动商品快照
- `product_operation_state`：选品推进状态、负责人、转链、入库状态

这个拆法和现有业务闭环是一致的，后续不建议再额外创建一张“商品库主表”替代它。

### 3. 当前最明显的缺口不是“没有表”，而是“字段口径还没完全收口”

重点缺口集中在：

- 达人公海/私海与多人认领规则的最终口径
- 订单是否稳定拿到“达人标识”，从而支撑独家达人重分配
- 商品审核补充字段、寄样要求、推广素材等字段的结构化程度
- 商家归属字段类型与实体类型不一致
- 一些 V2.2 规则目前仍偏“配置 + 服务层判断”，尚未完全沉淀为可审计数据字段

## 三、推荐数据域划分

### 1. 组织与权限域

- `sys_user`
- `sys_role`
- `sys_user_role`

用途：

- 员工账号、角色、数据范围
- 渠道短码与推广链路透传
- 招商/渠道/运营/管理员权限控制

### 2. 活动与商品域

- `colonel_activity`
- `product`
- `colonel_activity_product`
- `product_snapshot`
- `product_operation_state`
- `product_operation_log`

用途：

- 活动绑定
- 活动商品同步
- 招商分配/审核
- 商品加入共享商品库
- 商品详情补充信息沉淀

### 3. 推广与归因域

- `promotion_link`
- `pick_source_mapping`
- `colonelsettlement_order`

用途：

- 生成推广链接
- 维护 `pick_source -> channel/talent/product/activity` 映射
- 订单归因与业绩归属

### 4. 达人经营域

- `talent`
- `talent_claim`
- `talent_enrich_task`
- `talent_follow_record`

用途：

- 达人基础信息
- 公海/私海/多人认领
- 爬虫任务与失败重试
- 跟进记录

### 5. 寄样履约域

- `sample_request`
- `sample_status_log`

用途：

- 寄样申请
- 审核/发货/签收/交作业/关闭
- 限制规则判定与履约审计

### 6. 商家与独家规则域

- `merchant`
- `exclusive_talent`
- `exclusive_merchant`

用途：

- 商家被动建档
- 独家达人月度生效
- 独家商家月度生效

### 7. 配置与审计域

- `system_config`
- `operation_log`

用途：

- 业务规则配置
- 操作日志

## 四、核心表设计草案

以下字段不是逐列抄库，而是只保留和 V2.2 需求最相关的部分。

### 4.1 `sys_user`

用途：

- 员工账号主表
- 兼承推广透传短码与部门归属

当前已具备：

- `id`
- `username`
- `real_name`
- `dept_id`
- `channel_code`
- `status`

建议继续保持：

- `channel_code` 唯一，且保证 `pick_extra` 模板生成后不超长度限制
- `dept_id + role` 作为数据权限和负责人分配的基础口径

### 4.2 `colonel_activity`

用途：

- 团长活动主表
- 活动同步与活动级提成/归属的上游基础

关键字段：

- `activity_id`
- `activity_name`
- `shop_id`
- `shop_name`
- `colonel_buyin_id`
- `start_time`
- `end_time`
- `status`
- `last_sync_at`
- `extra_data`

建议：

- 保留 `activity_id` 业务唯一
- 活动级特殊规则优先走 `system_config` 或扩展 JSON，不建议过早拆多张子表

### 4.3 `product`

用途：

- 商品基础主数据

关键字段：

- `product_id`
- `name`
- `market_price`
- `discount_price`
- `cover`
- `detail_url`
- `first_cid ~ fourth_cid`
- `cos_ratio`
- `service_ratio`

说明：

- 这张表只承担基础商品资料，不承担“是否入共享商品库”的业务状态

### 4.4 `colonel_activity_product`

用途：

- 活动商品同步结果
- 招商审核入口数据

关键字段：

- `activity_id`
- `product_id`
- `title`
- `price`
- `cos_ratio`
- `service_ratio`
- `shop_id`
- `shop_name`
- `assignee_id`
- `sample_requirement`
- `promotion_info`
- `audit_status`
- `audit_time`
- `audit_remark`

当前适配度：

- 已经很接近 V2.2 的“活动绑定 -> 拉商品 -> 待分配 -> 待审核”模型

建议补强：

- `sample_requirement` 统一保存结构化 JSON
- `promotion_info` 统一保存审核补充资料
- 招商审核拒绝原因建议单列 `reject_reason`，不要完全塞进备注

推荐 JSON 结构：

```json
{
  "exclusive_price_note": "",
  "delivery_info": {
    "ship_from": "",
    "express_company": ""
  },
  "selling_points": [],
  "promotion_script": "",
  "traffic_support": true,
  "traffic_material_urls": [],
  "reward_note": "",
  "participation_requirements": [],
  "activity_time_text": "",
  "cue_card_assets": []
}
```

### 4.5 `product_snapshot`

用途：

- 保存活动商品当前展示快照
- 为活动商品页、商品详情页提供稳定展示来源

关键字段：

- `activity_id`
- `product_id`
- `title`
- `cover`
- `price`
- `shop_id`
- `shop_name`
- `sales`
- `detail_url`
- `raw_payload`
- `sync_time`

建议：

- 保留 `(activity_id, product_id)` 唯一约束
- `raw_payload` 继续保留，便于联调排查

### 4.6 `product_operation_state`

用途：

- 商品主链路状态机
- 商品入共享商品库状态
- 当前负责人、转链、审核载荷沉淀

关键字段：

- `activity_id`
- `product_id`
- `biz_status`
- `assignee_id`
- `audit_status`
- `audit_remark`
- `audit_payload`
- `promote_link`
- `short_link`
- `external_unique_id`
- `selected_to_library`
- `selected_at`
- `selected_by`

当前适配度：

- 已可承接“待审核/已通过/已分配/已转链/已跟进/已入商品库”

建议：

- 继续把商品“是否进入共享商品库”留在这张状态表
- `audit_payload` 推荐从 `TEXT` 逐步收口为结构化 JSONB；如果短期不改字段类型，至少约定 JSON schema

推荐 schema：

```json
{
  "exclusive_price_note": "",
  "delivery_info": "",
  "selling_points": "",
  "promotion_script": "",
  "traffic_support": false,
  "reward_note": "",
  "participation_requirements": "",
  "activity_time": "",
  "cue_cards": [],
  "sample_requirement_override": {
    "min_30day_sales": 30000,
    "min_level": "LV1",
    "other_conditions": "粉丝>=10000"
  }
}
```

## 五、推荐的关系链路

### 1. 商品上架链路

`colonel_activity`
-> `colonel_activity_product`
-> `product_snapshot`
-> `product_operation_state`
-> `product_operation_log`

说明：

- 活动商品来自活动同步
- 快照负责展示
- 状态表负责推进
- 日志表负责审计

### 2. 推广归因链路

`product_operation_state`
-> `promotion_link`
-> `pick_source_mapping`
-> `colonelsettlement_order`

说明：

- 推广链接是调用事实
- 映射表是归因查找入口
- 订单是最终业绩事实

### 3. 达人寄样链路

`talent`
-> `talent_claim`
-> `sample_request`
-> `sample_status_log`
-> `colonelsettlement_order`

说明：

- 寄样自动完成依赖订单回流命中

### 4. 独家规则链路

`colonelsettlement_order`
-> `exclusive_talent`
-> 渠道业绩重分配

`colonelsettlement_order`
-> `merchant`
-> `exclusive_merchant`
-> 招商业绩重分配

## 六、建议新增或调整的字段清单

### 1. 优先级 P1：应尽快收口

1. `merchant.owner_id / owner_dept_id`
   - 目标：统一为 `UUID`
   - 原因：实体与 SQL 类型不一致

2. `product_operation_state.audit_payload`
   - 目标：至少 schema 收口，理想状态改为 `JSONB`
   - 原因：当前承接商品审核补充信息，后续查询和校验会越来越多

3. `sample_request.extra_data`
   - 目标：明确保存寄样资格校验结果与未达标理由
   - 原因：后续测试和审计都需要追溯申请时口径

4. `talent_claim`
   - 建议补 `release_time`、`release_reason`
   - 原因：保护期释放当前更偏服务逻辑，审计证据不足

### 2. 优先级 P2：真实化阶段建议补

1. `talent_enrich_task.sec_uid`
2. `talent_enrich_task.response_payload`
3. `sample_request.address_book_id`（如果要做常用地址）
4. `colonelsettlement_order.settle_status` 或更明确的结算态字段映射
5. `promotion_link.request_payload`

## 七、不建议现在新增的表

以下内容当前不建议立即建新表：

### 1. “商品库主表”

原因：

- 当前 `product + product_snapshot + product_operation_state` 已承担职责
- 再建一张“共享商品库表”会和 `selected_to_library` 语义重复

### 2. “渠道业绩汇总表” / “招商业绩汇总表”

原因：

- 当前看板正在往 SQL 聚合口径收口
- 过早做冗余汇总表，会增加对账复杂度

### 3. “独家规则计算结果临时表”

原因：

- 月度生效结果已经有 `exclusive_talent`、`exclusive_merchant`
- 中间过程可先用任务日志或后续专项表，不急着前置

## 八、待澄清后再落库的点

### 1. 达人多人认领与公海显示口径

当前代码已按“多人可同时认领”推进，但需求原文存在冲突。

数据库层暂不建议把“公海/私海状态”硬写回 `talent` 主表，继续通过 `talent_claim` 聚合判断更稳。

### 2. 订单达人标识来源

独家达人规则依赖订单能关联达人。

如果真实 SDK 无法稳定返回达人标识，则需要重新定义：

- 独家达人规则是否延后
- 是否允许通过推广链路或其他中间关系推断

### 3. 寄样资格中的“近 30 天销售额”来源

如果当前只能稳定拿到粉丝和等级，数据库可以先保留标准快照字段，但规则实现需标注为“部分模拟/待真实化”。

### 4. 商品审核拒绝后的回退流向

如果后续要区分：

- 退回待分配
- 保留在招商待修正
- 彻底拒绝

则需要更明确的 `audit_status` 枚举和日志文案，不一定要新增表。

## 九、推荐的后续落地顺序

### 1. 先补迁移，不先大改实体

优先做：

- 商家归属字段类型统一
- 商品审核载荷 schema 收口
- 寄样申请校验快照沉淀

### 2. 再补接口文档

重点对齐：

- 商品审核通过请求体
- 寄样申请请求体
- 达人认领/释放响应体
- 数据看板筛选与业绩归属字段

### 3. 最后补测试矩阵

至少覆盖：

- 商品审核补充信息回写
- 推广链生成与映射入库
- 7 天寄样限制
- 30 天保护期释放
- 独家达人/独家商家月度生效
- 订单归因缺失场景

## 十、结论

从数据库设计角度看，当前项目并不缺主体表，缺的是口径收口。

更准确地说：

> V2.2 所需核心数据骨架已经基本存在，后续重点应放在复用现有表、补齐关键字段语义、统一类型与 JSON 结构，而不是推翻当前库重新建模。

当前最值得优先处理的数据库事项只有三件：

1. 收口商品审核补充字段的存储结构
2. 补齐寄样申请时的规则快照与审计信息
3. 统一商家归属字段类型，避免实体与 SQL 漂移
