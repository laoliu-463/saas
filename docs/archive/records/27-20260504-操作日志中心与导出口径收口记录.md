# 2026-05-04 操作日志中心与导出口径收口记录

## 1. 背景

在上一轮本地规则中心闭环完成后，`docs/10-V2.2场景覆盖矩阵.md` 中仍存在一组更偏后台治理的半闭环能力：

- 场景 35：管理员追溯操作日志
- 场景 19：寄样自动关闭文案与规则中心配置不一致
- 数据导出接口权限与数据范围口径存在偏差

本轮目标不是扩 SDK，而是继续把本地 Mock 下能独立闭环的治理能力补齐。

## 2. 本轮完成项

### 2.1 管理员统一操作日志中心

已新增：

- 后端 `operation_log` 实体 / Mapper / Service / Controller
- `OperationLogInterceptor`，统一沉淀 `POST / PUT / PATCH / DELETE` 关键操作
- 前端 `/system/operation-logs` 页面与系统管理菜单入口

当前口径：

- 默认查询最近 90 天日志
- 支持按操作人、模块、动作、请求方法、日期范围筛选
- 当前优先覆盖系统配置、用户、角色、寄样状态流转等关键变更类请求
- 商品详情 / 寄样详情的局部时间线继续保留，不被新日志中心替代

### 2.2 订单导出权限与数据范围收口

已修复：

- `/orders/exports` 仅允许 `admin / biz_leader / channel_leader`
- 导出接口后端补齐 `applyPageDataScope(...)`

结果：

- 普通成员即使知道接口路径，也不会再绕过数据范围直接导出更大数据集

### 2.3 寄样自动关闭原因文案动态化

已修复：

- `SampleLifecycleService` 自动关闭原因从固定“30 天 / 15 天”改为随传入规则动态生成

结果：

- 当管理员在 `/system/config` 修改寄样自动关闭天数时，寄样详情页与状态日志中的关闭原因将与实际规则一致

## 4. 当前结论

本地 Mock 基线下，系统管理已不再只有“用户 / 角色 / 配置”三块，而是补齐了统一的管理员日志追溯入口；同时把订单导出和寄样自动关闭这两个容易在验收时暴露口径问题的点一起收紧。

这意味着：

- 场景 35 可上调为“已覆盖”
- 场景 19 的剩余风险已从“规则没接上”收敛为“需要继续保留可见浏览器专项证据”
