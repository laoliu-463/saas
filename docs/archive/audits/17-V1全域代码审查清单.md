## 一、使用口径

本清单是代码审查执行文档，不是新增需求文档。

当前阶段按主干文档表述为：V1 功能冻结，real-pre 已具备业务可用性，但仍不能宣称生产发布；下一步重点是 `P3-6` 真实 `pick_source` 归因样本观察与 `P3-8` 异常分支证据补齐。代码改动必须先满足文档口径，不能为了审查模板中的全量 V1/V2.2 能力反向扩大当前阶段范围。

审查优先级：

1. `docs/README.md` 与 `docs/04-开发进度.md` 的当前状态。
2. `docs/03-Test与Real网关契约.md` 的 Test / Real Gateway 边界。
3. `docs/05-接口与数据模型.md` 的内部 API、DTO、DataScope、导出口径。
4. `docs/10-V2.2场景覆盖矩阵.md` 的场景状态与角色权限矩阵。
5. 当前代码和测试事实。

## 四、状态机审查清单

| 状态机 | 当前代码口径 | 必审断言 |
| --- | --- | --- |
| 商品业务状态 | `PENDING_AUDIT -> APPROVED/REJECTED -> ASSIGNED -> LINKED -> FOLLOWING`，`BOUND` 仅兼容读取 | 非法流转返回统一业务错误；并发更新走乐观锁；每次关键状态变化写操作日志 |
| 寄样状态 | `PENDING_AUDIT -> PENDING_SHIP -> SHIPPED -> PENDING_TASK -> FINISHED`，另有 `REJECTED/CLOSED` | 角色权限、前置状态、日志、自动完成和超时关闭一致；`CLOSED` 分支必须校验前置状态 |
| 达人认领 | 有效认领、释放、过期，支持多人有效认领 | 释放单个认领后仍有其他认领人时不得清空归属快照；跨组编辑/删除必须 403 |
| 订单归因 | `ATTRIBUTED / UNATTRIBUTED` + `attribution_remark` 诊断 | 未归因原因可解释；`unsafeBecauseCreatedAfterOrder` 不写成已归因，不进入业绩 |
| 独家达人 / 商家 | 月度计算后输出有效记录和状态页 VO | 不直接暴露 Entity 审计字段；V1 默认 `exclusive.enabled=false`，只有显式开启后才把当前有效独家记录用于归因覆盖 |

## 六、接口与异常审查清单

| 类别 | 必审断言 |
| --- | --- |
| 请求参数 | 写接口使用 DTO + `@Valid`，禁止直接暴露 Entity 写入敏感字段 |
| 响应结构 | 返回 VO / DTO，避免把 Entity 审计字段、Token、联系方式原文直接返回 |
| 分页 | 列表接口必须有 page/size 上限；前端分页默认 20 到 100 |
| 第三方异常 | Real Gateway 捕获上游异常后转业务异常或可解释状态，日志脱敏 |
| Token 过期 | 刷新链路、重新授权标记、定时刷新不得泄露 token 前后缀 |
| 限流 / 429 | 只允许白名单安全接口设计退避重试，不在通用 client 上无差别重试 |
| Webhook | 默认验签；事件幂等；重放只处理未完成或失败事件；原始 payload 日志脱敏 |

## 七、数据一致性审查清单

| 数据对象 | 必审断言 |
| --- | --- |
| `colonelsettlement_order` | `order_id` 去重；软删过滤；金额、结算时间、归因字段、DataScope 字段一致 |
| `pick_source_mapping` | 真实转链必须解析到有效 `pick_source` 才落库；NATIVE 映射按 `colonel_buyin_id + activity_id + product_id` 唯一候选 |

| `talent_claim` | 达人归属、DataScope 和多人认领均以有效认领记录为准 |
| `sample_request` / `sample_status_log` | 状态变化和自动任务要同步写日志，避免只有状态无轨迹 |
| `system_config` | 配置更新后缓存失效，默认值与文档一致 |
| `operation_log` | 导出、重算、配置修改、归因副作用、商品主链路关键动作可审计 |

## 八、验证门槛

文档确认后再进入代码阶段。代码阶段每轮改动至少满足：

1. 后端定向测试覆盖变更模块。
2. 涉及权限 / DataScope / DTO 的改动补 Controller 或 Service 单测。
3. 涉及前端交互的改动执行 `frontend npm run test` 或定向 Vitest，必要时补 Playwright。
4. 涉及 real-pre 的改动仅用只读脚本或人工确认后的清理流程，不自动创建不可回滚真实上游数据。
5. 若影响阶段结论，同步 `docs/04-开发进度.md`、`docs/05-接口与数据模型.md`、`docs/10-V2.2场景覆盖矩阵.md`。
