## 七、关键风险

### R1. 真实订单归因尚未闭环

真实转链已成立，但最新真实订单仍未携带 `pick_source / pick_extra`。当前库内真实订单多数为 `NO_PICK_SOURCE`，因此不能宣称“真实转链 -> 真实订单 -> 自动归因”已经闭环。

### R2. `pickSource` 复用会冲击唯一映射模型

real-pre 记录显示多个不同商品真实转链都返回同一个 `pickSource=v.MxZLIw`。当前 `pick_source_mapping` 对 `pick_source` 有唯一索引，`PickSourceMappingService.saveOrUpdate` 命中同一 `pick_source` 时会覆盖 `product_id / activity_id / promotion_link_id`。如果上游持续复用 `pickSource`，真实归因可能归到最后一次转链记录。

### R4. 商品详情 / SKU 不能用活动商品快照替代

活动商品快照能支撑商品卡片和业务详情 smoke，但不能证明 `product.detail` / SKU 明细已通。真实 raw probe 已返回权限不足，后续验收必须拿到上游商品详情和 SKU 字段样本。

### R5. Webhook 只是“接收层”覆盖

当前 Webhook 覆盖了签名、日志脱敏和快速返回，但没有完成真实控制台投递、事件幂等、业务消费或重放补偿。不能把它算作订单 / 商品变更回流闭环。

### R6. Talent / Logistics 真实链路尚未选型

达人和物流在本地业务中可用，但真实三方侧没有明确 RealGateway。尤其 `TalentApi` 的命名容易误导，它当前不是达人资料 API，而是转链能力的轻包装。

### R7. 本地真实联调配置敏感度较高

本机 `.env.real-pre` 存在真实上游联调用配置项。审计报告不记录具体值；后续必须确认该文件不进入仓库、不进入公开制品，若曾外流需要轮换。

## 八、建议收口顺序

1. 先修正文档口径：把 `docs/03` 的商品详情 / SKU 状态改为“代码已实现、权限待补齐”；把 real-pre 当前 profile 统一为 `real`。
2. 单独拆 `pickSource` 复用风险：调整映射唯一键、归因查找策略或引入 `pick_extra / promotion_link_id / product_id` 组合口径，避免真实订单归因被覆盖。
3. 申请并验证权限包：商品详情 / SKU、店铺侧订单。
4. Webhook 下一步只做接收事实不够，应补“真实投递 -> 幂等落库 -> 业务消费 -> 重放补偿”专项。
5. Talent / Logistics 不要混进 Douyin 主链路验收，应分别先做接口选型和 Gateway 契约。
6. M1.6 数据看板真实化需要先定 `createTime / settleTime` 默认口径，再做页面与指标验收。

## 九、结论清单

### 已覆盖

- 本地 Mock / test 核心闭环：商品、订单、寄样、达人、系统管理、认证安全。
- real-pre 真实上游主前置：Token、授权主体、活动、活动商品、真实转链、团长侧订单同步、订单列表、Dashboard createTime smoke。
- 抖店联调管理入口：`/system/douyin` 前端可见，E2E 已通过。

### 部分覆盖

- 真实订单归因：订单同步可用，但真实订单未携带归因参数。
- 商品详情 / SKU：代码有实现，权限和真实样本缺失。
- Webhook：接收和日志防护有，业务消费缺失。
- 数据看板真实化：createTime 可读，完整 M1.6 口径未封口。
- 达人真实资料：本地业务有，真实资料 Gateway 缺失。

### 未覆盖

- 真实物流网关。
- 店铺侧订单详情成功样本。
- Webhook 真实业务消费闭环。
- 限流 / 429 / 退避白名单验证。
- 真实订单携带 `pick_source / pick_extra` 后的归因成功样本。

### 最高优先级风险

1. `pickSource` 复用 + 唯一映射覆盖，可能导致真实商品级归因错误。
2. 商品详情 / 店铺订单权限包未开通前，不应宣称真实 SDK 全闭环。
3. real-pre 已是强联调环境，敏感配置和真实订单数据需要按准生产级别保护。
