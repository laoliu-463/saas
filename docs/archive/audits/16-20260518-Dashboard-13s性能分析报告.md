## P1 详细：活动商品下钻 SQL

```sql
-- 查出 20 个商品行，每行内嵌 4 个子查询
SELECT
  co.activity_id, co.product_id,
  -- 子查询 1：mapping 数量
  COALESCE((SELECT COUNT(DISTINCT psm.id)
    FROM pick_source_mapping psm
    WHERE psm.activity_id = co.colonel_activity_id
      AND psm.product_id = co.product_id
      AND psm.deleted = 0 AND psm.status = 1), 0) AS mapping_count,
  -- 子查询 2：promotion_link 数量
  COALESCE((SELECT COUNT(DISTINCT pl.id)
    FROM promotion_link pl
    WHERE pl.activity_id = co.colonel_activity_id
      AND pl.product_id = co.product_id
      AND pl.deleted = 0), 0) AS promotion_link_count
FROM colonelsettlement_order co
LEFT JOIN product_snapshot ps ON ...
LEFT JOIN product_operation_state pos ON ...
LEFT JOIN sys_user su ON ...
WHERE co.deleted = 0
GROUP BY co.colonel_activity_id, co.product_id
LIMIT 20 OFFSET 0
```

**问题**：20 行 × 4 子查询 = 约 80 次隐性查询，打在 `pick_source_mapping`、`promotion_link`、`product_snapshot`、`product_operation_state` 四张表上。

## 建议

|| 级别 | 建议 | 方向 |
||------|------|------|
| S1 | V2 必须 | 预聚合表：在订单入库时写入 `order_product_summary`，消除子查询 |
| S2 | V2 必须 | Redis 缓存 Dashboard 结果，TTL 5 分钟 |
| S3 | V2 应做 | SQL 重构：将相关子查询改写为 JOIN，预计算 mapping_count |
| S4 | V2 应做 | Prometheus 监控：记录 `/api/dashboard/summary` 耗时，超过 5s 告警 |
| S5 | V1 暂缓 | 分页改造：breakdown 接口支持分页，避免全量 GROUP BY |

## 关联

Dashboard 展示 7152 条订单 0% 归因（`attributedOrderCount=0`），根因是 `mappingCount=0`（没有推广链接生成记录），不是归因逻辑错误。需要业务方在测试环境真正走一遍"选品→转链→分享→下单"链路才能验证归因。
