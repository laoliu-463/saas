# 04-开发进度

更新时间：2026-05-18

## 一、总体结论

当前项目的更准确状态不是“继续新增功能”，而是：

> V1 功能建设已接近完成，当前重点是统一文档口径，明确 V1 交付合同、当前系统实现和 V2 后续规划；验收证据补齐放在下一阶段。

当前建议统一表述为：

- V1 主交付范围已基本明确
- 当前系统大部分 V1 能力已有对应实现或底座
- 抖音联盟主链路已有 real-pre 事实基础
- 现在优先做文档统一口径，不再让旧 V2.2、当前实现、验收记录互相打架

## 三、模块状态

| 模块 | V1 状态 | 当前实现情况 | 当前主要差距 | 后续处理 |
|---|---|---|---|---|
| 商品域 | 已实现待确认 | 活动、商品、商品库、转链、`pick_source_mapping` 已有实现 | 逐项业务口径仍需按 V1 范围表收口 | 继续对齐商品上架、默认招商归因口径 |
| 达人域 | 已实现待确认 | 达人录入、认领、保护期、地址、标签已有实现 | 达人外部数据仍以基础字段 + 手动兜底为主 | 保留爬虫增强为 V2 规划 |
| 寄样域 | 已实现待确认 | 申请、审核、发货、待交作业、完成已有实现 | 物流自动跟踪不能写成 V1 已完成 | 明确 V1 以手动物流为主 |
| 订单域 | 已实现待确认 | 订单同步、归因分析、订单详情已有实现 | 真实 `pick_source` 样本仍待后续补证 | 文档上先固定“默认归因 + 样本待补” |
| 业绩域 | 已实现待确认 | 双轨金额、基础提成、基础看板已有实现 | 毛利、独家覆盖、差异化提成不应混入 V1 | V1 只保留默认归因和基础提成 |
| 用户域 | 已实现待确认 | 角色、权限、数据范围、菜单已有实现 | 一人多角色口径需统一成“权限并集、数据范围最宽” | 在后续验收阶段专项确认 |
| 配置域 | 已实现待确认 | 寄样限制、保护期、全局提成等已有实现 | 需要明确哪些配置属于 V1 启用项 | 把独家阈值、差异化提成降到 V2 |
| 分析模块 | 简化实现 | 基础 Dashboard、基础明细、基础下钻可用 | 高级排行、趋势增强、复杂分析不作为 V1 阻断 | 固定基础分析口径即可 |
| 三方对接 | 部分完成 | 抖音联盟主链已有事实基础；物流 / 达人外部数据未完成真实联调 | 容易被误写成“全部已接通” | 统一写成“主链已接，增强项后置” |

## 四、当前优先级

### 1. 现在优先做的

1. 统一文档口径
2. 统一 V1 范围定义
3. 统一当前实现边界
4. 把 V2 能力从 V1 文档里降级为后续规划

### 2. 现在不优先做的

1. 继续扩新功能
2. 把所有历史联调细节塞回总纲文档
3. 把所有“代码里有入口”的能力都写成已验收

## 五、当前非阻断待收口项

以下项目需要在文档中说明边界，但不作为当前“文档统一口径”阶段的阻断项：

| 条目 | 当前口径 |
|---|---|
| 真实 `pick_source` 订单样本 | 作为后续验收补证项，先不用于推翻 V1 范围 |
| 物流 API 自动跟踪 | 保留底座，V1 默认手动物流 |
| Excel 批量导入物流单号 | 后置，不进 V1 P0 |
| 一人多角色 | 当前按“权限并集、数据范围最宽”写入口径，专项验收后再补证 |
| 高级排行榜 / 趋势图 | 简化版即可，不作为 V1 阻断 |
| 差异化提成 | 后续规划，不纳入当前 V1 |
