# 00-项目总览

更新时间：2026-05-18

## 一、当前项目定位

本项目是抖音团长内部 SaaS 系统。

当前项目的目标不是一次性交付完整 V2.2 规划，而是先把 **V1 基础业务闭环** 跑通，并把需求范围、业务口径、技术实现和后续规划拆清楚。

一句话定位：

> 本项目先交付 V1 三条主链，旧版 V2.2 作为背景参考，当前仓库实现以 Spring Boot 模块化单体为事实口径。

## 二、V1 三条业务主线

### 1. 渠道链

认领达人  
↓  
商品库选品  
↓  
复制推广链接  
↓  
生成 `pick_source` 映射  
↓  
申请寄样  
↓  
招商审核  
↓  
运营 / 商家发货  
↓  
订单同步  
↓  
订单默认归因到渠道  
↓  
业绩统计

### 2. 招商链

同步活动  
↓  
同步活动商品  
↓  
招商补充商品信息  
↓  
商品上架到商品库  
↓  
渠道推广  
↓  
订单同步  
↓  
默认招商归因 = 活动绑定招商  
↓  
招商业绩统计

### 3. 管理链

配置人员、角色、组织  
↓  
配置寄样限制、保护期、提成比例等规则  
↓  
商品、达人、寄样、订单、业绩读取规则  
↓  
不同角色看到不同数据范围

## 三、当前技术实现

当前项目实际技术栈如下：

| 层级 | 当前实现 |
|---|---|
| 后端 | Spring Boot 模块化单体 |
| 数据库 | PostgreSQL |
| 缓存 | Redis |
| 前端 | Vue 3 + Vite |
| 部署 | Docker Compose |
| 三方隔离 | Gateway 方式隔离 Mock / Test / Real |

补充说明：

- 新版需求包中的 `FastAPI + Celery` 是原始技术落地建议。
- 当前仓库实际实现是 `Spring Boot + PostgreSQL + Redis + Docker Compose`。
- 文档不能再写成“必须迁移 FastAPI / Celery”，只能写成“业务设计参考，当前技术实现采用 Spring Boot 模块化单体”。

## 五、V1 当前实现边界

### V1 主交付

- 商品、活动、推广链接、`pick_source_mapping`
- 达人录入、认领、保护期、地址、标签
- 寄样申请、审核、发货、待交作业、完成
- 订单同步、默认渠道归因、默认招商归因
- 双轨金额、基础提成、基础看板
- 用户、角色、数据范围、系统配置

### V1 简化

- 物流以手动录入和手动状态推进为主
- 达人外部信息以基础字段 + 手动兜底为主
- 提成配置以全局配置为主
- 看板以基础指标、基础下钻为主

### V2 / 后续规划

- 独家达人
- 独家商家
- 差异化提成
- 高级排行榜 / 趋势图 / 高级分析
- 爬虫增强
- 物流 API 自动跟踪

## 六、当前阶段判断

当前项目的工作重点已经不是“继续堆功能”，而是：

1. 先把所有文档统一成一个 V1 口径。
2. 把 V1 交付合同、当前项目实现、V2 后续规划明确拆开。
3. 在文档口径稳定后，再补 P0 验收步骤和证据索引。

因此当前阶段的更准确表述是：

> V1 功能建设已接近完成，当前优先级是文档统一口径，其次才是补齐验收证据。

## 七、一句话总结

> 本项目当前应按“V1 交付合同 + 当前 Spring Boot 实现 + V2 后续规划”三层理解，而不是把旧版 V2.2 规划、当前实现和验收清单混成一个口径。
