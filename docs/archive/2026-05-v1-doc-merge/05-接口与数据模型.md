# 05-接口与数据模型

更新时间：2026-05-18

## 一、文档使用口径

本文档用于描述**当前项目实际接口与数据模型**，不是旧版 V2.2 技术方案的直接复刻。

当前统一口径如下：

1. V1 范围以 [V1交付范围表](./requirements/2026-05-新版需求包/V1交付范围表.md) 为准。
2. 当前接口实现以 Spring Boot 仓库事实为准。
3. 旧版 `saas系统V2.2(定稿).md`、`技术落地设计.md` 中的 FastAPI / Celery 方案只作为背景参考。
4. 本文档优先区分：
   - V1 必需模型
   - V1 简化模型
   - V2 预留模型

## 二、V1 必需模型

| 模型 | 当前用途 |
|---|---|
| 用户 / 角色 / 组织 | 登录、权限、数据范围 |
| 活动 | 招商链活动同步与筛选 |
| 商品 | 商品库、活动商品、共享商品视图 |
| 达人 | 达人档案、地址、标签、认领入口 |
| 达人认领 | 公海 / 私海、保护期、归属判断 |
| 寄样申请 | 申请、审核、发货、待交作业、完成 |
| `pick_source_mapping` | 推广链接归因映射 |
| 订单 | 同步、默认渠道归因、默认招商归因 |
| 业绩 / 汇总 | 双轨金额、基础提成、基础 Dashboard |
| 系统配置 | 7 天限制、保护期、全局提成等 |
| 操作日志 | 后台关键操作留痕 |

## 三、V1 简化模型

## 七、命名与使用口径

### 1. 路径规则

- 资源名优先使用复数：`/orders`、`/samples`、`/talents`、`/users`、`/roles`
- 主链路路径体现层级关系：`/colonel/activities/{activityId}/products/{productId}`
- 子动作后缀统一使用 kebab-case：`audit-result`、`promotion-links`、`manual-fill`

### 2. 分页参数

当前存在两种命名：

- 业务接口部分仍使用 `page` + `pageSize`
- 业务接口部分已经使用 `page` + `size`

当前项目决定是：

- **暂不强行一次性改名**
- 已在 Swagger 中明确标注命名差异
- 后续是否统一为 `page` + `size`，以 `docs/archive/audits/接口整改-决策备忘录.md` 结论为准

### 3. 测试、联调、废弃接口展示规则

- `/test/**`：统一归类到 `测试工具`，接口名称前缀为 `[测试]`
- `/douyin/**`、`/douyin/webhooks/**`：统一归类到 `抖音联调`，接口名称前缀为 `[联调]`
- `/products/**`：统一归类到 `商品管理（兼容）`，其中 `/products` 为共享商品库，`/products/picks` 为旧版选品候选兼容入口
- `/test/**` 仅允许出现在 `local-mock`、`test` 或当前回归型 `real-pre` 环境，不作为未来纯真实 SDK 环境的默认能力

### 4. 枚举说明规则

- 已有代码常量、枚举或映射函数定义的，按代码说明
- 代码中没有明确定义的，Swagger 中统一写：
  - `待确认：取值含义请联系产品`

本文档同样遵守该规则，不臆造未确认枚举值。

### 5. V1 与兼容接口判读规则

- `V1 主链接口`：直接服务于渠道链、招商链、管理链的当前交付能力。
- `联调接口`：用于真实 SDK / Webhook / Token 运维，不等于全部属于 V1 主交付。
- `测试接口`：仅用于测试、播种、回归和链路推进，不纳入 V1 对外能力说明。
- `兼容 / 运维接口`：为旧页面兼容、历史运维面板或 V2 预留保留，不直接作为 V1 已交付结论。
- 看到接口存在时，必须回到 [V1交付范围表](./requirements/2026-05-新版需求包/V1交付范围表.md) 判断它是不是当前范围。

### 0. 商品管理（兼容）

```http
GET /api/products
GET /api/products/picks
POST /api/colonel/activities/{activityId}/products/{productId}/library-entry
```

说明：

- `GET /api/products/picks`：查询旧版本地选品候选列表，主要用于 Token 缺失时的降级展示

### 1. 认证中心

```http
POST /api/auth/login
POST /api/auth/refresh
POST /api/auth/logout
```

登录请求：

```json
{
  "username": "admin",
  "password": "admin123"
}
```

登录响应（`LoginResponse`）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `token` | String | Access Token（有效期 2 小时） |
| `refreshToken` | String | Refresh Token（有效期 7 天） |
| `expiresIn` | Long | Access Token 有效期秒数（7200） |
| `refreshExpiresIn` | Long | Refresh Token 有效期秒数（604800） |
| `accessTokenExpiresIn` | Long | Access Token 有效期秒数（与 `expiresIn` 一致） |
| `userId` | Long | 用户 ID |
| `username` | String | 用户名 |

Token 刷新请求：

```json
{
  "refreshToken": "eyJhbGciOi..."
}
```

登出请求：

```json
{
  "accessToken": "eyJhbGciOi...",
  "refreshToken": "eyJhbGciOi..."
}
```

说明：

- Access Token 有效期 2 小时，携带 `type=access` claim
- Refresh Token 有效期 7 天，携带 `type=refresh` claim，使用 SHA-256 哈希存储于 Redis
- 登出后 Access Token 和 Refresh Token 同时加入 Redis 黑名单，立即失效
- 黑名单 key 格式：`auth:blacklist:{tokenHash}`，过期后自动清理

#### DataScope 数据权限

系统支持三级数据权限过滤（`@DataScope` 注解 + `DataScopeAspect`）：

| 枚举值 | 说明 |
|---|---|
| `PERSONAL(1)` | 仅查看本人数据 |
| `DEPT(2)` | 查看本组数据 |
| `ALL(3)` | 查看全部数据 |

```http

GET  /api/orders/exports
GET  /api/operations/exclusive-talents
GET  /api/operations/exclusive-merchants

```

时间字段口径：

- `/api/dashboard/metrics`、`/api/data/orders`、`/api/orders/exports` 均支持 `timeField=createTime|settleTime`
- 默认 `createTime`，用于覆盖真实订单已同步但尚未结算的运营看板和明细核对场景
- 需要按结算收益复核时显式传 `timeField=settleTime`
- `/api/data/orders` 订单明细返回 `createTime` 与 `settleTime`；前端切换 `timeField=settleTime` 时应展示结算时间，未结算订单显示未结算，不使用前端伪造时间
- `/api/orders/exports` 导出列包含 `创建时间` 与 `结算时间`，筛选时间字段仍由 `timeField` 决定

说明：

- 当前项目不再提供订单手机号解密功能
- 订单页和订单详情仅展示上游已返回的联系电话字段；上游未返回时按“未返回”展示
- `/api/operations/exclusive-talents`、`/api/operations/exclusive-merchants` 属于独家相关监控 / 运维口子，当前不能作为 V1 主交付能力宣称完成
- 这两个 `exclusive-*` 接口对应的页面或前端封装，如继续保留，应统一按“运维兼容 / V2 预留”理解，而不是按“V1 已启用独家机制”理解
