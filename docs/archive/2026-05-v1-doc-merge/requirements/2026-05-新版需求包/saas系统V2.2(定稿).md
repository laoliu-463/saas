## 一、项目概述

### 1.1 项目定位
为抖音团长团队打造的一站式业务管理 SaaS 系统，整合商品管理、达人 CRM、寄样跟踪、业绩统计等核心功能。

### 1.2 使用规模

| 指标     | 数值          | 说明                     |
| -------- | ------------- | ------------------------ |
| 订单数据 | 5 万条/天     | 峰值，年增长约 1800 万条 |
| 达人数据 | 数千条        | 仅入库的达人             |
| 商品数据 | 数万条        | 持续增长                 |
| 并发用户 | 50 人同时使用 | 要求查询不卡顿           |

### 1.3 开发策略

| 阶段     | 目标                             | 说明                               |
| -------- | -------------------------------- | ---------------------------------- |
| **V0.5** | 本地开发，核心流程跑通           | 不上线，Mock 数据                  |
| **V1.0** | Bug 修复、性能优化后部署上线     | 接入真实 API，PostgreSQL，爬虫采集 |
| **V2.0** | 增加独家机制、达人分析等高级功能 | 差异化提成、高级看板               |

### 1.4 技术选型

| 层级     | 技术                            | 版本           | 说明                   |
| -------- | ------------------------------- | -------------- | ---------------------- |
| 前端     | Vue 3 + Naive UI + Vite         | Vue 3.4+       | 组件化开发             |
| 后端     | FastAPI + SQLAlchemy            | FastAPI 0.115+ | Python 3.13            |
| 数据库   | PostgreSQL                      | 15+            | 支持分区表             |
| 缓存     | Redis                           | 7.0+           | Token 缓存、限流       |
| 任务队列 | Celery + Redis                  | Celery 5.3+    | 异步同步订单、爬虫任务 |
| 爬虫     | Python Requests + BeautifulSoup | -              | 达人信息采集           |
| 开发环境 | Docker                          | -              | 一键启动               |
| 生产环境 | 云数据库托管（RDS）             | -              | 自动备份               |

## 二、用户角色与权限

### 2.1 岗位列表

| 角色         | 核心职责                         |
| ------------ | -------------------------------- |
| **管理员**   | 系统配置、人员管理、权限分配     |
| **招商组长** | 绑定活动、分配商品、管理招商团队 |
| **招商**     | 审核商品、补充信息、审核寄样     |
| **渠道组长** | 管理渠道团队、数据汇总           |
| **渠道**     | 达人对接、申请寄样、跟进产出     |
| **运营**     | 物流录入、辅助操作               |

### 2.2 权限设计原则

**核心逻辑：基于角色的数据范围控制 + 灵活的权限配置**

| 角色     | 数据范围                 | 核心操作                               |
| -------- | ------------------------ | -------------------------------------- |
| 管理员   | 全部数据                 | 所有操作                               |
| 招商组长 | 本组招商 + 自己的数据    | 活动绑定、商品分配、查看组内业绩       |
| 招商     | 自己负责的商品/活动/寄样 | 商品审核、寄样审核、查看自己品出单数据 |
| 渠道组长 | 本组渠道 + 自己的数据    | 查看组内业绩、管理组员                 |
| 渠道     | 自己的达人/寄样/业绩     | 达人认领、申请寄样、查看自己数据       |
| 运营     | 按需配置                 | 物流录入、发货操作                     |

### 2.3 权限配置方式

- 每个角色可单独配置**可见菜单**
- 每个角色可单独配置**数据范围**（仅自己/本组/全部）
- 每个角色可单独配置**操作权限**（增/删/改/查/导出）
- 支持**新增自定义角色**

### 2.4 可配置规则（管理员权限设置）

| 规则项                 | 默认值 | 说明                                     |
| ---------------------- | ------ | ---------------------------------------- |
| 寄样限制天数           | 7天    | 同一渠道+同一达人+同一商品的重复申请间隔 |
| 寄样限制开关           | 启用   | 是否启用寄样限制                         |
| 独家达人服务费占比阈值 | 70%    | 触发独家达人的占比条件                   |
| 独家达人月寄样数量阈值 | 10个   | 触发独家达人的月寄样条件                 |
| 独家商家服务费占比阈值 | 70%    | 触发独家商家的占比条件                   |
| 达人保护期             | 30天   | 认领后无产出自动释放时间                 |

### 2.5 导出权限

- **寄样台**：支持导出（用于同步商家）
- **数据平台其他模块**：初期不做导出，聚焦线上看板

## 三、核心功能模块

### 3.1 商品库

**定位**：渠道商务选品的入口，全员可见。

#### 功能清单

- **卡片展示**：图片、商品名称、商品ID、店铺、售价、佣金率、服务费率、寄样要求、活动信息
- **多维度筛选**：商品类目、佣金率范围、寄样门槛、支持投流、招商负责人
- **一键复制**：调用抖音官方转链接口，生成带归因参数的推广链接
- **商品详情页**：展示完整信息（寄样要求、投流信息、推广素材）
- **排序**：最新推广、总销量、价格、佣金比例
- **分页**：支持分页加载

#### 数据来源

| 数据类型     | 来源             | 说明                                           |
| ------------ | ---------------- | ---------------------------------------------- |
| 团长活动信息 | 抖音 API         | 查询机构下的团长活动列表                       |
| 活动商品列表 | 抖音 API         | 查询指定活动下的商品列表                       |
| 商品补充信息 | 招商审核时填写   | 寄样要求、投流信息、卖点话术等，存入本地数据库 |
| 商品详情     | 抖音 API（按需） | 同步商品最新价格、佣金率                       |

#### 审核流程

```
老板在官方后台创建活动
       ↓
招商组长在系统中同步活动列表（或手动绑定活动ID）
       ↓
系统自动同步该活动下的商品列表
       ↓
招商组长将商品批量分配给招商人员
       ↓
招商补充信息（寄样要求、话术等）后审核通过
       ↓
商品正式上架到商品库，对全员可见
```

### 3.2 活动与商品管理（招商端）

#### 招商组长操作

- **活动绑定**：输入活动 ID 或从列表选择，触发系统同步商品列表
- **商品分配**：将活动商品批量分配给招商人员，支持按商品数量均分或指定分配

#### 招商操作

- **待审核列表**：查看分配给自己的商品，支持按活动、时间筛选
- **审核通过**：补充以下信息后上架到商品库
  - 专属价说明、发货信息
  - 商品卖点、推广话术
  - 是否支持投流
  - 奖励说明、参与要求
  - 活动时间、手卡（素材图片/文档）
- **审核拒绝**：填写拒绝原因，商品退回，渠道不可见
- **寄样要求配置**：可覆盖全局默认标准（如近30天销售额≥3W、达人等级≥LV1）

#### 业绩归属

- 招商业绩按活动归属
- 订单通过 `商品 → 活动 → 招商负责人` 追溯业绩
- **注意**：独家商家规则优先于上述归属规则

### 3.3 达人 CRM

**定位**：渠道管理合作达人的核心工具。

#### 功能清单

- **达人添加**：渠道手动输入达人抖音号或分享链接
- **达人信息自动补全**：
  - 系统自动触发爬虫任务，采集达人基本信息
  - 自动填充：昵称、粉丝数、获赞数、关注数、作品数、IP属地
  - 采集失败时支持手动填写兜底
- **公海/私海**：
  - 未认领达人在公海，全员可见
  - 认领后进入私海，仅认领人可见（保护期内）
- **达人认领**：
  - 渠道从公海认领达人
  - **多人可同时认领同一达人，联系方式均可见**
  - 认领后进入 30 天保护期，保护期内仅认领人可见
  - 保护期内无产出，到期自动释放回公海
- **达人列表**：展示昵称、UID、粉丝数、等级、月销、合作次数、总产出、认领人、认领时间
- **数据刷新**：
  - 支持手动刷新（点击按钮重新爬取）
  - 定时任务每周批量更新活跃达人数据

#### 寄样限制规则

- **限制对象**：同一渠道商务 + 同一达人 + 同一商品
- **限制范围**：渠道商务级别，7 天内不允许该渠道商务对该达人+该商品发起新的寄样申请
- **例外情况**：被拒绝的申请不受 7 天限制
- **不受限角色**：组长、管理员不受此限制
- **可配置性**：限制规则（天数、开关）整合到管理员权限设置中，支持动态调整

### 3.4 独家达人机制

#### 触发条件（两个条件需同时满足）

| 条件       | 说明                                                       |
| ---------- | ---------------------------------------------------------- |
| 服务费占比 | 该达人产生的服务费 ≥ **本团该达人总服务费的 70%**（含70%） |
| 月寄样数量 | 该渠道给该达人的月寄样数量 ≥ 10 个                         |

寄样不满足团队长可审核成为独家达人

#### 寄样统计口径

- 寄样状态为 **“待发货”及之后**（已通过审核，寄样流程已启动即算）

#### 生效/退出机制

- 当月满足条件 → 整月享受独家资格
- 当月不满足条件 → **下月初生效**（失去独家资格）

#### 独家达人效果

- 该达人所有后续订单的服务费收益，全部归该渠道所有（无论订单通过谁的推广链接产生）

### 3.5 寄样台

**定位**：渠道申请寄样、招商审核、运营发货的全流程管理。

#### 功能清单

- **申请寄样**：渠道选择达人、商品，填写收货地址
- **申请校验**：
  - 达人是否符合商品寄样标准（基于爬虫采集的达人数据判断）
  - 若不符合，弹窗提醒且必须填写申请原因
  - 寄样限制校验：同一渠道+同一达人+同一商品，7 天内不重复申请
- **审核**：对应商品的招商审核（通过/拒绝），拒绝需填写原因
- **发货**：运营录入物流单号
  - V1.0：手动录入单号
  - V2.0：对接第三方物流 API（快递鸟/快递100）自动更新状态
- **交作业判断**：该达人通过该渠道的链接产生该商品的订单（≥1 单）即视为完成

#### 状态流转

| 状态     | 说明            | 触发操作                      |
| -------- | --------------- | ----------------------------- |
| 待审核   | 渠道已提交申请  | 招商审核                      |
| 待发货   | 招商已审核通过  | 运营录入物流单号              |
| 快递中   | 商家已发货      | 物流 API 更新或手动确认       |
| 待交作业 | 达人已签收      | 等待该渠道该商品出单          |
| 已完成   | 已出单（≥1 单） | 系统自动判断订单              |
| 已拒绝   | 招商拒绝申请    | 渠道可重新申请（不受7天限制） |
| 已关闭   | 超时30天未出单  | 系统自动关闭                  |

#### 筛选与批量操作

- 按状态、商品、达人、申请时间、招商负责人筛选
- 批量审核、批量导出（用于同步商家）

### 3.6 数据平台

**定位**：核心业绩看板，支撑管理层决策。

#### 功能清单

- **核心指标卡片**：
  - 总订单数、订单总额
  - 服务费收入、技术服务费
  - 服务费支出、服务费收益
  - 招商提成、渠道提成
  - 毛利
- **多维度筛选**：时间范围、订单状态、商品、达人、招商、渠道、活动
- **订单明细表**：展示订单详情，支持自定义表头、排序
- **数据导出**：仅管理员和组长可导出，普通成员仅看板查看

#### 业绩归属规则

1. **渠道业绩**：通过订单中的 `pick_source` 参数归因（由 `/buyin/instPickSourceConvert` 接口生成）
2. **招商业绩**：按商品关联的活动归属（独家商家规则优先）
3. **独家规则优先**：独家达人/商家 → 覆盖默认归属逻辑

#### 提成公式

```
服务费收益 = 订单金额 × 服务费率（以抖音接口返回的 service_fee 为准）
招商提成 = 服务费收益 × 招商提成比例
渠道提成 = 服务费收益 × 渠道提成比例
```

#### 业绩计算时间

- **实际提成/结算**：以订单**结算时间**为准（使用接口返回的 `settle_time`）
- **预估/看板数据**：以订单**创建时间**为准（用于实时监控，包含未结算订单）

#### 提成配置

- 支持差异化配置（按活动、商品、员工等维度）
- V0.5/V1.0 使用全局统一配置，数据库预留扩展字段
- V2.0 支持按活动/商品单独配置提成比例

### 3.7 招商独家商家机制

#### 触发条件

- 商家产生的服务费 ≥ 该招商个人总服务费的 **70%**（含70%）

#### 生效/退出机制

- 当月满足条件 → 整月享受独家资格
- 当月不满足条件 → **下月初生效**（失去独家资格）

#### 独家商家效果

- 该商家所有后续订单的服务费收益，全部归该招商所有（无论订单是通过哪个招商审核上架的）

#### 订单业绩归属规则（优先级从高到低）

1. 获取订单的商家 ID
2. 查询该商家是否有“生效中”的独家招商
3. 如果有 → 业绩归该独家招商
4. 如果没有 → 按原规则：商品 → 活动 → 招商负责人

#### 商家库建立

- 新建简单的商家信息表
- **数据来源**：通过订单同步接口，从订单详情中提取商家 ID 和商家名称，**被动填充**，无需主动同步全量商家

### 3.8 系统管理

#### 功能清单

- **员工管理**：添加/删除员工，分配角色，重置密码
- **角色权限**：定义各角色可见菜单、数据范围、操作权限
- **规则配置**：动态配置寄样限制天数、独家阈值等规则（界面化配置，无需改代码）
- **提成规则配置**：全局配置招商/渠道提成比例
- **寄样全局默认标准**：配置默认寄样要求（近30天销售额≥3W、达人等级≥LV1）
- **推广链接规则配置**：配置转链接口参数（`pick_extra` 生成规则）
- **操作日志**：记录关键操作（登录、上架、审核、导出），保留 90 天

## 四、数据库设计

### 4.1 设计原则

- 订单表按月分区，提升查询性能
- 关键字段建立索引（订单时间、渠道ID、商家ID、商品ID）
- 预留差异化配置扩展字段（JSONB 类型）
- 使用 UUID 作为主键（便于未来分库分表扩展）

### 4.2 核心表结构（逻辑设计）

| 表名                  | 说明               | 关键字段                                                     | 分区策略     |
| --------------------- | ------------------ | ------------------------------------------------------------ | ------------ |
| `users`               | 用户表             | 用户名、角色、组别                                           | -            |
| `roles`               | 角色表             | 角色名、权限配置（JSONB）                                    | -            |
| `system_configs`      | 系统配置表         | 配置键、配置值                                               | -            |
| `activities`          | 活动表             | 活动ID（抖音）、活动名称、时间范围                           | -            |
| `products`            | 商品表             | 商品ID、活动ID、招商负责人、补充信息（JSONB）                | -            |
| `talents`             | 达人表             | 昵称、UID、粉丝数、获赞数、作品数、IP属地、采集时间          | -            |
| `talent_claims`       | 达人认领记录表     | 达人ID、认领人、保护期截止时间                               | -            |
| `exclusive_talents`   | 独家达人记录表     | 达人ID、渠道ID、生效月份                                     | -            |
| `exclusive_merchants` | 独家商家记录表     | 商家ID、招商ID、生效月份                                     | -            |
| `pick_source_mapping` | 推广链接归因映射表 | pick_source、渠道ID、创建时间                                | -            |
| `sample_requests`     | 寄样申请表         | 达人ID、商品ID、申请人、状态、物流单号                       | -            |
| `orders`              | 订单表             | 订单ID、商品ID、商家ID、渠道ID、pick_source、金额、服务费、时间 | **按月分区** |
| `commissions`         | 提成配置表         | 用户ID、提成类型、比例、有效期                               | -            |
| `logs`                | 操作日志表         | 用户ID、操作类型、目标、详情（JSONB）                        | 按月分区     |

### 4.3 索引设计要点

| 表                  | 索引字段                 | 说明                 |
| ------------------- | ------------------------ | -------------------- |
| orders              | create_time, settle_time | 时间范围查询         |
| orders              | pick_source              | 渠道业绩归属（核心） |
| orders              | merchant_id              | 独家商家判断         |
| orders              | product_id               | 招商业绩归属         |
| products            | activity_id              | 活动商品查询         |
| products            | assignee_id              | 招商负责人筛选       |
| talents             | uid                      | 达人唯一标识         |
| sample_requests     | user_id, status          | 我的申请/待审核      |

### 5.3 推广链接转链与归因方案（核心）

#### 接口信息

| 项目           | 值                                                  |
| -------------- | --------------------------------------------------- |
| **接口名称**   | `buyin.instPickSourceConvert`                       |
| **功能**       | 商品选品来源转链                                    |
| **核心作用**   | 将普通商品链接转换为带 `pick_source` 参数的推广链接 |
| **归因有效期** | `pick_source` 参数有效期为 **3个月**                |

#### 请求参数

| 参数名        | 类型   | 必填 | 说明                                               |
| ------------- | ------ | ---- | -------------------------------------------------- |
| `product_url` | String | 是   | 商品链接                                           |
| `pick_extra`  | String | 否   | 自定义参数（只允许数字、字母和下划线，限制长度20） |

#### 响应参数

| 参数名        | 类型   | 说明                                      |
| ------------- | ------ | ----------------------------------------- |
| `product_url` | String | 转链后的商品链接，带有 `pick_source` 参数 |

#### 完整归因流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        推广链接生成环节                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  渠道商务在商品库点击"一键复制"                                  │
│       ↓                                                         │
│  系统调用 /buyin/instPickSourceConvert                          │
│       ├── product_url: 商品原始链接                             │
│       └── pick_extra: channel_{user_id}（限制20字符）           │
│       ↓                                                         │
│  抖音返回转链链接                                                │
│       └── product_url: ...&pick_source={抖音生成的归因码}       │
│       ↓                                                         │
│  系统存储映射关系：pick_source → channel_{user_id}              │
│       ↓                                                         │
│  渠道商务复制链接发给达人                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                        订单归因环节                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户通过链接下单                                                │
│       ↓                                                         │
│  抖音订单接口返回订单数据                                        │
│       └── 包含 pick_source 参数                                 │
│       ↓                                                         │
│  系统查询映射表                                                  │
│       └── pick_source → channel_{user_id}                       │
│       ↓                                                         │
│  业绩归属该渠道商务                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.4 订单同步策略

**同步方式**：Celery 定时任务

**同步频率**：每 5-10 分钟执行一次

**同步逻辑**：
1. 记录上次同步时间
2. 调用订单接口，按结算时间范围查询增量订单
3. 支持分页，循环拉取全部订单
4. 提取订单中的 `pick_source` 参数，通过映射表反查渠道 ID
5. 将订单数据写入本地 `orders` 分区表
6. 更新同步时间戳

**异常处理**：
- 接口超时：重试 3 次，间隔递增
- Token 过期：SDK 自动刷新后重试
- 频率限制：退避等待后重试

### 5.5 接口调用限制

| 限制项       | 建议值             | 说明             |
| ------------ | ------------------ | ---------------- |
| 转链接口调用 | 按需，单用户单商品 | 渠道点击时才调用 |
| 订单同步频率 | 5-10 分钟          | 避免频繁调用     |
| 爬虫请求间隔 | 3-6 秒             | 避免 IP 被封     |

## 六、达人信息爬虫方案

### 6.1 可行性确认 ✅

**结论：通过爬虫获取抖音达人基本信息完全可行，且有大量成熟案例。**

| 数据源         | 可实现数据                                   | 技术难度 | 稳定性 |

| 抖音网页版     | 昵称、粉丝数、获赞数、作品数、抖音号、IP属地 | 低       | 较高   |
| 抖音移动端接口 | 更完整数据（含等级、近30天销售额等）         | 高       | 一般   |
| 第三方聚合平台 | 达人详细分析数据                             | 中       | 需付费 |

> **推荐方案**：优先使用抖音网页版爬虫，技术门槛低、稳定性好、满足基础信息需求。

### 6.3 技术实现方案

#### 方案选型：Python + Requests（自研）

**技术栈**：Python + requests + fake-useragent

**核心实现逻辑**：

1. **接口分析**：抖音网页版存在公开接口，返回用户主页数据的 JSON 格式
   - 请求 URL 格式：`https://www.douyin.com/aweme/v1/web/user/profile/?sec_uid={sec_uid}`
   - 返回字段包含：昵称、粉丝数、获赞数、作品数等

2. **关键参数获取**：
   - `sec_uid`：用户唯一标识，可从用户主页 URL 或达人分享链接提取
   - Cookie：需携带登录态 Cookie

3. **反爬策略**：
   - 随机 User-Agent 轮换
   - 请求间隔 3-6 秒
   - 可选配置代理 IP 池

#### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                     达人信息采集流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  渠道录入达人抖音号/UID                                       │
│       ↓                                                     │
│  系统触发爬虫任务（Celery 异步）                              │
│       ↓                                                     │
│  爬虫模块执行：                                              │
│      ├── 构造请求 URL（用户主页 API）                        │
│      ├── 携带 Cookie 和随机 User-Agent                      │
│      ├── 发送请求获取 JSON 数据                              │
│      └── 解析并提取目标字段                                   │
│       ↓                                                     │
│  数据存入 talents 表                                        │
│       ↓                                                     │
│  更新采集时间和数据状态                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 采集频率策略

| 场景         | 频率     | 说明                     |
| ------------ | -------- | ------------------------ |
| 首次添加达人 | 立即采集 | 渠道录入后自动触发       |
| 定时刷新     | 每周一次 | 定时任务批量更新活跃达人 |
| 手动刷新     | 按需     | 渠道可手动点击刷新       |

### 6.4 兜底方案

爬虫采集失败时，系统提供手动录入兜底：

| 场景        | 处理方式                                |
| ----------- | --------------------------------------- |
| 爬虫超时    | 提示“采集超时”，允许手动填写            |

| IP 被封     | 自动切换代理 IP（V2.0），或提示稍后重试 |
| Cookie 过期 | 系统告警通知管理员更新                  |
