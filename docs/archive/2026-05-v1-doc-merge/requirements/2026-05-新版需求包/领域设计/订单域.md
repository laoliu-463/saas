## 订单域 - 完整设计方案（V1.6 正式版）

------

## 文档说明

本文档基于骨架设计 V1.4 和需求文档 V2.2 设计。

**定位**：管理订单数据的同步、存储、默认归因，为业绩域提供数据基础。

**原则**：

- 订单域**只存事实**（含接口返回的预估/结算双轨金额），不算提成、不应用独家规则
- 金额以抖音 `buyin.colonelMultiSettlementOrders` 返回为准，不做业务推导
- 独家覆盖、双轨提成/毛利由**业绩域**计算

**版本更新**：

- V1.5 修正：媒介改为渠道（pick_source 映射）
- V1.6 **新增：预估/有效（结算）双轨金额字段**；对齐看板「成交/预估」与「结算」两行；修正筛选参数命名；同步事件携带双轨金额

**版本状态**：✅ 可进入开发（首 sync 需校验 API 字段名，见 2.3 落地说明）

**关联文档**：`重要/抖音开放平台Python_SDK接口文档.md` §4.2.6、§9.4

## 一、领域定位

### 1.1 职责

| 职责         | 说明                            |
| ------------ | ------------------------------- |
| 同步抖音订单 | 定时从抖音API拉取订单数据       |
| 存储订单数据 | 订单表按月分区，支持高效查询    |
| 存储归因映射 | 存储pick_source与渠道的映射关系 |
| 提供订单查询 | 供业绩域、寄样域、前端查询订单  |
| 发出订单事件 | 订单同步后通知其他领域          |

### 1.2 边界

**管**：

- 订单数据的同步、存储、更新
- pick_source 归因映射存储
- 订单状态跟踪

**不管**：

- 独家规则判断（业绩域管）
- 业绩计算（业绩域管）
- 寄样完成判断（寄样域订阅订单事件自行判断）
- 提成配置（配置域管）

### 1.3 对外提供的能力

| 操作             | 调用方      | 说明                         |
| ---------------- | ----------- | ---------------------------- |
| 查询订单列表     | 前端/业绩域 | 按时间、渠道、商品等筛选     |
| 查询订单详情     | 前端        | 订单详细信息                 |
| 同步订单         | 定时任务    | 自动触发                     |
| 手动同步         | 管理员      | 手动触发同步                 |
| 获取订单默认归因 | 业绩域      | 获取订单的默认渠道和默认招商 |

### 1.4 发出的事件

| 事件       | 触发时机       | 携带数据                                                     |
| ---------- | -------------- | ------------------------------------------------------------ |
| 订单已同步 | 新订单插入或更新完成 | 见 §2.8 事件载荷（含双轨金额 + 默认归因） |
| 订单状态已变更 | 待结算→已结算/退款等 | 订单ID、旧状态、新状态、有效金额是否更新 |

### 1.5 订阅的事件

| 事件 | 来源 | 处理                       |
| ---- | ---- | -------------------------- |
| 无   | -    | 订单域不依赖其他领域的事件 |

## 二、核心概念

### 2.1 订单数据来源

| 项目     | 说明                                                     |
| -------- | -------------------------------------------------------- |
| API接口  | `buyin.colonelMultiSettlementOrders`（团长多重结算订单） |
| 同步频率 | 每5-10分钟执行一次                                       |
| 同步方式 | Celery定时任务，增量同步                                 |
| 数据范围 | 增量拉取；定时任务默认按**结算时间**窗口，看板查询另支持**付款时间**（`time_type`） |
| SDK 类   | `BuyinColonelMultiSettlementOrdersRequest`                 |

### 2.2 双轨金额（核心，对齐数据平台看板）

同一订单在接口中同时存在**两套金额**，对应看板每一指标下的两行：

| 看板展示行 | 业务含义           | 订单域存储前缀   | 典型来源                         |
| ---------- | ------------------ | ---------------- | -------------------------------- |
| 成交 / 预估 | 付款后统计口径     | `estimate_*` / `pay_*` | 接口预估类字段，待结算也有值     |
| 结算       | 官方结算到账口径   | `effective_*` / `settle_*` | 接口结算类字段，未结算时常为 0 |

**原则**：

- 不是用「付款时间 vs 结算时间」在同一金额上切两刀，而是**两套独立金额字段**
- 时间筛选（付款/结算）决定**哪些订单进入 cohort**；进入后分别 SUM 预估列、结算列
- 订单域原样落库；业绩域对两套金额各算一遍提成（见业绩域 §2.4）

### 2.3 订单状态

| 状态   | 说明                     | 抖音对应状态 |
| ------ | ------------------------ | ------------ |
| 待结算 | 订单已产生，未到结算时间 | 待结算       |
| 已结算 | 订单已结算，可计算业绩   | 已结算       |
| 已退款 | 订单已退款               | 已退款       |
| 已失效 | 订单无效                 | 已失效       |

### 2.4 抖音 API → 系统字段映射（`buyin.colonelMultiSettlementOrders`）

> 接口路径：`/buyin/colonelMultiSettlementOrders`  
> SDK：`BuyinColonelMultiSettlementOrdersRequest`  
> **金额单位**：接口一般为**分**，入库统一转为**元**（÷100）。  
> **落地校验**：首次联调将 `response.data.orders[0]` 原始 JSON 落日志，核对下列「抖音响应字段」列；若与主字段不一致，改用「常见别名」列并更新 SDK 文档 §9.4。

#### 2.4.1 身份与归因

| 系统字段 | 抖音响应字段（主） | 常见别名 | 说明 |
| -------- | ------------------ | -------- | ---- |
| `order_id` | `order_id` | - | 主键 |
| `activity_id` | `activity_id` | `ads_activity_id`, `ads_promotion_id` | 招商业绩追溯 |
| `product_id` | `product_id` | `item_id` | |
| `product_name` | `product_name` | `item_title` | 冗余展示 |
| `talent_id` | `author_buyin_id` | `author_id`, `talent_id`, `author_short_id` | 达人/推广者 |
| `talent_nickname` | `author_account` | `author_name`, `nickname` | |
| `partner_id` | `shop_id` | `merchant_id`, `store_id` | **合作方/店铺**（独家商家维度） |
| `partner_name` | `shop_name` | `merchant_name`, `store_name` | |
| `shop_id` | `shop_id` | - | 店铺（可与合作方相同） |
| `colonel_id` | `colonel_buyin_id` | `institution_id`, `colonel_id` | **团长**（机构侧，非内部招商） |

| `recruiter_attribution` | （派生） | - | `activity_owner` / `product_owner` / `unattributed` |

#### 2.4.2 双轨金额（对齐看板）

| 系统字段 | 看板指标 | 抖音响应字段（主） | 常见别名 | 说明 |
| -------- | -------- | ------------------ | -------- | ---- |
| `pay_amount` | 订单额·成交 | `pay_goods_amount` | `order_amount`, `total_pay_amount` | 付款/成交金额 |
| `settle_amount` | 订单额·结算 | `settled_goods_amount` | `settle_goods_amount`, `settle_amount` | 结算/有效订单额 |
| `estimate_service_fee` | 服务费收入·预估 | `estimated_commission` | `estimated_service_fee`, `estimate_institution_commission` | **预估服务费**（统计收入） |
| `effective_service_fee` | 服务费收入·结算 | `commission` | `real_commission`, `settled_commission`, `institution_commission` | **有效服务费**（到账） |
| `estimate_tech_service_fee` | 技术服务费·预估 | `estimated_tech_service_fee` | `estimate_platform_service_fee` | |
| `effective_tech_service_fee` | 技术服务费·结算 | `tech_service_fee` | `settled_tech_service_fee`, `platform_service_fee` | |
| `service_fee_rate` | - | `service_rate` | `commission_rate` | 服务费率（如有） |
| `commission_rate` | - | `commission_rate` | `cos_ratio` | 商品佣金率（如有） |

**待结算单**：`effective_*` 通常为 0 或空；`estimate_*` 有值。已结算后再次同步应**更新** `effective_*`，保留 `estimate_*` 快照。

#### 2.4.3 时间与状态

| 系统字段 | 抖音响应字段（主） | 常见别名 | 说明 |
| -------- | ------------------ | -------- | ---- |
| `order_status` | （映射） | - | 由 `flow_point` / `order_status` 映射为：待结算/已结算/已退款/已失效 |
| `flow_point` | `flow_point` | `order_status`, `status` | 原始状态码，建议冗余存储 |
| `pay_time` | `pay_success_time` | `pay_time`, `create_time` | 付款时间；看板「付款时间」筛选 |
| `settle_time` | `settle_time` | `settled_time` | 结算时间；看板「结算时间」筛选 |
| `delivery_time` | `confirm_receipt_time` | `receipt_time` | 收货时间（如有） |
| `expire_time` | `refund_time` | `invalid_time` | 失效/退款时间（如有） |
| `order_create_time` | `create_time` | `order_time` | 订单创建时间 |

#### 2.4.4 查询入参 `time_type`（与看板时间筛选一致）

| time_type（待官方确认） | 含义 | 看板对应 |
| ----------------------- | ---- | -------- |
| `1` 或 `pay` | 按付款时间筛订单 | 时间筛选 = **付款时间** |
| `2` 或 `settle` | 按结算时间筛订单 | 时间筛选 = **结算时间** |

定时增量同步建议：`time_type=结算`，`start_time`/`end_time` 为上次同步点至今。

### 2.5 归因逻辑（核心）

**核心原则**：订单数据本身已包含完整的归因信息，不需要逐步查找。

| 归因维度 | 默认归属                | 依据                                     |
| -------- | ----------------------- | ---------------------------------------- |
| **渠道** | 根据 `pick_source` 映射 | 订单中的pick_source → 映射表 → 渠道ID    |
| **招商** | 根据 `activity_id` + `product_id` 追溯 | 商品域：活动商品关联「实际负责人」优先，否则活动默认负责人 |

**关键说明**：

- 订单API返回的数据已经明确了订单来源（哪个活动、哪个渠道的链接）
- 订单域只负责存储**默认归因**（订单本身的数据）
- 独家规则的覆盖由**业绩域**在计算业绩时处理

### 2.6 独家规则覆盖（业绩域负责）

| 规则     | 作用维度 | 覆盖逻辑                 | 负责领域 |
| -------- | -------- | ------------------------ | -------- |
| 独家达人 | 渠道     | 将默认渠道替换为独家渠道 | 业绩域   |
| 独家商家 | 招商     | 将默认招商替换为独家招商 | 业绩域   |

**注意**：

- 独家达人和独家商家是**独立维度**，互不干扰
- 两者可以同时生效（渠道归独家渠道，招商归独家招商）
- 订单域只存储默认归因，不应用独家规则

### 2.7 归因方式枚举（订单域存储）

| 归因方式         | 说明                    | 存储位置 |
| ---------------- | ----------------------- | -------- |
| `pick_source`    | 通过pick_source映射归因 | 默认渠道 |
| `activity_owner` | 通过活动负责人归因      | 默认招商 |
| `unattributed`   | 无法归因                | -        |

**独家规则产生的归因方式由业绩域记录，订单域不存储。**

## 三、订单同步策略

### 3.1 同步流程

```plain
定时任务触发（每5-10分钟）
      ↓
获取上次同步时间戳
      ↓
调用 BuyinColonelMultiSettlementOrders
      ├── time_type = 结算时间（增量同步）
      ├── start_time / end_time = 上次同步点 ~ 当前
      ├── size + cursor 分页
      └── 失败重试 3 次
      ↓
遍历 orders[]（每条为一单）
      ↓
对每个订单：
  ├── 按 order_id 去重：存在则 UPDATE，不存在则 INSERT
  ├── 映射双轨金额（§2.4.2）：estimate_* + effective_*，分→元
  ├── 映射时间、flow_point → order_status
  ├── 默认渠道 = pick_source → 映射表
  ├── 默认招商 = 商品域 get_recruiter(activity_id, product_id)
  ├── 写入 orders 表（不计算提成）
  └── 发出「订单已同步」事件（§2.8 载荷）
      ↓
若 effective_service_fee 从 0 变为有值 → 可额外发「订单状态已变更」
      ↓
更新同步时间戳
```

### 3.2 同步参数

| 参数     | 说明               | 建议值                  |
| -------- | ------------------ | ----------------------- |
| 同步频率 | 定时任务执行间隔   | 5-10分钟                |
| 时间范围 | 每次查询的时间窗口 | 上次同步时间 ~ 当前时间 |
| 分页大小 | 每页订单数量       | 100条                   |
| 重试次数 | 失败重试           | 3次                     |
| 重试间隔 | 递增间隔           | 5s, 10s, 30s            |

### 3.3 订单去重

| 去重字段 | 说明                                     |
| -------- | ---------------------------------------- |
| 订单ID   | 抖音订单唯一标识                         |
| 处理方式 | 存在则更新（状态可能变化），不存在则插入 |

### 3.4 订单更新策略

| 场景         | 处理                           |
| ------------ | ------------------------------ |
| 新订单       | 插入新记录                     |
| 订单状态变更 | 更新状态字段，发出状态变更事件 |
| 退款/售后    | 更新订单状态和金额             |
