## 分析模块 - 完整设计方案（V1.0 正式版）

------

## 文档说明

本文档基于骨架设计 V1.5（第六章「数据看板方案」）和需求文档 V2.2 设计。

**定位**：独立分析模块，订阅各领域事件，维护汇总表，为数据平台看板提供高性能只读查询。

**原则**：

- 不修改业务领域数据，不重算业绩归属规则
- 以「业绩已计算」事件为主驱动，避免重复计算
- 前端看板默认查汇总表，明细下钻可走业务域接口

**版本状态**：✅ 所有问题已确认，可进入开发

## 一、模块定位

### 1.1 职责

| 职责 | 说明 |
| ---- | ---- |
| 订阅领域事件 | 异步消费，更新汇总表 |
| 维护日/月汇总表 | 业绩、寄样、商品等多维聚合 |
| 提供看板 API | 指标卡片、趋势、排行榜 |
| 双时间口径 | 结算时间（实际）与创建时间（预估） |
| 权限过滤 | 调用用户域 data_scope 后过滤结果 |

### 1.2 边界

**管**：

- 汇总表 schema 与增量更新逻辑
- 看板查询接口

**不管**：

- 单笔订单业绩计算（业绩域）
- 订单/寄样等业务 CRUD
- 用户认证（用户域）

### 1.3 对外提供的能力

| 操作 | 调用方 | 说明 |
| ---- | ------ | ---- |
| 看板指标卡片 | 数据平台 | 总订单、服务费、提成、毛利等 |
| 趋势图数据 | 数据平台 | 按日/周/月趋势 |
| 排行榜 | 数据平台 | 渠道/招商 Top N |
| 寄样统计 | 数据平台/管理端 | 申请量、完成率 |
| 商品统计 | 管理端 | 上架数、推广中等 |

### 1.4 发出的事件

| 事件 | 触发时机 | 说明 |
| ---- | -------- | ---- |
| 无 | - | V1.0 分析模块为末端消费者，不向外发业务事件 |

### 1.5 订阅的事件

| 事件 | 来源 | 处理 |
| ---- | ---- | ---- |
| 业绩已计算 | 业绩域 | **主驱动**：更新日业绩汇总、排行 |
| 订单已同步 | 订单域 | 更新实时预估汇总（按 create_time） |
| 寄样已提交 | 寄样域 | 寄样申请量 +1 |
| 寄样已通过 | 寄样域 | 通过量 +1 |
| 寄样已完成 | 寄样域 | 完成量 +1，完成率重算 |
| 商品已上架 | 商品域 | 可推广商品数 +1 |
| 商品已隐藏 | 商品域 | 可推广商品数 -1 |
| 活动同步完成 | 商品域 | 活动数统计刷新 |
| 达人已认领 | 达人域 | 私海达人数（可选 V1.0） |

## 二、架构

```plain
商品域 ──→ 事件 ──→
达人域 ──→ 事件 ──→
订单域 ──→ 事件 ──→  分析模块 Consumer（Celery / 内存队列）
寄样域 ──→ 事件 ──→       ↓
业绩域 ──→ 事件 ──→   汇总表（PostgreSQL）
                            ↓
                      看板 API（只读）
                            ↓
                      数据平台前端
```

| 组件 | 说明 |
| ---- | ---- |
| Event Consumer | 幂等消费，按 event_id 去重 |
| Aggregator | 增量 UPDATE 汇总行，非全表扫描 |
| Query API | 只读，带缓存（Redis 60s，可选） |

## 三、核心概念

### 3.1 双时间口径

| 口径 | 字段 | 数据来源 | 用途 |
| ---- | ---- | -------- | ---- |
| 实际/结算 | settle_time | 业绩已计算（is_valid=true） | 正式业绩、提成统计 |
| 预估/实时 | order_create_time | 订单已同步（含未结算） | 实时监控、预估看板 |

**两套汇总表分离**，避免混用：

- `agg_daily_performance_settle`
- `agg_daily_performance_create`

### 3.2 与业绩域分工

| 需求 | 使用模块 |
| ---- | -------- |
| 订单明细 + 准确提成 | 业绩域 `list_performance` |
| 管理层看板卡片/趋势 | 分析模块汇总表 |
| 单笔订单详情页 | 订单域 + 业绩域 |

分析模块**不重新实现**独家归属规则，只消费业绩域计算结果。

### 3.3 幂等与补偿

| 机制 | 说明 |
| ---- | ---- |
| 事件去重表 | processed_events(event_id) |
| 失败重试 | Celery 最多 3 次 |
| 日终对账 | 每日 04:00 用业绩域汇总校验 agg 表，差异告警 |
| 手动重建 | 管理员按日期范围重建汇总（管理接口） |

## 四、汇总表设计

### 4.1 日业绩汇总（结算口径）`agg_daily_performance_settle`

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| 日期 | date | 分区键 |
| 渠道ID | string | 可空（全站汇总时 NULL） |
| 招商ID | string | 可空 |
| 活动ID | string | 可空 |
| 商品ID | string | 可空 |
| 订单数 | int | |
| 订单金额 | decimal | |
| 服务费收入 | decimal | |
| 技术服务费 | decimal | |
| 招商提成 | decimal | |
| 渠道提成 | decimal | |
| 毛利 | decimal | |
| 更新时间 | datetime | |

**维度组合**：V1.0 至少支持 (date + channel_id)、(date + recruiter_id)、(date) 全站；活动/商品维度 V1.0 可选。

**主键**：(date, channel_id, recruiter_id, activity_id, product_id) 使用 COALESCE 哨兵值区分 NULL。

### 4.2 日业绩汇总（创建口径/预估）`agg_daily_performance_create`

结构同 4.1，数据来自「订单已同步」事件 + 订单金额/服务费字段；提成在订单结算后由「业绩已计算」回补或仅展示服务费不含提成（V1.0 预估卡片可只显示订单数/服务费收入）。

### 4.4 商品/活动快照 `agg_snapshot_products`

| 字段 | 说明 |

## 五、看板 API

### 5.1 指标卡片 `get_dashboard_cards`

**入参**：

```json
{
  "time_basis": "settle | create",
  "time_start": "2026-05-01",
  "time_end": "2026-05-17",
  "channel_id": "可选",
  "recruiter_id": "可选",
  "activity_id": "可选"
}
```

**出参**（与需求 3.6 对齐）：

| 指标 | 说明 |
| ---- | ---- |
| total_orders | 总订单数 |
| total_order_amount | 订单总额 |
| service_fee_income | 服务费收入 |
| tech_service_fee | 技术服务费 |
| service_fee_expense | 招商提成+渠道提成 |
| service_fee_profit | 服务费收益 |
| recruiter_commission | 招商提成 |
| channel_commission | 渠道提成 |
| gross_profit | 毛利 |

权限：请求前调用 `get_user_data_scope`，自动注入 channel_id/recruiter_id 过滤。

### 5.2 趋势 `get_dashboard_trend`

| 参数 | 说明 |
| ---- | ---- |
| metric | orders / service_fee / gross_profit 等 |
| granularity | day / week / month |
| time_basis | settle / create |

返回时间序列 `[{ date, value }]`。

### 5.3 排行榜 `get_dashboard_ranking`

| 参数 | 说明 |
| ---- | ---- |
| dimension | channel / recruiter |
| metric | service_fee / gross_profit / order_count |
| top_n | 默认 10 |
| time_range | 同卡片 |

### 5.4 寄样统计 `get_sample_stats`

| 指标 | 说明 |
| ---- | ---- |
| apply_count | 申请数 |
| complete_count | 完成数 |
| complete_rate | 完成率 |
| avg_days_to_complete | 平均交作业天数（V2.0） |

## 六、数据平台页面对接

### 6.1 页面结构（需求 3.6）

| 区域 | 数据来源 |
| ---- | -------- |
| 顶部指标卡片 | get_dashboard_cards |
| 趋势图 | get_dashboard_trend |

时间范围、订单状态、商品、达人、招商、渠道、活动——明细表走业务域；汇总卡片将筛选条件映射到 agg 表维度。

### 6.3 导出

| 功能 | 实现 |
| ---- | ---- |
| 看板数据导出 | V1.0 不做（需求：数据平台其他模块初期不导出） |
| 订单明细导出 | 订单域 export_orders，管理员/组长 |

## 七、Consumer 处理逻辑（业绩已计算）

```plain
收到 业绩已计算 事件
      ↓
若 is_valid = false → 对 settle 表做减量（冲正）
      ↓
否则按 settle_time 日期 + final_channel_id + final_recruiter_id
      ↓
UPSERT agg_daily_performance_settle 对应行：
  order_count += 1
  service_fee += ...
  recruiter_commission += ...
  channel_commission += ...
  gross_profit += ...
```
