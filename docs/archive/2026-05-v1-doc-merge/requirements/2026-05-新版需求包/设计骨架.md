## 抖音团长 SaaS 系统 - 总体骨架设计（V1.6）

------

## 文档说明

本文档是系统的**顶层架构设计**，只定义骨架，不包含细节。

**用途**：作为后续开发的纲领，保证系统可持续迭代。

**原则**：骨架定后不改，细节逐步填充。（注：V1.6 为全部领域设计完成后的对齐更新）

------

## 一、核心设计原则

| 原则           | 说明                             | 解决的问题           |
| -------------- | -------------------------------- | -------------------- |
| **领域划分**   | 按业务边界切分成7个独立领域      | 改一个功能不影响其他 |
| **事件驱动**   | 领域间通过事件通信，不直接调用   | 加新模块不改老代码   |
| **状态机**     | 每个领域内部用状态机管理核心实体 | 状态流转清晰         |
| **规则优先级** | 跨领域冲突时，用优先级表定胜负   | 业务逻辑不打架       |
| **分析独立**   | 数据看板用独立分析模块 + 汇总表  | 查询快、不影响业务   |

------

## 二、7个领域划分

| 领域       | 一句话职责                             | 核心实体                         |
| ---------- | -------------------------------------- | -------------------------------- |
| **商品域** | 管理活动、合作方、商品及它们的关联关系 | 活动、合作方、商品、活动商品关联 |
| **达人域** | 达人是谁、归谁管、独家资格、收货地址   | 达人、认领记录、独家达人         |
| **寄样域** | 样品怎么申请、怎么流转                 | 寄样申请                         |
| **订单域** | 订单从哪来、归谁（默认归因）           | 订单                             |
| **业绩域** | 谁赚了多少钱、凭什么规则（独家覆盖）   | 业绩记录、独家商家               |
| **用户域** | 谁能做什么、看什么数据                 | 用户、角色                       |
| **配置域** | 规则怎么配、怎么生效                   | 系统配置                         |

**独家资格分工（实现时勿重复）**：

| 类型     | 判定与落库     | 应用与计算         |
| -------- | -------------- | ------------------ |
| 独家达人 | 达人域         | 业绩域（覆盖渠道） |
| 独家商家 | 业绩域         | 业绩域（覆盖招商） |

**领域间关系**：互相独立，通过事件通信，不直接调用对方的数据库或接口。

------

## 三、领域间通信方式（事件驱动）

### 3.1 核心机制

- 一个领域发生重要的事 → 发出事件
- 其他领域想响应就订阅
- 发出者不知道谁在订阅

### 3.2 核心事件清单

| 事件             | 发出者 | 订阅者                   | 说明                              |
| ---------------- | ------ | ------------------------ | --------------------------------- |
| 订单已同步       | 订单域 | 寄样域、业绩域、分析模块 | 新订单到达，携带默认渠道/默认招商 |
| 寄样已提交       | 寄样域 | 业绩域                   | 新寄样申请                        |
| 寄样已通过       | 寄样域 | 业绩域                   | 寄样审核通过                      |
| 寄样已发货       | 寄样域 | -                        | 寄样已发货                        |
| 寄样已完成       | 寄样域 | 业绩域、达人域           | 寄样出单完成                      |
| 寄样已关闭       | 寄样域 | -                        | 超时关闭                          |
| 商品已上架       | 商品域 | 前端、寄样域             | 商品在商品库开始展示              |
| 商品已隐藏       | 商品域 | 前端                     | 商品从商品库移除                  |
| 商品负责人变更   | 商品域 | 业绩域                   | 个别品被重新分配                  |
| 活动同步完成     | 商品域 | 分析模块                 | 活动数据更新                      |
| 合作方同步完成   | 商品域 | 业绩域                   | 合作方数据更新                    |
| 达人已认领       | 达人域 | 前端                     | 达人被认领                        |
| 达人已释放       | 达人域 | 前端                     | 达人认领被释放                    |
| 达人标签更新     | 达人域 | 前端                     | 达人标签被修改                    |
| 达人地址更新     | 达人域 | 寄样域                   | 达人收货地址被修改                |
| 独家达人资格生效 | 达人域 | 业绩域                   | 达人获得独家资格                  |
| 独家达人资格失效 | 达人域 | 业绩域                   | 达人失去独家资格                  |
| 独家商家资格生效 | 业绩域 | -                        | 商家获得独家资格                  |
| 独家商家资格失效 | 业绩域 | -                        | 商家失去独家资格                  |

### 3.3 事件驱动说明

| 事件流向        | 说明                                                  |
| --------------- | ----------------------------------------------------- |
| 订单域 → 业绩域 | 订单同步后，业绩域根据订单默认归因 + 独家规则计算业绩 |
| 订单域 → 寄样域 | 订单同步后，寄样域检查是否有匹配的寄样申请，标记完成  |
| 达人域 → 业绩域 | 独家达人资格变化时，业绩域更新缓存                    |
| 商品域 → 业绩域 | 合作方/商品负责人变化时，业绩域更新缓存               |

### 3.4 扩展方式

加新领域时：

1. 订阅已有事件
2. 发出自己的新事件
3. 不修改已有领域

------

## 四、领域内部组织方式（状态机 + 对外接口）

### 4.1 每个领域的标准结构

| 组成部分     | 说明                             |
| ------------ | -------------------------------- |
| 对外查询接口 | 给前端和其他领域调用             |
| 对外命令接口 | 接收操作请求（创建、更新、删除） |
| 发出的事件   | 状态变化时通知其他领域           |
| 订阅的事件   | 响应其他领域的变化               |
| 内部状态机   | 管理核心实体的生命周期           |

### 4.2 核心实体状态机（简版）

| 实体         | 状态流转                                                     |
| ------------ | ------------------------------------------------------------ |
| 活动商品关联 | 待补充 → 可推广 →（展示中 / 已隐藏）                         |
| 达人认领记录 | 保护中 →（已释放）                                           |
| 寄样申请     | 待审核 → 待发货 → 快递中 → 待交作业 → 已完成 / 已拒绝 / 已关闭 |
| 订单         | 待结算 → 已结算 / 已退款 / 已失效                            |

### 4.3 扩展方式

加新状态时：只改本领域内部状态机，不影响其他领域。

------

## 五、规则优先级（冲突处理）

### 5.1 业绩归属规则（由业绩域执行）

订单业绩归属的最终确定规则（优先级从高到低）：

| 优先级 | 规则     | 作用维度  | 说明               |
| ------ | -------- | --------- | ------------------ |
| 1      | 独家达人 | 渠道      | 覆盖订单的默认渠道 |
| 2      | 独家商家 | 招商      | 覆盖订单的默认招商 |
| 3      | 默认归因 | 渠道/招商 | 订单域存储的默认值 |

**注意**：独家达人和独家商家是**独立维度**，互不干扰，可以同时生效。

### 5.2 寄样申请限制优先级（高到低）

| 优先级 | 检查项                         | 不通过处理     |
| ------ | ------------------------------ | -------------- |
| 1      | 商品是否已上架（商品库展示中） | 拒绝           |
| 2      | 7天内是否重复申请              | 拒绝           |
| 3      | 申请人是否是组长/管理员        | 跳过后续       |
| 4      | 达人是否符合寄样门槛（达人域） | 警告，允许申请 |

### 5.3 数据可见范围优先级（高到低）

| 优先级 | 规则                 |
| ------ | -------------------- |
| 1      | 管理员可见全部数据   |
| 2      | 组长可见本组数据     |
| 3      | 普通成员仅见自己数据 |

### 5.4 扩展方式

加新规则时：插入优先级表的对应位置，不影响其他规则。

------

## 六、数据看板方案（跨领域查询）

### 6.1 核心机制

- 设立独立的**分析模块**
- 分析模块订阅各领域事件
- 实时更新**汇总表**
- 前端查询只读汇总表

### 6.2 架构

```plain
商品域 ──→ 事件 ──→
达人域 ──→ 事件 ──→ 分析模块 ──→ 汇总表 ──→ 前端看板
订单域 ──→ 事件 ──→
寄样域 ──→ 事件 ──→
业绩域 ──→ 事件 ──→
```

### 6.3 扩展方式

加新看板指标时：

1. 汇总表加字段
2. 分析模块订阅新事件或已有事件
3. 不修改业务领域

## 七、核心业务链（验证骨架）

### 链1：渠道侧

```plain
达人域（认领达人）→ 多人可同时认领，各自保护期
    ↓
达人域（维护收货地址、标签）→ 渠道保存达人收货地址和标签
    ↓
商品域（商品库选品）→ 支持筛选、排序、置顶
    ↓
商品域（复制讲解）→ 调用转链接口，生成带 pick_source 的链接
    ↓
寄样域（快速寄样）→ 自动带入达人地址，选择规格、备注
    ↓
寄样域（审核/发货/物流）→ 招商审核、运营发货
    ↓
订单域（订单同步）→ 存储订单默认归因（pick_source → 渠道，活动ID → 招商）
    ↓
业绩域（业绩计算）→ 应用独家达人/独家商家规则，计算最终业绩和提成
```

### 链2：招商侧

```plain
商品域（API同步活动、合作方、商品）
    ↓
商品域（活动下绑定商品，关联合作方）
    ↓
商品域（补充信息 + 确认上架）→ 配置标签
    ↓
商品域（置顶）→ 负责的招商可置顶，上限10个，24小时
    ↓
寄样域（审核）→ 招商审核寄样申请，单次最多100条
    ↓
寄样域（导出）→ 导出寄样数据给商家
    ↓
订单域（订单同步）→ 默认招商归因
    ↓
业绩域（独家商家）→ 应用独家商家规则，覆盖默认招商
```

### 链3：管理侧

```plain
用户域（配置角色权限、渠道、招商）
    ↓
配置域（寄样限制天数、独家阈值、保护期、提成比例等）
    ↓
各领域（按权限过滤数据、按配置执行规则）
```

**三条链都能走通 → 骨架正确。**

## 九、部署架构

### 9.1 决策

**采用单体部署，模块化组织**

### 9.2 说明

- 7个领域作为代码模块（package/module）放在同一个项目中
- 模块间通过**内部事件接口**通信，不直接调用对方数据库
- 未来需要拆分为微服务时，只需将模块抽出，业务逻辑不用大改

### 9.3 技术栈概览（待技术落地设计细化）

| 层级     | 选型方向                      |
| -------- | ----------------------------- |
| 语言     | Go / Java / Python（FastAPI） |
| Web框架  | Gin / Spring Boot / FastAPI   |
| 数据库   | PostgreSQL                    |
| 事件     | 内存队列 / 消息队列           |
| 定时任务 | cron / Celery                 |
| 缓存     | Redis                         |
