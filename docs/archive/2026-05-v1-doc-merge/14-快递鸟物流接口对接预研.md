# 14-快递鸟物流接口对接预研

更新时间：2026-05-14

文档性质：预研文档，不是已接入确认。

本轮边界：

- 只阅读快递鸟官方文档和本地项目文档，形成对接计划。
- 不写业务代码，不修改寄样流程，不修改抖音联盟真实 SDK 主链路。
- 不伪造快递鸟接口返回，不把 Sandbox / 正式环境能力写成已验证。
- 本期上线不因物流自动化阻塞；当前仍以运营手动录入物流公司和单号、手动推进状态作为兜底事实来源。

## 一、当前结论

本项目第一阶段建议只预研并规划接入：

```text
快递鸟即时查询 API
```

不建议第一阶段直接接：

```text
电子面单 API
预约取件 API
物流跟踪订阅推送
在途监控高频监控
```

理由很简单：当前寄样台的真实业务动作是“运营手动录入物流公司 + 运单号，系统辅助查询轨迹，签收后推进到待交作业”。快递鸟在第一阶段只应该做轨迹查询和物流状态辅助推进，不能替代发货事实来源，也不能把寄样单直接推进到“已完成”。

## 三、官方接口能力梳理

本节只引用本轮实际打开的快递鸟官方页面。

| 能力 | 官方定位 | 关键官方事实 | 本项目判断 |
| --- | --- | --- | --- |
| 即时查询 API | 根据快递单号和快递公司编码实时查询物流轨迹，查询后即时返回状态。 | 官方页面写明按单个运单号查询、需指定快递公司编码、轨迹按发生时间升序排列、接口指令 `1002`、HTTP POST、测试地址和正式地址；每日调用次数不超过 500 次，超过建议接物流跟踪 API。 | 第一阶段推荐采用。 |
| 物流跟踪 API | 用户把运单信息订阅给快递鸟，快递鸟监控并推送轨迹直到签收。 | 订阅接口 `RequestType=1008`；推送接口由商户实现，推送查询结果 `RequestType=101`；官方说明适用于日查询量大于 500 次的网站和系统。 | 第一阶段不接。需要回调端点、订阅模型和推送幂等处理。 |
| 在途监控 API | 对物流订单进行实时监控，支持订阅推送，也可以即时获取订单当前实时状态。 | 官方页面写明全流程状态包含无轨迹、已揽收、在途中、已签收、问题件、到达派件城市、派件中、驿站签收、超时、拒签 / 退件等；增值版即时查询 `RequestType=8001`，订阅 `RequestType=8008`。 | 第一阶段不接。适合高频、全流程监控，不是当前 MVP。 |
| 单号识别 API | 输入运单号，返回可能的一个或多个快递公司。 | 官方页面写明只识别快递公司，不返回物流轨迹；`RequestType=2002`；失败时匹配结果为空。 | 可作为后续辅助能力；第一阶段仍要求运营手动选择物流公司编码。 |
| 电子面单 API | 电子面单下单，直接向快递公司请求下单 / 取号 / 打印相关信息。 | 官方页面写明这是下单接口，`RequestType=1007`；会把收发件人和货物信息发送到快递公司获取单号、模板等；`OrderCode` 不可重复提交。 | 第一阶段不接。当前寄样台不负责统一下单打单。 |

补充说明：

- 快递鸟官网 API 列表还展示预约取件、物流查询地图等能力；这些不属于本轮要求的 MVP 范围。
- 本轮在官网页面中只核到即时查询 API 的“每日不超过 500 次”限制，以及物流跟踪 API 适用于日查询量大于 500 次场景；未在已打开官方网页中核到统一每秒并发上限。若后续账号后台、合同或接口技术文档另给 SLA，应在 Sandbox 前补证。

## 四、本项目推荐方案

### 4.1 第一阶段采用即时查询 API

推荐第一阶段采用 `RequestType=1002` 的即时查询 API。

适配当前业务的原因：

1. 寄样单量和刷新频率通常低于高频电商物流监控。
2. 当前业务已经有人工录入物流公司和单号，不需要快递鸟负责下单。
3. 当前闭环只需要“录入后查一次、手动刷新查一次、低频定时查一次”。
4. 即时查询无需先建设回调接收、订阅状态机和推送幂等。
5. 即时查询失败可以自然降级为人工兜底，不影响寄样主流程。

### 4.2 不先接电子面单

电子面单 API 的核心是在线下单、取号和打印。它要求系统管理收发件人、货物、面单模板、快递账号和重复订单号等下单要素。

当前寄样台的第一阶段事实来源仍是：

```text
运营在线下或第三方平台完成发货
↓
运营把物流公司和单号录入系统
↓
系统查询轨迹并辅助推进状态
```

所以电子面单不是当前最小闭环。

### 4.3 不先接订阅推送

物流跟踪 API 和在途监控 API 都更偏持续监控，需要：

- 回调 URL；
- 推送验签；
- 推送幂等；
- 订阅状态；
- 失败重放；
- 签收后停止或归档策略。

这些能力可以后续建设，但不应该在第一阶段压进寄样台 MVP。

### 4.4 手动录入仍是事实来源

第一阶段必须明确：

- `companyCode` / `ShipperCode` 来自运营选择或系统映射；
- `trackingNo` / `LogisticCode` 来自运营录入；
- 快递鸟只负责轨迹查询和状态辅助推进；
- 查询失败、无轨迹、状态异常时，不自动推进寄样状态。

## 五、即时查询 API 对接要点

### 5.1 请求基础信息

| 项目 | 官方口径 |
| --- | --- |
| 请求方式 | `HTTP POST` |
| Content-Type | `application/x-www-form-urlencoded;charset=utf-8` |
| 测试地址 | `http://sandboxapi.kdniao.com:8080/kdniaosandbox/gateway/exterfaceInvoke.json` |
| 正式地址 | `https://api.kdniao.com/Ebusiness/EbusinessOrderHandle.aspx` |
| 接口指令 | `RequestType=1002` |
| 数据类型 | `DataType=2`，即 JSON |
| 查询粒度 | 单个运单号查询 |

### 5.2 系统级参数

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `RequestData` | 是 | 请求内容。JSON 格式，并按 UTF-8 URL 编码。 |
| `EBusinessID` | 是 | 快递鸟商户 ID。 |
| `RequestType` | 是 | 即时查询固定为 `1002`。 |
| `DataSign` | 是 | 数据签名。 |
| `DataType` | 否 / 建议传 | 官方写明 `2-json`；本项目建议固定传 `2`。 |

`DataSign` 官方规则：

```text
请求内容未编码字符串 + AppKey
↓
MD5
↓
Base64
↓
URL encode(utf-8)
```

注意：用于签名的是“未编码”的 `RequestData` 字符串；提交表单时 `RequestData` 和 `DataSign` 都需要按官方要求 URL 编码。

### 5.3 业务参数

最小请求体：

```json
{
  "OrderCode": "",
  "ShipperCode": "SF",
  "LogisticCode": "118650888018"
}
```

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `OrderCode` | 否 | 订单编号。本项目可放内部寄样单号或留空，第一阶段建议留空或仅用于排查关联。 |
| `ShipperCode` | 是 | 快递公司编码，如 `SF`、`EMS`、`YTO`、`STO`、`ZTO`、`YD`、`JD` 等。编码错误会返回失败。 |
| `LogisticCode` | 是 | 物流运单号。 |

快递公司编码策略：

1. 第一阶段要求运营录入时选择系统内维护的物流公司，系统映射到快递鸟 `ShipperCode`。
2. 不建议第一阶段只靠单号识别自动判断快递公司。
3. 若后续接单号识别 API，只把它作为“辅助推荐”，最终仍应由运营确认。

### 5.4 返回字段

| 字段 | 说明 |
| --- | --- |
| `EBusinessID` | 用户 ID。 |
| `OrderCode` | 订单编号。 |
| `ShipperCode` | 快递公司编码。 |
| `LogisticCode` | 物流运单号。 |
| `Success` | 查询是否成功。 |
| `Reason` | 失败原因。 |
| `State` | 物流状态。即时查询官方页面列出 `2`、`3`、`4`。 |
| `Traces` | 轨迹列表。 |
| `AcceptTime` | 轨迹时间。 |
| `AcceptStation` | 轨迹描述。 |
| `Remark` | 备注。 |

即时查询官方页面给出的 `State`：

| `State` | 官方含义 |
| --- | --- |
| `2` | 在途中 |
| `3` | 签收 |
| `4` | 问题件 |

边界说明：

- 即时查询返回无轨迹示例中 `Success=false` 且 `Traces=[]`，未给 `State`。
- 物流跟踪推送文档中还出现 `0-无轨迹`、`1-已揽收` 等更细状态；这些属于后续订阅 / 监控扩展口径，第一阶段不要混进即时查询 MVP 的验收标准。

## 六、状态映射

第一阶段建议建立独立物流状态，不直接复用寄样主状态。

| 快递鸟返回 | 内部物流状态建议 | 寄样状态影响 | 说明 |
| --- | --- | --- | --- |
| `Success=false` | `FAILED` | 不推进 | 记录失败原因，保留人工处理。 |
| `Success=true` 但 `State` 为空 | `UNKNOWN` | 不推进 | 可展示“暂无明确物流状态”。 |
| `State=2` | `IN_TRANSIT` | 保持或推进到“快递中” | 仅当寄样单已发货或待发货转发货场景明确时推进；避免状态回退。 |
| `State=3` | `SIGNED` | 推进到“待交作业” | 只表示物流签收，不表示寄样已完成。 |
| `State=4` | `EXCEPTION` | 不自动关闭 | 标记物流异常，等待运营处理。 |

关键原则：

1. 快递鸟只能辅助推进物流状态，不能直接把寄样单推进到 `FINISHED`。
2. “已完成”仍必须由订单出单 / 命中订单判断触发。
3. 查询失败、无轨迹、问题件不自动推进寄样状态。
4. 自动推进必须幂等，不能把人工已经推进过的状态回退。
5. 签收时间 `signedAt` 优先取签收轨迹节点的 `AcceptTime`；无法可靠解析时记录查询时间并标记来源。

## 七、系统设计建议

本节是后续设计建议，不是本轮代码改动。

### 7.1 Gateway 抽象

建议后续按现有项目分层继续设计：

```text
LogisticsGateway
├── TestLogisticsGateway
└── KdniaoLogisticsGateway
```

职责边界：

| 组件 | 职责 |
| --- | --- |
| `LogisticsGateway` | 对 Service 暴露统一物流查询契约，不暴露快递鸟原始字段。 |
| `TestLogisticsGateway` | 在 `test` 环境模拟无轨迹、在途中、已签收、问题件、查询失败。 |
| `KdniaoLogisticsGateway` | 在真实配置启用后调用快递鸟即时查询 API。 |

### 7.2 返回模型建议

建议统一返回 `LogisticsTrackResult`：

```text
companyCode
trackingNo
success
reason
externalState
internalStatus
signed
signedAt
traces
rawResponse
```

建议轨迹节点 `LogisticsTraceNode` 包含：

```text
acceptTime
acceptStation
remark
```

安全说明：

- `rawResponse` 只用于排障和取证，不应默认前台展示。
- 轨迹描述中可能包含派件员姓名、电话、地点等信息；落库和日志都要做脱敏或权限控制。
- `EBusinessID`、`AppKey`、`DataSign` 不允许进入普通业务日志。

### 7.3 状态推进建议

推荐流程：

```text
运营录入物流公司 + 单号
↓
系统保存为发货事实
↓
手动刷新或低频定时任务调用 LogisticsGateway
↓
把快递鸟状态映射为内部物流状态
↓
仅在安全条件满足时推进寄样状态
↓
记录操作日志 / 轨迹快照
```

第一阶段触发方式：

| 触发方式 | 建议 |
| --- | --- |
| 发货后立即查一次 | 可选，失败不影响发货成功。 |
| 页面手动刷新 | 推荐优先实现，便于运营感知。 |
| 定时任务刷新 | 小批量、低频，只扫快递中寄样单。 |
| 高频轮询 | 不做。 |
| 订阅推送 | 不做。 |

## 八、环境变量建议

第一阶段建议只保留即时查询所需配置：

```env
KDNIAO_ENABLED=false
KDNIAO_EBUSINESS_ID=
KDNIAO_APP_KEY=
KDNIAO_REQUEST_URL=http://sandboxapi.kdniao.com:8080/kdniaosandbox/gateway/exterfaceInvoke.json
KDNIAO_REQUEST_TYPE=1002
```

配置说明：

| 环境变量 | 说明 |
| --- | --- |
| `KDNIAO_ENABLED` | 是否启用快递鸟真实查询；默认必须为 `false`。 |
| `KDNIAO_EBUSINESS_ID` | 快递鸟商户 ID，不写入仓库。 |
| `KDNIAO_APP_KEY` | 快递鸟 AppKey，敏感凭据，不写入仓库，不进日志。 |
| `KDNIAO_REQUEST_URL` | Sandbox / 正式地址切换。Sandbox 先行，real-pre 小范围验证后再考虑正式地址。 |
| `KDNIAO_REQUEST_TYPE` | 即时查询固定 `1002`，保留为配置便于验收核对。 |

后续可按项目现有风格增加连接超时、读取超时、定时刷新开关、批量大小和失败退避配置，但不应在第一阶段引入复杂调度。
