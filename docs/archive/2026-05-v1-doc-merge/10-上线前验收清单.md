# 10-上线前验收清单

更新时间：2026-05-18

## 一、当前结论

## 二、V1 上线验收原则

1. **只按 `V1交付范围表` 验收**
   - 旧 V2.2、领域设计、技术设计里的增强项，不能自动转成 V1 上线条件。
2. **P0-1 到 P0-9 是当前上线判断主轴**
   - 只有 P0 项通过，才能判断 V1 是否具备上线条件。
3. **real-pre 证据优先**
   - 没有 real-pre 证据时，可以记为“待补”或“可用 local-mock 辅助补证”，但不能伪造通过。
4. **V1 简化项可以通过“明确简化边界”验收**
   - 不是所有能力都要做到完整设计强度。
5. **V2 后移项和已关闭项不参与本轮上线判断**
   - 不得在验收阶段重新回流为 V1 必交项。

## 四、P0-1 渠道链端到端

**验收目标**

渠道链要能从“认领达人”一路走到“产生可统计业绩”。

**覆盖范围**

- 达人认领
- 商品库选品
- 转链与 `pick_source_mapping`
- 寄样申请
- 招商审核
- 运营发货
- 订单同步
- 默认归因到渠道
- 业绩统计 / 看板展示

**通过标准**

- 渠道可以完成达人认领、商品选品和寄样申请
- 商品转链成功，`pick_source_mapping` 可查
- 寄样流转能进入发货、待交作业
- 订单同步成功并进入订单池
- 渠道默认归因结果和业绩展示一致

**当前判断**

- 当前已有主链 smoke 和可视化剧本证据
- 但仍受 P0-4、P0-6、P0-7 影响，不能单独提前宣称完全通过

---

## 五、P0-2 招商链端到端

**验收目标**

招商链要能从“同步活动 / 商品”一路走到“默认招商归因和业绩展示”。

**覆盖范围**

- 活动同步
- 活动商品同步
- 商品审核 / 上架
- 商品负责人 / 活动负责人链路
- 寄样审核
- 订单同步
- 默认招商归因
- 业绩统计

**通过标准**

- 招商组长可同步活动和活动商品
- 招商可完成商品审核 / 上架和寄样审核
- 商品能按当前 V1 口径进入商品库
- 订单进入系统后，默认招商归因与当前活动绑定招商一致
- 业绩列表 / 看板中的招商归属一致

**当前判断**

- 当前活动 / 商品 / 审核 / 页面链路已有证据
- 默认招商归因专项证据仍待补，不能提前判定通过

---

## 六、P0-3 管理链规则生效

**验收目标**

管理员调整规则后，规则要真正影响业务行为，而不是只改了配置表。

**覆盖范围**

- 用户与角色
- 数据范围
- 配置域
- 寄样限制
- 达人保护期
- 提成比例

**通过标准**

- 管理员可以修改配置
- 修改后规则在寄样、达人、业绩等链路中真实生效
- 配置变更有留痕
- 数据范围和权限边界不被破坏

**当前判断**

- 当前有配置能力、日志能力和权限能力
- 但缺“改前 / 改后”专项对照，因此暂不能通过

---

## 七、P0-4 `pick_source` 渠道归因正确

**验收目标**

证明真实推广链路下，订单能通过 `pick_source` 正确归因到默认渠道。

**通过标准**

- 至少 1 条真实订单带回 `pick_source`
- 该 `pick_source` 能命中 `pick_source_mapping`
- `mapping.created_at <= order.create_time`
- 订单默认渠道归因正确
- 订单列表、业绩、Dashboard 对应口径一致

**当前判断**

- 当前只有归因底座和无样本时的真实观测证据
- 当前状态应固定为：**待真实样本，不得伪造通过**

---

## 八、P0-5 活动默认招商归因正确

**验收目标**

证明当前 V1 口径下，默认招商归因只按活动绑定招商，不混入个别品负责人覆盖。

**通过标准**

- 活动 / 商品负责人链路清楚
- 订单进入系统后，默认招商归因与活动绑定招商一致
- 业绩列表、看板中的招商归属一致
- 明确排除 P-08 / O-04 的个别品负责人覆盖逻辑

**当前判断**

- 当前文档口径已收口
- 当前仍缺一份专项对表证据，暂不能通过

---

## 九、P0-6 寄样交作业自动完成

**验收目标**

证明寄样在订单命中后，会按 V1 规则自动从“待交作业”进入“已完成”。

**通过标准**

- 寄样申请能进入发货、待交作业状态
- 相关订单同步入库
- 系统正确识别寄样与订单的匹配关系
- 寄样状态自动进入“已完成”
- 页面、日志或数据库都能说明是“被订单命中后完成”

**当前判断**

- 当前已有发货和 `PENDING_TASK` 证据
- 仍缺 `completed_by_order` 样本，因此暂不能通过

## 十、P0-7 双轨金额与提成正确

**验收目标**

证明订单域双轨金额、业绩域提成、看板聚合三者口径一致。

**通过标准**

- 订单双轨金额字段正确
- 服务费收益公式正确
- 招商 / 渠道提成公式正确
- Dashboard Summary 与数据库聚合一致
- 至少有 1 份手算表能和系统结果对上

**当前判断**

- Dashboard 对账已有 PASS
- 手算表仍待补，因此只能算“部分通过”

---

## 十一、P0-8 寄样限制与豁免正确

**验收目标**

证明寄样规则不是静态文案，而是系统真实生效。

**覆盖范围**

- 7 天重复申请限制
- 组长 / 管理员豁免
- 配置修改后规则变化

**通过标准**

- 普通角色 7 天内重复申请被拦截
- 组长 / 管理员可以按规则豁免
- 修改限制配置后，规则前后表现发生变化

**当前判断**

- 当前有规则底座
- 仍缺真实前后对照和豁免样本，暂不能通过

---

## 十二、P0-9 数据范围正确

**验收目标**

证明不同角色只能看到该看到的数据，越权访问会被拦截。

**覆盖范围**

- `self / group / all`
- 管理员
- 招商组长 / 招商
- 渠道组长 / 渠道
- 运营
- 一人多角色特殊口径

**通过标准**

- 普通角色只见自身范围
- 组长只见组内范围
- 管理员可见全局范围
- 越权页面 / 接口被拦截
- 如果 U-09 进入当前 V1 口径，则需明确“权限并集、数据范围取最宽”

**当前判断**

- 基础多角色证据已有
- U-09 最终项目口径和专项样本仍待补

- P0-1 ~ P0-9 全部达到“通过”或“确认通过”

- P0-4 真实 `pick_source` 渠道归因样本仍缺
- P0-5 默认招商归因专项证据仍缺
- P0-6 `completed_by_order` 样本仍缺
- P0-7 手算表仍缺

- P0-9 U-09 口径和专项样本仍缺
