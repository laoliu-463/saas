# 02-V1业务流程与领域设计

更新时间：2026-05-19

> 本文档统一描述 **V1 三条业务链 + 领域职责 + 状态机 + 跨域关系**。  
> 范围边界以 [01-V1交付范围与边界](./01-V1交付范围与边界.md) 为准，本文不重新扩张范围。

## 一、当前结论

- V1 业务设计已经从“旧 V2.2 全量规划”切到“按当前三条主链收口”。
- 当前目标不是把所有历史设计原样实现，而是把业务规则压缩成可执行、可验收、可补证的主线。

## 二、三条业务链

### 1. 渠道链

达人认领  
→ 维护达人地址 / 标签  
→ 商品库选品  
→ 复制推广链接  
→ 生成 `pick_source_mapping`  
→ 申请寄样  
→ 招商审核  
→ 运营发货  
→ 订单同步  
→ 默认归因到渠道  
→ 业绩统计

**当前 V1 关键点**

- 达人认领是渠道入口
- 转链必须产生 `pick_source_mapping`
- 寄样不是孤立表单，而是有状态流转
- 订单只按默认渠道归因，不启用独家达人覆盖

### 2. 招商链

招商组长同步活动  
→ 同步活动商品 / 刷新状态  
→ 分配 / 补充商品信息  
→ 商品上架商品库  
→ 渠道推广  
→ 寄样审核  
→ 订单同步  
→ 默认招商归因 = 活动绑定招商  
→ 招商业绩统计

**当前 V1 关键点**

- 活动和活动商品是招商链起点
- 当前活动 / 商品同步口径先按活动列表查询、活动商品 `refresh=true` 强刷与抖音详情补水收口
- 商品上架后才进入共享商品库
- 默认招商归因按活动绑定招商收口
- 不把个别品负责人覆盖带回当前 P0

### 3. 管理链

管理员配置人员、角色、数据范围  
→ 配置寄样限制、保护期、提成比例、模板规则  
→ 商品 / 达人 / 寄样 / 订单 / 业绩读取规则  
→ 不同角色看到不同数据范围  
→ 规则变更影响实际业务动作

**当前 V1 关键点**

- 权限和数据范围是主链的一部分
- 配置必须影响业务，不是只存配置表
- 当前管理链只围绕 V1 必需规则收口

## 三、领域设计总览

| 领域 | V1 职责 | 当前口径 |
|---|---|---|
| 用户域 | 账号、角色、数据范围、下拉基础能力 | U-09 当前按“权限并集、数据范围取最宽”收口 |
| 配置域 | 寄样限制、保护期、全局提成、模板规则 | 接受当前实现路径，不强求事件中心形态 |
| 商品域 | 活动、商品、合作方、商品库、转链 | 个别品负责人覆盖不进 V1 |
| 达人域 | 达人档案、认领、保护期、地址、标签 | 标签最多 3 个需继续收口 |
| 寄样域 | 申请、审核、发货、待交作业、完成 | 物流 API 只保留底座 |
| 订单域 | 同步、双轨金额、默认归因、明细 | 真实 `pick_source` 样本仍待补 |
| 业绩域 | 默认归因业绩、双轨提成、列表和卡片 | 不混入独家覆盖 |
| 分析模块 | 指标卡片、时间筛选、明细下钻 | 高级分析后移 |

## 四、用户域

### V1 职责

- 登录 / 登出 / JWT
- 预置角色
- 自定义角色与数据范围
- 员工管理
- 部门 / 组别
- 渠道 / 招商 / 部门下拉
- 一人多角色

### 关键规则

- 数据范围只认 `self / group / all`
- 管理员可见全部
- 组长可见组内
- 普通成员仅见自身范围
- U-09 当前项目口径是：权限并集，数据范围取最宽

## 五、配置域

### V1 职责

- 寄样限制和开关
- 达人保护期
- 全局提成比例
- 默认寄样门槛
- 商品补充信息里的寄样门槛（商品级优先于全局默认）
- 模板与 `pick_extra` 规则
- 配置变更生效
- 配置变更留痕

### 关键规则

- 当前 V1 不启用独家阈值
- 当前 V1 不启用差异化提成
- 当前 V1 不启用预设标签库

## 六、商品域

### V1 职责

- 同步活动和活动商品
- 通过活动商品列表 / 详情带出合作方信息
- 招商审核和补充商品信息
- 商品库展示与筛选
- 商品排序 / 置顶
- 转链与 `pick_source_mapping`
- 快速寄样入口（当前以商品详情暴露寄样前置信息为主）

### 关键规则

- 先审核、后上架
- 商品库面向渠道共享
- 转链必须落 `pick_source_mapping`
- 当前 V1 主链按商品级别上架、转链、寄样、订单归因收口；`buyin.productSkus.v2` 只作为商品规格补充能力，优先级按 P1 处理，不并入订单 / 结算闭环
- 商品补充信息中的寄样要求是 V1 寄样门槛来源之一；当前只读取商品级门槛并回退全局默认，不新增复杂配置页、模板或多条件规则引擎
- 只有在寄样需要选择具体 SKU、商品详情需要展示规格价格、或业务需要按 SKU 区分价格 / 库存时，才继续补强到 SKU 维度
- 当前没有独立合作方列表 / 详情接口时，按活动商品列表筛选与商品详情聚合口径收口
- 快速寄样的真实提交动作仍在 `/samples/**`，商品域当前只承担入口与前置信息
- 当前 V1 不把个别品负责人覆盖带入默认招商归因

## 七、达人域

### V1 职责

- 达人查询 / 补全
- 公海 / 私海
- 多人同时认领
- 保护期和释放
- 认领人维护地址
- 标签
- 数据刷新

### 关键规则

- 多人认领不互斥
- 保护期读取配置域；认领写截止日、到期无产出释放；保护期内有出单则**跳过释放**（不延长 `protectedUntil`）
- 寄样收货信息 V1 支持渠道在寄样申请时手工填写覆盖；未填写时可读取认领记录收货信息，`addressSource` 标记为 `manual` 或 `claim`，real-pre 自动带入证据仍需专项补证
- 标签最多 3 个，后者覆盖（`PUT /api/talents/{id}/tags`）

## 八、寄样域

### V1 职责

- 渠道申请寄样
- 7 天限制和豁免
- 招商审核
- 物流录入
- 状态推进
- 订单命中后自动完成
- 筛选与导出

### 关键规则

- 普通角色受 7 天重复申请限制
- 7 天限制是渠道申请侧规则，豁免角色仅限管理员和渠道组长；招商组长可审核本组寄样，但不豁免重复申请限制
- 物流以手动录入和手动推进状态为主
- 当前 V1 只保留物流 API 底座
- 寄样资格校验读取商品级寄样门槛；商品未配置时读取全局默认门槛
- 订单触发交作业完成必须是已付款订单：渠道 + 达人 + 商品匹配，并且订单 `pay_time != null`
- 寄样列表 / 导出应带出合作方 ID、合作方名称、收货人、手机号、收货地址，并支持合作方名称与收货地址关键字筛选

## 九、订单域

### V1 职责

- 团长订单同步
- 结算事实同步
- 双轨金额
- `pick_source -> 默认渠道`
- 状态 / 退款 / 失效更新
- 列表 / 筛选 / 明细
- 导出
- 后续处理信号

### 关键规则

- 默认渠道归因依赖 `pick_source_mapping`
- 主订单事实源是 `buyin.instituteOrderColonel`，默认 `time_type=update`，用于维护订单状态、预估看板、退款 / 失效变化和寄样“已付款出单即完成”判断
- 结算事实源是 `buyin.colonelMultiSettlementOrders / 2704`，默认 `time_type=settle`，用于维护实际服务费、实际提成和结算看板
- 两条链路不得混用：不得把“结算按 `settle_time`”套到主订单增量同步上
- 当前 V1 不把独家达人 / 商家逻辑写入主链
- 当前 V1 不把个别品负责人覆盖写入默认招商
- O-07 导出仅按基础导出收口：当前筛选条件、最多 5000 行、管理员 / 招商组长 / 渠道组长可用、默认字段、记录操作日志；不做异步、全量、自定义列或看板整体导出

## 十、业绩域

### V1 职责

- 默认归因业绩
- 双轨提成
- 服务费收益口径
- 退款 / 失效冲正
- 业绩列表 / 指标卡片
- 提成比例读取配置域

### 关键规则

- `final_channel = default_channel`
- `final_recruiter = default_recruiter`
- 提成基数 = 服务费收益
- 实际结算金额进入展示和聚合前，必须先完成至少 1 条真实 2704 raw JSON 与后台金额交叉校验；未确认金额单位前，结算金额不作为上线可信展示
- 当前不启用独家达人 / 商家覆盖

## 十一、分析模块

### V1 职责

- 指标卡片
- 时间筛选
- 汇总更新
- 订单明细下钻

### 当前简化

- 趋势图只保留基础展示
- 排行榜只保留基础展示
- 寄样统计只保留基础统计
- 不做高级经营分析和整体导出

## 十二、跨域关系

| 来源领域 | 目标领域 | 关系 | 当前口径 |
|---|---|---|---|
| 用户域 | 全域 | 角色、数据范围、人员下拉 | 当前主链基础设施 |
| 配置域 | 达人域 | 保护期 | 当前必须生效 |
| 配置域 | 寄样域 | 7 天限制、超时规则 | 当前必须生效 |
| 配置域 | 商品域 | 模板和 `pick_extra` 规则 | 接受当前实现路径 |
| 商品域 | 订单域 | `pick_source_mapping`、活动 / 商品关系 | 支撑默认归因 |
| 达人域 | 寄样域 | 地址带入 | 需继续补证 |
| 订单域 | 寄样域 | 订单命中后寄样完成 | 需继续补证 |
| 订单域 | 业绩域 | 双轨金额、默认归属 | 当前事实来源 |
| 业绩域 | 分析模块 | 汇总与展示 | 当前看板口径来源 |

## 十三、状态机汇总

| 实体 | 状态流转 |
|---|---|
| 活动商品 / 商品展示 | 待补充 / 待审核 → 可推广 / 展示中 → 已隐藏 |
| 达人认领 | 保护中 → 已释放 |
| 寄样申请 | 待审核 → 待发货 → 快递中 → 待交作业 → 已完成 / 已拒绝 / 已关闭 |
| 订单 | 待结算 → 已结算 / 已退款 / 已失效 |

## 十四、当前不纳入 V1 主链的能力

- 独家达人跨域覆盖
- 独家商家跨域覆盖
- 差异化提成
- 自动物流轨迹
- Excel 批量物流
- 高级排行榜 / 高级分析
- 达人爬虫增强

## 十五、一句话结论

> 当前 V1 业务设计只围绕渠道链、招商链、管理链三条主线收口；独家、自动物流、高级分析等能力统一后移，不参与当前主线验收。
