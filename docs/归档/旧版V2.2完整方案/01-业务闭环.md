# 01-业务闭环

更新时间：2026-05-03

## 一、当前闭环目标

本项目当前最核心的业务目标是：

> 让商品、推广链接、订单、寄样、达人之间形成一条能够在 `local-mock` 基线稳定跑通、并可向 `real-pre` 浏览器回归环境与后续真实 Gateway 联调平滑扩展的闭环。

当前阶段不是追求复杂策略，而是先确保：

- 业务动作能走通
- 页面结果能看懂
- 出问题时能排查

## 二、当前核心闭环

### 0. 认证与安全闭环

登录 -> 获取 Access Token + Refresh Token -> 接口鉴权 -> Token 无感刷新 -> 登出 / Token 吊销

关键结果：

- 用户身份可验证，未登录不可访问业务接口
- Access Token 过期后 Refresh Token 自动续签，无需重新登录
- 登出后 Token 立即失效，不留安全隐患
- 数据权限按角色自动过滤（本人 / 本组 / 全部）

### 1. 商品管理 / 商品库闭环

活动列表 -> 商品列表 -> 组长分配审核人 -> 招商审核 -> 审核通过自动加入商品库 -> 商品库共享可见 -> 分配招商负责人 -> 复制推广链接 -> 生成归因映射

关键结果：

- 业务人员能选品
- `/product` 只承接已入库商品的共享商品库视图，对全员可见
- `/product/manage` 负责活动列表与活动下商品列表的流转入口
- 组长在活动商品列表中先分配审核负责人，招商在“我负责的商品”中完成审核；审核通过后会自动加入商品库，并继续沿用后续分配、推进判断、转链与达人跟进链路
- 选品完成后可沉淀为团队共享商品资产
- 系统能生成推广链接
- 后续订单可用于归因

### 2. 订单归因闭环

推广链接 -> Test 订单生成 -> 归因识别 -> 订单工作台 -> Dashboard

关键结果：

- 已归因订单能找到渠道
- 未归因订单能看到原因
- 看板能实时反映结果

### 3. 寄样履约闭环

申请寄样 -> 审核 -> 发货 -> 签收 -> 待交作业 -> 命中订单自动完成

关键结果：

- 寄样不是孤立表单
- 可以跟踪履约状态
- 出单后能自动闭环

### 4. 达人沉淀闭环

达人列表 -> 公海 / 私海 -> 认领 / 释放 -> 达人详情 -> 查看寄样与订单

关键结果：

- 达人不是静态档案
- 能与寄样、订单结果关联
- 能沉淀渠道长期资产

## 三、当前关键业务问题对应关系

### 1. 商品管理 / 商品库解决什么

解决：

- 商品从哪来
- 哪些商品只是候选，哪些已经沉淀为共享商品
- 谁负责跟进
- 如何转成可归因推广链接

### 2. 订单工作台解决什么

解决：

- 这单为什么已归因
- 这单为什么未归因
- `pick_source` 是什么
- 推广链路是否匹配成功
- 下一步怎么排查

### 3. 合作管理（寄样履约）解决什么

前台菜单口径已收敛为“合作管理”：业务角色看到“合作单”，运营角色看到“发货台”。`/sample` 已升级为合作单工作台，集中展示达人、商品、合作方、申请、招商、物流 / 进度和收货信息；`/ops/shipping` 继续保留旧路由，但复用同一个工作台的发货视角。当前底层仍复用寄样履约链路、`/samples` API、寄样状态机和既有 RBAC，不代表新增完整合作管理领域。

解决：

- 样品有没有申请
- 有没有审核
- 有没有发货 / 签收
- 有没有出单完成
- 发货单、物流单号、收货信息是否能在同一个合作单里追踪

### 4. 达人 CRM 解决什么

解决：

- 这个达人是谁
- 当前归谁
- 是否在公海
- 被谁认领
- 申请过哪些寄样
- 产生过哪些订单

## 四、当前标准状态流转

### 1. 商品状态

- `PENDING_AUDIT`
- `APPROVED`
- `REJECTED`
- `ASSIGNED`
- `LINKED`
- `FOLLOWING`

补充说明：
- `绑定活动` 当前作为商品主链路的元数据修正动作保留，但不再单独推进 `bizStatus`
- 历史上可能存在 `BOUND` 旧值，当前仅作为兼容读取口径，不再作为新写入状态

### 2. 寄样状态

- `PENDING_AUDIT`
- `PENDING_SHIP`
- `SHIPPED`
- `PENDING_TASK`
- `FINISHED`
- `REJECTED`
- `CLOSED`

### 3. 订单归因状态

- `ATTRIBUTED`
- `UNATTRIBUTED`

未归因重点原因：

- `NO_PICK_SOURCE`
- `MAPPING_NOT_FOUND`

## 五、当前系统如何证明闭环成立

可以通过以下演示路径证明：

1. `/login` 登录获取 Token（Access 2h + Refresh 7d）
2. `/dev/test` 初始化数据
3. `/product` 复制推广链接
3. `/dev/test` 生成已归因与未归因订单
4. `/orders` 查看订单和详情排查
5. `/dashboard` 查看统计变化
6. `/sample` 查看寄样状态推进与自动完成
7. `/talent` 查看达人关联寄样和订单
8. 调用 `/auth/logout` 验证 Token 吊销，登出后接口返回 401

## 六、当前边界

当前闭环明确不包含：

- 人工补归因
- 批量导出
- 复杂提成结算
- 真实达人爬虫
- 高级达人分析

## 七、当前结论

从业务闭环角度看，当前系统已经不是“商品库搭骨架”阶段，而是已经进入：

> 本地 Mock 核心闭环完成，进入真实 SDK 联调准备阶段

补充说明：

- 当前本地人工联调默认口径已经统一为 `local-mock`
- `test` 继续用于自动化测试 / 隔离测试栈
- `real-pre` 当前用于浏览器回归、权限验收、部署形态验证，并作为真实 SDK 联调前的独立拓扑基线

这也是后续真实 SDK 联调和前端体验优化的基础。
