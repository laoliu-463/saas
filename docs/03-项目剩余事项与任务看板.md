## 八、当前性能观察

### PERF-01：`GET /api/dashboard/summary` 在 test 环境耗时约 13 秒

- 观察时间：2026-05-18（大数据量 test 环境）；2026-05-19 小数据量 smoke 约 29ms~63ms
- 证据来源：`runtime/qa/out/e2e-base-smoke-20260518-203042/api-responses/04-dashboard-summary.json`；对比 `runtime/qa/out/e2e-base-smoke-20260519-083947/`
- 根因（已闭环分析，非功能 bug）：
  - `DashboardService.getSummary()` 顺序执行多段聚合 SQL；瓶颈在 `loadActivityProductBreakdown()` 对 `pick_source_mapping` / `promotion_link` 等的 **N+1 相关子查询**（约 7152 订单、20 行下钻 × 4 子查询）。
  - 详见 [tech-debt-dashboard-13s.md](./tech-debt-dashboard-13s.md)、[archive/audits/16-20260518-Dashboard-13s性能分析报告.md](./archive/audits/16-20260518-Dashboard-13s性能分析报告.md)
- 当前判断：
  - V1 **不阻断上线**；属架构性查询成本，随订单量线性恶化
  - V2 方向：预聚合表 / Redis 缓存 / 改写 JOIN 消除相关子查询
- 后续动作：smoke 继续记录耗时；V2 排期优化，不把 13s 写成 P0 功能失败
