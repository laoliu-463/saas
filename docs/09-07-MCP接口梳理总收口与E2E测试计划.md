# 09-07-MCP接口梳理总收口与E2E测试计划

更新时间：2026-05-18

> 本文档把 09-01 ~ 09-06 的 MCP 接口梳理收口成 **直接可落地的 E2E 测试计划**。  
> 从本轮开始，**MCP / OAS 继续作为接口事实来源，Playwright + TypeScript 作为测试执行器**；不再规划 Apifox 作为测试执行器。

## 一、收口结论

1. 当前 V1 主链的接口事实底稿已经齐备：
   - [09-01-MCP商品主链接口梳理](./09-01-MCP商品主链接口梳理.md)
   - [09-02-MCP订单归因接口梳理](./09-02-MCP订单归因接口梳理.md)

   - [09-04-MCP配置规则接口梳理](./09-04-MCP配置规则接口梳理.md)
   - [09-05-MCP寄样接口梳理](./09-05-MCP寄样接口梳理.md)
   - [09-06-MCP达人接口梳理](./09-06-MCP达人接口梳理.md)
2. 当前仓库已经有可复用的 E2E 资产，不需要重新找执行器：
   - `playwright.config.ts`
   - `tests/e2e/**/*.spec.ts`
   - `runtime/qa/*.cjs`
   - `README-e2e.md`
3. 本轮决策已经明确：
   - **保留 MCP / OAS** 作为接口、字段、角色、断言的事实来源
   - **停止 Apifox 场景执行规划**
   - **统一切换到 Playwright + TypeScript**

   - API E2E
   - 浏览器 E2E

   - 真实样本补证
5. `completed_by_order`、`pick_source` 真实归因、默认招商归因、双轨金额手算、保护期变化、多角色数据范围，仍不能只靠接口存在判定通过。

## 三、E2E 总体方案

### 1. 执行器

- 推荐执行器：**Playwright + TypeScript**
- 主执行面：
  - 浏览器 E2E：`@playwright/test`
  - API E2E：`request.newContext()` / `APIRequestContext`
- 保留现有根目录 `playwright.config.ts` 与 `tests/e2e/**` 为第一阶段执行面

### 2. 接口事实来源

- `business` OAS：V1 主验收接口事实源
- `sdk-debug` OAS：真实抖音联调辅助源
- MCP / OAS 负责提供：
  - path / method
  - request / response schema
  - 角色注解
  - 字段命名与状态枚举

### 3. 证据分层

1. **API E2E 自动化**
   - 适合登录、配置、达人 CRUD、寄样申请、订单查询、Dashboard 接口断言
2. **浏览器 E2E 自动化**
   - 适合菜单可见性、角色越权、地址带入、审核发货入口、Dashboard 页面展示
3. **DB / 日志补证**
   - 适合 `pick_source_mapping`、`completed_by_order`、保护期变化、提成对账、配置留痕
4. **真实样本补证**
   - 适合 `pick_source` 真实归因、默认招商归因、真实订单触发完成、手算表

### 4. 环境分工

| 环境 | 主要用途 | 说明 |
|---|---|---|
| `test` | 可控 API / 浏览器 E2E | 适合回归、重复执行、规则前后对照 |
| `real-pre` | 真实联调与最终补证 | 适合 P0 最终闭证、真实订单和真实角色样本 |

### 5. 输出约束

- 证据输出目录继续沿用：`runtime/qa/out/<suite>-<timestamp>/`
- 本轮 **不跑测试**
- 本轮 **不改业务代码**
- 本轮 **不新增 V1 需求**
- 本轮 **不把 V2 后移项拉回 V1**

### 目录职责

| 目录 | 作用 |
|---|---|
| `config/` | 环境变量、项目矩阵、执行参数 |
| `fixtures/` | 测试账号、活动、商品、达人、寄样等样本定义 |
| `helpers/api/` | 登录、转链、下单查询、寄样推进等 API helper |
| `helpers/ui/` | 页面对象、导航、截图、可见性断言 |
| `helpers/db/` | `pick_source_mapping`、`sample_status_log`、`talent_claim` 等补证查询 |
| `helpers/oas/` | 从 OAS 读取 path / schema / 枚举的事实辅助层 |
| `helpers/evidence/` | 输出 JSON、截图、HTML 摘要、DB 对照结果 |
| `manifests/` | 场景清单、哪些场景必须 real-pre 执行 |
| `specs/smoke/` | 最小冒烟，例如登录、首页、健康检查 |
| `specs/domains/` | 商品、订单、达人、寄样、配置等领域脚本 |
| `specs/p0/` | P0-1 ~ P0-9 的主验收脚本 |
| `specs/real-pre/` | 真实联调、真实样本专项脚本 |
| `runners/` | 场景编排、证据收集、批量执行入口 |

## 六、可自动化 / 需 DB 补证 / 需真实样本分类

### 1. 可 API E2E 自动测

- `POST /auth/login`
- `GET /menus/tree`
- `GET /roles/enabled`
- `GET /users/assignable`
- `/configs/**`
- `/talents/**` CRUD、认领、释放、公私海列表
- `POST /samples`
- `PUT /samples/{id}/status`
- `GET /samples/{id}/status-logs`
- `POST /orders/sync`
- `GET /orders`
- `GET /orders/{orderId}`
- `GET /dashboard/summary`
- `GET /dashboard/metrics`
- `GET /data/orders`

### 2. 需浏览器 E2E 的场景

- 菜单可见性与页面入口
- 角色越权拦截
- 商品审核与加入商品库的页面操作
- 寄样审核、发货、待交作业页面状态
- 公海 / 私海视图切换
- 达人选择后寄样地址是否自动带入
- Dashboard / Data 页面展示与交互筛选

### 3. 必须 DB / 日志补证

- `pick_source_mapping` 是否命中且时间早于订单创建
- `completed_by_order` 是否由订单同步触发
- `sample_status_log` 是否记录自动完成
- `talent_claim.expire_time` / `protectedUntil` 是否按规则变化
- 配置改前改后是否真正生效，并有 `operation-logs`
- Dashboard / 订单 / 提成聚合对账

### 4. 必须真实样本

- `P0-4` 真实 `pick_source` 订单归因
- `P0-5` 默认招商归因
- `P0-6` 真实订单触发 `completed_by_order`
- `P0-7` 双轨金额与提成手算闭证
- `P0-9` U-09 一人多角色专项样本
