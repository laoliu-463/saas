# 物流接口适配性调研

更新时间：2026-05-14 19:47:54 +08:00

## 1. 调研范围与结论先看

本次只做抖店/抖音开放平台物流能力适配性调研，不改业务代码。

本轮结论先给清楚：

1. **抖音/抖店平台确实有物流相关接口。**
   但这些接口主要是围绕 **店铺订单发货、抖音电子面单、店铺侧物流轨迹、即时零售仓配/运力** 展开，不是围绕团长寄样单展开。
2. **当前团长 SaaS 的 real-pre 授权主体，不具备直接使用这批主流物流接口的调用条件。**
   当前 real-pre 已确认的真实授权主体是 **机构/团长侧**，不是店铺/商家侧。
3. **当前寄样台不应该优先接抖店物流接口。**
   因为寄样台当前处理的是内部 `sample_request` 寄样履约，不是抖店店铺订单，也不是抖音电子面单取号链路。
4. **当前阶段最合理方案仍是：手动录入物流公司/单号作为主方案；若要增强，再接快递100/快递鸟这类通用物流轨迹查询。**
5. **本期上线不阻塞。**
   仓库当前口径已经把“未接真实物流 API、运营手动录入物流单号和手动推进状态”作为可上线兜底验收结论固化。

## 4. 授权主体判断

## 4.1 官方层面的授权主体判断

本轮可以明确分成两类：

### A. 团长/机构侧

当前项目 real-pre 已验证通过的主链路是：

- `buyin.institutionInfo`
- `alliance.instituteColonelActivityList`
- `buyin.instituteOrderColonel`

这属于 **机构 / 团长侧授权能力**。

### B. 物流/发货/电子面单/即时零售侧

官方物流相关接口普遍呈现以下信号：

1. 文档元数据 `authSubject=[3]` 或依赖店铺 token；
2. 权限包分类集中在：
   - `商家后台系统`
   - `订单管理`
   - `电商ERP`
   - `打单发货-微应用`
   - `仓储管理`
   - `即时零售ERP`
3. FAQ 明确要求：
   - 同一店铺 token
   - 同一电子面单店铺
   - 店铺联系小二开白
   - ebill 账号授权

所以它们整体上应归为：

- **店铺授权 / 商家授权**
- 部分特殊场景还会涉及：
  - **ebill 账号授权**
  - **物流侧开白**
  - **厂家/供应商授权**

而不是机构/团长授权。

## 4.2 当前 real-pre 是否具备调用条件

**结论：不具备。**

原因有四个：

1. **当前真实主体不是店铺主体。**
   当前 real-pre 已确认的是 `institution + colonel` 主体，不是某个店铺订单主体。
2. **当前主链路是精选联盟/团长链路，不是店铺订单链路。**
   仓库内也一直按“联盟/团长链路”和“店铺侧接口”分开推进。
3. **已有本地文档证明店铺侧接口当前仍是权限不足口径。**
   仓库历史记录已明确：
   - `order.searchList / order.orderDetail` 是店铺侧接口；
   - 当前 real-pre 对店铺侧接口仍不是既有主链路能力。
4. **就算勉强补权限，也仍然和当前寄样台业务对象不匹配。**
   寄样台处理的是本地 `sample_request`，不是抖店消费者订单，也不是抖音电子面单订单。

## 5. 寄样台是否有必要接抖店物流接口

**结论：当前寄样台没有必要直接接抖店物流接口。**

### 5.1 业务对象不匹配

当前寄样台是：

- 渠道/运营内部寄样履约；
- 运营手动录入物流公司和单号；
- 手动推进“待发货/快递中/签收/待交作业”等状态。

而抖店物流接口处理的是：

- 店铺订单发货；
- 抖音电子面单取号；
- 店铺订单物流轨迹；
- 即时零售门店/仓/骑手配送。

两边的主对象完全不同。

### 5.2 技术前提不匹配

要用抖店这批物流接口，通常至少需要其中之一：

- 店铺授权
- 商家授权
- 电子面单店铺 token
- ebill 账号
- 即时零售仓配主体

当前寄样台没有这些前提。

### 5.3 数据前提不匹配

官方 `/logistics/trackNoRouteDetail` 明确要求：

- 单号来自抖音电子面单；
- 查询 token 和取号 token 是同一店铺。

而当前寄样台录入的是“任意快递公司 + 任意人工单号”，天然不满足这个前提。

## 6. 如果不能直接接，应该怎么做

### 6.1 本期建议

**继续使用“手动录入物流信息 + 手动推进寄样状态”作为主方案。**

理由：

1. 这已经是当前仓库正式固化的验收结论；
2. 与当前寄样台业务模型完全匹配；
3. 不依赖新增店铺授权、ebill 账号或物流侧开白；
4. 不阻塞当前上线。

### 6.2 若要做增强，优先级建议

**优先接通用第三方物流查询，不优先接抖店物流发货。**

推荐顺序：

1. **保留手动录入作为发货事实来源**
   - 运营录入 `物流公司 + 单号`
2. **再接快递100 / 快递鸟这类通用轨迹查询**
   - 用于自动刷新“已揽收/运输中/签收”等轨迹；
   - 可作为寄样台的“自动查询增强”，不是发货主数据源；
3. **只有当未来寄样链路变成“抖店店铺真实订单/电子面单出号链路”时，才再评估接抖店物流接口**

### 6.3 为什么第三方查询比抖店物流更合适

因为第三方物流查询的前提通常只是：

- 快递公司
- 运单号

它不要求：

- 店铺 token
- 抖音电子面单取号来源
- 店铺授权主体
- 即时零售仓配主体

这与当前寄样台形态更贴合。

因为当前 real-pre 主体是 **机构/团长侧**，不是 **店铺/商家侧**。

### 8.3 当前寄样台是否应该接

**当前不应该优先接抖店物流接口。**

寄样台更适合：

- 手动录入物流信息；
- 如需增强，再接通用物流轨迹查询。

### 8.4 本期上线是否阻塞

**不阻塞。**

当前仓库已经明确把“物流 API 暂未接入、运营手动录入物流单号与手动推进状态”作为可上线兜底口径。

## 10. 快递鸟即时查询API字段定义（2026-05-14 补充）

> 来源：[快递鸟即时查询API](https://www.kdniao.com/api-track)
>
> 调研背景：根据第 6.2 节建议，如需增强寄样台物流轨迹查询能力，建议优先接通用第三方物流查询（如快递100/快递鸟）。以下为快递鸟即时查询API的完整字段定义，供后续对接参考。

### 10.1 Traces（物流轨迹节点）完整字段定义

| 字段名 | 类型 | 中文含义 | 必须 |
|--------|------|----------|------|
| **AcceptTime** | String | 时间（轨迹发生时间，格式：YYYY/MM/DD HH:mm:ss） | 是 |
| **AcceptStation** | String | 描述（轨迹发生地点/状态描述） | 是 |
| **Remark** | String | 备注（附加说明，可为空） | 否 |

### 10.3 State（物流状态码）

| 状态码 | 含义 |
|--------|------|
| 2 | 在途中 |
| 3 | 签收 |
| 4 | 问题件 |

### 10.4 API基础信息

| 项目 | 值 |
|------|-----|
| 接口指令 | 1002 |
| 请求方式 | HTTP POST (application/x-www-form-urlencoded;charset=utf-8) |
| 测试地址 | http://sandboxapi.kdniao.com:8080/kdniaosandbox/gateway/exterfaceInvoke.json |
| 正式地址 | https://api.kdniao.com/Ebusiness/EbusinessOrderHandle.aspx |
| 数据类型 | 仅支持 JSON |
| 每日调用限制 | 即时查询API不超过500次/天，超过请对接物流跟踪API |

### 10.5 请求参数

| 参数名 | 类型 | 说明 | 必须 |
|--------|------|------|------|
| OrderCode | String | 订单编号 | 否 |
| ShipperCode | String | 快递公司编码（如 SF、YTO、STO） | 是 |
| LogisticCode | String | 物流单号 | 是 |

### 10.6 对接注意事项

1. **轨迹顺序**：返回的物流跟踪信息按照发生时间升序排列
2. **快递公司编码**：需先确认快递公司编码，错误会导致返回失败
3. **调用频率**：即时查询API每日限制500次，超限需对接物流跟踪API
4. **全流程跟踪**：如需无限次查询，需对接在途监控API
