# 01-业务闭环

更新时间：2026-04-27

## 一、当前闭环目标

本项目当前最核心的业务目标是：

> 让商品、推广链接、订单、寄样、达人之间形成一条能够在 `test` 环境稳定跑通的闭环。

当前不是追求复杂策略，而是先确保：

- 业务动作能走通
- 页面结果能看懂
- 出问题时能排查

## 二、当前核心闭环

### 1. 商品库闭环

活动商品 -> 商品列表 -> 商品详情 -> 复制推广链接 -> 生成归因映射

关键结果：

- 业务人员能选品
- 系统能生成推广链接
- 后续订单可用于归因

### 2. 订单归因闭环

推广链接 -> Test 订单生成 -> 归因识别 -> 订单工作台 -> Dashboard

关键结果：

- 已归因订单能找到渠道
- 未归因订单能看到原因
- 看板能实时反映结果

### 3. 寄样履约闭环

申请寄样 -> 审核 -> 发货 -> 签收 -> 待交作业 -> 命中订单自动完成

关键结果：

- 寄样不是孤立表单
- 可以跟踪履约状态
- 出单后能自动闭环

### 4. 达人沉淀闭环

达人列表 -> 公海 / 私海 -> 认领 / 释放 -> 达人详情 -> 查看寄样与订单

关键结果：

- 达人不是静态档案
- 能与寄样、订单结果关联
- 能沉淀渠道长期资产

## 三、当前关键业务问题对应关系

### 1. 商品库解决什么

解决：

- 商品从哪来
- 谁负责跟进
- 如何转成可归因推广链接

### 2. 订单工作台解决什么

解决：

- 这单为什么已归因
- 这单为什么未归因
- `pick_source` 是什么
- 推广链路是否匹配成功
- 下一步怎么排查

### 3. 寄样台解决什么

解决：

- 样品有没有申请
- 有没有审核
- 有没有发货 / 签收
- 有没有出单完成

### 4. 达人 CRM 解决什么

解决：

- 这个达人是谁
- 当前归谁
- 是否在公海
- 被谁认领
- 申请过哪些寄样
- 产生过哪些订单

## 四、当前标准状态流转

### 1. 商品状态

- `PENDING_AUDIT`
- `APPROVED`
- `REJECTED`
- `BOUND`
- `ASSIGNED`
- `LINKED`
- `FOLLOWING`

### 2. 寄样状态

- `PENDING_AUDIT`
- `PENDING_SHIP`
- `SHIPPED`
- `PENDING_TASK`
- `FINISHED`
- `REJECTED`
- `CLOSED`

### 3. 订单归因状态

- `ATTRIBUTED`
- `UNATTRIBUTED`

未归因重点原因：

- `NO_PICK_SOURCE`
- `MAPPING_NOT_FOUND`

## 五、当前系统如何证明闭环成立

可以通过以下演示路径证明：

1. `/dev/test` 初始化数据
2. `/product` 复制推广链接
3. `/dev/test` 生成已归因与未归因订单
4. `/orders` 查看订单和详情排查
5. `/dashboard` 查看统计变化
6. `/sample` 查看寄样状态推进与自动完成
7. `/talent` 查看达人关联寄样和订单

## 六、当前边界

当前闭环明确不包含：

- 人工补归因
- 批量导出
- 复杂提成结算
- 真实达人爬虫
- 高级达人分析

## 七、当前结论

从业务闭环角度看，当前系统已经不是“商品库搭骨架”阶段，而是已经进入：

> Test MVP 核心闭环完成阶段

这也是后续真实 SDK 联调和前端体验优化的基础。
