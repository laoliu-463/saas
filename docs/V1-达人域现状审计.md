## 三、七问专项答复

### 1. 多人同时认领是否真正互斥？

**不互斥，设计如此。** `uk_tc_active` 是 `(talent_id, user_id)` 组合唯一，允许多个不同用户在同一达人上拥有各自的有效认领记录。Redis 锁保证同一用户对同一达人的认领操作串行化（防并发写入冲突），但不阻止不同用户同时认领。

### 2. 保护期变更是否即时生效？

**是。** `BusinessRuleConfigService` 使用 `ConcurrentHashMap.computeIfAbsent` 缓存 + `SysConfigService` CRUD 时调用 `invalidate(key)` 清缓存。新认领时调用 `getProtectDays()` 读取最新值。已有的 `protected_until` 不会追溯修改（符合预期）。

### 3. 释放任务是否真的检查产出？

**是。** `releaseExpiredClaims(now)` 对每条过期认领调用 `hasOutputSinceClaim(talentId, userId, claimTime)`，查询认领后是否有订单记录。有产出则跳过释放、保留认领；无产出才设置 `status=CLAIM_STATUS_EXPIRED`。

### 4. 出单重置保护期的调用链是否完整？

**是。**
```
OrderSyncPersistenceService.afterOrderPersisted(order)
  → talentService.extendProtectionOnOrder(order)
    → 按 payTime(优先)/createTime + getTalentProtectionDays() 计算新 protectedUntil
    → 批量更新该达人所有 active claims 的 protected_until
```
此链路在订单同步入库后自动触发，无需人工干预。

### 5. 达人列表 DataScope 是否安全？

**功能安全，但实现路径不统一。** `TalentService.page()` 手动接收 `DataScope`/`userId`/`deptId` 参数并实现过滤。Controller 层从 `@RequestAttribute` 取值（JwtAuthInterceptor 注入），有基本信任。但与订单域/寄样域使用 `@DataScope` + `DataScopeAspect` AOP 自动追加条件的模式不一致，存在参数伪造的理论风险（需同时绕过 JWT 校验，实际风险极低）。

### 6. 标签是否受认领关系约束？

**是。** `TalentTagService.replaceTags()` 和 `addTags()` 要求当前用户对该达人有 active 认领（admin 豁免）。`findActiveClaim(talentId, userId)` 查询认领记录，无 active 认领则抛出 `BizException("没有该达人的认领关系")`。

### 7. 独家达人机制是否影响 V1 订单归因？

**不影响。** `evaluateExclusive` 方法仅做条件判断（服务费占比 >= 70% AND 月度样品 >= 10），不改变任何认领状态或 `protected_until`。V1 范围内，独家评估结果不参与 `AttributionService` 的归因计算。月度调度（每月 1 号 3:00/3:30）与其他日级任务不冲突。
