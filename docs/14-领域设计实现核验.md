本文件是文档与静态代码核验，不等同于本地全量跑测报告。

## 二、核验方法

核验来源：

- 需求原文：[13-领域设计总览](./13-领域设计总览.md) 所列七个领域文档。
- 阶段事实：[04-开发进度](./04-开发进度.md)、[10-V2.2场景覆盖矩阵](./10-V2.2场景覆盖矩阵.md)、[10-上线前验收清单](./10-上线前验收清单.md)。
- 当前代码抽样：`backend/src/main/java/com/colonel/saas/**`、`frontend/src/**`。
- 验收口径：开发角度看功能是否闭环，用户角度看业务是否顺手。

核验状态定义：

| 状态 | 含义 |
| ---- | ---- |
| 已实现 | 设计中的关键能力已有代码、页面或文档证据支撑，并有测试/验收记录 |
| 部分实现 | 主链路可用，但字段、权限、边界、三方样本或用户体验未完全覆盖 |
| 未实现/未进入阶段 | 当前代码或主干文档未能证明已落地，或被明确列为后续阶段 |

## 四、逐域实现核验

### 一、用户域

需求入口：[01-用户域](./requirements/domain-design/01-用户域.md)

当前判断：**部分实现**。

已落地能力：

- 认证基础能力已具备：登录、刷新 Token、登出。
- 当前用户接口族已补齐：`GET /users/current`、`PUT /users/current/password`、`GET /users/current/data-scope`、`POST /users/current/permissions/check`，并已提供前端 API 封装。
- 前端个人中心 `/profile` 已迁移到当前用户接口族，可查看当前账号、数据范围和角色权限，支持个人改密与权限自检。
- 人员主数据接口已补齐基础下拉：`GET /users/master-data/channels`、`GET /users/master-data/recruiters`、`GET /users/master-data/group-members`，并已提供前端 API 封装。
- 商品分配弹窗已迁移到人员主数据接口，负责人候选通过 `/users/master-data/recruiters` 获取。
- 订单筛选已迁移到人员主数据接口，渠道负责人候选通过 `/users/master-data/channels` 获取，招商负责人候选通过 `/users/master-data/recruiters` 获取。
- 寄样台筛选已迁移到人员主数据接口，渠道负责人候选通过 `/users/master-data/channels` 获取，招商负责人候选通过 `/users/master-data/recruiters` 获取；后端寄样分页/导出支持按渠道负责人和商品招商负责人过滤。
- 登录成功、登录失败、登录锁定、账号停用拒绝和登出已写入 `operation_log`；个人改密已由 `UserDomainService.changePassword` 写入同一操作日志中心。
- 认证审计仅记录用户名、动作、结果和错误原因，不记录密码或 token 原文。
- 系统用户、角色、部门、菜单管理已有后端 Controller 与前端管理页面。
- `@RequireRoles`、`DataScope`、菜单权限与角色权限已在 P1/P2 多轮审查中补强。
- 系统操作日志已有管理入口。

主要缺口：

- 更多业务页权限判断仍需逐步迁移到统一用户域 API。
- 角色数据范围目前主要围绕本人/本组/全部落地，原需求中更完整的组、部门、人员主数据和 `data_scope JSONB` 配置 UI 未完全闭环。
- 前端用户管理仍偏基础账号维护，用户多角色/部门协作和更多业务权限判断迁移仍需补齐。
- 组织架构与业务域下拉的稳定契约仍需进一步统一。

交付影响：

- V1 基础 RBAC、当前用户接口、人员主数据下拉和认证安全审计可交付。
- 若按用户域 V1.0 原文验收，不能宣称用户域完整实现。

### 二、商品域

需求入口：[02-商品域](./requirements/domain-design/02-商品域.md)

当前判断：**核心主链路部分闭环**。

已落地能力：

- 商品库、活动商品、商品审核、分配、转链、归因映射和商品详情已有主链路实现。
- 前端已补商品筛选、复制讲解、短链、内部寄样入口、`supportsAds`、置顶标识、置顶/取消置顶按钮等 UX 能力。
- 服务端商品库已补 `salesRange`、转链状态、联盟状态、佣金、寄样规则、负责人、系统标签、推进判断等筛选参数。
- 后端已补商品置顶服务和接口，前端已接商品库卡片与商品管理表格操作，支持 24 小时有效、每用户最多 10 个置顶。
- 商家型合作方列表、详情和商品列表已补 `GET /merchants`、`GET /merchants/{id}`、`GET /merchants/{id}/products`，聚合活动商品快照与商家沉淀数据，前端 `product.ts` 已提供对应 API。
- 商品域快速寄样已按 V1 内部寄样口径收口：商品卡按钮显示“内部寄样”，跳转 `/sample/apply`，并通过寄样单 `extra_data.applySource=INTERNAL_QUICK_SAMPLE` 留痕。
- real-pre 商品链路审计已证明活动商品同步可用，商品快照可入库，商品库主数据可持续积累。

主要缺口：

- 团长型合作方未实现；抖店外部 `quick_sample_apply` 不作为 V1 已接能力。
- 商品域原文中的合作方独立管理、活动商品排序和人工置顶、商品库展示规则仍未全部形成稳定后台配置能力。
- real-pre 真实新单 `pick_source` 样本仍待观察，不能把“真实订单归因闭环”说满。
- 合作方独立管理、活动商品排序和人工置顶的完整前后端验收仍需补齐。

交付影响：

- 商品主链路可以进入交付准备。
- 若按商品域 V2.3 原文验收，团长型合作方、外部抖店寄样和部分筛选排序仍是缺口。

### 三、达人域

需求入口：[03-达人域](./requirements/domain-design/03-达人域.md)

当前判断：**基础 CRM 与经营台一期已实现，完整达人域未闭环**。

已落地能力：

- 达人列表、详情、认领、释放、黑名单、手动补录、刷新、同步资料等基础 CRM 能力已实现。
- 多人认领释放一致性已修复，达人归属快照按有效认领重新生成。
- 达人写接口已改为 DTO/VO，避免直接暴露 Entity 和敏感字段。
- 渠道专员/渠道组长/管理员对象级权限已多轮补强。
- 达人详情 DTO 已返回 `tags`、`tagUpdatedBy`、达人历史默认地址和当前认领记录地址；前端详情弹窗已接入标签编辑和当前认领地址编辑，快速寄样会优先带入当前认领人的地址。
- `GET /talents/status-transitions` 已输出公海、我的私海、他人已认领、已释放、黑名单，以及认领、继续认领、释放、订单重置保护期、保护期到期自动释放等状态流转矩阵；前端 `talent.ts` 已提供 `getTalentStatusTransitions` API。

主要缺口：

- 达人域原文中的独家达人完整判定、服务费占比、月寄样阈值、组长审核闭环，不属于当前 V1 默认主链路；当前独家相关能力默认由 `exclusive.enabled=false` 关闭。
- 真实外部达人采集能力仍受三方接口/风控限制，当前以手动补录兜底。
- 达人标签已记录最近更新人 `tag_updated_by`，后端限制最多 3 个；保存语义为覆盖当前标签列表。
- 达人地址已补 `talent_claim.recipient_name/recipient_phone/recipient_address` 认领人维度字段；维护地址时要求当前登录人存在有效认领记录，历史 `talent.shipping_recipient_*` 仅作为兼容回填。
- 达人公开池/私海池的业务口径和状态流转矩阵已可用；后续仍需将独家达人专项继续收口。

交付影响：

- 基础达人 CRM 可交付。
- 独家达人机制不能按原文宣称完整实现，只能说“有代码底座和默认关闭开关，V1 不作为启用口径”。

### 四、寄样域

需求入口：[04-寄样域](./requirements/domain-design/04-寄样域.md)

当前判断：**主流程较完整，状态机矩阵已固化**。

已落地能力：

- 申请、审核、发货、签收、待交作业、作业完成、自动闭环已有实现。
- `SampleLifecycleService` 自动完成待交作业已有事务边界修复。
- 寄样申请商品上下文、达人详情快速寄样、物流手动兜底等用户动作已补齐。
- 批量审核/拒绝/发货、导出、物流刷新等敏感动作已补权限注解和测试记录。
- 前端寄样台已接入待审核批量通过/批量拒绝按钮，运营发货台已接入批量发货。
- 寄样申请已支持结构化收货人、手机号、地址，前端提交 `recipientName/recipientPhone/recipientAddress`，后端写入 `sample_request.recipient_*` 并在 VO 返回。
- `GET /samples/status-transitions` 已暴露完整状态流转矩阵，覆盖动作别名、前置状态、后置状态、内部状态、角色、必填字段、错误文案和批量接口；前端已封装 `getSampleStatusTransitions`。
- 导出角色矩阵已按寄样域原文收口：管理员、招商组长、招商专员、运营可导出；渠道组长和渠道专员不可导出。前端寄样台与运营发货台共用 `canExportSamplesByRole`。

主要缺口：

- 真实物流 API 尚未在 sandbox/real-pre 完成调用，当前仍按手动物流兜底。
- 寄样域与商品域/达人域/订单域的事件化通信在代码中更多表现为服务调用和同步副作用，尚未形成完整领域事件总线。

交付影响：

- V1 寄样主链路和状态流转矩阵可以交付。
- 真实物流仍需在 V2 或上线前规则确认中补齐。

### 五、订单域

需求入口：[05-订单域](./requirements/domain-design/05-订单域.md)

当前判断：**同步/查询/归因/双轨金额底座已实现**。

已落地能力：

- 订单同步、幂等 claim、按订单号查回、已有订单更新等持久化能力已具备；Real Gateway 的常规时间范围同步已切到 `buyin.colonelMultiSettlementOrders`，旧 `buyin.instituteOrderColonel` 仅保留 RAW 探针对照。
- 订单归因支持原生团长字段、`pick_source` / `pick_extra` 回落、NATIVE 映射和冲突保护。
- 订单同步成功后可补推广映射、沉淀商家、触发寄样自动完成等副作用。
- Dashboard 与订单库口径已有 real-pre 对账证据。
- `OrderDualTrackAmountResolver` 已解析上游 `pay_amount/settle_amount/estimate_service_fee/effective_service_fee/estimate_tech_service_fee/effective_tech_service_fee`，并在订单更新时保留预估快照。
- 订单详情 DTO、`OrderQueryService` 与前端 `OrderDetail.amount` 类型已返回成交、结算、预估服务费、有效服务费、预估技术服务费和有效技术服务费双轨字段。
- `OrderSyncedEvent` 已携带订单同步后的双轨金额、达人 UID 和原始 `extraData` 载荷，供业绩记录、看板聚合和达人保护期重置订阅。

主要缺口：

- real-pre 真实 `pick_source` 新单样本仍在观察，现有证据不能证明真实新单归因已 100% 闭环。
- `buyin.colonelMultiSettlementOrders` 当前仍存在非空结算样本等待项，需要真实样本继续核验字段完整性。
- 原文中的退款/结算变更独立事件和导出 CSV 完整字段表仍需继续对照真实样本和前端展示做字段级核验；数据页导出已与列表同口径支持 `orderId/status/talentId/merchantId/timeField/startDate/endDate`。
- 独家覆盖按当前 V1 口径默认关闭，订单域也不应承担独家业务计算。

交付影响：

- 订单域底座可进入交付准备。
- 真实订单样本和结算样本未补齐前，不能对外宣称“真实订单归因全场景已完成验收”。

### 六、业绩域

需求入口：[06-业绩域](./requirements/domain-design/06-业绩域.md)

当前判断：**看板与业绩记录底座已实现，完整业绩域未完全实现**。

已落地能力：

- Dashboard Summary、活动商品看板、诊断 drilldown 和数据库对账已有通过记录。
- 订单归因后的 `user_id`、渠道归属、GMV、预估/结算类指标已支撑看板展示。
- 提成计算已支持全局配置、活动覆盖，以及 `commissions` 表中的 global/activity/product/user 差异化规则。
- 独家商家/独家达人有代码底座，但 V1 默认关闭。
- `PerformanceRecord`、`PerformanceCalculationService`、`PerformanceRecordSyncListener` 和 `PerformanceMetricsQueryService` 已补业绩记录、订单同步后计算、双轨提成和 Data/Dashboard 聚合底座。
- `alter-dual-track-performance-20260522.sql`、`init-db.sql`、`migrate-all.sql` 已包含订单双轨金额字段与 `performance_records` 表结构。

主要缺口：

- 原文中的独家覆盖层、退款反冲、管理员重算仍未形成完整领域闭环。
- `dashboard_performance_daily` 事件汇总代码已有底座，已先跳过既有订单更新事件以避免重复累计；但仍按当前日期汇总，正式用于业绩或看板前需要继续明确日期口径和完整重算策略。
- 独家商家资格和独家达人资格不应在 V1 默认启用，否则会与当前上线口径冲突。
- 差异化提成规则来源已明确，完整重算和退款反冲仍需继续收口。

交付影响：

- V1 可按“订单归因 + Dashboard 对账 + `performance_records` 双轨业绩记录底座”交付。
- 不能按业绩域 V1.1 原文宣称完整业绩结算系统已实现。

### 七、配置域

需求入口：[07-配置域](./requirements/domain-design/07-配置域.md)

当前判断：**基础 KV、规则配置、配置变更明细和配置源清单部分实现**。

已落地能力：

- 系统配置 Controller、配置分组、强 KV 校验和管理员权限边界已具备。
- 寄样门槛、保护期、超时、独家阈值等关键规则已有配置化记录。
- 达人标签预设库已通过 `talent.preset_tags` 配置、`GET /talents/preset-tags` 接口和达人详情预设多选闭环，后端保存仍执行最多 3 个标签和非预设拒绝规则。
- 差异化提成规则已通过 `commissions` 表、`/commission-rules` 管理接口和 `/system/commission-rules` 页面管理，业绩计算按 `product → activity → user → global` 优先级解析规则。
- prod/real/pre 环境保护和危险配置 fail-fast 已补齐。
- 配置创建、更新、删除已写入 `system_config_change_log`，记录配置 ID、配置键、动作、旧值、新值、来源、操作者和变更时间。
- `SysConfigService` 在配置变更后会继续触发业务配置缓存失效。
- `docs/06-部署与对接计划.md` 已补“关键配置源清单”，区分环境变量/启动配置、`system_config` KV 业务配置和代码常量/固定规则。

主要缺口：

- 原文中的配置变更事件尚未形成独立领域事件。
- 复制模板、`pick_extra` 模板、默认寄样标准和达人标签预设库已进入通用配置中心并有后端强校验；尚无专用规则中心页面和配置变更领域事件。
- 部分配置能力仍散落在业务代码、环境变量和 KV 配置中；当前已有来源清单，后续需要统一配置中心页面视角。

交付影响：

- V1 基础配置管理可用。
- 若按配置域 V1.0 原文验收，配置变更明细、配置源清单、达人标签预设库和差异化提成规则管理已补；模板、领域事件和业务规则 UI 仍需继续收口。
