# 23-七域需求缺口收口清单

更新时间：2026-05-23

## 一、定位

本文档承接 [13-领域设计总览](./13-领域设计总览.md) 与 [14-领域设计实现核验](./14-领域设计实现核验.md)，把七个领域设计的“当前缺口”拆成可排期、可验收的收口任务。

本文档只做任务口径收敛，不改变当前 V1 交付判断：

- V1 仍按“基础业务闭环 + TEST/mock 验收 + real-pre 主链路证据”交付。
- 独家达人、独家商家、完整业绩域、完整用户域组织/人员主数据，默认进入 V2 或专项收口。
- 真实 `pick_source` 新单样本、多结算非空样本属于上游样本等待项，不用 Mock 数据伪造通过。

## 二、优先级定义

| 优先级 | 含义 | 处理方式 |
| ------ | ---- | -------- |
| P0-口径 | 不一定要马上写代码，但上线/汇报前必须说清楚 | 写入验收口径、老板汇报和发版说明 |
| P1-交付前 | 影响用户完整操作或生产解释成本 | 优先排入近期开发或专项验证 |
| P2-V2 | 原始领域设计要求完整，但不阻塞 V1 | 进入 V2 迭代 |
| OBS-观察 | 依赖真实上游样本或外部权限 | 保留自动观察脚本和证据目录 |

## 五、推荐执行顺序

1. 先做 P0-口径确认：保证老板汇报、验收报告、发版说明不会把未完成能力说满。
2. 寄样域 `SAMPLE-01/SAMPLE-04` 已完成，继续推进真实物流 API 验证；用户域人员主数据已覆盖商品分配、订单筛选和寄样台筛选，个人中心已接当前用户接口族，后续继续迁移更多业务权限判断和组员协作场景。
3. 商品域 `PRODUCT-06/07`（展示去重 + 类目/核心筛选）已完成，下一步推进 `PRODUCT-08`（快速寄样增强、复制模板、基础事件）；`PRODUCT-01/03/04/05` 已完成；`PRODUCT-09`（完整规则引擎、团长主数据、Outbox）归 V2 专项。
4. 订单域保持 `ORDER-01/02` 自动观察；`ORDER-02` 代码主路径已切到多结算接口，拿到真实非空样本后再补字段级验收。
5. 配置变更日志、配置缓存失效、配置源清单、模板类配置和达人标签预设库已补；达人独家、业绩域完整重算和配置事件统一进入 V2 专项，不混入 V1 默认上线口径。

## 六、本轮不建议立即做的事

- 不建议为了验收伪造真实 `pick_source` 新单样本。
- 不建议在 V1 临时打开 `exclusive.enabled=true`。
- 不建议把 Dashboard 对账通过包装成完整业绩域完成。
- 不建议在未取得真实 sandbox/real-pre 物流样本前宣称真实物流 API 已闭环。
- 不建议把真实达人采集爬虫能力继续写成已稳定接入。
