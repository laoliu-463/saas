# 09-02-MCP订单归因接口梳理

更新时间：2026-05-18

> 本文档是 [09-MCP接口梳理-第一版](./09-MCP接口梳理-第一版.md) 的第二份拆分稿，专门收口 **转链 -> `pick_source` -> 订单 -> Dashboard** 链路。  
> 本轮只读取 **business 离线 OAS / 前端调用 / 后端代码**，不改业务代码，不跑测试，不把 V2 后移项重新拉回 V1。

## 一、梳理范围

本轮只拆以下 5 组：

1. 订单同步接口
2. 订单列表 / 详情 / 筛选接口
3. 渠道归因字段与未归因原因
4. 默认招商归因字段
5. Dashboard 汇总接口

统一边界：

- **主事实源**：`SAAS-business-offline` -> `docs/openapi/openapi-business.json`
- **辅助事实源**：前端 `src/api/order.ts`、`src/api/dashboard.ts`、`src/api/data.ts` 与 `orders / dashboard / data` 页面；后端 controller / service / DTO / gateway
- **不纳入主链**：
  - `/douyin/**`
  - V2 后移的独家归因、差异化提成、高级看板
  - 把“接口存在”直接写成“验收通过”

## 二、当前结论

1. 订单回流主入口已经明确为 `POST /orders/sync`，且当前是 `ADMIN` 专属接口。
2. 订单主工作台当前以 `GET /orders`、`GET /orders/{orderId}`、`GET /orders/stats`、`GET /orders/filter-options` 为主。
3. 未归因订单排查主入口是 `GET /orders/unattributed`；`GET /orders/order-attribution-unattributed` 更像摘要排查辅助页。
4. Dashboard 主汇总当前以 `GET /dashboard/summary` 为准；`GET /dashboard/activity-products` 是活动商品维度下钻；`GET /dashboard/metrics`、`GET /data/orders`、`GET /dashboard/order-attribution-summary` 更偏分析 / 辅助。
5. 当前 **没有** 独立的 business OAS 接口直接查询 `pick_source_mapping`。
6. 当前订单结果接口只直接暴露最终落库归属字段：
   - `channelUserId`
   - `colonelUserId`
   当前没有独立 `defaultChannelId / finalChannelId / defaultRecruiterId / finalRecruiterId` 分拆读接口。
7. `payTime` 当前不是 business OAS 里的独立字段；可读时间字段主要是：
   - `createTime`
   - `settleTime`
   真实抖音回流里 `pay_success_time` 当前会被归并映射到 `createTime` 口径。
8. `GET /products/{productId}/promotion-links/history` 仍可作为 `pick_source` 历史证据入口，但它属于旧 `/products/**` 兼容链，不是 V1 订单主入口。

## 三、状态标记

| 状态 | 含义 |
|---|---|
| `已确认主链` | 已在当前 business OAS、controller、前端链路中形成一致口径 |
| `待核对` | 当前有实现迹象，但还不足以直接写成已确认主链 |
| `辅助接口` | 对排查、补水、修复有用，但不是订单归因主链的直接主入口 |
| `sdk-debug 辅助` | 仅在 `/douyin/**` 调试域使用，不进入本轮主表 |
| `V1 不采用` | 当前 V1 不以该接口 / 能力作为主链验收口径 |

## 七、与主文档口径核对

### 1. 对照 `docs/01-V1交付范围与边界.md`

当前订单归因链与范围文档 **一致**：

1. `O-04` 仍然关闭，没有因为本轮梳理被重新拉回 V1。
2. `Y-04` 已按 2026-06-05 用户决策纳入 V1 经营指标；`Y-05 / A-08` 仍然不作为 V1 主链。
3. `O-07` 仍按简化项处理，只保留基础导出接口，不升格为主验收；边界固定为当前筛选条件、最多 5000 行、管理员 / 招商组长 / 渠道组长、默认字段、记录操作日志。
4. `final_channel = default_channel`、`final_recruiter = default_recruiter` 仍是当前范围口径；只是 business OAS 没有把它拆成两套字段直接返回。
5. 2704 金额字段在真实 raw JSON 对表完成前不得作为结算金额上线可信展示；结算事实记录需保留 `raw_payload` 和 `source_amount_unit`。

### 2. 对照 `docs/02-V1业务流程与领域设计.md`

当前订单归因链与流程文档 **基本一致**：

1. 商品转链之后，确实通过 `pick_source_mapping` 支撑默认渠道归因。
2. 订单同步后，当前订单详情 / 列表 / Dashboard 都能看到归因结果。
3. 默认招商归因当前仍按“活动绑定招商”口径理解；代码层来源是商品运营状态里的 `assigneeId`。

   - 没有独立 `pick_source_mapping` 读接口
   - 没有 `default / final` 字段分拆读接口
   - `payTime` 没有单独暴露

## 八、归因链路说明

1. **转链阶段**
   - 商品主链接口 `POST /colonel/activities/{activityId}/products/{productId}/promotion-links` 返回 `pickSource`，并在服务层写入 `promotion_link` 与 `pick_source_mapping`。
2. **订单回流阶段**
   - `POST /orders/sync` 从抖音回流订单，写入 `orderId`、`productId`、`pickSource`、`orderAmount`、`settleColonelCommission`、`createTime`、`settleTime` 等基础字段。
3. **默认渠道归因阶段**
   - `AttributionService` 先用 `pickSource / pickExtra / shortId` 查 `pick_source_mapping`，拿到渠道侧 `userId`。
4. **默认招商归因阶段**
   - 同一服务再用 `activityId + productId` 对应的商品运营状态 `assigneeId` 作为默认招商归因来源。
5. **落账结果阶段**
   - 最终落到订单结果里的，是当前可直接读取的 `channelUserId`、`colonelUserId`、`attributionStatus`、`attributionRemark` 等字段。
6. **排查与补证阶段**
   - 如果要证明 `pick_source -> mapping -> order -> dashboard` 闭环，目前必须组合：
     - 转链回包
     - 推广历史
     - 订单详情 / 未归因排查
     - Dashboard 汇总
     - 数据库专项对表

## 九、Dashboard 汇总链路说明

1. **主汇总口径**
   - `GET /dashboard/summary` 是当前首页 Dashboard 主事实源，直接参数只有 `startTime / endTime`。
2. **活动商品下钻**
   - `GET /dashboard/activity-products` 负责把汇总拆到 `activityId + productId` 粒度，直接参数是 `page / size / startTime / endTime`，继续筛活动或商品要使用返回的 `drillDownQuery` 去查订单。
3. **归因摘要辅助**
   - `GET /dashboard/order-attribution-summary` 可用于快速看近 30 天归因摘要，但不是首页主口径。
4. **数据页辅助**
   - `GET /dashboard/metrics` 和 `GET /data/orders` 更偏分析页，不替代主看板；`metrics` 只支持 `timeField`，`data/orders` 使用 `startDate / endDate`。
5. **V1 当前收口**
   - P0-7 当前以 `orders/{orderId}`、`dashboard/summary`、必要时补 `dashboard/metrics` / `data/orders` 和数据库手算表为准。
   - `grossProfit` 已按 2026-06-05 用户决策纳入 V1 经营毛利指标；不得扩展成财务结算或多账期治理。
