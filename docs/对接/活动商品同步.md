# 活动商品同步

> 范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 对接目标

- [V1 必做] 从抖音 / 抖店侧同步活动商品，落入商品域，支撑招商和渠道推广。
- [V1 必做] 同步结果必须能通过商品 API、数据库和日志验证。

## 当前事实

- [V1 必做] 活动商品同步属于商品域能力。
- [V1 必做] 第三方字段进入内部模型时必须保留来源和必要原始标识。
- [V1 简化] V1 优先保证 P0 商品字段、筛选、列表和转链所需字段。
- [V2 预留] 更完整的商品展示规则、推荐排序和多平台商品源后续治理。

## 数据流

1. [V1 必做] 管理端调用 `POST /api/colonel/activities/{activityId}/products/sync` 或定时任务触发活动商品同步。
2. [V1 必做] 手动触发接口只提交后台任务并立即返回，前端提示后台同步中，不等待第三方同步完成。
3. [V1 必做] 后端 Gateway 调用第三方接口。
4. [V1 必做] 商品域标准化字段并入库；上游商品自身 `status=1/推广中` 且本地未拒绝、未暂停、未过期时自动进入商品库候选。
5. [V1 必做] 活动刷新后执行商品库展示状态 repair，再执行展示规则去重重算；历史 `PENDING_AUDIT/selected_to_library=false` 的推广中商品不得依赖裸 SQL 修复。
6. [V1 必做] 前端通过内部 API 查询商品。
7. [V1 必做] 验收通过 API/SQL/日志证明数据来自同步结果。

## 同步保护策略

- [V1 必做] 活动商品分页同步默认保持串行；手动一键同步可按商品状态 `0/1/2/3/4/6` 做受控分片并发，首包预检按同一并发度执行。
- [V1 简化] 优先状态同步（默认 `0/1`）在首包返回 `total` 且无游标时，按 `retrieveMode=0` 使用页码分页并受控并发拉取各页；若上游返回游标、分页参数异常或页请求失败，则回退既有游标分片路径，不改变同步结果语义。
- [V1 必做] 状态分片并发必须先做首包预检：只有各状态返回可用 `total`，且汇总行数不超过 `maxRowsPerActivity`、汇总页数不超过 `maxPagesPerActivity` 时才启用；否则回退默认串行同步，避免大活动越过安全上限。
- [V1 必做] 状态分片同步的 `maxRowsPerActivity` 是跨状态分片的全局预算，优先同步同样适用；并发分片通过原子预算共同消耗，不能按每个状态重复取满上限。
- [V1 必做] 单页请求间隔通过 `product.activity.sync.page-interval-ms` 配置，运行时约束在 300ms~1000ms。
- [V1 必做] 手动同步状态分片并发度通过 `product.sync.activityProduct.manual-status-partition-parallelism` 配置，运行时约束在 1~6，默认 `2`；并发只用于后台任务，不允许前端直接调用第三方接口。
- [V1 必做] 真实上游若拒绝某个状态过滤（例如返回 `50002 参数校验失败`），状态分片预检必须回退串行全量同步，不得让手动同步任务失败。
- [V1 必做] 单页失败最多按 `product.activity.sync.max-retries` 重试，默认 3 次，指数退避上限 5s。
- [V1 必做] 遇到 429 / 限流类错误时停止当前批次并记录接口名、activityId、page、size、耗时、错误码和错误信息。
- [V1 必做] 当前没有 buyin QPS 压测证据，不得把上述保护策略写成“QPS 已验证”。

## 并发与恢复边界

- [V1 必做] 定时活动商品同步按活动分批并发，`PRODUCT_ACTIVITY_SYNC_PARALLELISM` 运行时约束为 `1~2`，默认 `2`；每批等待全部活动完成后才进入下一批，配置为 `1` 可回退串行。
- [V1 必做] 定时同步与手动同步共享 Redis owner 租约槽 `product:backfill:concurrency:slots` / `product:backfill:concurrency:leases`，跨实例真实上游总并发默认不超过 2；每个活动使用独立 owner，防止并发任务错误复用同一租约。
- [V1 必做] 定时同步继续持有 `PRODUCT_BACKFILL_GLOBAL` 和 `PRODUCT_ACTIVITY_SYNC` 全局锁，并对每个活动持有 activity owner 锁；并发只发生在不同活动之间，不允许同一活动重复写入，也不与完整 backfill 并发。
- [V1 必做] 手动活动同步按 `activityId` 使用 owner 化活动锁；不再依赖覆盖所有活动的 `PRODUCT_BACKFILL_GLOBAL` 锁，因此不同活动可以受控并行。
- [V1 必做] 手动同步使用 Redis owner 租约槽 `product:backfill:concurrency:slots` / `product:backfill:concurrency:leases` 限制跨实例并发；默认最大并发为 2，配置项为 `PRODUCT_SYNC_ACTIVITY_PRODUCT_MANUAL_UPSTREAM_MAX_CONCURRENCY`。
- [V1 必做] 槽位释放按 owner 幂等执行；stale reconcile 在原子标记任务为 `ABANDONED` 后回收活动锁和并发槽。完整 backfill 仍保留全局独占锁，避免本次改造扩大其写入范围。
- [V1 简化] 单活动状态分片并发默认配置为 2，运行时仍受首包预检和 1~6 上限约束；只有在真实上游出现限流或异常时，才通过 `PRODUCT_SYNC_ACTIVITY_PRODUCT_MANUAL_STATUS_PARTITION_PARALLELISM` 灰度降回 1。
- [V1 必做] 定时活动级并发与单活动状态分片并发不能相乘：定时任务调用默认串行分页刷新；手动任务才允许状态分片并发，二者最终都受共享 Redis 上游槽限制。

## 验收证据

- [V1 必做] 第三方请求 / 响应摘要。
- [V1 必做] 手动触发接口返回 `syncStatus=ACCEPTED/RUNNING`。
- [V1 必做] 商品表或商品 API 返回同步商品。
- [V1 必做] `ProductDisplayRule` 日志输出 activityId、relations、productIds、displaying、hidden、queryCostMs、calcCostMs、updateCostMs、totalCostMs。
- [V1 必做] 商品库巡检接口或只读 SQL 能发现 `status=1` 但未入库 / 未展示的状态断链。
- [V1 必做] 商品列表 E2E 可展示同步结果。
- [V1 必做] real-pre 空数据必须记录真实响应，不手工伪造。

## 不做

- [V1 不做] 不让前端直接调用第三方商品接口。
- [V1 不做] 不把 mock 商品作为 real-pre 同步成功证据。
