# 活动商品同步

> 范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 对接目标

- [V1 必做] 从抖音 / 抖店侧同步活动商品，落入商品域，支撑招商和渠道推广。
- [V1 必做] 同步结果必须能通过商品 API、数据库和日志验证。

## 当前事实

- [V1 必做] 活动商品同步属于商品域能力。
- [V1 必做] 第三方字段进入内部模型时必须保留来源和必要原始标识。
- [V1 简化] V1 优先保证 P0 商品字段、筛选、列表和转链所需字段。
- [V2 预留] 更完整的商品展示规则、推荐排序和多平台商品源后续治理。

## 数据流

1. [V1 必做] 管理端调用 `POST /api/colonel/activities/{activityId}/products/sync` 或定时任务触发活动商品同步。
2. [V1 必做] 手动触发接口只提交后台任务并立即返回，前端提示后台同步中，不等待第三方同步完成。
3. [V1 必做] 后端 Gateway 调用第三方接口。
4. [V1 必做] 商品域标准化字段并入库；上游商品自身 `status=1/推广中` 且本地未拒绝、未暂停、未过期时自动进入商品库候选。
5. [V1 必做] 活动刷新后执行商品库展示状态 repair，再执行展示规则去重重算；历史 `PENDING_AUDIT/selected_to_library=false` 的推广中商品不得依赖裸 SQL 修复。
6. [V1 必做] 前端通过内部 API 查询商品。
7. [V1 必做] 验收通过 API/SQL/日志证明数据来自同步结果。

## 同步保护策略

- [V1 必做] 活动商品分页同步默认保持串行；手动一键同步可按商品状态 `0/1/2/3/4/6` 做受控分片并发，每个状态分片内部仍串行分页。
- [V1 必做] 状态分片并发必须先做首包预检：只有各状态返回可用 `total`，且汇总行数不超过 `maxRowsPerActivity`、汇总页数不超过 `maxPagesPerActivity` 时才启用；否则回退默认串行同步，避免大活动越过安全上限。
- [V1 必做] 单页请求间隔通过 `product.activity.sync.page-interval-ms` 配置，运行时约束在 300ms~1000ms。
- [V1 必做] 手动同步状态分片并发度通过 `product.sync.activityProduct.manual-status-partition-parallelism` 配置，运行时约束在 1~6，默认 `1`（关闭并发）；并发只用于后台任务，不允许前端直接调用第三方接口。
- [V1 必做] 真实上游若拒绝某个状态过滤（例如返回 `50002 参数校验失败`），状态分片预检必须回退串行全量同步，不得让手动同步任务失败。
- [V1 必做] 单页失败最多按 `product.activity.sync.max-retries` 重试，默认 3 次，指数退避上限 5s。
- [V1 必做] 遇到 429 / 限流类错误时停止当前批次并记录接口名、activityId、page、size、耗时、错误码和错误信息。
- [V1 必做] 当前没有 buyin QPS 压测证据，不得把上述保护策略写成“QPS 已验证”。

## 验收证据

- [V1 必做] 第三方请求 / 响应摘要。
- [V1 必做] 手动触发接口返回 `syncStatus=ACCEPTED/RUNNING`。
- [V1 必做] 商品表或商品 API 返回同步商品。
- [V1 必做] `ProductDisplayRule` 日志输出 activityId、relations、productIds、displaying、hidden、queryCostMs、calcCostMs、updateCostMs、totalCostMs。
- [V1 必做] 商品库巡检接口或只读 SQL 能发现 `status=1` 但未入库 / 未展示的状态断链。
- [V1 必做] 商品列表 E2E 可展示同步结果。
- [V1 必做] real-pre 空数据必须记录真实响应，不手工伪造。

## 不做

- [V1 不做] 不让前端直接调用第三方商品接口。
- [V1 不做] 不把 mock 商品作为 real-pre 同步成功证据。
