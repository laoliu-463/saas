## 五、P0-9 权限口径确认单

### 1. 非 admin `menus/tree=0`

- 当前状态：待确认，不能写通过。
- 当前证据：
  - `runtime/qa/out/e2e-rbac-20260519-133713/summary.json` 与 `report.md` 已确认：admin `GET /api/menus/tree` 返回 `12` 个菜单，`bizStaff / channelStaff / ops / bizLeader / channelLeader` 均为 `0`，本轮统一按 `OBSERVE` 记录，不计 FAIL。
  - `runtime/qa/out/e2e-p0-regression-20260519-134453/gaps.md` 已把该项固定为 `P0-9 menus/tree=0` 待确认缺口，下一步动作是“补浏览器菜单可见性、菜单 seed/bootstrap 与权限配置复核”。
  - `docs/archive/records/21-20260503-可见浏览器综合验收报告.md` 已留档本地 Mock / real-pre 基线下的角色页面可见性结果：`biz_staff /data`、`channel_staff /sample/apply`、`ops_staff /ops/shipping` 均能打开并完成页面级演示，说明“非 admin 菜单树为 0”当前更像菜单 seed/bootstrap 或环境基线问题，而不是已确认的主链业务阻断。
- 风险：
  - 非管理员菜单入口在不同环境之间可能出现“接口 200 但菜单为空”的不一致，影响角色页面可发现性和部署后稳定性。
  - 在没有补浏览器菜单证据和 `role_menu` / seed 复核前，不能把“菜单权限已通过”写进验收结论。
- 是否阻断 V1：
  - 不阻断当前单账号主链演示，也不阻断已有角色页面打开结论。
  - 但阻断 `P0-9` 作为“菜单权限已闭证”写法。
- 建议决策：`可带风险上线`
  - V1 继续保留限制说明：该项按菜单 seed/bootstrap / 权限配置专项挂起。
  - 后续只补菜单可见性与配置复核，不把本项改写成“已通过”。

### 2. `ops_staff configured / all` 冲突

- 当前状态：待确认，不能写通过。
- 当前证据：
  - `runtime/qa/out/e2e-rbac-20260519-133713/report.md` 与 `api-responses/38-ops-staff-configured-conflict.json` 已明确记录 `OPS_STAFF_CONFIGURED_ALL_CONFLICT`：`ops_staff` 登录返回 `roleCodes=ops_staff`、`dataScope=all`，本轮按 `SKIP` 保留，不计通过。
  - `docs/09-04-MCP配置规则接口梳理.md` 已确认：`ops_staff` 初始化 `data_scope=1`，但 `AuthService` 登录 / 刷新时强制覆盖成 `dataScope=3`；当前未发现 `system_config` 或角色配置项可以解释这条覆盖。
  - 同一轮 `E2E-RBAC-01` 同时确认：`ops_staff` 的 `GET /api/samples` 可用，但 `GET /api/orders`、`GET /api/dashboard/summary`、`GET /api/talents` 仍按越权预期记为 `PASS_DENIED_AS_EXPECTED`，说明当前风险主要是“数据范围口径和文档不一致”，不是“运营已拿到全部业务接口放行”。
- 风险：
  - 文档口径、seed 数据和运行时行为不一致，后续很容易把“固定 all”误写成“configured 可配”。
  - 如果后续新增接口只依赖 `dataScope` 而没有角色级拦截，运营视角可能出现超预期放量。
- 是否阻断 V1：
  - 不阻断当前运营单角色主链使用，也不阻断本轮 `GO WITH RISK` 放行判断。
  - 但阻断把“运营 configured 数据范围已确认”写成通过。
- 建议决策：`可带风险上线`
  - V1 书面口径先按“`ops_staff` 当前实现视为固定 `all`，接口权限继续以角色注解 / 业务接口放行为准”收口。
  - 若后续业务仍坚持“运营 = configured”，再单独升级为 `V1 必须修`，本轮不改业务代码。
