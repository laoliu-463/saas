- `real-pre` 受控部署不等于正式生产全量上线。
- `PENDING` / `BLOCKED` 不得写成 `PASS`。

## 18. 后续部署步骤

服务器侧建议按如下顺序执行：

```bash
sudo mkdir -p /opt/saas/app /opt/saas/env /opt/saas/logs /opt/saas/backups /opt/saas/runtime/qa/out
sudo chown -R "$USER":"$USER" /opt/saas

cd /opt/saas
git clone https://github.com/laoliu-463/saas.git app
cd /opt/saas/app
git checkout feature/auth-system
git pull --ff-only

cp .env.real-pre.example /opt/saas/env/.env.real-pre
chmod 600 /opt/saas/env/.env.real-pre
vi /opt/saas/env/.env.real-pre

ENV_FILE=/opt/saas/env/.env.real-pre ./scripts/deploy-real-pre.sh
./scripts/health-check.sh
```

部署后执行：

```bash
curl -fsS http://127.0.0.1:8081/api/system/health
curl -fsS http://127.0.0.1:3001/healthz
npm run e2e:real-pre:p0:preflight
npm run e2e:real-pre:roles
npm run e2e:real-pre:p0
```

## 19. 回滚方案

优先使用脚本回滚：

```bash
ENV_FILE=/opt/saas/env/.env.real-pre ./scripts/rollback-real-pre.sh
```

如涉及数据库变更，必须先确认已有备份，再按迁移和备份编号执行，禁止无备份破坏性操作。
