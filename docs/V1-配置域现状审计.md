## 三、五问专项答复

### 1. 配置是否统一走 `system_config` 表？

**大部分是，提成和归因例外。**

| 配置类别 | 存储位置 | 访问路径 |
|----------|----------|----------|
| 寄样（5 键） | `system_config` | `BusinessRuleConfigService` → `SystemConfigMapper` |
| 达人（3 键） | `system_config` | `BusinessRuleConfigService` → `SystemConfigMapper` |
| 提成（4+ 键） | `system_config` | `CommissionService` → **直接 JdbcTemplate** |
| 归因规则 | Spring `@Value` | `PickSourceMappingService` → application.yml |

提成键（`commission.*`）虽然数据存在 `system_config` 表，但 `CommissionService` 绕过了 `BusinessRuleConfigService` 的缓存和验证层。

### 2. 配置变更是否即时生效？

**是（两条路径均支持）。**

- **寄样/达人配置**：`SysConfigService` CRUD → `businessRuleConfigService.invalidate(key)` → 下次读取时 `ConcurrentHashMap.computeIfAbsent` 重新加载
- **提成配置**：`CommissionService` 无缓存，每次 `loadRatio()` 直接查询 DB，天然实时生效
- **归因规则**：Spring `@Value` 需重启生效（V1 接受）

### 3. 配置验证机制是否完整？

**部分完整。**

| 类别 | 有验证？ | 验证方式 |
|------|----------|----------|
| 寄样/达人（8 键） | **是** | `ConfigDefinitionRegistry`：类型强制转换 + 范围校验 + JSON 结构校验 |
| 提成比例 | **否** | 仅 `>= 0` 检查（`CommissionService.queryRatio` L132），无上限 |
| 归因规则 | **否** | Spring 属性注入，无运行时校验 |

### 4. 敏感配置是否脱敏？

**是。**

- `SysConfigService.isSensitive()`：基于关键词启发式（`key`、`secret`、`password`、`token`）匹配
- `ConfigDefinitionRegistry.isSensitive()`：标记敏感定义
- `/configs/grouped` 返回给前端时，敏感值被掩码

### 5. 配置 CRUD 权限是否正确？

**是。**

- `SysConfigController` 类级 `@RequireRoles({RoleCodes.ADMIN})`
- `/configs/grouped` 方法级 override 为全部 6 类角色（只读分组查询）
- 配置增删改仅 admin 可操作

## 五、P0 验收用例可达性（6 用例）

| 用例 | 期望 | 当前可达性 | 说明 |
|------|------|------------|------|
| 改寄样7天限制为14天 | 即时生效 | **可达** | admin 改 `sample.restrict_days=14`；`invalidate()` 清缓存；下次申请用新值 |
| 关闭寄样7天限制 | 申请不受限 | **可达** | admin 改 `sample.restrict_enabled=false`；`isSampleRestrictEnabled()` 返回 false |

| 改招商提成为20% | 即时生效 | **可达** | admin 改 `commission.business_default_ratio=0.20`；`CommissionService` 无缓存，下次计算用新值 |
| 非 admin 改配置 | 拒绝 | **可达** | `@RequireRoles(ADMIN)` 拦截；非 admin 返回 403 |
| 查看配置变更日志 | 有记录 | **可达** | admin 查 `/operation-logs`；配置变更由 `OperationLogInterceptor` 记录 |
