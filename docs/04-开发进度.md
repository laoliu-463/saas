## 六、当前风险

### 1. 环境口径风险

- 文档与脚本仍有部分旧表述使用 `test` 或 `dev` 描述本地演示环境
- 若不继续清理旧文档和兼容 profile，新人仍可能绕回 `dev`
- 若不统一，后续联调、排障和新人接手会持续混乱

### 2. 真实联调风险

- token 与授权口径尚未实测
- 限流、空数据、错误码分支尚未全量验证

### 3. 演示数据风险

- 某些异常样本还需要继续补齐
- P0 测试数据仍建议继续收口

### 4. 本地外部依赖噪音风险

- 本地 Mock 阶段仍出现第三方爬虫类日志噪音
- 容易干扰“是否真的接了真实接口”的判断

## 七、下一步建议

建议按以下顺序推进：

1. 完成 P1-1 文档与脚本收口：明确 `local-mock` 是本地人工联调唯一默认入口
2. 进入 P1-2：固定 Mock 数据基线，确认 reset / seed 后结果稳定
3. 进入 P1-3：治理本地第三方噪音日志
4. 进入 P1-4：固化本地 Mock 标准联调流程
5. 暂停无序扩功能，继续保护本地 Mock 演示基线

## 八、本地优化执行口径

当前第三方 SDK 无法稳定联调时，后续开发默认按以下口径推进：

### 1. 当前优化目标

不是继续无序扩功能，而是基于现有 `test / local-mock` 主链路做两类优化：

- **业务逻辑真实性优化**：让页面和流程更接近真实团长 / 招商团队的工作方式
- **前端设计可读性优化**：让当前页面更容易支持“推进、排查、复盘”

### 2. 业务逻辑优化重点

优先补强“结果闭环已经存在，但过程证据偏薄”的部分，尤其是：

- 商品由谁接手、何时接手
- 商品推进过程中的关键动作时间线
- 商品推进优先级与人工判断痕迹
- 订单结果对前置动作的可复盘能力

当前阶段建议先从商品主链路入手，不优先扩更多独立模块。

### 3. 前端设计优化重点

前端优化不以“页面更花”为目标，而以“业务信息更顺手”作为标准，优先看：

- 商品详情页是否能看清当前责任人、最近动作和推进状态
- 商品列表是否能快速扫出“谁在跟、最近有没有动作”
- 订单详情页是否更容易理解归因原因和排查路径
- 寄样、达人、订单之间的关联入口是否顺手

### 4. 当前本地优化顺序

1. P0 本地收口已完成，保住当前可演示主链路
2. 后续新增优化必须服务“推进、排查、复盘”，不再无序扩模块
3. 优先进入环境治理、权限口径统一和真实联调准备
