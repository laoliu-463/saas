# 12 文档编码乱码问题分析报告

更新时间：2026-04-27

## 一、问题背景

2026-04-27，项目发现 `docs/` 目录下几乎所有 Markdown 文档以及部分 SQL 脚本的中文内容出现大面积乱码（Mojibake），表现为类似 `澧為噺鑴氭湰`、`閫傜敤` 等无意义字符。

## 二、根本原因分析

经过技术排查，确认该问题属于 **“双重编码损坏”**，具体链路如下：

1. **初始状态**：文档以标准 **UTF-8** 编码保存，中文正常。
2. **误读取（Interpretation Error）**：某个工具（可能是 Windows PowerShell 默认配置或旧版编辑器）在读取 UTF-8 字节流时，错误地使用了 **GBK (CP936)** 编码进行解码。
   - 示例：`增量` 的 UTF-8 字节为 `E5 A2 9E E9 87 8F`。
   - 误用 GBK 解码后，这 6 个字节被识别为 `澧為噺`。
3. **二次污染（Re-saving Error）**：系统在保存修改时，将这些“乱码字符”再次以 **UTF-8** 编码写入磁盘。
   - 结果：原始的“增量”字节被替换成了“澧為噺”对应的 UTF-8 字节。
4. **不可逆性**：由于在 GBK 解码过程中存在信息丢失或偏移，这种损坏是**无法通过简单的编码切换还原**的，必须通过人工比对代码逻辑进行重写。

## 三、环境触发因素

- **OS 差异**：Windows 系统默认区域设置（System Locale）通常为 GBK，而开发工具链通常为 UTF-8。
- **Shell 默认值**：Windows PowerShell 的默认 `$OutputEncoding` 和 `Get-Content` 编码在旧版本中并非 UTF-8。
- **协作冲突**：在不同编码环境的工具（如 AI 助手、本地 IDE、命令行脚本）交叉修改文件时，若其中一方未强制声明 UTF-8，极易触发该问题。

## 四、预防措施与规范

为杜绝此类问题再次发生，所有参与开发的人员与工具必须遵循以下规范：

### 1. IDE 配置 (VS Code / IntelliJ)
- 强制设置全局编码为 `UTF-8`。
- 禁用 `Auto Guess Encoding`（自动猜测编码），防止误判。
- 统一使用 `LF` 作为换行符，避免 `CRLF` 在不同系统间引起冲突。

### 2. Windows 命令行配置
在 PowerShell 中建议执行以下命令，确保终端与文件流使用 UTF-8：
```powershell
$OutputEncoding = [System.Text.Encoding]::UTF8
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
```

### 3. Git 配置
防止 Git 在处理中文路径和内容时出现转义：
```bash
git config --global core.quotepath false
git config --global gui.encoding utf-8
git config --global i18n.commit.encoding utf-8
git config --global i18n.logoutputencoding utf-8
```

### 4. 工具使用准则
- **禁止使用记事本 (Notepad.exe)** 修改任何项目文件。
- 在 AI 助手修改文件后，开发者应立即观察编辑器底部的状态栏，确认编码仍为 **UTF-8**。
- 若发现乱码，应立即撤销操作，严禁在乱码基础上继续保存，防止污染扩散。

## 五、处理结论

目前所有受影响的文档已由 Antigravity 助手根据项目事实库完成人工修复。当前 `docs/` 目录已恢复至稳定状态。
