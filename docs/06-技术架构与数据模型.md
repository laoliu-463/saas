# 06-技术架构与数据模型

更新时间：2026-05-20

> 本文档描述 **当前仓库真实技术实现**。  
> 历史需求包中的 `FastAPI + Celery`、原始数据库设计、旧 V2.2 技术方案只作为背景参考，不作为当前 V1 强制迁移目标。

## 一、当前结论

- 当前项目以 **Spring Boot 模块化单体** 为事实口径。
- 当前验收按业务闭环、数据口径和真实运行结果进行，不按历史技术方案原样复刻进行。
- 所有技术、接口、模型说明都必须服从 [01-V1交付范围与边界](./01-V1交付范围与边界.md)。

## 二、当前实际技术栈

| 层级 | 当前实现 |
|---|---|
| 后端 | Spring Boot |
| 数据库 | PostgreSQL |
| 缓存 | Redis |
| 前端 | Vue 3 + Vite + Pinia + Naive UI |

| 三方接入 | Gateway 契约隔离 Test / Real |

### 背景说明

| 历史方案 | 当前处理 |
|---|---|
| FastAPI | 仅作为原始设计背景 |
| SQLAlchemy / Alembic | 仅作为原始设计背景 |

| Spring Boot 现实现 | 作为当前 V1 唯一技术事实 |

## 三、总体架构

当前项目采用 **前后端分离的模块化单体架构**：

- 前端只调用内部 `/api`
- 后端负责业务编排和权限控制
- 第三方能力统一通过 Gateway 隔离
- PostgreSQL 保存业务事实
- Redis 用于缓存、黑名单、幂等和运行态辅助

一句话：

> 一套业务逻辑，按环境切换 Gateway 实现，不把 Test / Real 差异渗透进业务主链。

## 四、后端分层

### 1. Controller

- 暴露内部 API
- 处理请求参数
- 返回统一响应

### 2. Service

- 承担核心业务编排
- 串联商品、达人、寄样、订单、业绩等主线

### 3. Gateway

- 吸收第三方平台差异
- 对上提供统一 DTO
- 对下适配 Test / Real 具体实现

### 4. Repository / 数据访问层

- 落库
- 查询
- 为排查与回放提供数据基础

### 5. 认证与权限（横切）

- `JwtAuthInterceptor`
- `JwtTokenProvider`
- `TokenBlacklistService`
- `@RequireRoles`
- `@DataScope`

## 五、网关契约原则

### 当前约束

1. Service 只依赖 Gateway 接口
2. 不允许在业务 Service 里写 `if test` / `if real`
3. 不允许为了 Real 接口去污染前端字段结构
4. 第三方原始结构必须先转换，再向上提供

### 当前覆盖的三方能力

| 网关 | 当前口径 |
|---|---|
| 授权 / Token | Test / Real 都已接入 |
| 活动 / 商品 / SKU 补充 | Test / Real 都已接入；SKU 当前通过 `DouyinProductGateway.queryProductSkus` 适配 `buyin.productSkus.v2` |
| 订单同步 | Test / Real 都已接入 |
| 转链 | Test / Real 都已接入 |
| 物流 | V1 默认手动物流，真实物流 API 只保留底座 |

## 六、前端结构

前端围绕业务页面组织：

- `/product`
- `/orders`
- `/dashboard`
- `/sample`
- `/talent`
- `/system`

当前原则：

- 只调用内部 API
- 不直连第三方
- 页面围绕业务闭环，而不是围绕技术结构组织

## 七、核心模块与领域边界

| 模块 / 领域 | 当前职责 |
|---|---|
| 用户域 | 登录、角色、数据范围、下拉基础能力 |
| 配置域 | 7 天限制、保护期、提成比例、模板规则 |
| 商品域 | 活动、活动商品、合作方、商品库、转链、SKU 补充 |
| 达人域 | 达人档案、认领、地址、标签、保护期 |
| 寄样域 | 申请、审核、发货、待交作业、完成 |
| 订单域 | 同步、双轨金额、默认归因、筛选明细 |
| 业绩域 | 默认归因业绩、双轨提成、列表卡片 |
| 分析模块 | 汇总展示、基础指标、明细下钻 |

商品域当前第三方事实口径补充：

- 活动商品同步主接口为 `alliance.colonelActivityProduct`
- 活动商品落库按三层职责收口：`colonel_activity_product` 只保留 `activity_id + product_id + shop_id + status` 等关联事实；展示快照走 `product_snapshot`；审核、入库、分配、置顶等操作状态统一沉淀到 `product_operation_state`
- 商品级寄样门槛读取 `product_operation_state.audit_payload` 中的商品补充信息字段，当前消费 `sampleThresholdSales` / `sampleThresholdLevel` / `sampleThresholdRemark` 等键；没有商品级门槛时回退 `sample.default_standard`
- 商品库读接口为 `GET /products`（旧兼容 Controller 但承载当前共享商品库读能力），支持 `keyword/status/categoryId/shopId/tag` 筛选，并在返回前按同一 `product_id` 去重展示
- 合作方读接口为 `GET /merchants`，从 `merchant` 表分页返回合作方 / 店铺数据；合作方维护类操作仍不进入 V1 主链
- 商品 SKU 补充接口为 `buyin.productSkus.v2`（路径 `/buyin/productSkus/v2`，文档 ID `6468`，名称“查询商品 SKU”）
- `buyin.productSkus.v2` 属于商品域补充能力，当前按 P1 管理，不并入订单 / 结算主同步链路
- `buyin.productSkus.v2` 不提供 `settled_goods_amount`、`commission`、`settle_time` 等结算口径；这些字段仍以 `buyin.colonelMultiSettlementOrders` 为补充样本来源

订单域当前第三方事实口径补充：

- 主同步接口为 `buyin.instituteOrderColonel`
- 按订单号定向补拉也走 `buyin.instituteOrderColonel`
- O-01 已拆成两条链路：主订单同步默认 `time_type=update`，用于订单事实、状态变化、预估看板和寄样出单判断；O-01-S 结算事实同步默认 `time_type=settle`，用于结算服务费、实际提成和结算看板
- `buyin.colonelMultiSettlementOrders` 是结算事实源，不替代 `buyin.instituteOrderColonel` 的主订单事实源
- `buyin.colonelMultiSettlementOrders` 的结算事实已显式落 `colonel_order_settlement`，当前映射包含 `colonel_name`、`service_fee_rate`、`commission_rate`、`phase_id`、预估 / 结算服务费、技术服务费等补充字段；`upstream_key` 使用 `order_id + phase_id + settle_time + pay_time` 防止分期结算覆盖。
- 2704 金额字段必须保留 `raw_payload` 并记录 `source_amount_unit`；真实 raw JSON 未对表前写入 `UNVERIFIED`，Dashboard / 数据页结算事实聚合只消费已验证单位（`FEN`、`YUAN_STRING`、`YUAN_NUMBER`）
- 结算事实同步的页级上游拉取失败会按 5s / 10s / 30s 最多重试 3 次，仍失败则保留锁释放与失败上抛，不推进 last sync time。

## 八、核心数据模型

### 1. V1 必需模型

| 模型 | 当前用途 |
|---|---|
| 用户 / 角色 / 组织 | 登录、权限、数据范围 |
| 活动 / 活动商品 | 招商链起点 |
| 商品 / 商品状态 | 商品库、审核、转链、展示 |
| 合作方 | 商品侧基础实体 |
| 达人 / 达人认领 | 公海 / 私海、保护期、归属 |
| 寄样申请 | 状态流转主实体 |
| `pick_source_mapping` | 推广映射 |
| 订单 | 同步、默认归因、明细 |
| `colonel_order_settlement` | 2704 结算事实、实际服务费 / 提成补证；未验证金额单位前不作为可信金额展示 |
| 业绩 / 汇总 | 双轨金额、提成、Dashboard |
| 系统配置 | 业务规则 |
| 操作日志 | 关键操作留痕 |

### 2. V1 简化模型

| 模型 | 当前口径 |
|---|---|
| 物流信息 | 手动录入公司、单号、状态推进 |

| 多角色能力 | 权限并集、数据范围取最宽 |
| 商品 SKU 快照 | 当前不入订单 / 结算主链；如商品详情、寄样或价格库存需要按 SKU 维度展示，再扩展本地快照模型（如 `product_skus`） |

## 九、关键数据关系

### 1. 商品与活动

- 商品归属于活动
- 商品可被审核、分配、转链
- 关系事实与操作状态分离：活动商品关系读 `colonel_activity_product`，展示读 `product_snapshot`，P-09/P-05 相关状态读 `product_operation_state`
- 寄样门槛属于商品补充信息读取能力，不新增独立复杂门槛配置表；优先级为商品级补充信息 > 全局默认配置

### 2. 转链与归因映射

- 推广链接生成后写入推广状态
- 归因标识写入 `pick_source_mapping`

### 3. 订单与默认归因

- 订单通过 `pick_source` 匹配默认渠道
- 默认招商按活动绑定招商收口

### 4. 寄样与订单

- 寄样记录与商品、达人关联
- 订单命中后可自动完成寄样单
- 自动完成条件固定为 `channel_user_id + talent_uid/author_id + product_id` 匹配，寄样单处于 `PENDING_HOMEWORK`，且订单 `pay_time != null`
- `pay_time` 为空的匹配订单只可作为负向样本，不允许推进寄样完成

### 5. 业绩与看板

- 业绩域产出默认归因和双轨提成
- 分析模块读取汇总或聚合结果展示

## 十、接口与返回口径

当前统一口径如下：

- 前端业务接口统一通过 `/api`
- 认证统一为 `Authorization: Bearer <token>`
- 成功响应统一包装
- 分页响应统一包装
- 主链接口优先服务三条业务链
- 联调接口、测试接口、兼容接口不自动等于 V1 主交付

## 十一、当前已知技术边界

- 不做微服务拆分
- 不做复杂调度中心
- 不做完整财务结算引擎
- 不做平台化复杂权限矩阵
- 不把 FastAPI / Celery 方案重新拉回当前改造目标

## 十二、一句话结论

> 当前 V1 技术架构的核心不是“迁移到历史理想方案”，而是“在现有 Spring Boot 单体上稳定承载三条主链、统一数据口径并完成验收补证”。
