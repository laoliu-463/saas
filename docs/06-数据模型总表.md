# 数据模型总表

> 范围标记：`V1 必做`、`V1 简化`、`V1 不做`、`V2 预留`、`历史归档`。

## 建模原则

- [V1 必做] 订单域只存事实，不计算提成，不应用独家覆盖。
- [V1 必做] 业绩域负责最终归属、提成、冲正、双轨金额计算。
- [V1 必做] 配置域只存配置事实和变更审计，不执行规则。
- [V1 必做] 分析模块只读汇总表，不重算业绩归属。
- [V1 必做] 数据表字段级事实以数据库迁移、实体类和 SQL 查询结果为准。

## 数据模型总表

| 模型 / 表族 | 所属领域 | 存储事实 | 禁止承载 | 证据方式 | 范围 |
| --- | --- | --- | --- | --- | --- |
| 用户、角色、组织、菜单 | 用户域 | 用户身份、权限、组织和菜单 | 业务归属计算 | 用户 API、权限测试 | V1 必做 |
| 配置、规则、变更日志 | 配置域 | 配置值、规则参数、变更审计 | 具体业务执行 | 配置 API、审计表 | V1 必做 |
| 商品库、活动商品 | 商品域 | 商品、活动、价格、佣金展示输入 | 最终业绩 | 商品 API、同步日志 | V1 必做 |
| `pick_source_mapping` | 商品域 | 转链结果、达人/商品/渠道映射 | 最终提成结论 | 转链 API、映射表 | V1 必做 |
| 达人、标签、地址、关系 | 达人域 | 达人基础资料、标签、地址、渠道关系 | 订单最终归属 | 达人 API、操作日志 | V1 必做 |
| 寄样申请、寄样状态 | 寄样域 | 申请、审批、发货、交作业状态 | 业绩计算 | 寄样 API、状态日志 | V1 必做 |
| 订单、订单明细 | 订单域 | 抖店订单事实、支付金额、商品、时间 | 提成、独家覆盖 | 订单 API、同步日志 | V1 必做 |
| 退款 / 售后 | 订单域 | 退款事实、售后状态、退款金额 | 冲正结果 | 退款表、同步日志 | V1 必做 |
| 业绩明细 | 业绩域 | 最终归属、提成、冲正、双轨金额、经营毛利 | 第三方原始事实采集 | 业绩 API、计算日志 | V1 必做 |
| 业绩汇总 | 业绩域 | 按人、组织、周期的汇总结果和经营指标 | 重新归因逻辑 | 汇总表、dashboard API | V1 必做 |
| 分析快照 / 报表 | 分析模块 | 只读汇总、经营指标矩阵、导出快照 | 修改订单或业绩事实 | 看板 API、导出文件 | V1 简化 |
| Outbox / 事件记录 | 基础设施 | 事件投递事实、状态、重试 | 业务事实唯一来源 | Outbox 表、日志 | V1 简化 |
| 操作审计 | 基础设施 | 实际状态变化、批量任务汇总、结构化错误与链路 | 高频无状态变化明细 | `operation_log` 月分区、`error_code/error_message/trace_id` | V1 必做 |

## 历史模型处理

- [V1 必做] `colonel_activity` 保存活动级招商组长分配事实：`recruiter_user_id`、`recruiter_dept_id`、`assigned_at`、`assigned_by`；该事实级联到活动商品的 `product_operation_state.assignee_id`。
- [V1 必做] `colonel_activity.activity_status_code` / `activity_status_text` 只保存抖店活动状态；活动状态不得写入或推导 `product_operation_state.selected_to_library`、`biz_status`、`audit_status`、`display_status` 等商品状态事实。

- [历史归档] 旧 V2.2 数据库设计草案保留在 `docs/archive/records/19-V2.2数据库设计草案.md` 和 `docs/归档/旧版V2.2完整方案/`。
- [V1 必做] `operation_log` 通过 `V20260719_001__operation_log_observability.sql` 增加结构化错误码与 trace ID；保留期清理必须调用 `cleanupOldPartitions` 按完整月分区 DROP，不执行大表逐行 DELETE。
- [V1 不做] 旧文档中的 FastAPI / Celery / Python 调度模型不作为当前数据模型事实。
- [V2 预留] 跨服务拆分后的事件表、消息表、审计湖和 BI 宽表另行设计。
